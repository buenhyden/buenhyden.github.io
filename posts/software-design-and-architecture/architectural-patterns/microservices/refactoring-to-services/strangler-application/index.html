Strangler Application#

Strangler Application 패턴은 마이크로서비스 아키텍처로의 전환을 위한 중요한 리팩토링 전략이다.

Strangler Application 패턴은 기존의 모놀리식 애플리케이션을 점진적으로 새로운 마이크로서비스 기반 시스템으로 대체하는 방법이다.
이 패턴의 이름은 호주의 열대우림에서 볼 수 있는 스트랭글러 무화과나무에서 유래했다. 이 나무는 숙주 나무를 감싸고 자라다가 결국 숙주를 대체하는데, 이와 유사하게 새로운 서비스가 기존 시스템을 점진적으로 대체한다.

Strangler Application 패턴은 레거시 시스템을 현대화하는 효과적인 방법으로, 리스크를 관리하면서 점진적으로 마이크로서비스 아키텍처로 전환할 수 있게 해준다. 이 패턴을 통해 기업은 기존 시스템의 가치를 유지하면서 새로운 기술과 아키텍처의 이점을 활용할 수 있다.

주요 특징#

  1. 점진적 전환: 시스템의 일부분씩 마이크로서비스로 전환하여 리스크를 최소화한다.
  2. 공존: 기존 시스템과 새로운 마이크로서비스가 일정 기간 동안 공존한다.
  3. 기능 이전: 기존 기능을 새로운 서비스로 점진적으로 이전한다.
  4. 프록시 사용: 요청을 적절한 시스템(기존 또는 새로운)으로 라우팅하는 프록시 계층을 사용한다.

구현 단계#

  1. 변환 (Transform): 기존 애플리케이션의 특정 기능을 식별하고 이를 새로운 마이크로서비스로 재구현한다.
  2. 공존 (Coexist): 프록시 계층(예: API Gateway)을 도입하여 요청을 기존 시스템 또는 새로운 마이크로서비스로 라우팅한다.
  3. 제거 (Eliminate): 새로운 서비스가 안정화되면 기존 시스템의 해당 기능을 제거한다.

적용 방법#

  1. 기능 식별 및 분리:
    • 기존 모놀리식 애플리케이션에서 마이크로서비스로 분리할 기능이나 모듈을 식별한다.
  2. 프록시 또는 파사드 계층 설정:
    • 클라이언트 요청을 가로채어 기존 시스템과 새로운 마이크로서비스로 라우팅할 수 있는 프록시나 파사드 계층을 구축한다. 이를 통해 클라이언트는 변경 사항을 인지하지 못한 채 서비스를 이용할 수 있다.
  3. 기능 구현 및 전환:
    • 식별된 기능을 새로운 마이크로서비스로 구현하고, 프록시를 통해 해당 기능에 대한 요청을 새로운 서비스로 라우팅한다.
  4. 반복 및 최적화:
    • 이 과정을 반복하여 점진적으로 기존 시스템의 기능을 새로운 마이크로서비스로 이전한다. 모든 기능이 이전되면, 기존 모놀리식 시스템은 폐기할 수 있다.

장점#

  1. 리스크 감소: 전체 시스템을 한 번에 교체하는 것보다 위험이 적다.
  2. 점진적 현대화: 시스템을 계속 운영하면서 현대화할 수 있다.
  3. 유연성: 새로운 기술과 아키텍처를 단계적으로 도입할 수 있다.
  4. 비즈니스 연속성: 서비스 중단 없이 마이그레이션을 수행할 수 있다.

주의사항#

  1. 복잡성 관리: 두 시스템의 공존 기간 동안 복잡성이 증가할 수 있다.
  2. 데이터 일관성: 두 시스템 간의 데이터 동기화에 주의해야 한다.
  3. 성능 고려: 프록시 계층으로 인한 지연을 최소화해야 한다.
