Application Metrics#

마이크로서비스 아키텍처(MSA) 패턴에서 Observability는 매우 중요한 개념이며, 그 중 Application Metrics는 핵심적인 요소이다.
Application Metrics는 애플리케이션의 성능과 동작을 이해하고 문제를 해결하는 데 필수적인 도구이다.

Application Metrics는 애플리케이션의 성능, 동작, 그리고 건강 상태를 나타내는 수치화된 데이터이다.
이는 개별 작업에 대한 통계를 수집하고, 이를 중앙 집중식 메트릭 서비스에 집계하여 보고 및 경고를 제공하는 패턴을 말한다.

Application Metrics는 MSA 환경에서 Observability를 달성하는 데 핵심적인 역할을 한다.
이를 통해 개발자와 운영팀은 복잡한 시스템을 더 잘 이해하고, 문제를 신속하게 해결하며, 시스템의 성능을 지속적으로 개선할 수 있다.

Application Metrics의 중요성#

  1. 시스템 이해: 복잡한 마이크로서비스 환경에서 각 서비스의 동작을 이해하는 데 도움을 준다.
  2. 문제 해결: 성능 저하나 오류의 원인을 신속하게 파악할 수 있게 해준다.
  3. 예측적 분석: 시스템의 동향을 파악하여 잠재적 문제를 사전에 예측할 수 있다.
  4. 최적화: 성능 병목 현상을 식별하고 시스템을 최적화하는 데 필요한 정보를 제공한다.

주요 Application Metrics#

  1. 서비스 레벨 메트릭:

    • 응답 시간: 평균, 95번째 백분위수, 최대 응답 시간
    • 처리량: 초당 요청 수 또는 분당 트랜잭션 수
    • 오류율: 실패한 요청의 비율과 오류 유형

  2. 리소스 사용률:

    • CPU 사용률: 서비스가 소비하는 CPU 자원의 비율.
    • 메모리 사용량: 서비스가 사용하는 메모리의 양.
    • 디스크 I/O: 디스크 입출력 활동의 빈도와 속도.
    • 네트워크 대역폭 사용률

  3. 의존성 메트릭:

    • 서비스 간 통신 빈도
    • 외부 API 호출 성능
    • 데이터베이스 쿼리 효율성

  4. 비즈니스 메트릭:

    • 사용자 세션 수: 활성 사용자 세션의 수.
    • 거래 또는 주문 수: 특정 기간 동안 처리된 거래 또는 주문의 수.
    • 사용자 활동, 트랜잭션 볼륨, 수익 등 비즈니스 관련 지표

Application Metrics 구현 방법#

  1. 계측 라이브러리 사용:

    • Coda Hale/Yammer Java Metrics Library
    • Prometheus 클라이언트 라이브러리

  2. 메트릭 집계 모델:

    • Push 모델: 서비스가 메트릭을 메트릭 서비스로 푸시
    • Pull 모델: 메트릭 서비스가 서비스로부터 메트릭을 가져옴

  3. 중앙 집중식 메트릭 저장소:

    • Prometheus, Grafana, ELK 스택 등의 도구 활용

모범 사례#

  1. 아키텍처 고려: 특정 마이크로서비스 설계와 잠재적 약점에 대한 통찰력을 제공하는 메트릭에 집중.
  2. 자동화된 알림: 중요한 임계값을 초과할 때 자동 알림 설정.
  3. 상관 관계 분석: 로그, 메트릭, 트레이스 간의 상관 관계를 분석하여 전체적인 시스템 상태 파악.
  4. 동적 임계값 설정: 시스템의 변화에 따라 자동으로 조정되는 동적 임계값 사용.

도전 과제#

  1. 데이터 과부하: 너무 많은 메트릭은 분석을 복잡하게 만들 수 있다.
  2. 메트릭 정확성: 일관되고 정확한 메트릭 수집이 중요하다.
  3. 알림 피로: 과도한 알림은 중요한 문제를 놓칠 수 있게 한다.
  4. 확장성: 대규모 시스템에서 메트릭 수집 및 분석의 확장이 필요하다.
