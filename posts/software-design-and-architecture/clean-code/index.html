클린 코드 (Clean Code)#

클린 코드는 단순히 작동하는 코드를 넘어, 이해하기 쉽고, 유지보수가 용이하며, 오류가 적고, 확장 가능한 코드를 의미한다.
로버트 C. 마틴(Robert C. Martin)의 저서 “Clean Code: A Handbook of Agile Software Craftsmanship"에서 체계적으로 정리된 이 개념은 오늘날 소프트웨어 개발의 핵심 가치로 자리 잡았다.
코드는 컴퓨터뿐만 아니라 다른 개발자들이 읽고 이해해야 하는 것이기 때문에, 클린 코드는 소프트웨어 개발 조직과 프로젝트의 장기적 성공에 중요한 역할을 한다.

클린 코드는 단순한 미학적 선호가 아니라 소프트웨어의 유지보수성, 확장성, 신뢰성을 위한 실질적인 접근 방식이다.
의미 있는 이름, 작고 집중된 함수, 적절한 주석, 일관된 형식, 적절한 추상화 등의 원칙을 따르면 더 나은 코드를 작성할 수 있다.

클린 코드를 작성하는 것은 기술뿐만 아니라 태도의 문제이기도 하다.
코드를 더 깨끗하게 남기려는 책임감과 전문성을 가지고 접근하는 것이 중요하다.
로버트 마틴의 말처럼 “전문가는 깨끗한 코드를 작성한다"는 점을 기억하자!

Clean Code의 중요성#

  1. 가독성과 유지보수성 향상: 깨끗한 코드는 다른 개발자들이 쉽게 이해하고 수정할 수 있게 한다.
  2. 팀 협업 개선: 일관된 코딩 표준을 따르면 팀원 간 의사소통과 협업이 더 원활해진다.
  3. 디버깅과 문제 해결 용이: 명확하고 단순한 구조는 이슈를 더 쉽게 찾고 해결할 수 있게 한다.
  4. 코드 품질과 신뢰성 향상: 잘 구조화된 코드는 오류 발생 위험을 줄이고 더 높은 품질의 소프트웨어로 이어진다.

클린 코드의 핵심 원칙#

의미 있는 이름 사용 (Meaningful Names)#

좋은 이름은 코드의 의도를 분명히 드러내고 문서화의 필요성을 줄인다.

주요 가이드라인:

  • 의도를 분명히 드러내는 이름: 변수, 함수, 클래스 이름은 존재 이유, 수행 기능, 사용 방법을 명확히 표현해야 한다.

// 좋은 예

let distanceInKilometers;
  • 발음하기 쉬운 이름: 코드에 대해 대화할 때 발음하기 쉬운 이름이 커뮤니케이션을 원활하게 한다.

// 좋은 예

# 나쁜 예: 여러 작업을 수행하는 큰 함수
def process_user_data(user_data):
    # 데이터 검증
    if not user_data.get('name'):
        raise ValueError("이름이 필요합니다")
    if not user_data.get('email'):
        raise ValueError("이메일이 필요합니다")
    
    # 데이터 정제
    name = user_data['name'].strip()
    email = user_data['email'].lower().strip()
    
    # 데이터베이스 저장
    user = User(name=name, email=email)
    db.session.add(user)
    db.session.commit()
    
    # 이메일 전송
    send_welcome_email(email, name)
    
    return user

# 좋은 예: 작고 단일 책임을 가진 함수들
def validate_user_data(user_data):
    if not user_data.get('name'):
        raise ValueError("이름이 필요합니다")
    if not user_data.get('email'):
        raise ValueError("이메일이 필요합니다")

def clean_user_data(user_data):
    return {
        'name': user_data['name'].strip(),
        'email': user_data['email'].lower().strip()
    }

def save_user(name, email):
    user = User(name=name, email=email)
    db.session.add(user)
    db.session.commit()
    return user

def process_user_data(user_data):
    validate_user_data(user_data)
    clean_data = clean_user_data(user_data)
    user = save_user(clean_data['name'], clean_data['email'])
    send_welcome_email(clean_data['email'], clean_data['name'])
    return user

  • 한 가지 작업만 수행: 함수는 한 가지 작업만 수행해야 하며, 그 작업을 잘 수행해야 한다. 단일 책임 원칙(SRP)의 함수 레벨 적용이다.

  • 함수 이름은 수행하는 작업을 설명해야 함: 함수 이름만 보고도 무슨 일을 하는지 알 수 있어야 한다.

형식 지정 (Formatting)#

// 나쁜 예: 데이터 구조 노출
class Point {
    public double x;

}

// 좋은 예: 캡슐화
class Point {
    private double x;
    private double y;
    
    public Point(double x, double y) {
        this.x = x;
        this.y = y;
    }
    
    public double distanceTo(Point other) {
        return Math.sqrt(Math.pow(this.x - other.x, 2) + Math.pow(this.y - other.y, 2));
    }
    
    // 필요한 경우 게터와 세터 제공
    public double getX() { return x; }
    public double getY() { return y; }
}
  • 객체 vs 자료 구조: 객체는 동작을 숨기고 데이터를 노출하지 않는 반면, 자료 구조는 데이터를 노출하고 의미 있는 동작은 없다. 상황에 맞게 적절히 선택한다.
  • 디미터 법칙(Law of Demeter): 객체는 자신이 직접 알고 있는 객체와만 대화해야 한다. “한 번의 점만 사용하라"는 원칙이다.

// 나쁜 예: 오류 코드 반환
public int withdraw(Account account, int amount) {
    if (amount > account.getBalance()) {
        return -1; // 잔액 부족 오류 코드
    }
    account.deductFromBalance(amount);
    return 0; // 성공 코드
}

// 좋은 예: 예외 사용
public void withdraw(Account account, int amount) throws InsufficientFundsException {
    if (amount > account.getBalance()) {
        throw new InsufficientFundsException();
    }
    account.deductFromBalance(amount);
}

  • 언체크 예외 사용: 체크 예외(checked exceptions)는 의존성 문제를 일으킬 수 있으므로 언체크 예외(unchecked exceptions)를 선호하는 것이 낫다.

        users.put(id, user);
    }
    
    public User findUser(int id) {
        return users.get(id);
    }

}

테스트 (Tests)#

깨끗한 테스트는 가독성이 높고, 빠르게 실행되며, 독립적이어야 한다.

주요 가이드라인:

  • TDD 규칙: 테스트를 실패하게 만드는 코드를 작성한 후, 테스트를 통과하는 코드를 작성하고, 마지막으로 코드를 리팩토링한다.
  • 가독성 있는 테스트: 테스트는 이해하기 쉽게 작성해야 한다.

// 좋은 예: 가독성 있는 테스트
@Test
public void shouldWithdrawMoneyWhenEnoughBalance() {
    // 준비(Arrange)
    Account account = new Account(100);
    
    // 실행(Act)
    account.withdraw(50);
    
    // 확인(Assert)
    assertEquals(50, account.getBalance());
}
  • 개념 당 하나의 assert: 각 테스트는 하나의 개념만 테스트해야 한다.
  • F.I.R.S.T. 원칙: 테스트는 빠르고(Fast), 독립적이며(Independent), 반복 가능하고(Repeatable), 자체 검증이 가능하며(Self-validating), 적시에 작성되어야(Timely) 한다.

냄새와 휴리스틱 (Smells and Heuristics)#

코드 냄새(Code Smells)는 더 깊은 문제의 징후일 수 있는 코드의 특성이다.

주요 가이드라인:

  • 중복 코드(DRY - Don’t Repeat Yourself): 중복 코드는 버그의 온상이며 유지보수를 어렵게 만든다.

// 나쁜 예: 중복 코드
function validateUsername(username) {
    if (username.length < 3) {
        throw new Error("사용자 이름은 3자 이상이어야 합니다");
    }
    // 추가 검증…
}

function validateUserCredentials(username, password) {
    if (username.length < 3) { // 중복된 검증 로직
        throw new Error("사용자 이름은 3자 이상이어야 합니다");
    }
    // 비밀번호 검증…
}

// 좋은 예: 중복 제거
function validateUsername(username) {
    if (username.length < 3) {
        throw new Error("사용자 이름은 3자 이상이어야 합니다");
    }
    // 추가 검증…
}

function validateUserCredentials(username, password) {
    validateUsername(username);
    // 비밀번호 검증…
}

  • 함수와 인수: 함수는 작아야 하고 매개변수는 적어야 한다.
  • KISS(Keep It Simple, Stupid): 복잡한 해결책보다 단순한 해결책을 선호한다.
  • YAGNI(You Aren’t Gonna Need It): 필요할 때까지 기능을 추가하지 않는다.
  • 불필요한 추상화 피하기: 필요한 만큼만 추상화하고, 과도한 추상화는 피한다.

클린 코드의 실천 방법#

  1. 리팩토링 습관화
    코드를 처음부터 완벽하게 작성하는 것은 불가능하다.
    작동하는 코드를 작성한 후 지속적으로 리팩토링하는 습관을 들인다.

  2. 코드 리뷰
    동료의 코드를 리뷰하고 피드백을 주고받으면서 더 나은 코드 작성법을 배울 수 있다.

  3. 테스트 주도 개발(TDD)
    테스트를 먼저 작성하면 설계를 개선하고 코드 품질을 높일 수 있다.

  4. 지속적인 학습
    프로그래밍 기술과 모범 사례는 계속 발전한다. 새로운 도구와 기술을 학습하여 코드 품질을 향상시킨다.
