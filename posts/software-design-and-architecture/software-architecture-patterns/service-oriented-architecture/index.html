Service-oriented Architecture#

**Service-Oriented Architecture(SOA)**는 기업의 복잡한 IT 시스템을 비즈니스 기능 단위의 서비스로 모듈화하고, 이를 조합해 유연한 애플리케이션을 구축하는 소프트웨어 설계 패턴이다.
대규모 분산 시스템에서 재사용성과 상호 운용성을 극대화하는 데 목적을 둔다.

SOA는 기업 IT 시스템의 유연성을 높이는 전략적 도구이지만, 초기 설계 단계에서 명확한 서비스 경계 정의와 ESB 관리 계획 수립이 필수적이다.
특히 금융, 의료 등 복잡한 업무 프로세스가 존재하는 분야에서 효과적이며, 최근에는 MSA로 진화하면서 클라우드 환경에 적합한 형태로 재편되고 있다. 성공적인 SOA 구축을 위해서는 조직 차원의 아키텍처 거버넌스와 지속적인 모니터링 체계 구축이 동반되어야 한다.

SOA의 핵심 개념#

  1. 정의 및 목적

    • 비즈니스 중심 분할: 애플리케이션을 독립적인 서비스로 분할 (예: 결제, 고객 관리).
    • 표준화 인터페이스: 웹 서비스(SOAP/WSDL)나 REST API 등을 통해 플랫폼 독립적 통신 지원.
    • 느슨한 결합(Loosely Coupled): 서비스 간 의존성을 최소화하여 변경 영향도를 낮춤.

  2. 주요 구성 요소

구성 요소역할예시 기술
서비스 제공자(Provider)비즈니스 로직 구현AWS Lambda, SAP Adapter
서비스 소비자(Consumer)서비스 호출웹/모바일 애플리케이션
서비스 레지스트리(Registry)서비스 검색 및 관리UDDI, 서비스 카탈로그
ESB(Enterprise Service Bus)메시지 라우팅, 변환MuleSoft, IBM Integration Bus

    // 서비스 계약 정의 예시
public interface OrderService {
    OrderResponse createOrder(OrderRequest request);
    OrderStatus getOrderStatus(String orderId);
    void cancelOrder(String orderId);
}

// 서비스 구현
@Service
public class OrderServiceImpl implements OrderService {
    private final OrderRepository orderRepository;
    private final PaymentService paymentService;

    @Override
    public OrderResponse createOrder(OrderRequest request) {
        // 주문 생성 로직
        Order order = new Order(request);
        orderRepository.save(order);

        // 결제 서비스 호출
        PaymentResult payment = paymentService.processPayment(
            order.getId(), 
            request.getPaymentDetails()
        );

        return new OrderResponse(order, payment);
    }
}

public interface PaymentGateway {

// 구체적인 구현
@Service
public class StripePaymentGateway implements PaymentGateway {
    private final StripeClient stripeClient;

    @Override
    public PaymentResult processPayment(PaymentRequest request) {
        // Stripe 결제 처리 로직
        return stripeClient.charge(request);
    }
}

@Service
public class PayPalPaymentGateway implements PaymentGateway {
    private final PayPalClient paypalClient;

    @Override
    public PaymentResult processPayment(PaymentRequest request) {
        // PayPal 결제 처리 로직
        return paypalClient.makePayment(request);
    }
}

@Service
public class ServiceRegistry {
    private final Map<String, ServiceEndpoint> services = new ConcurrentHashMap<>();

    public void register(String serviceName, ServiceEndpoint endpoint) {
        services.put(serviceName, endpoint);
    }

    public ServiceEndpoint discover(String serviceName) {
        return services.get(serviceName);
    }
}

    // 서비스 조합 예시
@Service
public class OrderProcessingService {
    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final ShippingService shippingService;

    public OrderResult processOrder(OrderRequest request) {
        // 재고 확인
        if (!inventoryService.checkAvailability(request.getItems())) {
            throw new InsufficientInventoryException();
        }

        // 주문 생성
        OrderResponse order = orderService.createOrder(request);

        // 배송 처리
        ShippingResponse shipping = shippingService.arrangeShipping(order);

        return new OrderResult(order, shipping);
    }
}

SOA의 아키텍처 모델#

  1. 발전 단계

    1. Fundamental SOA: 기존 시스템을 서비스화해 통합 (P2P 연결).
    2. Networked SOA: ESB 도입으로 중앙 집중식 관리 및 복잡성 해소.
    3. Process-Enabled SOA: BPM(Business Process Management)과 연동해 업무 프로세스 자동화.

  2. 표준 기술 스택

주요 특징:#

  1. 서비스 재사용성:
    • 각 서비스는 독립적으로 설계되어 다양한 애플리케이션에서 재사용될 수 있다. 이를 통해 개발 효율성을 높이고 중복 개발을 방지한다.
  2. 느슨한 결합:
    • 서비스 간의 의존성을 최소화하여, 한 서비스의 변경이 다른 서비스에 미치는 영향을 줄인다. 이를 통해 시스템의 유연성과 유지보수성을 향상시킨다.
  3. 표준화된 인터페이스:
    • SOAP, JSON, REST 등 표준 프로토콜을 사용하여 서비스 간 통신을 수행한다. 이를 통해 이기종 시스템 간의 상호 운용성을 보장한다.
  4. 엔터프라이즈 서비스 버스(ESB):
    • 중앙 허브 역할을 하는 ESB를 통해 서비스 간 통신, 메시지 라우팅, 데이터 변환 등을 관리한다. 이를 통해 시스템 통합을 용이하게 한다.

장단점 분석#

장점#

  • 재사용성: 한 번 개발된 서비스는 다중 애플리케이션에서 활용 가능 (예: 고객 조회 서비스).
  • 유연한 통합: 이기종 시스템 간 데이터 교환 용이 (은행 계좌 시스템 ↔ 결제 게이트웨이).
  • 비용 효율성: 레거시 시스템 재활용으로 인프라 투자 절감.

단점#

  • 성능 오버헤드: XML 기반 메시징으로 인한 지연.
  • 복잡한 관리: ESB 설정 및 서비스 버전 관리 어려움.
  • 벤더 종속성: 특정 ESB 솔루션에 대한 의존성 발생.

실제 적용 사례#

  1. 금융 분야: 하나은행

    • 문제점: 기존 시스템의 데이터 중복 및 유지보수 비용 증가.
    • 해결책: 15개 핵심 컴포넌트를 SOA로 전환, ESB를 통해 계좌 조회/이체 서비스 통합.
    • 효과: 트랜잭션 처리 시간 40% 단축.

  2. 제조업: U-헬스케어 시스템

    • 과제: 다양한 장치 간 데이터 호환성 부재.
    • 구현: SOAP 기반 서비스로 혈압/혈당 데이터 통합 모니터링.
    • 결과: 병원-보험사 간 실시간 데이터 공유 체계 구축.

SOA Vs 마이크로서비스(MSA) 비교#

기준SOAMSA
구성 단위대형 비즈니스 서비스소규모 기능 단위
통신 방식ESB 의존API Gateway + 경량 프로토콜(HTTP/JSON)
데이터 관리공유 데이터베이스서비스별 독립 DB
적합 시나리오엔터프라이즈 레거시 통합클라우드 네이티브 애플리케이션[8][31]

구현 전략#

  1. 서비스 식별 방법

    • 비즈니스 역량 매핑: 조직의 핵심 업무 프로세스를 서비스 후보군으로 추출 (예: 주문 처리, 재고 관리).
    • 기존 시스템 분석: 레거시 코드를 모듈화해 서비스 경계 정의.

  2. 성능 최적화

    • 메시지 압축: JSON 사용으로 XML 대비 70% 크기 감소.
    • 캐싱 전략: 자주 호출되는 서비스 결과를 인메모리 DB에 저장.
