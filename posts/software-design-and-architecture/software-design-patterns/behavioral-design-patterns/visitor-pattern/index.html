Visitor Pattern#

Visitor Pattern은 객체 구조와 알고리즘을 분리하는 행위 디자인 패턴이다.
이 패턴의 주요 목적은 기존 객체 구조를 변경하지 않고 새로운 동작을 추가할 수 있게 하는 것이다.

Visitor Pattern의 주요 구성 요소#

  1. Visitor (방문자) 인터페이스:

    • 객체 구조의 각 ConcreteElement에 대한 visit() 메서드를 선언한다.

  2. ConcreteVisitor (구체적인 방문자):

    • Visitor 인터페이스를 구현하여 각 ConcreteElement에 대한 구체적인 동작을 정의한다.

  3. Element (요소) 인터페이스:

    • accept(Visitor) 메서드를 선언하여 Visitor 객체를 받아들인다.

  4. ConcreteElement (구체적인 요소):

    • Element 인터페이스를 구현하고, accept() 메서드에서 visitor.visit(this)를 호출한다.

  5. ObjectStructure (객체 구조):

    • Element 객체들의 집합을 관리하고, Visitor가 각 요소를 방문할 수 있도록 한다.

Visitor Pattern의 동작 원리#

  1. Client가 ConcreteVisitor 객체를 생성하고 ObjectStructure에 전달한다.
  2. ObjectStructure는 각 Element에 대해 accept() 메서드를 호출한다.
  3. 각 Element는 visitor.visit(this)를 호출하여 자신을 방문자에게 전달한다.
  4. ConcreteVisitor는 전달받은 Element에 대한 특정 동작을 수행한다.

Visitor Pattern의 장점#

  1. 새로운 동작 추가가 용이: 새로운 ConcreteVisitor를 추가하여 기존 객체 구조를 변경하지 않고 새로운 동작을 추가할 수 있다.
  2. 관련 동작 그룹화: 특정 기능과 관련된 코드를 ConcreteVisitor에 모아 관리할 수 있다.
  3. 객체 구조와 알고리즘 분리: 데이터 구조와 처리 로직을 분리하여 단일 책임 원칙을 준수한다.

Visitor Pattern의 단점#

  1. 객체 구조 변경의 어려움: 새로운 ConcreteElement 추가 시 모든 Visitor 인터페이스와 구현체를 수정해야 한다.
  2. 캡슐화 위반 가능성: Visitor가 ConcreteElement의 내부 상태에 접근해야 할 경우 캡슐화가 깨질 수 있다.
  3. 복잡성 증가: 객체 간의 상호작용이 복잡해질 수 있어 코드 이해와 유지보수가 어려워질 수 있다.

사용 예시#

Visitor Pattern은 다음과 같은 상황에서 유용하게 사용될 수 있다:

  1. 복잡한 객체 구조에 대한 다양한 연산 수행 (예: 파일 시스템 탐색)
  2. 컴파일러의 추상 구문 트리 처리
  3. 보고서 생성이나 데이터 내보내기 기능 구현
  4. GUI 요소에 대한 다양한 동작 정의 (예: 렌더링, 이벤트 처리)

// Element 인터페이스
interface Shape {
    void accept(ShapeVisitor visitor);
}

// ConcreteElement 클래스들
class Circle implements Shape {
    private double radius;
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    public double getRadius() {
        return radius;
    }
    
    @Override
    public void accept(ShapeVisitor visitor) {
        visitor.visit(this);
    }
}

// Visitor 인터페이스
interface ShapeVisitor {
    void visit(Circle circle);
    void visit(Rectangle rectangle);
}

// ConcreteVisitor 클래스
class AreaCalculator implements ShapeVisitor {
    @Override
    public void visit(Circle circle) {
        double area = Math.PI * circle.getRadius() * circle.getRadius();
        System.out.println("원의 넓이: " + area);
    }
    
    @Override
    public void visit(Rectangle rectangle) {
        double area = rectangle.getWidth() * rectangle.getHeight();
        System.out.println("직사각형의 넓이: " + area);
    }
}

from abc import ABC, abstractmethod

# Visitor 인터페이스
class Visitor(ABC):
    @abstractmethod
    def visit_concrete_element_a(self, element):
        pass

    @abstractmethod
    def visit_concrete_element_b(self, element):
        pass

# ConcreteVisitor 클래스
class ConcreteVisitor1(Visitor):
    def visit_concrete_element_a(self, element):
        print(f"{element.feature()}를 활용한 ConcreteVisitor1의 연산 수행")

    def visit_concrete_element_b(self, element):
        print(f"{element.feature()}를 활용한 ConcreteVisitor1의 연산 수행")

# Element 인터페이스
class Element(ABC):
    @abstractmethod
    def accept(self, visitor):
        pass

# ConcreteElementA 클래스
class ConcreteElementA(Element):
    def accept(self, visitor):
        visitor.visit_concrete_element_a(self)

    def feature(self):
        return "ConcreteElementA의 기능"

# ConcreteElementB 클래스
class ConcreteElementB(Element):
    def accept(self, visitor):
        visitor.visit_concrete_element_b(self)

    def feature(self):
        return "ConcreteElementB의 기능"

# 클라이언트 코드
def client_code(elements, visitor):
    for element in elements:
        element.accept(visitor)

# 사용 예시
elements = [ConcreteElementA(), ConcreteElementB()]
visitor = ConcreteVisitor1()
client_code(elements, visitor)
