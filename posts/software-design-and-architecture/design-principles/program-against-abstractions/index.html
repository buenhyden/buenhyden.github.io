Program Against Abstractions#

“추상화에 대한 프로그래밍”(Program Against Abstractions)은 소프트웨어 설계의 핵심 원칙으로, 구체적인 구현보다는 추상적인 인터페이스나 기본 타입에 의존하여 코드를 작성해야 한다는 개념이다.
이 원칙은 “구현이 아닌 인터페이스에 대해 프로그래밍하라”(Program to an Interface, Not an Implementation)라는 표현으로도 널리 알려져 있다.

이 원칙의 핵심은 코드가 특정 구현의 세부 사항에 의존하는 대신 더 일반적이고 안정적인 추상화에 의존해야 한다는 것이다. 이를 통해 시스템은 더 유연해지고, 변경에 더 견고해지며, 재사용성이 향상된다.

“추상화에 대한 프로그래밍"은 소프트웨어 설계에서 중요한 원칙이다.
이 원칙을 따르면 코드의 결합도가 낮아지고, 유연성이 증가하며, 테스트와 유지 관리가 더 쉬워진다. 이 원칙은 SOLID 원칙, 디자인 패턴, 그리고 현대적인 소프트웨어 아키텍처의 많은 측면과 밀접하게 연관되어 있다.

추상화의 형태#

소프트웨어 개발에서 추상화는 여러 형태로 나타날 수 있다:

  1. 인터페이스 (Interfaces): 객체가 구현해야 하는 메소드의 계약을 정의한다.
  2. 추상 클래스 (Abstract Classes): 일부 구현을 제공하면서 확장을 위한 템플릿을 정의한다.
  3. 프로토콜 (Protocols): 특히 Swift나 Objective-C와 같은 언어에서 객체가 준수해야 하는 메소드 집합을 정의한다.
  4. 일반 타입 (Generic Types): 다양한 데이터 타입에 대해 작동할 수 있는 알고리즘이나 데이터 구조를 정의한다.
  5. 함수형 인터페이스 (Functional Interfaces): 특히 함수형 프로그래밍에서 함수의 시그니처를 정의한다.

이 원칙의 이점#

추상화에 대한 프로그래밍은 다음과 같은 중요한 이점을 제공한다:

결합도 감소 (Reduced Coupling)#

코드가 특정 구현보다 추상화에 의존할 때, 구현 세부 사항의 변경이 전체 시스템에 미치는 영향이 줄어든다. 이는 시스템의 다른 부분이 서로 덜 얽혀있게 만들어 변경이 더 쉬워진다.

// 구현에 의존 (좋지 않음)

// 추상화에 의존 (더 좋음)

위 예제에서, 두 번째 접근 방식은 필요에 따라 ArrayList에서 LinkedList로 구현을 쉽게 변경할 수 있도록 한다.

유연성 증가 (Increased Flexibility)#

# 인터페이스/추상화를 활용한 전략 패턴
class PaymentProcessor:
    def __init__(self, payment_strategy):
        self.payment_strategy = payment_strategy
        
    def process_payment(self, amount):
        return self.payment_strategy.process(amount)

# 다양한 구현
class CreditCardPayment:
    def process(self, amount):
        print(f"{amount}원이 신용카드로 처리되었습니다.")
        
class PayPalPayment:
    def process(self, amount):
        print(f"{amount}원이 PayPal로 처리되었습니다.")

# 사용 예시
processor = PaymentProcessor(CreditCardPayment())
processor.process_payment(10000)

# 쉽게 전략 변경 가능
processor.payment_strategy = PayPalPayment()
processor.process_payment(10000)

// 인터페이스에 의존
public class OrderService {
    private PaymentGateway paymentGateway;
    
    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }
    
    public boolean processOrder(Order order) {
        // 주문 로직
        return paymentGateway.processPayment(order.getTotal());
    }
}

// 테스트
public void testOrderProcessing() {
    // 목 구현 생성
    PaymentGateway mockGateway = mock(PaymentGateway.class);
    when(mockGateway.processPayment(anyDouble())).thenReturn(true);
    
    OrderService service = new OrderService(mockGateway);
    Order testOrder = new Order(100.0);
    
    assertTrue(service.processOrder(testOrder));
    verify(mockGateway).processPayment(100.0);
}

// 추상 클래스 정의
public abstract class Shape {
    public abstract double CalculateArea();
}

// 확장 가능
public class Circle : Shape {
    private double radius;
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    public override double CalculateArea() {
        return Math.PI * radius * radius;
    }
}

public class Rectangle : Shape {
    private double width;
    private double height;
    
    public Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }
    
    public override double CalculateArea() {
        return width * height;
    }
}

// 인터페이스 정의
interface Logger {
    log(message: string): void;
}

// 구현체
class ConsoleLogger implements Logger {
    log(message: string): void {
        console.log(message);
    }
}

class FileLogger implements Logger {
    private filePath: string;
    
    constructor(filePath: string) {
        this.filePath = filePath;
    }
    
    log(message: string): void {
        // 파일에 로깅 로직
        console.log(`Writing to ${this.filePath}: ${message}`);
    }
}

// 인터페이스에 의존하는 서비스
class UserService {
    private logger: Logger;
    
    constructor(logger: Logger) {
        this.logger = logger;
    }
    
    createUser(username: string): void {
        // 사용자 생성 로직
        this.logger.log(`사용자 생성됨: ${username}`);
    }
}

// 사용 예시
const consoleLogger = new ConsoleLogger();
const userService = new UserService(consoleLogger);
userService.createUser("홍길동");

// 로깅 전략 변경
const fileLogger = new FileLogger("/var/log/users.log");
const fileBasedUserService = new UserService(fileLogger);
fileBasedUserService.createUser("김철수");

// 인터페이스
public interface Vehicle {
    void drive();
}

// 구현체
public class Car implements Vehicle {
    @Override
    public void drive() {
        System.out.println("자동차를 운전합니다");
    }
}

public class Truck implements Vehicle {
    @Override
    public void drive() {
        System.out.println("트럭을 운전합니다");
    }
}

// 팩토리
public class VehicleFactory {
    public static Vehicle createVehicle(String type) {
        if ("car".equalsIgnoreCase(type)) {
            return new Car();
        } else if ("truck".equalsIgnoreCase(type)) {
            return new Truck();
        }
        throw new IllegalArgumentException("알 수 없는 차량 유형: " + type);
    }
}

// 클라이언트 코드
public class Client {
    public static void main(String[] args) {
        // 추상화에 의존
        Vehicle vehicle = VehicleFactory.createVehicle("car");
        vehicle.drive();
    }
}

// React에서의 추상화 - 컴포넌트 인터페이스
class Button extends React.Component {
  render() {
    return <button onClick={this.props.onClick}>{this.props.label}</button>;
  }
}

// 다양한 버튼 스타일 (구현)
class PrimaryButton extends Button {
  render() {
    return <button className="primary" onClick={this.props.onClick}>{this.props.label}</button>;
  }
}

class SecondaryButton extends Button {
  render() {
    return <button className="secondary" onClick={this.props.onClick}>{this.props.label}</button>;
  }
}

// 사용 (추상화에 의존)
function App() {
  return (
    <div>
      <Button onClick={() => console.log('클릭됨')} label="기본 버튼" />
      <PrimaryButton onClick={() => console.log('기본 클릭')} label="주요 버튼" />
      <SecondaryButton onClick={() => console.log('보조 클릭')} label="보조 버튼" />
    </div>
  );
}

주의사항 및 도전과제#

추상화에 대한 프로그래밍은 강력하지만, 몇 가지 주의해야 할 점이 있다:

  1. 과도한 추상화 피하기
    너무 많은 추상화 계층은 코드를 불필요하게 복잡하게 만들 수 있다. 항상 추상화의 비용과 이점을 고려해야 한다.

  2. 적절한 추상화 수준 선택
    추상화 수준이 너무 높으면 유연성은 증가하지만 명확성이 감소할 수 있다. 반면 추상화 수준이 너무 낮으면 코드 재사용이 제한될 수 있다.

  3. 누수된 추상화 (Leaky Abstractions)
    모든 추상화는 어느 정도 “누수"될 수 있다. 이는 기본 구현의 세부 사항이 추상화 계층을 통해 “누출"되는 경우를 말한다. 이러한 상황에서는 추상화의 이점이 감소한다.
