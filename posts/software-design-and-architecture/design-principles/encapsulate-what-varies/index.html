Encapsulate What Varies

변경 가능성이 높은 부분을 식별하고 이를 캡슐화하여 나머지 코드에 미치는 영향을 최소화하는 접근 방식이다.

Encapsulate What Varies#

소프트웨어 설계에서 “가변성 캡슐화(Encapsulate What Varies)” 원칙은 변경 가능성이 높은 부분을 식별하고 이를 캡슐화하여 나머지 코드에 미치는 영향을 최소화하는 접근 방식이다. 이 원칙은 객체지향 설계의 기본 개념 중 하나로, 유지보수성, 확장성, 재사용성을 높이는 데 중요한 역할을 한다.

원칙의 정의와 목적#

“가변성 캡슐화” 원칙은 간단히 말해 “변화하는 것을 캡슐화하라"는 지침입니다. 이는 소프트웨어에서 변경될 가능성이 높은 부분을 식별하고, 그것을 나머지 시스템으로부터 분리하여 향후 변경이 시스템 전체에 미치는 영향을 최소화하는 것을 의미한다.

이 원칙의 주요 목적은 다음과 같다:

  1. 변경의 영향 범위 제한: 변경 사항이 시스템의 제한된 부분에만 영향을 미치도록 한다.
  2. 코드 재사용 촉진: 변경되지 않는 부분은 재사용할 수 있다.
  3. 유지보수성 향상: 변경이 필요할 때 시스템의 작은 부분만 수정하면 된다.
  4. 확장성 증대: 기존 코드를 변경하지 않고 새로운 기능을 추가할 수 있다.

원칙의 역사와 배경#

“가변성 캡슐화” 원칙은 1994년 출판된 “디자인 패턴: 재사용 가능한 객체지향 소프트웨어의 요소(Design Patterns: Elements of Reusable Object-Oriented Software)“에서 GoF(Gang of Four)에 의해 공식화되었다. 그들은 이 책에서 다양한 디자인 패턴을 설명하면서 공통적인 원칙으로 “가변성 캡슐화"를 강조했다.

이 원칙은 객체지향 프로그래밍의 핵심 개념인 캡슐화(encapsulation)에서 진화했다. 전통적인 캡슐화가 데이터와 그 데이터를 조작하는 메서드를 함께 묶는 것에 초점을 맞췄다면, “가변성 캡슐화"는 이 개념을 확장하여 변경 가능성이 있는 모든 측면(알고리즘, 의존성, 구현 세부사항 등)을 캡슐화하는 것을 강조한다.

가변성 캡슐화의 핵심 개념#

변화하는 부분 식별하기#

가변성 캡슐화의 첫 번째 단계는 변경될 가능성이 높은 부분을 식별하는 것이다.
이는 다음과 같은 요소들을 포함할 수 있다:

  • 알고리즘이나 비즈니스 규칙: 시간이 지남에 따라 변경될 수 있는 계산 로직이나 비즈니스 정책
  • 외부 시스템과의 상호작용: API, 데이터베이스, 파일 시스템 등 외부 의존성
  • 사용자 인터페이스 요소: 디자인이나 사용자 경험이 변경될 수 있는 UI 컴포넌트
  • 하드웨어나 환경 의존성: 다른 플랫폼이나 환경에서 다르게 동작해야 하는 코드

추상화를 통한 분리#

변화하는 부분을 식별한 후에는 이를 추상화하여 나머지 시스템에서 분리한다. 이는 주로 다음과 같은 방법으로 이루어진다:

  • 인터페이스 정의: 변화하는 부분에 대한 안정적인 인터페이스를 정의한다.
  • 추상 클래스 사용: 공통 동작을 정의하고 변화하는 부분은 하위 클래스에서 구현한다.
  • 컴포지션 활용: 변화하는 부분을 별도의 클래스로 추출하고 합성을 통해 사용한다.

변화에 대한 대비#

가변성 캡슐화는 미래의 변화에 대비하는 것이다. 이는 다음과 같은 방식으로 이루어진다:

  • 확장 지점 제공: 시스템이 향후 확장될 수 있는 지점을 명확히 정의한다.
  • 플러그인 아키텍처: 새로운 기능이나 변형을 플러그인 형태로 추가할 수 있게 한다.
  • 설정 기반 변경: 코드 변경 없이 설정만으로 동작을 변경할 수 있게 한다.

// 전략 인터페이스
interface PaymentStrategy {
    void pay(int amount);
}

// 구체적인 전략 구현
class CreditCardPayment implements PaymentStrategy {
    private String cardNumber;
    
    CreditCardPayment(String cardNumber) {
        this.cardNumber = cardNumber;
    }
    
    public void pay(int amount) {
        System.out.println(amount + " paid with credit card " + cardNumber);
    }
}

class PayPalPayment implements PaymentStrategy {
    private String email;
    
    PayPalPayment(String email) {
        this.email = email;
    }
    
    public void pay(int amount) {
        System.out.println(amount + " paid with PayPal to " + email);
    }
}

// 컨텍스트 클래스
class ShoppingCart {
    private PaymentStrategy paymentStrategy;
    
    void setPaymentStrategy(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }
    
    void checkout(int amount) {
        paymentStrategy.pay(amount);
    }
}

// 사용 예
ShoppingCart cart = new ShoppingCart();
cart.setPaymentStrategy(new CreditCardPayment("1234-5678-9012-3456"));
cart.checkout(100);

// 다른 결제 방법으로 전환
cart.setPaymentStrategy(new PayPalPayment("example@email.com"));
cart.checkout(200);

이 예시에서 ShoppingCart 클래스는 결제 처리 방법의 변화를 캡슐화하기 위해 PaymentStrategy 인터페이스를 사용합니다. 새로운 결제 방법이 추가되더라도 ShoppingCart 클래스를 수정할 필요 없이 새로운 전략 구현체만 추가하면 됩니다.

// 추상 템플릿 클래스
abstract class DataProcessor {
    // 템플릿 메서드
    public final void processData() {
        readData();
        processSpecificData();  // 변화하는 부분
        writeData();
    }
    
    private void readData() {
        System.out.println("Reading data from source");
    }
    
    // 변화하는 부분을 추상 메서드로 선언
    protected abstract void processSpecificData();
    
    private void writeData() {
        System.out.println("Writing processed data");
    }
}

// 구체적인 구현
class XMLDataProcessor extends DataProcessor {
    @Override
    protected void processSpecificData() {
        System.out.println("Processing XML data");
    }
}

class JSONDataProcessor extends DataProcessor {
    @Override
    protected void processSpecificData() {
        System.out.println("Processing JSON data");
    }
}

// 사용 예
DataProcessor xmlProcessor = new XMLDataProcessor();
xmlProcessor.processData();

DataProcessor jsonProcessor = new JSONDataProcessor();
jsonProcessor.processData();

이 패턴에서는 데이터 처리의 일반적인 단계(읽기, 처리, 쓰기)는 고정되어 있지만, 구체적인 처리 방법은 데이터 형식에 따라 달라질 수 있습니다. 템플릿 메서드 패턴은 이러한 변화를 캡슐화하여 각 데이터 형식별 처리기가 자신의 방식으로 데이터를 처리할 수 있게 합니다.

// 제품 인터페이스
interface Vehicle {
    void drive();
}

// 구체적인 제품
class Car implements Vehicle {
    public void drive() {
        System.out.println("Driving a car");
    }
}

class Motorcycle implements Vehicle {
    public void drive() {
        System.out.println("Riding a motorcycle");
    }
}

// 추상 팩토리
abstract class VehicleFactory {
    // 템플릿 메서드
    public void deliverVehicle() {
        Vehicle vehicle = createVehicle();  // 팩토리 메서드
        vehicle.drive();
    }
    
    // 팩토리 메서드 - 변화하는 부분
    protected abstract Vehicle createVehicle();
}

// 구체적인 팩토리
class CarFactory extends VehicleFactory {
    @Override
    protected Vehicle createVehicle() {
        return new Car();
    }
}

class MotorcycleFactory extends VehicleFactory {
    @Override
    protected Vehicle createVehicle() {
        return new Motorcycle();
    }
}

// 사용 예
VehicleFactory factory = new CarFactory();
factory.deliverVehicle();

factory = new MotorcycleFactory();
factory.deliverVehicle();

팩토리 메서드 패턴은 객체 생성 로직의 변화를 캡슐화합니다. 새로운 종류의 차량이 추가되더라도 기존 코드를 수정할 필요 없이 새로운 제품 클래스와 팩토리 클래스만 추가하면 됩니다.

// 컴포넌트 인터페이스
interface Coffee {
    double getCost();
    String getDescription();
}

// 구체적인 컴포넌트
class SimpleCoffee implements Coffee {
    public double getCost() {
        return 1.0;
    }
    
    public String getDescription() {
        return "Simple coffee";
    }
}

// 데코레이터 추상 클래스
abstract class CoffeeDecorator implements Coffee {
    protected Coffee decoratedCoffee;
    
    CoffeeDecorator(Coffee coffee) {
        this.decoratedCoffee = coffee;
    }
    
    public double getCost() {
        return decoratedCoffee.getCost();
    }
    
    public String getDescription() {
        return decoratedCoffee.getDescription();
    }
}

// 구체적인 데코레이터
class MilkDecorator extends CoffeeDecorator {
    MilkDecorator(Coffee coffee) {
        super(coffee);
    }
    
    public double getCost() {
        return super.getCost() + 0.5;
    }
    
    public String getDescription() {
        return super.getDescription() + ", with milk";
    }
}

class SugarDecorator extends CoffeeDecorator {
    SugarDecorator(Coffee coffee) {
        super(coffee);
    }
    
    public double getCost() {
        return super.getCost() + 0.2;
    }
    
    public String getDescription() {
        return super.getDescription() + ", with sugar";
    }
}

// 사용 예
Coffee coffee = new SimpleCoffee();
System.out.println(coffee.getDescription() + ": $" + coffee.getCost());

coffee = new MilkDecorator(coffee);
System.out.println(coffee.getDescription() + ": $" + coffee.getCost());

coffee = new SugarDecorator(coffee);
System.out.println(coffee.getDescription() + ": $" + coffee.getCost());

데코레이터 패턴은 객체의 확장 방법의 변화를 캡슐화한다.
새로운 기능이나 속성을 추가할 때 기존 코드를 수정하지 않고 새로운 데코레이터만 추가하면 된다.

// 인터페이스
public interface UserRepository {
    User findById(Long id);
    void save(User user);
}

// 구현체 1: JPA 기반
@Repository
public class JpaUserRepository implements UserRepository {
    @PersistenceContext
    private EntityManager entityManager;
    
    @Override
    public User findById(Long id) {
        return entityManager.find(User.class, id);
    }
    
    @Override
    public void save(User user) {
        entityManager.persist(user);
    }
}

// 구현체 2: MongoDB 기반
@Repository
public class MongoUserRepository implements UserRepository {
    @Autowired
    private MongoTemplate mongoTemplate;
    
    @Override
    public User findById(Long id) {
        return mongoTemplate.findById(id, User.class);
    }
    
    @Override
    public void save(User user) {
        mongoTemplate.save(user);
    }
}

// 서비스 클래스
@Service
public class UserService {
    private final UserRepository userRepository;
    
    @Autowired
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
    
    public User getUser(Long id) {
        return userRepository.findById(id);
    }
}

이 예시에서 UserService는 구체적인 저장소 구현에 의존하지 않고 UserRepository 인터페이스에 의존한다.
데이터베이스 기술이 변경되더라도 UserService는 수정할 필요가 없다.

// 플러그인 인터페이스
public interface CompilerPlugin {
    void compile(SourceFile file);
    boolean supportsLanguage(String language);
}

// Java 컴파일러 플러그인
public class JavaCompilerPlugin implements CompilerPlugin {
    @Override
    public void compile(SourceFile file) {
        // Java 파일 컴파일 로직
    }
    
    @Override
    public boolean supportsLanguage(String language) {
        return "java".equalsIgnoreCase(language);
    }
}

// Python 컴파일러 플러그인
public class PythonCompilerPlugin implements CompilerPlugin {
    @Override
    public void compile(SourceFile file) {
        // Python 파일 컴파일 로직
    }
    
    @Override
    public boolean supportsLanguage(String language) {
        return "python".equalsIgnoreCase(language);
    }
}

// 플러그인 매니저
public class CompilerManager {
    private List<CompilerPlugin> plugins = new ArrayList<>();
    
    public void registerPlugin(CompilerPlugin plugin) {
        plugins.add(plugin);
    }
    
    public void compileFile(SourceFile file) {
        for (CompilerPlugin plugin : plugins) {
            if (plugin.supportsLanguage(file.getLanguage())) {
                plugin.compile(file);
                return;
            }
        }
        throw new UnsupportedLanguageException(file.getLanguage());
    }
}

이 아키텍처에서는 새로운 언어 지원을 추가할 때 기존 코드를 수정하지 않고 새로운 플러그인만 구현하여 등록하면 된다.

public class ConfigManager {
    private Properties properties = new Properties();
    
    public ConfigManager(String configFile) throws IOException {
        try (InputStream input = new FileInputStream(configFile)) {
            properties.load(input);
        }
    }

        return properties.getProperty(key, defaultValue);
    }
    
    public int getIntProperty(String key, int defaultValue) {
        String value = properties.getProperty(key);
        if (value == null) {
            return defaultValue;
        }
        return Integer.parseInt(value);
    }
}

가변성 캡슐화의 장점#

  1. 변경의 격리
    변화하는 부분을 캡슐화하면 변경이 시스템의 나머지 부분에 미치는 영향을 최소화할 수 있다. 이는 유지보수 비용을 줄이고 버그 발생 가능성을 낮춘다.

  2. 코드 재사용 증가
    변하지 않는 부분은 다양한 맥락에서 재사용할 수 있다. 예를 들어, 템플릿 메서드 패턴에서 공통 알고리즘 구조는 여러 구현에서 재사용된다.

  3. 확장성 향상
    가변성 캡슐화는 기존 코드를 수정하지 않고 새로운 기능을 추가할 수 있게 한다. 이는 “개방-폐쇄 원칙(OCP)“을 지원하며, 시스템을 더 유연하고 확장 가능하게 만든다.

  4. 테스트 용이성
    변화하는 부분을 캡슐화하면 해당 부분을 독립적으로 테스트하기 쉬워진다. 예를 들어, 전략 패턴에서는 각 전략을 개별적으로 테스트할 수 있다.

  5. 병렬 개발 지원
    가변성 캡슐화는 팀 간의 명확한 경계를 설정하여 병렬 개발을 용이하게 한다.
    각 팀은 인터페이스에 따라 자신의 구성 요소를 독립적으로 개발할 수 있다.

가변성 캡슐화의 도전과 주의사항#

  1. 과도한 추상화
    너무 많은 추상화는 코드를 복잡하게 만들고 이해하기 어렵게 만들 수 있다. 변화할 가능성이 낮은 부분까지 캡슐화하면 불필요한 복잡성이 증가한다.

  2. 성능 영향
    추상화 계층을 추가하면 성능에 영향을 미칠 수 있다. 예를 들어, 인터페이스를 통한 메서드 호출은 직접 호출보다 약간 더 느릴 수 있다.

  3. 설계 시간 증가
    가변성을 식별하고 적절한 추상화를 설계하는 데는 시간과 노력이 필요하다. 이는 초기 개발 시간을 증가시킬 수 있다.

  4. 과도한 유연성
    때로는 유연성보다 명확성이 더 중요할 수 있다. 모든 것을 변경 가능하게 만들면 코드의 흐름을 이해하기 어려울 수 있다.

가변성 캡슐화의 적용 전략#

변화 패턴 식별#

가변성 캡슐화를 적용하려면 먼저 시스템에서 변화하는 부분을 식별해야 한다.
이는 다음과 같은 질문을 통해 이루어질 수 있다:

  • 어떤 부분이 가장 자주 변경되는가?
  • 어떤 부분이 다양한 구현을 가질 수 있는가?
  • 어떤 기능이 향후 확장될 가능성이 높은가?
  • 어떤 요구사항이 가장 불확실한가?

적절한 추상화 수준 선택#

변화하는 부분을 식별한 후에는 적절한 추상화 수준을 선택해야 한다.
이는 다음과 같은 요소에 따라 달라질 수 있다:

  • 변화의 자연스러운 경계
  • 시스템의 복잡도
  • 팀의 역량과 경험
  • 성능 요구사항
  • 개발 일정

점진적 리팩토링#

가변성 캡슐화는 종종 점진적인 리팩토링을 통해 적용된다.
기존 코드에서 변화하는 부분을 식별하고, 이를 적절한 추상화로 리팩토링하는 과정이다다:

  1. 변화하는 부분 식별
  2. 적절한 추상화(인터페이스, 추상 클래스 등) 설계
  3. 기존 코드를 리팩토링하여 변화하는 부분을 추상화 뒤로 캡슐화
  4. 필요한 구현체 생성
  5. 테스트를 통한 검증

디자인 패턴 활용#

앞서 살펴본 것처럼, 많은 디자인 패턴은 가변성 캡슐화 원칙을 구현하는 방법을 제공한다.
상황에 맞는 패턴을 선택하여 적용할 수 있다:

  • 전략 패턴: 알고리즘의 변화를 캡슐화
  • 템플릿 메서드 패턴: 알고리즘 구조 내의 특정 단계의 변화를 캡슐화
  • 팩토리 메서드/추상 팩토리 패턴: 객체 생성 로직의 변화를 캡슐화
  • 데코레이터 패턴: 객체 책임의 변화를 캡슐화
  • 어댑터 패턴: 인터페이스의 변화를 캡슐화
  • 옵저버 패턴: 객체 간 의존성과 통신 방식의 변화를 캡슐화
  • 명령 패턴: 요청의 처리 방식의 변화를 캡슐화
  • 상태 패턴: 객체 상태에 따른 동작의 변화를 캡슐화

public interface StorageService {
    void store(String key, byte[] data);
    byte[] retrieve(String key);
    void delete(String key);
}

// AWS S3 구현
public class S3StorageService implements StorageService {
    private final AmazonS3 s3Client;
    private final String bucketName;
    
    // 생략...
    
    @Override
    public void store(String key, byte[] data) {
        s3Client.putObject(bucketName, key, new ByteArrayInputStream(data), new ObjectMetadata());
    }
    
    @Override
    public byte[] retrieve(String key) {
        S3Object object = s3Client.getObject(bucketName, key);
        return IOUtils.toByteArray(object.getObjectContent());
    }
    
    @Override
    public void delete(String key) {
        s3Client.deleteObject(bucketName, key);
    }
}

// 파일 시스템 구현
public class FileSystemStorageService implements StorageService {
    private final Path rootLocation;
    
    // 생략...
    
    @Override
    public void store(String key, byte[] data) {
        try {
            Files.write(rootLocation.resolve(key), data);
        } catch (IOException e) {
            throw new StorageException("Failed to store file", e);
        }
    }
    
    @Override
    public byte[] retrieve(String key) {
        try {
            return Files.readAllBytes(rootLocation.resolve(key));
        } catch (IOException e) {
            throw new StorageException("Failed to retrieve file", e);
        }
    }
    
    @Override
    public void delete(String key) {
        try {
            Files.delete(rootLocation.resolve(key));
        } catch (IOException e) {
            throw new StorageException("Failed to delete file", e);
        }
    }
}

// 서비스에서 사용
@Service
public class DocumentService {
    private final StorageService storageService;
    
    @Autowired
    public DocumentService(StorageService storageService) {
        this.storageService = storageService;
    }
    
    public void saveDocument(String id, byte[] content) {
        storageService.store("documents/" + id, content);
    }
}

// 기능 플래그 서비스
public interface FeatureFlagService {
    boolean isEnabled(String featureName);
    boolean isEnabled(String featureName, String userId);
}

// 구현
@Service
public class FeatureFlagServiceImpl implements FeatureFlagService {
    private final FeatureFlagRepository repository;
    
    @Autowired
    public FeatureFlagServiceImpl(FeatureFlagRepository repository) {
        this.repository = repository;
    }
    
    @Override
    public boolean isEnabled(String featureName) {
        FeatureFlag flag = repository.findByName(featureName);
        return flag != null && flag.isEnabled();
    }
    
    @Override
    public boolean isEnabled(String featureName, String userId) {
        FeatureFlag flag = repository.findByName(featureName);
        if (flag == null || !flag.isEnabled()) {
            return false;
        }
        
        // 사용자별 점진적 출시 로직
        if (flag.getRolloutPercentage() < 100) {
            int hash = Math.abs(userId.hashCode()) % 100;
            return hash < flag.getRolloutPercentage();
        }
        
        return true;
    }
}

// 서비스에서 사용
@Service
public class PaymentService {
    private final FeatureFlagService featureFlagService;
    private final LegacyPaymentProcessor legacyProcessor;
    private final NewPaymentProcessor newProcessor;
    
    // 생략...
    
    public PaymentResult processPayment(String userId, Payment payment) {
        if (featureFlagService.isEnabled("new-payment-processor", userId)) {
            return newProcessor.process(payment);
        } else {
            return legacyProcessor.process(payment);
        }
    }
}

이 예시에서는 새로운 결제 처리 기능의 출시를 점진적으로 제어할 수 있다. 코드 변경 없이 구성만으로 어떤 사용자가 어떤 버전의 기능을 경험할지 결정할 수 있다.
