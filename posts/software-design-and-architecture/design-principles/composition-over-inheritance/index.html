Composition Over Inheritance

상속 대신 객체 조합(Composition)을 우선시함으로써, 시스템의 복잡성을 관리하고 변화에 대응하기 용이한 설계를 가능하게 한다.

Composition Over Inheritance#

객체지향 프로그래밍에서 코드 재사용과 모듈화를 위한 두 가지 주요 방법이 있다: 상속(inheritance)과 합성(composition).
‘합성 우선 상속’ 원칙은 많은 경우에 상속보다 객체 합성을 선호해야 한다는 설계 지침이다. 이 원칙은 객체지향 설계의 유연성, 유지보수성, 확장성을 높이는 데 중요한 역할을 한다.

Composition Over Inheritance는 객체지향 설계에서 유연성과 유지보수성을 극대화할 수 있다.
상속 대신 객체 조합(Composition)을 우선시함으로써, 시스템의 복잡성을 관리하고 변화에 대응하기 용이한 설계를 가능하게 한다.

그러나 모든 설계 결정과 마찬가지로, 합성과 상속 중 어떤 것을 선택할지는 특정 맥락과 요구사항에 따라 달라진다. 진정한 “is-a” 관계가 있거나 다형성이 주요 목표인 경우에는 상속이 적절할 수 있다.

상속과 합성의 기본 개념#

특성상속(Inheritance)컴포지션(Composition)
관계is-a (예: Car is-a Vehicle)has-a (예: Car has-a Engine)
결합도강한 결합 (부모 클래스 변경 영향 큼)느슨한 결합 (컴포넌트 독립적)
유연성컴파일 타임에 관계 고정런타임에 동적 관계 설정 가능
재사용클래스 계층 구조에 의존독립적인 컴포넌트 재사용

상속(Inheritance)#

상속은 한 클래스가 다른 클래스의 특성(속성과 메서드)을 물려받는 메커니즘이다.
이는 “is-a” 관계를 표현한다.

// 상속 예시
class Animal {
    void breathe() {
        System.out.println("Breathing");
    }
}

class Dog extends Animal {
    void bark() {
        System.out.println("Woof");
    }
}

// 사용
Dog dog = new Dog();
dog.breathe(); // Animal로부터 상속
dog.bark();    // Dog에 정의됨

이 예시에서 Dog는 Animal이다(Dog is an Animal). Dog 클래스는 Animal 클래스의 모든 특성을 상속받고, 여기에 자신만의 기능을 추가한다.

합성(Composition)#

합성은 한 클래스가 다른 클래스의 인스턴스를 포함하는 방식이다. 이는 “has-a” 관계를 표현한다.

// 합성 예시
class Engine {
    void start() {
        System.out.println("Engine started");
    }
}

class Car {
    private Engine engine;
    
    Car() {
        this.engine = new Engine();
    }
    
    void start() {
        engine.start();
        System.out.println("Car started");
    }
}

// 사용
Car car = new Car();
car.start();

이 예시에서 Car는 Engine을 가지고 있다(Car has an Engine). Car 클래스는 Engine의 인스턴스를 내부에 포함하고, Engine의 메서드를 사용하여 자신의 기능을 구현한다.

합성 우선 상속 원칙의 배경#

이 원칙은 “Design Patterns: Elements of Reusable Object-Oriented Software”(1994년)에서 GoF(Gang of Four)에 의해 처음 체계적으로 설명되었다. 그들은 “가능하면 상속보다 객체 합성을 선호하라"고 주장했다. 이후 이 개념은 객체지향 설계의 중요한 지침이 되었다.

이 원칙은 또한 리스코프 치환 원칙(LSP)과 관련이 있다. 상속은 부모 클래스와 자식 클래스 사이에 강한 결합을 만들기 때문에, 상속을 잘못 사용하면 LSP 위반으로 이어질 수 있다.

상속의 문제점#

합성 우선 상속 원칙이 등장한 이유는 상속이 가진 여러 문제점 때문이다.

강한 결합도#

상속은 부모 클래스와 자식 클래스 사이에 강한 결합을 만든다. 부모 클래스의 변경이 모든 자식 클래스에 영향을 미친다.

class Stack {
    protected List<Object> elements = new ArrayList<>();
    
    public void push(Object item) {
        elements.add(item);
    }
    
    public Object pop() {
        if (elements.isEmpty()) {
            throw new EmptyStackException();
        }
        return elements.remove(elements.size() - 1);
    }
    
    public int size() {
        return elements.size();
    }
}

// Stack을 확장하는 MyStack
class MyStack extends Stack {
    public void pushAll(Collection<?> items) {
        for (Object item : items) {
            push(item);
        }
    }
}

// 나중에 Stack 클래스가 변경됨
class Stack {
    // 변경: elements를 private으로 변경
    private List<Object> elements = new ArrayList<>();
    
    // 나머지 메서드는 동일
}

이 변경으로 인해 MyStack 클래스의 pushAll 메서드는 더 이상 작동하지 않게 된다.

다중 상속의 제한#

많은 언어(Java, C# 등)는 단일 상속만 지원한다.
이는 여러 소스에서 기능을 상속받아야 할 경우 심각한 제한이 된다.

계층 구조의 경직성#

상속은 컴파일 타임에 결정되므로, 런타임에 객체의 행동을 변경하기 어렵다. 이는 설계의 유연성을 제한한다.

합성의 장점#

합성은 이러한 문제점을 해결하고 더 유연한 설계를 제공한다.

느슨한 결합도#

합성은 객체 간에 느슨한 결합을 만든다.
각 클래스는 독립적으로 존재하며, 서로의 내부 구현에 덜 의존한다.

// 느슨한 결합을 위한 인터페이스 정의
interface Engine {
    void start();
}

class GasolineEngine implements Engine {
    public void start() {
        System.out.println("Gasoline engine started");
    }
}

class ElectricEngine implements Engine {
    public void start() {
        System.out.println("Electric engine started");
    }
}

class Car {
    private Engine engine;
    
    // 의존성 주입을 통해 어떤 Engine 구현체도 받을 수 있음
    Car(Engine engine) {
        this.engine = engine;
    }
    
    void start() {
        engine.start();
        System.out.println("Car started");
    }
}

// 사용
Car gasolineCar = new Car(new GasolineEngine());
Car electricCar = new Car(new ElectricEngine());

이 설계에서는 Engine의 구현이 변경되어도 Car 클래스는 영향을 받지 않는다.

car.changeEngine(new ElectricEngine());
car.start();  // 이제 전기 엔진으로 시작

단일 책임 원칙 준수#

합성은 각 클래스가 단일 책임을 가지도록 장려한다. 이는 SOLID 원칙의 ‘S’(Single Responsibility Principle)를 준수하는 데 도움이 된다.

interface Swimmer {
    void swim();
}

interface Flyer {
    void fly();
}

class SwimImplementation implements Swimmer {
    public void swim() {
        System.out.println("Swimming");
    }
}

class FlyImplementation implements Flyer {
    public void fly() {
        System.out.println("Flying");
    }
}

// Duck은 수영과 비행 능력을 모두 가짐
class Duck {
    private Swimmer swimmingBehavior;
    private Flyer flyingBehavior;
    
    Duck() {
        this.swimmingBehavior = new SwimImplementation();
        this.flyingBehavior = new FlyImplementation();
    }
    
    void swim() {
        swimmingBehavior.swim();
    }
    
    void fly() {
        flyingBehavior.fly();
    }
}

상속이 적합한 경우#

abstract class Shape {
    abstract double area();
}

class Circle extends Shape {
    private double radius;
    
    Circle(double radius) {
        this.radius = radius;
    }
    
    @Override
    double area() {
        return Math.PI * radius * radius;
    }
}

class Rectangle extends Shape {
    private double width;
    private double height;
    
    Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }
    
    @Override
    double area() {
        return width * height;
    }
}

이 예시에서 Circle과 Rectangle은 명확하게 Shape이다. 상속을 사용하는 것이 자연스럽다.

코드 재사용이 아닌 다형성을 위한 경우#

상속의 주요 목적이 코드 재사용이 아니라 다형성(polymorphism)을 달성하기 위한 것이라면, 상속이 적절할 수 있다.

베이스 클래스가 안정적인 경우#

부모 클래스가 잘 설계되어 있고 자주 변경되지 않는다면, 취약한 베이스 클래스 문제의 위험이 낮아진다.

합성과 상속의 균형 잡기#

실제 소프트웨어 설계에서는 합성과 상속을 적절히 균형 있게 사용하는 것이 중요하다.

// 인터페이스 정의
interface Animal {
    void makeSound();
}

// 행동 구현 클래스
class Barking {
    void bark() {
        System.out.println("Woof");
    }
}

class Meowing {
    void meow() {
        System.out.println("Meow");
    }
}

// 인터페이스 구현과 합성 결합
class Dog implements Animal {
    private Barking barkBehavior = new Barking();
    
    public void makeSound() {
        barkBehavior.bark();
    }
}

class Cat implements Animal {
    private Meowing meowBehavior = new Meowing();
    
    public void makeSound() {
        meowBehavior.meow();
    }
}

이 접근 방식은 인터페이스를 통한 다형성의 이점과 합성을 통한 유연성을 모두 제공한다.

설계 결정 가이드라인#

합성과 상속 중 선택할 때 고려해야 할 몇 가지 질문:

  1. 이 관계가 진정한 “is-a” 관계인가? 만약 그렇다면 상속을 고려한다.
  2. 다른 클래스의 기능 일부만 필요한가? 그렇다면 합성이 더 적합하다.
  3. 런타임에 동작을 변경해야 하는가? 합성이 더 나은 선택이다.
  4. 베이스 클래스가 자주 변경되는가? 자주 변경된다면 상속은 위험할 수 있다.
  5. 다중 소스에서 동작을 상속받아야 하는가? 합성이 필요할 것이다.

// 전략 인터페이스
interface PaymentStrategy {
    void pay(int amount);
}

// 구체적인 전략 구현
class CreditCardPayment implements PaymentStrategy {
    private String cardNumber;
    
    CreditCardPayment(String cardNumber) {
        this.cardNumber = cardNumber;
    }
    
    public void pay(int amount) {
        System.out.println(amount + " paid with credit card " + cardNumber);
    }
}

class PayPalPayment implements PaymentStrategy {
    private String email;
    
    PayPalPayment(String email) {
        this.email = email;
    }
    
    public void pay(int amount) {
        System.out.println(amount + " paid with PayPal to " + email);
    }
}

// 합성을 사용하는 컨텍스트 클래스
class ShoppingCart {
    private PaymentStrategy paymentStrategy;
    
    void setPaymentStrategy(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }
    
    void checkout(int amount) {
        paymentStrategy.pay(amount);
    }
}

// 사용 예
ShoppingCart cart = new ShoppingCart();
cart.setPaymentStrategy(new CreditCardPayment("1234-5678-9012-3456"));
cart.checkout(100);

// 다른 결제 방법으로 전환
cart.setPaymentStrategy(new PayPalPayment("example@email.com"));
cart.checkout(200);

이 패턴은 상속 없이 여러 결제 전략을 구현하고 런타임에 전환할 수 있게 한다.

// 기본 인터페이스
interface Coffee {
    double getCost();
    String getDescription();
}

// 기본 구현체
class SimpleCoffee implements Coffee {
    public double getCost() {
        return 1.0;
    }
    
    public String getDescription() {
        return "Simple coffee";
    }
}

// 데코레이터 기본 클래스
abstract class CoffeeDecorator implements Coffee {
    protected Coffee decoratedCoffee;
    
    CoffeeDecorator(Coffee coffee) {
        this.decoratedCoffee = coffee;
    }
    
    public double getCost() {
        return decoratedCoffee.getCost();
    }
    
    public String getDescription() {
        return decoratedCoffee.getDescription();
    }
}

// 구체적인 데코레이터들
class MilkDecorator extends CoffeeDecorator {
    MilkDecorator(Coffee coffee) {
        super(coffee);
    }
    
    public double getCost() {
        return super.getCost() + 0.5;
    }
    
    public String getDescription() {
        return super.getDescription() + ", with milk";
    }
}

class SugarDecorator extends CoffeeDecorator {
    SugarDecorator(Coffee coffee) {
        super(coffee);
    }
    
    public double getCost() {
        return super.getCost() + 0.2;
    }
    
    public String getDescription() {
        return super.getDescription() + ", with sugar";
    }
}

// 사용 예
Coffee coffee = new SimpleCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);

System.out.println(coffee.getDescription()); // "Simple coffee, with milk, with sugar"
System.out.println(coffee.getCost());        // 1.7

이 패턴은 합성과 약간의 상속을 결합하지만, 주요 동작은 합성을 통해 이루어진다.

import javax.swing.*;
import java.awt.*;

public class SimpleWindow {
    public static void main(String[] args) {
        // 메인 프레임 생성
        JFrame frame = new JFrame("Simple Window");
        frame.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        frame.setSize(300, 200);
        
        // 패널 생성
        JPanel panel = new JPanel();
        
        // 버튼 생성
        JButton button = new JButton("Click Me");
        
        // 라벨 생성
        JLabel label = new JLabel("Hello, World!");
        
        // 합성: 패널에 컴포넌트 추가
        panel.add(button);
        panel.add(label);
        
        // 합성: 프레임에 패널 추가
        frame.add(panel);
        
        // 프레임 표시
        frame.setVisible(true);
    }
}

이 예시에서 JFrame은 JPanel을 포함하고, JPanel은 JButton과 JLabel을 포함한다. 이는 모두 상속보다 합성을 통해 이루어진다.
