DRY(Don’t Repeat Yourself) Principle#

DRY(Don’t Repeat Yourself) 원칙은 소프트웨어 개발의 기본 지침으로, 중복을 제거하고 코드의 유지보수성, 가독성, 신뢰성을 향상시키는 데 중요한 역할을 한다.
이 원칙은 “모든 지식은 시스템 내에서 단 한 번만, 단 한 곳에서만 표현되어야 한다” 는 개념을 중심으로 한다.

그러나 DRY 원칙은 맹목적으로 적용해서는 안 된다. 과도한 추상화, 조기 추상화, 불필요한 의존성 등의 위험을 고려해야 한다. DRY를 효과적으로 적용하려면 맥락을 고려하고, 점진적으로 리팩토링하며, 테스트로 변경 사항을 뒷받침하고, 공통점과 차이점을 명확히 식별해야 한다.

현대적인 소프트웨어 개발에서 DRY는 여전히 중요하지만, 마이크로서비스, 분산 시스템 및 대규모 애플리케이션에서는 상황에 맞게 적용해야 한다. WET(Write Everything Twice)와 AHA(Avoid Hasty Abstractions) 같은 보완적인 접근 방식도 고려할 가치가 있다.

DRY 원칙의 정의와 역사#

정의#

DRY 원칙은 간단히 말해 “모든 지식은 시스템 내에서 단 한 번만, 단 한 곳에서만 표현되어야 한다"는 것이다. 이는 코드, 데이터, 비즈니스 로직, 문서 등 소프트웨어 시스템의 모든 측면에 적용된다.

앤디 헌트(Andy Hunt)와 데이브 토머스(Dave Thomas)는 1999년 그들의 저서 “실용주의 프로그래머(The Pragmatic Programmer)“에서 이 원칙을 다음과 같이 설명했다:

“모든 지식 조각은 시스템 내에서 단 한 번만, 명확하고 권위 있게 표현되어야 한다.”

역사적 배경#

DRY 원칙은 공식적으로는 “실용주의 프로그래머"에서 처음 명명되고 체계화되었지만, 그 개념 자체는 그 이전부터 존재했다. 프로그래밍의 초기부터 개발자들은 코드 중복을 피하고 재사용 가능한 구성 요소를 만드는 것의 중요성을 인식했다.

DRY 원칙은 다음과 같은 소프트웨어 개발의 진화와 함께 발전했다:

  • 구조적 프로그래밍의 등장과 서브루틴(함수) 사용
  • 모듈식 프로그래밍의 발전
  • 객체지향 프로그래밍의 캡슐화와 상속 개념
  • 컴포넌트 기반 개발과 서비스 지향 아키텍처

    public double calculateDiscount(Order order) {

        }
        return 0;
    }
}

class ReportGenerator {
    public double estimateDiscount(double amount) {
        if (amount > 1000) {
            return amount * 0.1; // 동일한 할인 규칙이 중복됨
        }
        return 0;
    }
}

표현 중복#

UI 컴포넌트, 문서, 주석 등에서 동일한 정보가 반복되는 경우.

“단 한 번, 단 한 곳에"의 의미#

DRY 원칙의 핵심은 모든 개념이 코드베이스에서 단 한 번만 표현되어야 한다는 것이다.
이는 다음을 의미한다:

  • 특정 기능이나 알고리즘은 한 곳에서만 구현되어야 한다.
  • 비즈니스 규칙은 단일 권위 있는 소스에서 정의되어야 한다.
  • 상수, 구성 값 등은 한 곳에서 정의되고 참조되어야 한다.

// 기본 검증 기능을 제공하는 추상 클래스
abstract class BaseValidator {
    protected boolean validateEmail(String email) {
        return email != null && email.contains("@");
    }
    
    protected boolean validateName(String name) {
        return name != null && name.length() >= 2;
    }
}

// 상속을 통한 재사용
class UserValidator extends BaseValidator {
    public boolean validateUser(User user) {
        return validateEmail(user.getEmail()) && validateName(user.getName());
    }
}

// 구성(합성)을 통한 재사용
class ProductValidator {
    private BaseValidator validator;
    
    public ProductValidator(BaseValidator validator) {
        this.validator = validator;
    }
    
    public boolean validateProduct(Product product) {
        return validator.validateName(product.getName());
    }
}

// 제네릭 타입을 사용한 재사용 가능한 검증 유틸리티
class ValidationResult<T> {
    isValid: boolean;
    errors: Record<keyof T, string>;
    
    constructor(isValid: boolean, errors: Record<keyof T, string> = {} as Record<keyof T, string>) {
        this.isValid = isValid;
        this.errors = errors;
    }
}

// 다양한 타입에 적용 가능한 검증 로직
function validateRequired<T>(object: T, fields: Array<keyof T>): ValidationResult<T> {
    const errors = {} as Record<keyof T, string>;
    let isValid = true;
    
    for (const field of fields) {
        if (!object[field]) {
            errors[field] = `${String(field)} is required`;
            isValid = false;
        }
    }
    
    return new ValidationResult<T>(isValid, errors);
}

// 다양한 객체 타입에 동일한 검증 로직 적용
const userResult = validateRequired(user, ['name', 'email']);
const productResult = validateRequired(product, ['name', 'price']);

DRY 원칙의 장점#

DRY 원칙을 따르면 다양한 이점을 얻을 수 있다:

  1. 유지보수성 향상
    코드 중복이 줄어들면 변경이 필요할 때 한 곳만 수정하면 되므로 유지보수가 더 쉬워진다. 이는 특히 대규모 시스템이나 장기 프로젝트에서 중요하다.

  2. 버그 감소
    동일한 로직이 여러 곳에서 반복되면 한 곳에서는 버그를 수정하고 다른 곳에서는 누락될 가능성이 높다. DRY 원칙을 따르면 이러한 불일치를 방지할 수 있다.

  3. 코드 가독성 향상
    잘 추상화된 코드는 일반적으로 더 읽기 쉽고 이해하기 쉽다. 코드의 의도와 기능이 명확하게 나타난다.

  4. 코드 크기 감소
    중복을 제거하면 전체 코드베이스의 크기가 줄어들어 개발, 테스트, 배포 및 학습이 더 쉬워진다.

  5. 테스트 용이성
    DRY 코드는 일반적으로 더 모듈화되어 있어 단위 테스트가 더 쉽다. 로직이 여러 곳에 흩어져 있지 않으므로 테스트 범위를 더 쉽게 보장할 수 있다.

  6. 쉬운 리팩토링
    중복이 적은 코드는 리팩토링이 더 쉽다. 변경의 영향 범위가 더 작고 예측 가능하기 때문이다.

DRY 적용의 현실적 도전과 주의사항#

DRY 원칙은 강력하지만, 과도하게 또는 부적절하게 적용하면 문제가 발생할 수 있다:

  1. 과도한 추상화
    DRY를 과도하게 적용하면 과도한 추상화로 이어질 수 있으며, 이는 코드를 더 복잡하고 이해하기 어렵게 만들 수 있다.

// 너무 일반화된 함수는 이해하기 어려울 수 있음
public <T, U, V> V processGenericEntity(
    T entity, 
    Function<T, U> transformer, 
    BiFunction<U, Map<String, Object>, V> processor,
    Map<String, Object> context) {

    U transformed = transformer.apply(entity);
    return processor.apply(transformed, context);
}

  2. 조기 추상화
    패턴이 완전히 확립되기 전에 추상화하면 잘못된 추상화가 발생할 수 있다. 이는 장기적으로 더 많은 문제를 야기할 수 있다.

  3. 불필요한 의존성
    DRY를 위해 공유 구성 요소를 만들면 불필요한 의존성이 도입될 수 있다. 이는 시스템의 다른 부분 간에 과도한 결합을 초래할 수 있다.

  4. WET(Write Everything Twice) 원칙
    때로는 “Write Everything Twice”(또는 “We Enjoy Typing”) 접근 방식이 더 적절할 수 있다. 이는 패턴이 세 번 이상 반복될 때까지 코드 중복을 허용하는 것을 제안한다. 이 접근 방식은 추상화하기 전에 패턴이 실제로 반복되는지 확인하는 데 도움이 된다.

  5. AHA(Avoid Hasty Abstractions) 원칙
    켄트 C. 도즈(Kent C. Dodds)가 제안한 “서두른 추상화를 피하라(Avoid Hasty Abstractions)” 원칙은 중복이 항상 나쁜 것은 아니며, 잘못된 추상화가 중복보다 더 해롭다고 제안한다.

  6. 맥락 고려의 중요성
    코드가 유사해 보이더라도 다른 맥락에서 사용되거나 다른 이유로 변경될 수 있다면, 추상화하기 전에 주의해야 한다.

DRY vs. WET vs. AHA#

시간이 지남에 따라 대안적인 접근 방식이 등장했다:

  • WET(Write Everything Twice 또는 We Enjoy Typing): 과도한 추상화보다 약간의 중복이 낫다는 아이디어를 제안한다.
  • AHA(Avoid Hasty Abstractions): 추상화를 서두르지 말고, 패턴이 명확해질 때까지 중복을 허용하는 것을 권장한다.

DRY 원칙을 효과적으로 적용하기 위한 실용적인 지침#

DRY 원칙을 적용할 때 다음 지침이 도움이 될 수 있습니다:

  1. 맥락 고려하기
    코드가 유사해 보이더라도 다른 맥락에서 사용되거나 다른 이유로 변경될 수 있다면, 추상화하기 전에 주의해야 한다.

  2. 점진적으로 리팩토링하기
    대규모 리팩토링 대신 점진적으로 중복을 제거하는 것이 좋다. 모든 중복을 한 번에 제거하려고 하지 않는다.

  3. 테스트로 뒷받침하기
    리팩토링 전에 테스트를 작성하고, 코드를 DRY하게 만든 후에도 동일한 테스트가 통과하는지 확인한다.

  4. 공통점과 차이점을 식별하기
    코드가 얼마나 유사한지, 그리고 어떤 면에서 다른지 명확히 파악한다. 이는 적절한 추상화 수준을 결정하는 데 도움이 된다.

  5. DRY를 코드 너머로 확장하기
    DRY 원칙을 코드뿐만 아니라 문서, 빌드 스크립트, 배포 구성 등에도 적용한다.
