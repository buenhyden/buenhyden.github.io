YAGNI(You Ain’t Gonna Need It) Principle#

YAGNI(You Ain’t Gonna Need It)는 소프트웨어 개발에서 중요한 원칙으로, “당신은 그것이 필요하지 않을 것이다"라는 의미를 담고 있다. 이 원칙은 개발자가 현재 필요하지 않은 기능이나 코드를 미리 구현하지 말아야 한다고 제안한다.

YAGNI 원칙은 익스트림 프로그래밍(XP, Extreme Programming)의 창시자인 론 제프리스(Ron Jeffries)와 다른 XP 선구자들에 의해 1990년대에 공식화되었다. 이들은 미래에 필요할지도 모른다는 예측에 기반하여 기능을 구현하는 것이 종종 낭비가 된다는 점을 깨달았다.

현재 요구사항에 집중하고 미래에 대한 과도한 예측을 피함으로써, 개발자는 더 단순하고, 유지보수하기 쉬우며, 실제 가치를 제공하는 코드를 작성할 수 있다.

그러나 YAGNI는 무조건적으로 적용해서는 안 된다. 좋은 아키텍처 결정, 비용-이익 분석, 실증적 경험을 고려하여 균형 있게 적용해야 한다. 지속적인 리팩토링과 함께 YAGNI를 적용할 때, 소프트웨어는 변화하는 요구사항을 유연하게 수용하면서도 불필요한 복잡성을 피할 수 있다.

YAGNI 원칙의 핵심 철학#

YAGNI 원칙의 핵심 사상은 다음과 같다:

  1. 현재 요구사항에 집중하라: 지금 당장 필요한 것만 개발하고, 미래에 필요할 ‘수도’ 있는 것들은 구현하지 마라.
  2. 미래 예측의 어려움 인정: 소프트웨어 요구사항은 자주 변경되며, 미래를 정확히 예측하는 것은 거의 불가능하다.
  3. 낭비 감소: 결국 사용되지 않는 코드를 작성하는 시간과 노력을 절약할 수 있다.
  4. 복잡성 최소화: 불필요한 코드는 시스템의 복잡성을 증가시키고 유지보수를 어렵게 만든다.

YAGNI와 애자일 방법론의 관계#

YAGNI는 애자일 방법론, 특히 익스트림 프로그래밍의 핵심 원칙 중 하나이다.
애자일의 기본 철학 중 하나는 “변화에 대응하는 것이 계획을 따르는 것보다 가치 있다"는 것이다.
YAGNI는 이러한 철학을 코드 수준에서 구체화한 것으로 볼 수 있다.

애자일 방법론에서 YAGNI를 지원하는 다른 관행들:

  • 점진적 개발: 작은 증분으로 소프트웨어를 개발하여 현재 필요한 기능에 집중한다.
  • 지속적인 리팩토링: 코드를 깔끔하게 유지하면서 새로운 요구사항을 수용할 수 있도록 한다.
  • 사용자 스토리 기반 개발: 실제 사용자 요구사항에 기반한 개발을 촉진한다.

def calculate_employee_salary(base_salary, hours_worked, overtime_hours=0, bonus=0, 
                           tax_rate=0.2, include_pension=False, pension_rate=0.05,
                           include_healthcare=False, healthcare_cost=100):
    """
    직원 급여 계산 - 현재는 기본 급여와 근무 시간만 사용하지만,
    미래에 추가 기능이 필요할 수 있으므로 많은 옵션을 추가
    """
    salary = base_salary * hours_worked
    
    if overtime_hours > 0:
        salary += overtime_hours * base_salary * 1.5
    
    if bonus > 0:
        salary += bonus
    
    gross_salary = salary
    
    if include_pension:
        pension = gross_salary * pension_rate
        salary -= pension
    
    if include_healthcare:
        salary -= healthcare_cost
    
    tax = salary * tax_rate
    net_salary = salary - tax
    
    return {
        "gross_salary": gross_salary,
        "pension": pension if include_pension else 0,
        "healthcare": healthcare_cost if include_healthcare else 0,
        "tax": tax,
        "net_salary": net_salary
    }

YAGNI의 이점#

  1. 개발 시간 단축: 실제로 필요한 기능에만 집중하여 제품을 더 빨리 출시할 수 있다.
  2. 코드 복잡성 감소: 불필요한 코드가 없어 시스템이 더 간단해지고 이해하기 쉬워진다.
  3. 기술 부채 감소: 사용되지 않는 코드는 유지보수 비용을 증가시키는 기술 부채가 될 수 있다.
  4. 설계 유연성 향상: 실제 요구사항이 명확해질 때까지 설계 결정을 미룸으로써, 후에 더 적합한 설계를 선택할 수 있는 유연성을 확보한다.
  5. 리소스 효율성: 개발자의 시간과 노력을 현재 가치를 제공하는 기능에 집중할 수 있다.

YAGNI 적용 시 고려사항과 균형점#

YAGNI는 강력한 원칙이지만, 무조건적으로 적용해서는 안 되며 다음 사항들을 고려해야 한다:

  1. 아키텍처 고려사항
    좋은 아키텍처 결정은 미래 변경을 수용하기 쉽게 만든다. 따라서 기본적인 아키텍처 설계에는 일정 수준의 미래 예측이 필요할 수 있다.

    YAGNI를 따르되 = "미래에 필요할지도 모르는 구체적인 기능을 미리 구현하지 마라"
기본 아키텍처 설계는 = "변화를 수용하기 쉬운 유연한 기반을 마련하라"

  2. 비용-이익 분석
    기능 추가의 현재 비용 vs 나중에 추가할 때의 비용을 비교해야 한다:

    • 지금 구현 비용이 적고, 나중에 추가 비용이 매우 크다면, 선제적 구현이 합리적일 수 있다.
    • 반대로, 지금 구현 비용이 크고 나중에 추가 비용이 비슷하다면, YAGNI를 따르는 것이 좋다.

  3. 실증적 사례 기반 결정
    팀의 경험과 프로젝트 이력을 기반으로 결정을 내려야 한다:

    • “우리가 이전에 이와 유사한 상황에서 실제로 X 기능이 필요했는가?”
    • “다른 프로젝트에서 이 패턴이 반복적으로 등장했는가?”

  4. 지속적인 리팩토링
    YAGNI를 성공적으로 적용하려면 지속적인 리팩토링이 중요하다:

    • 코드를 깔끔하게 유지하여 새로운 요구사항을 수용하기 쉽게 만든다.
    • 설계 품질을 유지하면서 기능을 추가할 수 있어야 한다.

YAGNI 적용을 위한 실용적인 지침#

  1. 요구사항에 충실하기
    “이건 좋은 기능이니까 추가하자"가 아닌 “이 기능이 현재 요구사항에 명시되어 있는가?“라고 질문하라.

  2. MVP (Minimum Viable Product) 접근법 채택
    최소한의 기능으로 제품을 출시하고, 사용자 피드백을 기반으로 반복적으로 개선하하라.

  3. 스토리 분할하기
    큰 기능을 작은 사용자 스토리로 분할하여 한 번에 하나씩 구현하라.

  4. 코드 검토에서 YAGNI 적용하기
    코드 리뷰 시 “이 코드가 현재 요구사항을 해결하는 데 필요한가?“라는 질문을 항상 하라.

  5. 기술 부채 관리하기
    YAGNI가 기술 부채로 이어지지 않도록, 지속적인 리팩토링과 코드 품질 유지에 투자하라.

실제 업계 사례#

  1. 아마존의 API 우선 접근법
    아마존은 “필요할 때만 구축"하는 API 우선 접근법을 채택했다. 새 기능을 개발하기 전에 실제 고객 사용 사례를 철저히 이해하고, 검증된 요구사항이 있을 때만 구현한다.

  2. 페이스북의 “Move Fast and Break Things”
    페이스북의 초기 모토는 YAGNI 철학을 반영한다. 완벽한 장기적 솔루션보다는 현재 문제를 해결하는 것에 집중하고, 필요에 따라 반복적으로 개선했다.

  3. 구글의 프로토타입 접근법
    구글은 종종 최소한의 기능으로 제품을 출시하고 사용자 피드백을 기반으로 발전시키는 전략을 사용한다. 구글 웨이브(Google Wave)와 같은 프로젝트는 과도하게 많은 기능을 미리 구현했다가 실패한 반면, Gmail은 기본 기능으로 시작하여 점진적으로 발전했다.
