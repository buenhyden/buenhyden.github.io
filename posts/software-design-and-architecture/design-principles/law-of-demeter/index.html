Law of Demeter#

데메테르 법칙은 객체 지향 프로그래밍에서 중요한 소프트웨어 설계 원칙으로, “최소 지식 원칙”(Principle of Least Knowledge)이라고도 불린다. 이 원칙은 객체 간의 결합도를 낮추고 소프트웨어의 유지보수성을 향상시키는 것을 목표로 한다.

데메테르 법칙은 객체 지향 설계에서 결합도를 낮추고 캡슐화를 강화하는 중요한 원칙이다.
모든 상황에서 엄격하게 적용하기보다는 소프트웨어의 유지보수성, 가독성, 그리고 견고성을 향상시키는 방향으로 지침으로 활용하는 것이 좋다. 복잡한 객체 그래프를 통한 탐색을 피하고 객체에게 직접 책임을 부여함으로써 더 모듈화되고 유연한 코드를 작성할 수 있다.

기본 개념#

데메테르 법칙은 1987년 노스이스턴 대학교의 데메테르 프로젝트에서 처음 소개되었다. 이 법칙의 핵심 아이디어는 간단하다: “객체는 자신과 직접적으로 관련된 객체와만 대화해야 한다.”

더 구체적으로, 메서드 M이 객체 O의 일부라면 M은 다음 객체들의 메서드만 호출해야 한다:

  1. O 자신 (this, self)
  2. M의 매개변수로 전달된 객체
  3. M 내에서 생성된 객체
  4. O의 직접적인 컴포넌트 객체 (인스턴스 변수)
  5. 전역 변수에 저장된 객체 (신중하게 사용해야 함)

// 데메테르 법칙 위반
public void processPayment() {
    Customer customer = getCustomer();
    Wallet wallet = customer.getWallet();
    Money money = wallet.getMoney();
    money.deduct(amount);
}

위 코드에서는 객체의 내부 구조를 깊이 탐색하고 있다. 객체 그래프를 따라 여러 단계를 거쳐 다른 객체에 접근하는 “기차 충돌” (train wreck) 패턴이 발생한다.

개선된 사례#

// 데메테르 법칙 준수
public void processPayment() {
    Customer customer = getCustomer();
    customer.pay(amount);
}

개선된 버전에서는 Customer 객체에 pay 메서드를 추가하여 내부 구현 세부 사항을 캡슐화했다.
이제 Customer 클래스는 자체적으로 Wallet과 Money 객체와 상호작용하는 책임을 갖는다.

이점#

  1. 결합도 감소: 객체 간의 의존성이 줄어들어 시스템의 유연성이 향상된다.
  2. 캡슐화 강화: 객체의 내부 구조를 숨겨 구현 세부 사항의 변경이 시스템 전체에 미치는 영향을 최소화한다.
  3. 유지보수성 향상: 코드가 더 모듈화되어 변경이 필요할 때 영향 범위가 줄어든다.
  4. 코드 가독성 개선: “기차 충돌” 패턴을 피함으로써 코드가 더 명확해진다.
  5. 테스트 용이성: 객체 간의 결합도가 낮아져 단위 테스트가 더 쉬워진다.

단점 및 고려사항#

  1. 래퍼 메서드 증가: 때로는 단순히 위임하는 메서드들이 많이 생길 수 있다.
  2. 성능 오버헤드: 추가 메서드 호출이 많아져 약간의 성능 저하가 발생할 수 있다.
  3. 실용적 균형: 모든 상황에서 엄격하게 적용하기보다는 상황에 맞게 판단해야 한다.
