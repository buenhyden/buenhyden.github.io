SOLID Principles#

SOLID 원칙은 객체 지향 프로그래밍(OOP)에서 가장 중요한 설계 원칙 모음으로, 유지보수가 용이하고 확장 가능한 소프트웨어를 만들기 위한 지침이다.
이 원칙들은 로버트 C. 마틴(Robert C. Martin, ‘Uncle Bob’으로도 알려진)이 2000년대 초반에 제안했으며, 마이클 페더스(Michael Feathers)가 SOLID라는 약어로 정리했다.

SOLID는 다음 다섯 가지 원칙의 첫 글자를 따서 만들어졌다:

  • Single Responsibility Principle (단일 책임 원칙)
  • Open/Closed Principle (개방/폐쇄 원칙)
  • Liskov Substitution Principle (리스코프 치환 원칙)
  • Interface Segregation Principle (인터페이스 분리 원칙)
  • Dependency Inversion Principle (의존성 역전 원칙)

단일 책임 원칙 (Single Responsibility Principle, SRP)#

핵심 개념#

“하나의 클래스는 오직 하나의 책임만 가져야 한다.”

이 원칙은 클래스가 변경되어야 하는 이유는 오직 한 가지여야 한다고 주장한다.
‘책임’은 ‘변경의 이유’로 정의될 수 있다.
클래스가 여러 책임을 가지면, 그 책임들이 서로 얽혀 하나의 책임을 변경하면 다른 책임에 영향을 미칠 수 있다.

이 Employee 클래스는 세 가지 책임을 가지고 있다:

  1. 직원 정보 관리
  2. 급여 계산
  3. 데이터베이스 조작

    }

// 데이터베이스 관리 책임

    public void save(Employee employee) {
        // 데이터베이스에 직원 정보 저장
    }

이제 각 클래스는 단일 책임을 가지므로, 한 영역의 변경이 다른 영역에 영향을 미치지 않는다.

이점#

  • 코드가 더 명확하고 이해하기 쉬워진다.
  • 클래스가 더 작고 집중되어 있다.
  • 변경의 영향 범위가 제한된다.
  • 테스트하기 쉬워진다.
  • 재사용성이 향상된다.

2. 개방/폐쇄 원칙 (Open/Closed Principle, OCP)#

핵심 개념#

“소프트웨어 엔티티(클래스, 모듈, 함수 등)는 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.”

이 원칙은 기존 코드를 변경하지 않고도 시스템의 동작을 확장할 수 있어야 한다고 주장한다.
새로운 기능을 추가하기 위해 기존 코드를 수정하는 것이 아니라, 새로운 코드를 작성하여 기능을 확장해야 한다.

public class Rectangle {
    public double width;
    public double height;
}

public class AreaCalculator {
    public double calculateArea(Object shape) {
        if (shape instanceof Rectangle) {
            Rectangle rectangle = (Rectangle) shape;
            return rectangle.width * rectangle.height;
        }
        else if (shape instanceof Circle) {
            Circle circle = (Circle) shape;
            return Math.PI * circle.radius * circle.radius;
        }
        return 0;
    }
}

여기서 AreaCalculator는 새로운 도형이 추가될 때마다 수정되어야 한다. 이는 OCP를 위반한다.

public interface Shape {
    double calculateArea();
}

public class Rectangle implements Shape {
    private double width;
    private double height;
    
    public Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }
    
    @Override
    public double calculateArea() {
        return width * height;
    }
}

public class Circle implements Shape {
    private double radius;
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    @Override
    public double calculateArea() {
        return Math.PI * radius * radius;
    }
}

public class AreaCalculator {
    public double calculateArea(Shape shape) {
        return shape.calculateArea();
    }
}

이제 새로운 도형(예: Triangle)을 추가하려면 Shape 인터페이스를 구현하는 새 클래스만 만들면 된다. AreaCalculator 클래스는 수정할 필요가 없다.

이점#

  • 코드의 유연성이 향상된다.
  • 코드 수정의 위험이 감소한다.
  • 기존 코드의 안정성이 유지된다.
  • 유지보수가 용이해진다.
  • 확장이 쉬워진다.

리스코프 치환 원칙 (Liskov Substitution Principle, LSP)#

핵심 개념#

“상위 타입의 객체를 하위 타입의 객체로 치환해도 프로그램의 정확성이 유지되어야 한다.”

이 원칙은 바바라 리스코프(Barbara Liskov)가 1987년에 소개했다. 하위 클래스는 상위 클래스의 행동 방식을 유지하면서 확장해야 한다는 것을 의미한다. 다시 말해, 하위 클래스는 상위 클래스의 계약을 준수해야 한다.

public class Rectangle {
    protected double width;
    protected double height;
    
    public void setWidth(double width) {
        this.width = width;
    }
    
    public void setHeight(double height) {
        this.height = height;
    }
    
    public double getArea() {
        return width * height;
    }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(double width) {
        this.width = width;
        this.height = width;  // 정사각형은 가로와 세로가 같아야 함
    }
    
    @Override
    public void setHeight(double height) {
        this.width = height;  // 정사각형은 가로와 세로가 같아야 함
        this.height = height;
    }
}

public interface Shape {
    double getArea();
}

public class Rectangle implements Shape {
    private double width;
    private double height;
    
    public Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }
    
    public void setWidth(double width) {
        this.width = width;
    }
    
    public void setHeight(double height) {
        this.height = height;
    }
    
    @Override
    public double getArea() {
        return width * height;
    }
}

public class Square implements Shape {
    private double side;
    
    public Square(double side) {
        this.side = side;
    }
    
    public void setSide(double side) {
        this.side = side;
    }
    
    @Override
    public double getArea() {
        return side * side;
    }
}

이제 Square는 Rectangle의 하위 클래스가 아니라 별도의 클래스이다. 두 클래스 모두 Shape 인터페이스를 구현하며, 각각 자신의 동작 방식을 가진다.

이점#

  • 상속 계층이 더 명확해진다.
  • 부모 클래스와 자식 클래스 간의 일관성이 유지된다.
  • 다형성을 안전하게 활용할 수 있다
  • 예측 가능한 동작을 보장한다.
  • 버그와 에러의 가능성이 감소합니다

인터페이스 분리 원칙 (Interface Segregation Principle, ISP)#

핵심 개념#

“클라이언트는 자신이 사용하지 않는 인터페이스에 의존하도록 강요받지 않아야 한다.”

이 원칙은 큰 인터페이스를 더 작고 특정한 여러 인터페이스로 분리하는 것이 좋다고 주장한다.
클라이언트가 실제로 필요한 메서드만 알게 하여 불필요한 의존성을 줄인다.

public interface Worker {
    void work();
    void eat();
    void sleep();
}

public class Human implements Worker {
    @Override
    public void work() {
        // 작업 수행 로직
    }
    
    @Override
    public void eat() {
        // 식사 로직
    }
    
    @Override
    public void sleep() {
        // 수면 로직
    }
}

public class Robot implements Worker {
    @Override
    public void work() {
        // 작업 수행 로직
    }
    
    @Override
    public void eat() {
        // 로봇은 먹지 않음 - 빈 구현
    }
    
    @Override
    public void sleep() {
        // 로봇은 자지 않음 - 빈 구현
    }
}

Robot 클래스는 eat()과 sleep() 메서드를 구현해야 하지만, 로봇은 이러한 행동을 하지 않는다. 이는 ISP를 위반한다.

이제 Robot 클래스는 자신에게 필요한 Workable 인터페이스만 구현하면 된다.

이점#

  • 인터페이스가 더 집중되고 명확해진다.
  • 불필요한 메서드 구현이 필요하지 않다.
  • 클라이언트 코드의 결합도가 감소한다.
  • 코드가 더 모듈화된다.
  • 변경의 영향 범위가 제한된다.

의존성 역전 원칙 (Dependency Inversion Principle, DIP)#

핵심 개념#

“고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다. 추상화는 세부 사항에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다.”

이 원칙은 전통적인 의존성 방향(고수준 모듈이 저수준 모듈에 의존)을 역전시켜, 두 모듈 모두 추상화에 의존하도록 하는 것이다. 이를 통해 시스템의 결합도를 낮추고 유연성을 높인다.

public class LightBulb {
    public void turnOn() {
        // 전구를 켜는 로직
    }
    
    public void turnOff() {
        // 전구를 끄는 로직
    }
}

public class Switch {
    private LightBulb bulb;
    
    public Switch() {
        this.bulb = new LightBulb();  // 구체적인 구현에 직접 의존
    }
    
    public void operate() {
        // 스위치 작동 로직
        bulb.turnOn();
    }
}

여기서 Switch(고수준 모듈)는 LightBulb(저수준 모듈)에 직접 의존한다. 이는 DIP를 위반한다.

public interface Switchable {
    void turnOn();
    void turnOff();
}

public class LightBulb implements Switchable {
    @Override
    public void turnOn() {
        // 전구를 켜는 로직
    }
    
    @Override
    public void turnOff() {
        // 전구를 끄는 로직
    }
}

public class Fan implements Switchable {
    @Override
    public void turnOn() {
        // 선풍기를 켜는 로직
    }
    
    @Override
    public void turnOff() {
        // 선풍기를 끄는 로직
    }
}

public class Switch {
    private Switchable device;  // 추상화에 의존
    
    public Switch(Switchable device) {
        this.device = device;  // 의존성 주입
    }
    
    public void operate() {
        // 스위치 작동 로직
        device.turnOn();
    }
}

이제 Switch와 LightBulb 모두 Switchable 인터페이스(추상화)에 의존한다. Switch는 의존성 주입을 통해 어떤 Switchable 구현체와도 작동할 수 있다.

이점#

  • 모듈 간의 결합도가 감소한다.
  • 코드가 더 유연해진다.
  • 테스트하기 쉬워진다.
  • 시스템의 확장성이 향상된다.
  • 변경의 영향 범위가 제한된다.

// 3. 리스코프 치환 원칙 (LSP)
// 하위 타입은 상위 타입을 대체할 수 있습니다

public abstract class Discount {
    public abstract double applyDiscount(double price);
    
    // 모든 할인은 음수가 될 수 없다는 계약
    public final boolean isValidDiscount(double price, double discount) {
        return (price - discount) >= 0;
    }
}

public class PercentageDiscount extends Discount {
    private double percentage;
    
    public PercentageDiscount(double percentage) {
        this.percentage = percentage;
    }
    
    @Override
    public double applyDiscount(double price) {
        double discount = price * percentage / 100;
        if (isValidDiscount(price, discount)) {
            return price - discount;
        }
        return price;
    }
}

public class FixedDiscount extends Discount {
    private double amount;
    
    public FixedDiscount(double amount) {
        this.amount = amount;
    }
    
    @Override
    public double applyDiscount(double price) {
        if (isValidDiscount(price, amount)) {
            return price - amount;
        }
        return price;
    }
}

// 5. 의존성 역전 원칙 (DIP)
// 고수준 모듈과 저수준 모듈 모두 추상화에 의존

public class OrderService {
    private final OrderProcessor orderProcessor;
    private final PaymentValidator paymentValidator;
    private final InventoryManager inventoryManager;
    
    // 의존성 주입
    public OrderService(OrderProcessor orderProcessor,
                        PaymentValidator paymentValidator,
                        InventoryManager inventoryManager) {
        this.orderProcessor = orderProcessor;
        this.paymentValidator = paymentValidator;
        this.inventoryManager = inventoryManager;
    }
    
    public boolean placeOrder(Order order, Payment payment) {
        // 재고 확인
        for (OrderItem item : order.getItems()) {
            if (!inventoryManager.checkStock(item.getProduct(), item.getQuantity())) {
                return false;
            }
        }
        
        // 결제 검증
        if (!paymentValidator.validatePayment(payment)) {
            return false;
        }
        
        // 주문 처리
        orderProcessor.processOrder(order);
        
        // 재고 업데이트
        for (OrderItem item : order.getItems()) {
            inventoryManager.updateStock(item.getProduct(), item.getQuantity());
        }
        
        return true;
    }
}

SOLID 원칙의 이점#

SOLID 원칙을 적용하면 다음과 같은 이점을 얻을 수 있다:

  1. 유지보수성 향상: 코드가 더 모듈화되고 결합도가 낮아져 변경이 더 쉬워진다.
  2. 확장성 향상: 기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있다.
  3. 테스트 용이성: 각 컴포넌트를 독립적으로 테스트하기 쉬워진다.
  4. 재사용성 향상: 컴포넌트가 잘 정의된 단일 책임을 가지므로 재사용하기 쉬워진다.
  5. 코드 품질 향상: 코드가 더 명확하고, 이해하기 쉬우며, 유지보수하기 쉬워진다.
  6. 기술 부채 감소: 잘 설계된 코드는 시간이 지남에 따라 기술 부채가 덜 쌓인다.

SOLID 원칙의 적용 시 고려사항#

SOLID 원칙을 적용할 때 다음 사항을 고려해야 한다:

  1. 과도한 추상화 피하기: SOLID 원칙을 과도하게 적용하면 불필요한 복잡성이 증가할 수 있다.
  2. 실용적 접근: 프로젝트의 규모와 요구사항에 맞게 SOLID 원칙을 적용해야 한다.
  3. 균형 유지: 여러 원칙 사이의 균형을 맞추는 것이 중요하다. 때로는 한 원칙을 완벽하게 따르기 위해 다른 원칙을 희생해야 할 수도 있다.
  4. 점진적 적용: 기존 코드베이스에 SOLID 원칙을 적용할 때는 점진적으로 개선하는 것이 좋다.
  5. 도구와 프레임워크 활용: 많은 현대적인 프레임워크와 도구는 SOLID 원칙을 따르도록 설계되어 있다. 이러한 도구를 활용하면 SOLID 원칙을 더 쉽게 적용할 수 있다.
