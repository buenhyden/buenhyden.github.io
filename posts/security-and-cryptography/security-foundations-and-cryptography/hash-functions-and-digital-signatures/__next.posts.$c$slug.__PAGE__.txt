## Definition

`Hash Functions & Digital Signatures`는 데이터의 **무결성(Integrity)**과 **진본성(Authenticity)**을 보장하는 두 가지 핵심 암호 기술입니다. 임의 길이의 데이터를 고정 길이의 고유값으로 변환하는 단방향 해시 함수와, 특정 주체가 해당 데이터를 생성했음을 수학적으로 증명(비부인성)하는 디지털 서명의 원리와 활용 범위를 다룹니다.

## Scope & Boundaries

- **포함 범위**:
  - **해시 알고리즘**: SHA-256, SHA-3의 특성 및 MD5/SHA-1의 취약성(Collision) 분석.
  - **패스워드 해싱**: 단순 해시의 한계와 전용 알고리즘(Argon2, bcrypt, scrypt)의 필요성.
  - **HMAC (Hash-based Message Authentication Code)**: 키가 있는 해시를 통한 데이터 무결성 및 인증 서비스.
  - **디지털 서명**: 서명 생성(Sign) 및 검증(Verify) 프로세스(RSA-PSS, ECDSA, Ed25519).
- **제외 범위**:
  - **블록체인 합의 알고리즘**: PoW 등 해시를 이용한 경제적 합의 모델의 상세 상세 (L1/Distributed Systems 담당).
  - **특정 SDK 코드**: 특정 언어(Java, Go 등)의 암호화 라이브러리 API 사용법 나열.
- **경계 근거**: [P3] CyBOK - Identity & Authentication, [P1] CS2023 - Security: Hash Functions.

## Counterexample

- **해시 = 암호화 오해**: 해시는 '단방향(One-way)' 함수이므로 복호화(Decryption)가 불가능합니다. 해시 값을 원래 데이터로 되돌리는 것은 암호학적 정의에 어긋나며, 레인보우 테이블이나 브루트포스를 통한 '역산 시도'는 가능할 뿐입니다.
- **Salt 없는 패스워드 저장**: 동일한 패스워드가 동일한 해시 값을 갖게 방치하는 것은 설계적 결함입니다. 솔트(Salt) 없는 해시는 현대 보안 표준에서 수용되지 않습니다.

## Prerequisites

- `Symmetric & Asymmetric Algorithms`: 비대칭키 쌍(Public/Private)의 기본 개념 이해.
- `Mathematics & Computing Logic`: 비둘기집 원리(충돌 가능성)와 확률적 안정성.

## Learning Map

1.  **돌이킬 수 없는 흐름 (Hash)**: 왜 해시는 거꾸로 풀 수 없는지, '지문'으로서의 역할을 배웁니다.
2.  **비밀번호의 요새 (Password Protection)**: 단순 해시를 넘어 CPU/Memory 비용을 강제하는 패스워드 전용 해싱을 익힙니다.
3.  **검증된 진실 (HMAC)**: 메시지가 중간에 바뀌지 않았음을 증명하는 '암호학적 봉인' 기법을 분석합니다.
4.  **디지털 인감 (Digital Signature)**: 내가 보냈음을 부인할 수 없게 만드는 서명의 법적/기술적 가치를 연구합니다.

## 기본

### 해시 함수의 3대 보안 특성

- **무엇 + 왜**
  - 해시 함수가 암호학적으로 유효하려면 아래 3가지 성질을 반드시 만족해야 합니다.
- **반드시 알아야 하는 개념**
  - Pre-image Resistance (역상 저항성): 고유값만 보고 원본 데이터를 유추하기 어려워야 함.
  - Second Pre-image Resistance (제2 역상 저항성): 특정 데이터와 같은 해시를 갖는 다른 데이터를 찾기 어려워야 함.
  - Collision Resistance (충돌 저항성): 똑같은 해시 값을 갖는 임의의 두 데이터를 찾기 어려워야 함.

## 권장

### 패스워드 보안과 Argon2

- **무엇 + 왜**
  - 사용자 패스워드는 GPU를 통한 고속 해시 연산 공격에 취약하므로, 의도적으로 연산을 느리게 만드는 알고리즘이 필요합니다.
- **반드시 알아야 하는 개념**
  - Salt: 사용자마다 고유한 난수 값을 해시 전에 추가하여 레인보우 테이블 공격을 무력화.
  - Argon2id: 메모리 하드(Memory-hard) 기능을 갖춘 현대적 패스워드 해싱 표준.
  - Key Stretching: 해시 연산을 수만 번 반복하여 브루트포스 비용을 높이는 기술.

## 실무

### 디지털 서명 워크플로우와 비부인성

- **무엇 + 왜**
  - 비대칭키를 활용해 메시지의 출처를 명확히 하고, 보낸 사람이 나중에 발송 사실을 부인할 수 없게 만듭니다.
- **반드시 알아야 하는 개념**
  - Sign Process: 원본의 해시 값을 본인의 '개인키'로 암호화하여 첨부.
  - Verify Process: 보낸 사람의 '공개키'로 서명을 복호화하여 추출된 해시 값과 직접 계산한 해시 값을 비교.
  - Ed25519: 현대 시스템에서 선호되는 고성능/고보안성 타원곡선 서명 알고리즘.

## 심화

### 머클 트리(Merkle Tree)와 무결성 증명

- **무엇 + 왜**
  - 대량의 데이터나 로그 파일의 무결성을 효율적으로(Log N 비용) 검증하기 위한 트리 구조를 연구합니다.
- **반드시 알아야 하는 개념**
  - Leaf Node & Root Hash: 개별 데이터의 해시를 층층이 쌓아 단 하나의 루트 해시로 요약.
  - Merkle Proof: 전체 데이터를 다 받지 않고도 내가 가진 조각이 전체에 포함되는지 증명하는 기법 (Git, Blockchain 필수).
  - Birthday Attack: 확률론에 근거하여 해시 충돌을 찾는 가장 효율적인 방법론의 한계 이해.

17:T243a,<h2 id="definition">Definition</h2>
<p><code>Hash Functions &#x26; Digital Signatures</code>는 데이터의 **무결성(Integrity)**과 **진본성(Authenticity)**을 보장하는 두 가지 핵심 암호 기술입니다. 임의 길이의 데이터를 고정 길이의 고유값으로 변환하는 단방향 해시 함수와, 특정 주체가 해당 데이터를 생성했음을 수학적으로 증명(비부인성)하는 디지털 서명의 원리와 활용 범위를 다룹니다.</p>
<h2 id="scope-boundaries">Scope &#x26; Boundaries</h2>
<ul>
<li><strong>포함 범위</strong>:
<ul>
<li><strong>해시 알고리즘</strong>: SHA-256, SHA-3의 특성 및 MD5/SHA-1의 취약성(Collision) 분석.</li>
<li><strong>패스워드 해싱</strong>: 단순 해시의 한계와 전용 알고리즘(Argon2, bcrypt, scrypt)의 필요성.</li>
<li><strong>HMAC (Hash-based Message Authentication Code)</strong>: 키가 있는 해시를 통한 데이터 무결성 및 인증 서비스.</li>
<li><strong>디지털 서명</strong>: 서명 생성(Sign) 및 검증(Verify) 프로세스(RSA-PSS, ECDSA, Ed25519).</li>
</ul>
</li>
<li><strong>제외 범위</strong>:
<ul>
<li><strong>블록체인 합의 알고리즘</strong>: PoW 등 해시를 이용한 경제적 합의 모델의 상세 상세 (L1/Distributed Systems 담당).</li>
<li><strong>특정 SDK 코드</strong>: 특정 언어(Java, Go 등)의 암호화 라이브러리 API 사용법 나열.</li>
</ul>
</li>
<li><strong>경계 근거</strong>: [P3] CyBOK - Identity &#x26; Authentication, [P1] CS2023 - Security: Hash Functions.</li>
</ul>
<h2 id="counterexample">Counterexample</h2>
<ul>
<li><strong>해시 = 암호화 오해</strong>: 해시는 '단방향(One-way)' 함수이므로 복호화(Decryption)가 불가능합니다. 해시 값을 원래 데이터로 되돌리는 것은 암호학적 정의에 어긋나며, 레인보우 테이블이나 브루트포스를 통한 '역산 시도'는 가능할 뿐입니다.</li>
<li><strong>Salt 없는 패스워드 저장</strong>: 동일한 패스워드가 동일한 해시 값을 갖게 방치하는 것은 설계적 결함입니다. 솔트(Salt) 없는 해시는 현대 보안 표준에서 수용되지 않습니다.</li>
</ul>
<h2 id="prerequisites">Prerequisites</h2>
<ul>
<li><code>Symmetric &#x26; Asymmetric Algorithms</code>: 비대칭키 쌍(Public/Private)의 기본 개념 이해.</li>
<li><code>Mathematics &#x26; Computing Logic</code>: 비둘기집 원리(충돌 가능성)와 확률적 안정성.</li>
</ul>
<h2 id="learning-map">Learning Map</h2>
<ol>
<li><strong>돌이킬 수 없는 흐름 (Hash)</strong>: 왜 해시는 거꾸로 풀 수 없는지, '지문'으로서의 역할을 배웁니다.</li>
<li><strong>비밀번호의 요새 (Password Protection)</strong>: 단순 해시를 넘어 CPU/Memory 비용을 강제하는 패스워드 전용 해싱을 익힙니다.</li>
<li><strong>검증된 진실 (HMAC)</strong>: 메시지가 중간에 바뀌지 않았음을 증명하는 '암호학적 봉인' 기법을 분석합니다.</li>
<li><strong>디지털 인감 (Digital Signature)</strong>: 내가 보냈음을 부인할 수 없게 만드는 서명의 법적/기술적 가치를 연구합니다.</li>
</ol>
<h2 id="h2">기본</h2>
<h3 id="3">해시 함수의 3대 보안 특성</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>해시 함수가 암호학적으로 유효하려면 아래 3가지 성질을 반드시 만족해야 합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Pre-image Resistance (역상 저항성): 고유값만 보고 원본 데이터를 유추하기 어려워야 함.</li>
<li>Second Pre-image Resistance (제2 역상 저항성): 특정 데이터와 같은 해시를 갖는 다른 데이터를 찾기 어려워야 함.</li>
<li>Collision Resistance (충돌 저항성): 똑같은 해시 값을 갖는 임의의 두 데이터를 찾기 어려워야 함.</li>
</ul>
</li>
</ul>
<h2 id="h2-1">권장</h2>
<h3 id="argon2">패스워드 보안과 Argon2</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>사용자 패스워드는 GPU를 통한 고속 해시 연산 공격에 취약하므로, 의도적으로 연산을 느리게 만드는 알고리즘이 필요합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Salt: 사용자마다 고유한 난수 값을 해시 전에 추가하여 레인보우 테이블 공격을 무력화.</li>
<li>Argon2id: 메모리 하드(Memory-hard) 기능을 갖춘 현대적 패스워드 해싱 표준.</li>
<li>Key Stretching: 해시 연산을 수만 번 반복하여 브루트포스 비용을 높이는 기술.</li>
</ul>
</li>
</ul>
<h2 id="h2-2">실무</h2>
<h3 id="h3">디지털 서명 워크플로우와 비부인성</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>비대칭키를 활용해 메시지의 출처를 명확히 하고, 보낸 사람이 나중에 발송 사실을 부인할 수 없게 만듭니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Sign Process: 원본의 해시 값을 본인의 '개인키'로 암호화하여 첨부.</li>
<li>Verify Process: 보낸 사람의 '공개키'로 서명을 복호화하여 추출된 해시 값과 직접 계산한 해시 값을 비교.</li>
<li>Ed25519: 현대 시스템에서 선호되는 고성능/고보안성 타원곡선 서명 알고리즘.</li>
</ul>
</li>
</ul>
<h2 id="h2-3">심화</h2>
<h3 id="merkle-tree">머클 트리(Merkle Tree)와 무결성 증명</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>대량의 데이터나 로그 파일의 무결성을 효율적으로(Log N 비용) 검증하기 위한 트리 구조를 연구합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Leaf Node &#x26; Root Hash: 개별 데이터의 해시를 층층이 쌓아 단 하나의 루트 해시로 요약.</li>
<li>Merkle Proof: 전체 데이터를 다 받지 않고도 내가 가진 조각이 전체에 포함되는지 증명하는 기법 (Git, Blockchain 필수).</li>
<li>Birthday Attack: 확률론에 근거하여 해시 충돌을 찾는 가장 효율적인 방법론의 한계 이해.</li>
</ul>
</li>
</ul>
