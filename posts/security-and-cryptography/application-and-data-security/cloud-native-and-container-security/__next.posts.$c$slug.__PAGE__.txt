## Definition

`Cloud-Native & Container Security`는 클라우드 컴퓨팅과 가상화 기술이 결합된 현대적 애플리케이션 환경을 보호하기 위한 보안 체계입니다. 전용 서버 환경과 달리 동적으로 생성되고 사라지는 **컨테이너(Container)**의 이미지 무결성을 보장하고, **쿠버네티스(Kubernetes)**와 같은 오케스트레이션 계층에서의 접근 통제 및 네트워크 격리 전략을 다룹니다.

## Scope & Boundaries

- **포함 범위**:
  - **이미지 보안**: 베이스 이미지 선택, 레이어 보안 취약점 스캔(Trivy), 이미지 서명(Cosign).
  - **런타임 보안**: 컨테이너 격리(Namespace, Cgroups), Read-only Root FS, Capability 제한.
  - **오케스트레이션 보안**: K8s RBAC, Network Policies, Admission Controllers(OPA, Kyverno).
  - **플랫폼 보호**: CWPP(Cloud Workload Protection), CSPM(Cloud Security Posture Management).
- **제외 범위**:
  - **클라우드 계정 관리**: 루트 계정 MFA 설정 등 범용 IAM 상세 (Identity, Access & Trust 담당).
  - **물리적 가상화 기술**: Xen/KVM 하이퍼바이저 수준의 커널 코드 상세 (Operating Systems 담당).
- **경계 근거**: [P3] CyBOK v1.1 - Infrastructure Security: Cloud, [I1] NIST SP 800-190: Application Container Security Guide.

## Counterexample

- **컨테이너 = VM 오해**: 컨테이너는 호스트 OS의 커널을 공유하므로 VM보다 격리 수준이 낮습니다. "컨테이너니까 안전하다"는 생각은 커널 취약점을 통한 컨테이너 탈옥(Escape) 공격에 무방비한 상태를 만듭니다.
- **이미지 스캔 = 완벽 보호 오해**: 이미지를 스캔하여 알려진 취약점(CVE)을 고치는 것은 시작일 뿐입니다. 설정 오류(Misconfiguration)나 런타임 공격은 스캔만으로는 막을 수 없으며, 가시성(Observability) 도구가 병행되어야 합니다.

## Prerequisites

- `Operating Systems`: 리눅스 네임스페이스(Namespace), 컨트롤 그룹(Cgroups)의 원리 이해.
- `System Architecture & High Availability`: 마이크로서비스 및 분산 인프라에 대한 기본 지식.

## Learning Map

1.  **가두리의 설계 (Container Isolation)**: 커널 위에서 어떻게 프로세스들을 안전하게 격리하는지 원리를 배웁니다.
2.  **공급망의 신뢰 (Image Security)**: 내가 만든 이미지가 정말 안전한지, 중간에 바뀌지는 않았는지 검증하는 법을 익힙니다.
3.  **오케스트레이터의 규칙 (K8s Security)**: 커다란 컨테이너 집합체인 쿠버네티스를 안전하게 운영하는 권한 체계를 분석합니다.
4.  **클라우드 가시성 (CWPP/CSPM)**: 수만 개의 자원이 오가는 클라우드에서 '틀린 그림 찾기'를 자동화하는 법을 연구합니다.

## 기본

### 컨테이너 격리와 보안 기초

- **무엇 + 왜**
  - 가상화 기술의 핵심인 격리 요소들을 파악하여 공격자가 인접 컨테이너나 호스트로 넘어오지 못하게 합니다.
- **반드시 알아야 하는 개념**
  - Namespaces: 프로세스별로 네트워크, 마운트, 사용자 ID 등을 독립적으로 보이게 함.
  - Cgroups: 특정 컨테이너가 CPU나 메모리를 독점하여 시스템 전체를 마비시키는 현상 방지.
  - Non-root User: 컨테이너 내부 프로세스를 루트 권한이 아닌 일반 사용자 권한으로 구동하는 습관.

## 권장

### 이미지 무결성과 스캐닝 기술

- **무엇 + 왜**
  - 애플리케이션의 집인 '이미지' 자체에 폭탄(취약점)이 들어있지 않은지 배포 전 검증합니다.
- **반드시 알아야 하는 개념**
  - Distroless Image: 실행에 꼭 필요한 최소한의 파일만 담아 공격 면(Package 등)을 줄인 이미지.
  - Image Scanning: 빌드 시점에 패키지 매니저 DB를 조회하여 Critical 취약점 유무를 체크.
  - Multi-stage Build: 빌드 도구와 최종 실행 결과물을 분리하여 최종 이미지 크기와 위험을 축소.

## 실무

### 쿠버네티스(K8s) 보안 통제 아키텍처

- **무엇 + 왜**
  - 대규모 환경에서는 수동 설정이 불가능하므로, 정책(Policy) 기반의 자동화된 통제 환경을 구축합니다.
- **반드시 알아야 하는 개념**
  - RBAC (Role-Based Access Control): "누가 어떤 API를 호출할 수 있는가"에 대한 엄격한 제한.
  - Network Policy: 파드(Pod) 간의 통신을 L3/L4 수준에서 직접 허용/차단하는 방화벽 역할.
  - Admission Controllers: API 서버에 요청이 도달했을 때 보안 규칙(예: 루트 실행 금지)에 어긋나면 거부하는 게이트키퍼.

## 심화

### 런타임 위협 탐지와 eBPF 활용

- **무엇 + 왜**
  - 프로그램이 실행 중일 때 벌어지는 미묘한 이상 징후를 성능 저하 없이 실시간으로 감시하는 하이엔드 기술을 탐구합니다.
- **반드시 알아야 하는 개념**
  - eBPF (extended Berkeley Packet Filter): 커널 코드를 수정하지 않고 샌드박스 안정성을 유지하며 모든 시스템 행위를 가시화.
  - CWPP (Cloud Workload Protection Platform): 런타임 공격(리버스 쉘 등)을 탐지하고 컨테이너를 강제 동결/삭제하는 플랫폼.
  - Infrastructure as Code (IaC) Scanning: Terraform이나 K8s YAML 파일에 적힌 보안 취약 설정을 배포 전에 자동 차단.

17:T2522,<h2 id="definition">Definition</h2>
<p><code>Cloud-Native &#x26; Container Security</code>는 클라우드 컴퓨팅과 가상화 기술이 결합된 현대적 애플리케이션 환경을 보호하기 위한 보안 체계입니다. 전용 서버 환경과 달리 동적으로 생성되고 사라지는 **컨테이너(Container)**의 이미지 무결성을 보장하고, **쿠버네티스(Kubernetes)**와 같은 오케스트레이션 계층에서의 접근 통제 및 네트워크 격리 전략을 다룹니다.</p>
<h2 id="scope-boundaries">Scope &#x26; Boundaries</h2>
<ul>
<li><strong>포함 범위</strong>:
<ul>
<li><strong>이미지 보안</strong>: 베이스 이미지 선택, 레이어 보안 취약점 스캔(Trivy), 이미지 서명(Cosign).</li>
<li><strong>런타임 보안</strong>: 컨테이너 격리(Namespace, Cgroups), Read-only Root FS, Capability 제한.</li>
<li><strong>오케스트레이션 보안</strong>: K8s RBAC, Network Policies, Admission Controllers(OPA, Kyverno).</li>
<li><strong>플랫폼 보호</strong>: CWPP(Cloud Workload Protection), CSPM(Cloud Security Posture Management).</li>
</ul>
</li>
<li><strong>제외 범위</strong>:
<ul>
<li><strong>클라우드 계정 관리</strong>: 루트 계정 MFA 설정 등 범용 IAM 상세 (Identity, Access &#x26; Trust 담당).</li>
<li><strong>물리적 가상화 기술</strong>: Xen/KVM 하이퍼바이저 수준의 커널 코드 상세 (Operating Systems 담당).</li>
</ul>
</li>
<li><strong>경계 근거</strong>: [P3] CyBOK v1.1 - Infrastructure Security: Cloud, [I1] NIST SP 800-190: Application Container Security Guide.</li>
</ul>
<h2 id="counterexample">Counterexample</h2>
<ul>
<li><strong>컨테이너 = VM 오해</strong>: 컨테이너는 호스트 OS의 커널을 공유하므로 VM보다 격리 수준이 낮습니다. "컨테이너니까 안전하다"는 생각은 커널 취약점을 통한 컨테이너 탈옥(Escape) 공격에 무방비한 상태를 만듭니다.</li>
<li><strong>이미지 스캔 = 완벽 보호 오해</strong>: 이미지를 스캔하여 알려진 취약점(CVE)을 고치는 것은 시작일 뿐입니다. 설정 오류(Misconfiguration)나 런타임 공격은 스캔만으로는 막을 수 없으며, 가시성(Observability) 도구가 병행되어야 합니다.</li>
</ul>
<h2 id="prerequisites">Prerequisites</h2>
<ul>
<li><code>Operating Systems</code>: 리눅스 네임스페이스(Namespace), 컨트롤 그룹(Cgroups)의 원리 이해.</li>
<li><code>System Architecture &#x26; High Availability</code>: 마이크로서비스 및 분산 인프라에 대한 기본 지식.</li>
</ul>
<h2 id="learning-map">Learning Map</h2>
<ol>
<li><strong>가두리의 설계 (Container Isolation)</strong>: 커널 위에서 어떻게 프로세스들을 안전하게 격리하는지 원리를 배웁니다.</li>
<li><strong>공급망의 신뢰 (Image Security)</strong>: 내가 만든 이미지가 정말 안전한지, 중간에 바뀌지는 않았는지 검증하는 법을 익힙니다.</li>
<li><strong>오케스트레이터의 규칙 (K8s Security)</strong>: 커다란 컨테이너 집합체인 쿠버네티스를 안전하게 운영하는 권한 체계를 분석합니다.</li>
<li><strong>클라우드 가시성 (CWPP/CSPM)</strong>: 수만 개의 자원이 오가는 클라우드에서 '틀린 그림 찾기'를 자동화하는 법을 연구합니다.</li>
</ol>
<h2 id="h2">기본</h2>
<h3 id="h3">컨테이너 격리와 보안 기초</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>가상화 기술의 핵심인 격리 요소들을 파악하여 공격자가 인접 컨테이너나 호스트로 넘어오지 못하게 합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Namespaces: 프로세스별로 네트워크, 마운트, 사용자 ID 등을 독립적으로 보이게 함.</li>
<li>Cgroups: 특정 컨테이너가 CPU나 메모리를 독점하여 시스템 전체를 마비시키는 현상 방지.</li>
<li>Non-root User: 컨테이너 내부 프로세스를 루트 권한이 아닌 일반 사용자 권한으로 구동하는 습관.</li>
</ul>
</li>
</ul>
<h2 id="h2-1">권장</h2>
<h3 id="h3-1">이미지 무결성과 스캐닝 기술</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>애플리케이션의 집인 '이미지' 자체에 폭탄(취약점)이 들어있지 않은지 배포 전 검증합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>Distroless Image: 실행에 꼭 필요한 최소한의 파일만 담아 공격 면(Package 등)을 줄인 이미지.</li>
<li>Image Scanning: 빌드 시점에 패키지 매니저 DB를 조회하여 Critical 취약점 유무를 체크.</li>
<li>Multi-stage Build: 빌드 도구와 최종 실행 결과물을 분리하여 최종 이미지 크기와 위험을 축소.</li>
</ul>
</li>
</ul>
<h2 id="h2-2">실무</h2>
<h3 id="k8s">쿠버네티스(K8s) 보안 통제 아키텍처</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>대규모 환경에서는 수동 설정이 불가능하므로, 정책(Policy) 기반의 자동화된 통제 환경을 구축합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>RBAC (Role-Based Access Control): "누가 어떤 API를 호출할 수 있는가"에 대한 엄격한 제한.</li>
<li>Network Policy: 파드(Pod) 간의 통신을 L3/L4 수준에서 직접 허용/차단하는 방화벽 역할.</li>
<li>Admission Controllers: API 서버에 요청이 도달했을 때 보안 규칙(예: 루트 실행 금지)에 어긋나면 거부하는 게이트키퍼.</li>
</ul>
</li>
</ul>
<h2 id="h2-3">심화</h2>
<h3 id="ebpf">런타임 위협 탐지와 eBPF 활용</h3>
<ul>
<li><strong>무엇 + 왜</strong>
<ul>
<li>프로그램이 실행 중일 때 벌어지는 미묘한 이상 징후를 성능 저하 없이 실시간으로 감시하는 하이엔드 기술을 탐구합니다.</li>
</ul>
</li>
<li><strong>반드시 알아야 하는 개념</strong>
<ul>
<li>eBPF (extended Berkeley Packet Filter): 커널 코드를 수정하지 않고 샌드박스 안정성을 유지하며 모든 시스템 행위를 가시화.</li>
<li>CWPP (Cloud Workload Protection Platform): 런타임 공격(리버스 쉘 등)을 탐지하고 컨테이너를 강제 동결/삭제하는 플랫폼.</li>
<li>Infrastructure as Code (IaC) Scanning: Terraform이나 K8s YAML 파일에 적힌 보안 취약 설정을 배포 전에 자동 차단.</li>
</ul>
</li>
</ul>
