## 1. 개요 (Overview)

`Event Sourcing & CQRS Mechanics`는 정보의 '현재 값'이 아닌 '변화의 기록(History)'을 시스템의 물리적 SSOT(Single Source of Truth)로 삼는 설계 기전을 다룹니다. 또한 데이터를 기록하는 경로(Command)와 관찰하는 경로(Query)를 물리적으로 분리하여, 각각의 하드웨어 부하 특성(I/O, Memory)에 맞게 독립적으로 최적화하는 역학을 연구합니다. 데이터의 '결과'가 아닌 '생성 과정'을 물리적으로 소유함으로써 얻는 무한한 복원력과 통찰력을 학습합니다.

## 2. 범위 및 경계 (Scope & Boundaries)

- **포함 범위 (In-Scope)**:
  - 이벤트 소싱: 어펜드 온리(Append-only) 영속성 저장구조와 로그 재생(Replay) 물리.
  - CQRS (Command Query Responsibility Segregation): 명령 모델과 조회 모델의 물리적 객체 및 저장소 분리.
  - 상태 복원: 대규모 로그의 물리적 재생 지연을 해결하는 스냅샷(Snapshot) 기전.
  - 프로젝션 (Projection): 이벤트 스트림을 물리적 읽기 최적화 테이블(Materialized View)로 변환하는 동역학.
- **제외 범위 (Out-of-Scope)**:
  - 단순 객체-관계 매핑(ORM) 튜닝: 이는 백엔드 상세 구현(Pillar 09) 영역입니다.
  - 전역 메시지 큐 설정: 이는 `SN07-04-01 Message Queues`로 위임합니다.
- **경계 근거**: [P1] CS2023 Systems Engineering, [P5] SWEBOK v4.0 Architectural Design.

## 3. 반례 (Counterexample)

- **'Event Sourcing is just Logging' 오해**: 로깅은 문제 발생 시 사람이 읽기 위해 텍스트를 남기는 행위이지만, 이벤트 소싱은 **'비즈니스 상태를 복구하기 위한 핵심 데이터 경로'**입니다. 로그가 유실되면 시스템 상태를 물리적으로 복원할 수 없습니다. 따라서 이벤트 저장소는 일반 로그 서버보다 훨씬 높은 물리적 신뢰성과 정렬 보장성이 요구됩니다.
- **'CQRS requires two Databases' 오해**: CQRS는 책임의 분리이지, 반드시 물리적인 두 대의 DB 서버를 뜻하지 않습니다. 하나의 DB 내에서도 '쓰기 전용 테이블'과 '조회 전용 뷰'를 엄격히 나누는 것으로도 CQRS의 논리적 이점을 얻을 수 있습니다. 다만, 극단적인 병목 해결을 위해 저장소까지 물리적으로 찢는 것이 실무적 권장 사항일 뿐입니다.

## 5. 학습 지도 (Learning Map)

1. 시간의 축적: 현재 통장 잔고가 아닌, '입금'과 '출금'이라는 물리적 사건들을 로그 파일에 차곡차곡 쌓는 법을 배웁니다.
2. 거울의 분리: 쓰는 공간과 읽는 공간을 물리적으로 떼어놓아, 독점적 락(Lock) 경합을 근본적으로 제거하는 구조를 이해합니다.
3. 기억의 압축: 10년 치 이벤트를 읽는 대신, '어제까지의 스냅샷' 한 장으로 상태를 즉시 복원하는 물리적 최적화를 관찰합니다.
4. 상(Image)의 투영: 원시 이벤트 스트림을 검색 엔진이나 통계 테이블로 물리적으로 동기화(Projection)하는 파이프라인을 분석합니다.

## 6. 단계별 학습 토픽 (Learning Topics)

### 기본 (Mandatory): 변화의 물리적 영속화 (Event Persistence)

- **What**: 모든 상태 변경을 '사실(Fact)'로서 불변 로그에 어펜드(Append)하는 이벤트 소싱의 기초를 학습합니다.
- **Why**: 데이터 변경의 모든 맥락을 물리적으로 기억하여, 완벽한 사후 추적(Auditing)과 신뢰를 확보하기 위함입니다.
- **Concepts**: 사실의 기록, 불변성(Immutability), 어카운팅 물리학, 재생 속도.
- **How**: 데이터 삭제 대신 '취소 이벤트'를 물리적으로 하나 더 쌓아 시스템의 엔트로피를 관리하는 과정을 추적합니다. (Evidence: [P1])
- **Implement**: 로컬 디스크 파일 기반의 단순 이벤트 저장소(Event Store) 모형을 구축합니다.
- **Validate**: 시스템 전원이 꺼진 후, 저장된 이벤트 로그만으로 '마지막 유효 상태'를 물리적 시차 없이 복원 가능한가?

### 권장 (Recommended): 읽기/쓰기 경로의 격리 (CQRS Dynamics)

- **What**: 서비스의 쓰기 로직과 복잡한 조회 쿼리를 개별 물리 객체와 저장소로 분리하는 CQRS 패턴을 학습합니다.
- **Why**: 수천 명의 독자가 한 명의 저자를 방해하지 않도록, 물리적 리소스(CPU/Disk) 경합을 원천 차단하기 위함입니다.
- **Concepts**: Command Model, Query Model, 동기화 지연 물리, 단일 진실원(SSOT).
- **How**: 쓰기 요청이 발생했을 때, 비동기 브로커를 거쳐 읽기 전담 DB의 물리적 블록이 업데이트되는 타임라인을 분석합니다. (Evidence: [P5], [P1])
- **Implement**: 도메인 객체를 명령 전용과 조회 전용 DTO로 물리적 분리 설계합니다.
- **Validate**: 읽기 모델의 동기화 지연(Stale Data)이 비즈니스 허용 한계(예: 3초)를 물리적으로 넘지 않음을 보장할 수 있는가?

### 실무 (Practical): 스냅샷과 프로젝션 (Optimization Mechanics)

- **What**: 로그 재생 성능을 높이는 스냅샷 기법과, 이벤트 스트림을 다중 형태로 투영하는 프로젝션을 다룹니다.
- **Why**: 이벤트 수가 수억 건으로 늘어날 때 발생하는 물리적 '복구 지체'를 막고 다양한 데이터 뷰를 제공하기 위함입니다.
- **Concepts**: 체크포인트 물리, 프로젝터(Projector), 비동기 모델 고도화.
- **How**: 1,000만 번째 이벤트 발생 시, 현재 상태를 바이너리로 덤프(Dump)하여 디스크에 쓰는 물리적 배치(Batch) 과정을 설계합니다. (Evidence: [P5])
- **Implement**: Redis를 활용한 '실시간 읽기 프로젝션' 엔진 혹은 자동 스냅샷 생성기를 구축합니다.
- **Validate**: 스냅샷 생성 중에 시스템이 멈추지 않도록 하는 '비차단 스냅샷(Non-blocking)'의 물리 구현이 유효한가?

### 심화 (Advanced): 메타 모니터링과 이벤트 스키마 진화 (Schema Physics)

- **What**: 시간이 지남에 따라 변하는 이벤트 스키마의 버전 관리와 분산된 이벤트 시스템의 물리적 감시를 분석합니다.
- **Why**: 5년 전에 기록된 낡은 포맷의 이벤트도 현재 시스템에서 물리적으로 로드 및 해석 가능해야 하기 때문입니다.
- **Concepts**: 이벤트 버전 업, 업캐스팅(Upcasting), 인과관계 추적(Correlation ID).
- **How**: 이벤트 구조가 변경되었을 때, 과거 데이터를 물리적으로 변환(Migration)하지 않고 런타임에 해석기로 처리하는 물리적 비용을 계측합니다. (Evidence: [P1], [P3])
- **Implement**: 분산 트레이싱이 통합된 '이벤트 감사 트레일' 시각화 도구 혹은 버전 관리 정책을 제안합니다.
- **Validate**: 특정 이벤트가 유실되었을 때, 전체 읽기 모델의 물리적 정합성을 검증하고 자가 치유(Self-healing)하는 경로를 제시할 수 있는가?

16:T31f7,<h2 id="1-overview">1. 개요 (Overview)</h2>
<p><code>Event Sourcing &#x26; CQRS Mechanics</code>는 정보의 '현재 값'이 아닌 '변화의 기록(History)'을 시스템의 물리적 SSOT(Single Source of Truth)로 삼는 설계 기전을 다룹니다. 또한 데이터를 기록하는 경로(Command)와 관찰하는 경로(Query)를 물리적으로 분리하여, 각각의 하드웨어 부하 특성(I/O, Memory)에 맞게 독립적으로 최적화하는 역학을 연구합니다. 데이터의 '결과'가 아닌 '생성 과정'을 물리적으로 소유함으로써 얻는 무한한 복원력과 통찰력을 학습합니다.</p>
<h2 id="2-scope-boundaries">2. 범위 및 경계 (Scope &#x26; Boundaries)</h2>
<ul>
<li><strong>포함 범위 (In-Scope)</strong>:
<ul>
<li>이벤트 소싱: 어펜드 온리(Append-only) 영속성 저장구조와 로그 재생(Replay) 물리.</li>
<li>CQRS (Command Query Responsibility Segregation): 명령 모델과 조회 모델의 물리적 객체 및 저장소 분리.</li>
<li>상태 복원: 대규모 로그의 물리적 재생 지연을 해결하는 스냅샷(Snapshot) 기전.</li>
<li>프로젝션 (Projection): 이벤트 스트림을 물리적 읽기 최적화 테이블(Materialized View)로 변환하는 동역학.</li>
</ul>
</li>
<li><strong>제외 범위 (Out-of-Scope)</strong>:
<ul>
<li>단순 객체-관계 매핑(ORM) 튜닝: 이는 백엔드 상세 구현(Pillar 09) 영역입니다.</li>
<li>전역 메시지 큐 설정: 이는 <code>SN07-04-01 Message Queues</code>로 위임합니다.</li>
</ul>
</li>
<li><strong>경계 근거</strong>: [P1] CS2023 Systems Engineering, [P5] SWEBOK v4.0 Architectural Design.</li>
</ul>
<h2 id="3-counterexample">3. 반례 (Counterexample)</h2>
<ul>
<li><strong>'Event Sourcing is just Logging' 오해</strong>: 로깅은 문제 발생 시 사람이 읽기 위해 텍스트를 남기는 행위이지만, 이벤트 소싱은 **'비즈니스 상태를 복구하기 위한 핵심 데이터 경로'**입니다. 로그가 유실되면 시스템 상태를 물리적으로 복원할 수 없습니다. 따라서 이벤트 저장소는 일반 로그 서버보다 훨씬 높은 물리적 신뢰성과 정렬 보장성이 요구됩니다.</li>
<li><strong>'CQRS requires two Databases' 오해</strong>: CQRS는 책임의 분리이지, 반드시 물리적인 두 대의 DB 서버를 뜻하지 않습니다. 하나의 DB 내에서도 '쓰기 전용 테이블'과 '조회 전용 뷰'를 엄격히 나누는 것으로도 CQRS의 논리적 이점을 얻을 수 있습니다. 다만, 극단적인 병목 해결을 위해 저장소까지 물리적으로 찢는 것이 실무적 권장 사항일 뿐입니다.</li>
</ul>
<h2 id="4-prerequisites">4. 선행 지식 (Prerequisites)</h2>
<ul>
<li><code>Distributed Transactions (SN07-02-04)</code>: 결과적 일관성과 정합성 트레이드오프 이해. (Basic 반영)</li>
<li><code>Database Durability (Pillar 06.01)</code>: Write-Ahead Log(WAL)의 물리적 선행 기록 원리. (Practical 반영)</li>
<li><code>Domain-Driven Design (SN07-04-03)</code>: 애그리거트(Aggregate)와 도메인 이벤트의 논리 기초. (Recommended 반영)</li>
</ul>
<h2 id="5-learning-map">5. 학습 지도 (Learning Map)</h2>
<ol>
<li>시간의 축적: 현재 통장 잔고가 아닌, '입금'과 '출금'이라는 물리적 사건들을 로그 파일에 차곡차곡 쌓는 법을 배웁니다.</li>
<li>거울의 분리: 쓰는 공간과 읽는 공간을 물리적으로 떼어놓아, 독점적 락(Lock) 경합을 근본적으로 제거하는 구조를 이해합니다.</li>
<li>기억의 압축: 10년 치 이벤트를 읽는 대신, '어제까지의 스냅샷' 한 장으로 상태를 즉시 복원하는 물리적 최적화를 관찰합니다.</li>
<li>상(Image)의 투영: 원시 이벤트 스트림을 검색 엔진이나 통계 테이블로 물리적으로 동기화(Projection)하는 파이프라인을 분석합니다.</li>
</ol>
<h2 id="6-learning-topics">6. 단계별 학습 토픽 (Learning Topics)</h2>
<h3 id="mandatory-event-persistence">기본 (Mandatory): 변화의 물리적 영속화 (Event Persistence)</h3>
<ul>
<li><strong>What</strong>: 모든 상태 변경을 '사실(Fact)'로서 불변 로그에 어펜드(Append)하는 이벤트 소싱의 기초를 학습합니다.</li>
<li><strong>Why</strong>: 데이터 변경의 모든 맥락을 물리적으로 기억하여, 완벽한 사후 추적(Auditing)과 신뢰를 확보하기 위함입니다.</li>
<li><strong>Concepts</strong>: 사실의 기록, 불변성(Immutability), 어카운팅 물리학, 재생 속도.</li>
<li><strong>How</strong>: 데이터 삭제 대신 '취소 이벤트'를 물리적으로 하나 더 쌓아 시스템의 엔트로피를 관리하는 과정을 추적합니다. (Evidence: [P1])</li>
<li><strong>Implement</strong>: 로컬 디스크 파일 기반의 단순 이벤트 저장소(Event Store) 모형을 구축합니다.</li>
<li><strong>Validate</strong>: 시스템 전원이 꺼진 후, 저장된 이벤트 로그만으로 '마지막 유효 상태'를 물리적 시차 없이 복원 가능한가?</li>
</ul>
<h3 id="recommended-cqrs-dynamics">권장 (Recommended): 읽기/쓰기 경로의 격리 (CQRS Dynamics)</h3>
<ul>
<li><strong>What</strong>: 서비스의 쓰기 로직과 복잡한 조회 쿼리를 개별 물리 객체와 저장소로 분리하는 CQRS 패턴을 학습합니다.</li>
<li><strong>Why</strong>: 수천 명의 독자가 한 명의 저자를 방해하지 않도록, 물리적 리소스(CPU/Disk) 경합을 원천 차단하기 위함입니다.</li>
<li><strong>Concepts</strong>: Command Model, Query Model, 동기화 지연 물리, 단일 진실원(SSOT).</li>
<li><strong>How</strong>: 쓰기 요청이 발생했을 때, 비동기 브로커를 거쳐 읽기 전담 DB의 물리적 블록이 업데이트되는 타임라인을 분석합니다. (Evidence: [P5], [P1])</li>
<li><strong>Implement</strong>: 도메인 객체를 명령 전용과 조회 전용 DTO로 물리적 분리 설계합니다.</li>
<li><strong>Validate</strong>: 읽기 모델의 동기화 지연(Stale Data)이 비즈니스 허용 한계(예: 3초)를 물리적으로 넘지 않음을 보장할 수 있는가?</li>
</ul>
<h3 id="practical-optimization-mechanics">실무 (Practical): 스냅샷과 프로젝션 (Optimization Mechanics)</h3>
<ul>
<li><strong>What</strong>: 로그 재생 성능을 높이는 스냅샷 기법과, 이벤트 스트림을 다중 형태로 투영하는 프로젝션을 다룹니다.</li>
<li><strong>Why</strong>: 이벤트 수가 수억 건으로 늘어날 때 발생하는 물리적 '복구 지체'를 막고 다양한 데이터 뷰를 제공하기 위함입니다.</li>
<li><strong>Concepts</strong>: 체크포인트 물리, 프로젝터(Projector), 비동기 모델 고도화.</li>
<li><strong>How</strong>: 1,000만 번째 이벤트 발생 시, 현재 상태를 바이너리로 덤프(Dump)하여 디스크에 쓰는 물리적 배치(Batch) 과정을 설계합니다. (Evidence: [P5])</li>
<li><strong>Implement</strong>: Redis를 활용한 '실시간 읽기 프로젝션' 엔진 혹은 자동 스냅샷 생성기를 구축합니다.</li>
<li><strong>Validate</strong>: 스냅샷 생성 중에 시스템이 멈추지 않도록 하는 '비차단 스냅샷(Non-blocking)'의 물리 구현이 유효한가?</li>
</ul>
<h3 id="advanced-schema-physics">심화 (Advanced): 메타 모니터링과 이벤트 스키마 진화 (Schema Physics)</h3>
<ul>
<li><strong>What</strong>: 시간이 지남에 따라 변하는 이벤트 스키마의 버전 관리와 분산된 이벤트 시스템의 물리적 감시를 분석합니다.</li>
<li><strong>Why</strong>: 5년 전에 기록된 낡은 포맷의 이벤트도 현재 시스템에서 물리적으로 로드 및 해석 가능해야 하기 때문입니다.</li>
<li><strong>Concepts</strong>: 이벤트 버전 업, 업캐스팅(Upcasting), 인과관계 추적(Correlation ID).</li>
<li><strong>How</strong>: 이벤트 구조가 변경되었을 때, 과거 데이터를 물리적으로 변환(Migration)하지 않고 런타임에 해석기로 처리하는 물리적 비용을 계측합니다. (Evidence: [P1], [P3])</li>
<li><strong>Implement</strong>: 분산 트레이싱이 통합된 '이벤트 감사 트레일' 시각화 도구 혹은 버전 관리 정책을 제안합니다.</li>
<li><strong>Validate</strong>: 특정 이벤트가 유실되었을 때, 전체 읽기 모델의 물리적 정합성을 검증하고 자가 치유(Self-healing)하는 경로를 제시할 수 있는가?</li>
</ul>
