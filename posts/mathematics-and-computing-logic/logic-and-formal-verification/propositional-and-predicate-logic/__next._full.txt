## Overview

`Propositional & Predicate Logic(명제 및 술어 논리)`는 명제를 기본 단위로 하는 명제 논리와 개체 간의 관계 및 한정사(Quantifier)를 포함하는 술어 논리를 포괄합니다. 이는 컴퓨팅 시스템의 상태 명세(Specification), 프로그램 정당성 증명, 그리고 보안 정책 분석의 가장 기초가 되는 형식적 언어이자 추론 체계입니다. 현대의 정적 분석 도구와 SAT/SMT Solver의 내부 엔진은 바로 이 논리 체계에 기반하고 있으며, 가상 공간의 모든 상태 전이가 모순 없이 작동하도록 강제하는 '논리적 물리 법칙' 역할을 합니다.

## Scope & Boundaries

- **포함 범위 (In-Scope)**:
    - 명제 논리 기초: 논리 연결사(¬, ∧, ∨, →, ↔), 진리표, 논리적 동치 및 대우 (CS2023: MS-Logic)
    - 보편적 성질: 항진식(Tautology), 만족 가능성(Satisfiability, SAT), 표준형(CNF/DNF) 변환 물리
    - 술어 논리 확장: 변수, 술어(Predicate), 한정사(∀, ∃), 자유 변수와 바인딩 범위(Scope)
    - 추론 및 검증: 자연 연역(Natural Deduction), 추론 규칙(Modus Ponens), 타당성(Validity) 및 건전성 기초
    - 실무 자동화: SAT/SMT Solver의 원리 및 비즈니스 규칙의 만족 가능성 판별 (P3: Formal Methods)
- **제외 범위 (Out-of-Scope)**:
    - 하드웨어 게이트 수준의 불 대수 최적화는 `02. Boolean Algebra & Circuit Logic`에서 다룹니다.
    - 시간 논리(LTL/CTL) 및 TLA+ 도구 활용은 `03. Formal Specifications & TLA+`로 위임합니다.
- **경계 근거**: [P1] CS2023 Mathematical Structures, [P3] CyBOK Formal Methods for Security.

## Counterexample

- **'단순 if-else 문' 오해**: 코드의 조건 분기를 아는 것과, 결합된 수천 개의 정책 간의 모순을 논리적으로 추론하는 것은 별개입니다. 논리는 코드의 실행이 아니라 '상태 공간의 진리성'을 규명하며, 인간의 직관이 놓치는 'Hidden Path'를 찾아냅니다.
- **'단순 진리표 작성기' 오해**: 진리표는 도구일 뿐입니다. 중요한 것은 복잡한 보안 요구사항을 논리식으로 '환원(Reduction)'하여 SAT 문제로 정의하고 자동화된 수치로 증명하는 아키텍처적 모델링 역량입니다.

## Prerequisites

1. `01.01. Set Theory & Relations`: 집합의 포함 관계와 논리적 함축($\subseteq$ vs $\implies$)의 기하적 대응 (Rationale: 논리 연산의 대상 정의).
2. `01. Logic & Formal Verification (ROOT)`: 논리가 컴퓨팅 신뢰성(Safety)에 기여하는 물리적 역할 인지.
3. `Pillar 05. Data Architecture & Database`: SQL의 `WHERE` 절에 투영된 술어 논리 기반 필터링 물리 (Rationale: 술어와 데이터의 실무 매핑).
4. `Pillar 09. Software Engineering & Architecture`: RBAC(역할 기반 접근 제어) 정책의 논리적 상충 문제 제기.
5. `Mathematical Induction 기초`: 논리식의 구조적 귀납법(Structural Induction) 증명을 위한 기반 지식.
6. `Variable Scoping (Programming)`: 코드에서의 변수 유효 범위와 논리에서의 한정사 바인딩 범위 매핑.
7. `Inference Rules basics`: 전제로부터 결론을 도출하는 3단 논법의 최소 논리 지식.

## Learning Map

1. **Step 1 (기호 형식화)**: 자연어 명세를 논리 기호로 변환하고, 진리표를 통해 물리적 모든 상태 조합에 대한 진위값을 확정합니다.
2. **Step 2 (구조적 단순화)**: 드모르간의 법칙 등을 적용하여 복잡한 정책식을 CNF 표준형으로 정규화하고 중복된 제약 조건을 제거합니다.
3. **Step 3 (범위의 확장)**: 한정사를 도입하여 '모든 사용자' 또는 '최소 하나의 관리자'와 같은 개체 집합에 대한 전역적 제약 조건을 모델링합니다.
4. **Step 4 (자동화된 검증)**: 정책간의 모순(Contradiction)을 SAT Solver로 자동 탐지하고, 건전한 추론을 통해 보안 취약점을 논리적으로 증명합니다.

## Learning Topics

### 기본 (Basic): 명제 논리의 구문과 표준형
- **What + Why**: 모든 컴퓨터 프로그램의 기초 제약 조건을 엄밀한 기호로 정의하고, 기계가 처리하기 가장 효율적인 '표준형'으로 변환하는 법을 배웁니다.
- **What to Learn**:
    - [Concepts]: 명제(Atom), 연결사(¬, ∧, ∨, →), 진위값(Truth Value), 논리적 동치, 드모르간 법칙, CNF(Conjunctive Normal Form), 리터럴(Literal).
    - [Skills]: 자연어의 기호화, 진리표 작성, 복잡한 식의 CNF 변환, 진리표를 통한 결과값 예측.
    - [Failure Modes]: $\implies$ 연산자의 대우(Contrapositive)와 역(Converse)을 혼동하여 비논리적 결론 도출.
- **How to Learn**:
    1. 개별 명제를 기호 $P, Q$로 치환하는 연습을 수행합니다.
    2. 진리표를 직접 그려 모든 연산자의 결과값을 물리적으로 확인합니다.
    3. 복잡한 식을 분배 법칙과 드모르간 법칙을 써서 절(Clause)들의 곱으로 리팩토링합니다.
    - [Practice]: 서로 상충하는 두 비즈니스 요구사항을 논리식으로 표현하고 진리표로 모순 지점 찾기.
- **Validate**: 모든 입력 조합에 대해 논리식의 결과가 진리표와 100% 일치함을 보장할 수 있는가?

### 권장 (Recommended): 술어 논리와 시스템 인바리언트
- **What + Why**: 개별 인스턴스가 아닌 '집합 전체'에 적용되는 시스템의 핵심 규칙인 인바리언트(Invariant)를 보편 한정사를 통해 기술하는 법을 학습합니다.
- **What to Learn**:
    - [Concepts]: 술어(Predicate), 한정사(∀, ∃), 정의역(Domain), 자유 변수(Free Var), 바인딩(Binding), 한정사의 부정(Duality).
    - [Skills]: "모든 유저는 유효한 토큰을 가져야 한다"와 같은 전역 규칙 명세, 한정사의 Scope 설정, 중첩 한정사 분석.
    - [Failure Modes]: ∀x ∃y와 ∃y ∀x의 순서를 뒤바꾸어 '어떤' 것이 '모든' 것에 대응되는 물리적 의미를 오해함.
- **How to Learn**:
    1. 술어 함수 $P(x)$를 정의하고 $x$의 범위(Domain)를 명시적으로 확정합니다.
    2. 보편 한정사(∀) 문장의 부정이 왜 존재 한정사(∃)와 NOT의 결합인지 물리적으로 유도합니다.
    3. TLA+나 SQL 제약 조건에서 사용되는 논리적 한정사 표현을 수집합니다.
    - [Practice]: "사이트의 모든 작성자는 적어도 하나의 게시글을 소유한다"를 술어 논리로 명세하기.
- **Validate**: 복잡한 한정사 식이 적용되는 개체의 범위를 정확히 한정하고, 예외 케이스(Counter-example)를 논리적으로 제시할 수 있는가?

### 실무 (Practical): SAT/SMT Solver와 정책 검증
- **What + Why**: 수천 개의 정책이 결합되었을 때 발생하는 로직 상충을 Z3와 같은 Solver를 통해 기계적으로 탐지하고 최적의 설정을 찾는 실무 능력을 배양합니다.
- **What to Learn**:
    - [Concepts]: 만족 가능성(Satisfiability), 타당성(Validity), DPLL 알고리즘 기초, 유닛 전파(Unit Propagation), SMT(Satisfiability Modulo Theories).
    - [Tools]: Z3 Solver (Microsoft), Python z3-solver 라이브러리.
    - [Skills]: 비즈니스 규칙의 SAT 문제 환원(Reduction), UNSAT Core(모순의 원인) 추출, SMT 비트 벡터 산술 활용.
- **How to Learn**:
    1. Z3 Python API를 설치하고 간단한 명제 논리식을 Solver에 입력합니다.
    2. `solver.check()` 결과가 `unsat`인 경우, 어떤 제약 조건들이 충돌하는지 식별합니다.
    3. AWS IAM 정책이나 Kubernetes RBAC 설정을 논리식으로 치환하여 권한 과잉을 검사합니다.
    - [Practice]: 스케줄링 제약 조건(A는 B와 같은 시간에 안 됨 등)들을 SAT 문제로 풀어 최적해 도출하기.
- **Validate**: Solver의 'Unknown' 결과를 방지하기 위해 제약 조건의 경계를 명확히 하고, 도출된 해가 실제 물리 시스템에서 유효한가?

### 심화 (Advanced): 메타 논리와 검증의 논리적 한계
- **What + Why**: 정적 분석 도구나 컴파일러가 잡아낼 수 있는 버그와 잡아낼 수 없는 버그의 논리적 경계인 '결정 불가능성'을 이해하여 검증 거버넌스를 수립합니다.
- **What to Learn**:
    - [Concepts]: 건전성(Soundness), 완전성(Completeness), 결정 가능성(Decidability), 처치-튜닝 테제, 반결정성(Semi-decidability).
    - [Skills]: 정적 분석 도구의 오탐(False Positive)과 미탐(False Negative) 원인 분석, 증명 부하(Proof Burden) 추산.
    - [Failure Modes]: 모든 버그를 수학적으로 잡을 수 있다고 맹신하여 검증 도구의 결과를 비판 없이 수용함.
- **How to Learn**:
    1. 술어 논리 추론 체계가 '참인 명제를 항상 찾아낼 수 있는지(완전성)'에 대한 괴델의 이론을 고찰합니다.
    2. 튜링 기계의 정지 문제(Halting Problem)와 정적 분석의 한계 사이의 논리적 연결을 추적합니다.
    3. 실제 프로젝트에서 '어디까지 정형 검증할 것인가'에 대한 비용 대비 효과(ROI) 분석 기준을 마련합니다.
    - [Practice]: 특정 정적 분석 도구가 특정 패턴의 버그를 못 잡는 이유를 결정 불가능성 이론으로 보고서 작성.
- **Validate**: 분석 도구의 결과를 논리적 한계 관점에서 해석하고, 인간의 개입이 필요한 '논리적 예외' 영역을 명확히 구분할 수 있는가?

27:T4e13,<h2 id="overview">Overview</h2>
<p><code>Propositional &#x26; Predicate Logic(명제 및 술어 논리)</code>는 명제를 기본 단위로 하는 명제 논리와 개체 간의 관계 및 한정사(Quantifier)를 포함하는 술어 논리를 포괄합니다. 이는 컴퓨팅 시스템의 상태 명세(Specification), 프로그램 정당성 증명, 그리고 보안 정책 분석의 가장 기초가 되는 형식적 언어이자 추론 체계입니다. 현대의 정적 분석 도구와 SAT/SMT Solver의 내부 엔진은 바로 이 논리 체계에 기반하고 있으며, 가상 공간의 모든 상태 전이가 모순 없이 작동하도록 강제하는 '논리적 물리 법칙' 역할을 합니다.</p>
<h2 id="scope-boundaries">Scope &#x26; Boundaries</h2>
<ul>
<li><strong>포함 범위 (In-Scope)</strong>:
<ul>
<li>명제 논리 기초: 논리 연결사(¬, ∧, ∨, →, ↔), 진리표, 논리적 동치 및 대우 (CS2023: MS-Logic)</li>
<li>보편적 성질: 항진식(Tautology), 만족 가능성(Satisfiability, SAT), 표준형(CNF/DNF) 변환 물리</li>
<li>술어 논리 확장: 변수, 술어(Predicate), 한정사(∀, ∃), 자유 변수와 바인딩 범위(Scope)</li>
<li>추론 및 검증: 자연 연역(Natural Deduction), 추론 규칙(Modus Ponens), 타당성(Validity) 및 건전성 기초</li>
<li>실무 자동화: SAT/SMT Solver의 원리 및 비즈니스 규칙의 만족 가능성 판별 (P3: Formal Methods)</li>
</ul>
</li>
<li><strong>제외 범위 (Out-of-Scope)</strong>:
<ul>
<li>하드웨어 게이트 수준의 불 대수 최적화는 <code>02. Boolean Algebra &#x26; Circuit Logic</code>에서 다룹니다.</li>
<li>시간 논리(LTL/CTL) 및 TLA+ 도구 활용은 <code>03. Formal Specifications &#x26; TLA+</code>로 위임합니다.</li>
</ul>
</li>
<li><strong>경계 근거</strong>: [P1] CS2023 Mathematical Structures, [P3] CyBOK Formal Methods for Security.</li>
</ul>
<h2 id="counterexample">Counterexample</h2>
<ul>
<li><strong>'단순 if-else 문' 오해</strong>: 코드의 조건 분기를 아는 것과, 결합된 수천 개의 정책 간의 모순을 논리적으로 추론하는 것은 별개입니다. 논리는 코드의 실행이 아니라 '상태 공간의 진리성'을 규명하며, 인간의 직관이 놓치는 'Hidden Path'를 찾아냅니다.</li>
<li><strong>'단순 진리표 작성기' 오해</strong>: 진리표는 도구일 뿐입니다. 중요한 것은 복잡한 보안 요구사항을 논리식으로 '환원(Reduction)'하여 SAT 문제로 정의하고 자동화된 수치로 증명하는 아키텍처적 모델링 역량입니다.</li>
</ul>

<h2 id="learning-map">Learning Map</h2>
<ol>
<li><strong>Step 1 (기호 형식화)</strong>: 자연어 명세를 논리 기호로 변환하고, 진리표를 통해 물리적 모든 상태 조합에 대한 진위값을 확정합니다.</li>
<li><strong>Step 2 (구조적 단순화)</strong>: 드모르간의 법칙 등을 적용하여 복잡한 정책식을 CNF 표준형으로 정규화하고 중복된 제약 조건을 제거합니다.</li>
<li><strong>Step 3 (범위의 확장)</strong>: 한정사를 도입하여 '모든 사용자' 또는 '최소 하나의 관리자'와 같은 개체 집합에 대한 전역적 제약 조건을 모델링합니다.</li>
<li><strong>Step 4 (자동화된 검증)</strong>: 정책간의 모순(Contradiction)을 SAT Solver로 자동 탐지하고, 건전한 추론을 통해 보안 취약점을 논리적으로 증명합니다.</li>
</ol>

<h3 id="basic">기본 (Basic): 명제 논리의 구문과 표준형</h3>
<ul>
<li><strong>What + Why</strong>: 모든 컴퓨터 프로그램의 기초 제약 조건을 엄밀한 기호로 정의하고, 기계가 처리하기 가장 효율적인 '표준형'으로 변환하는 법을 배웁니다.</li>
<li><strong>What to Learn</strong>:
<ul>
<li>[Concepts]: 명제(Atom), 연결사(¬, ∧, ∨, →), 진위값(Truth Value), 논리적 동치, 드모르간 법칙, CNF(Conjunctive Normal Form), 리터럴(Literal).</li>
<li>[Skills]: 자연어의 기호화, 진리표 작성, 복잡한 식의 CNF 변환, 진리표를 통한 결과값 예측.</li>

</li>
<li><strong>How to Learn</strong>:
<ol>
<li>개별 명제를 기호 <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>P</mi><mo separator="true">,</mo><mi>Q</mi></mrow><annotation encoding="application/x-tex">P, Q</annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8778em;vertical-align:-0.1944em;"></span><span class="mord mathnormal" style="margin-right:0.13889em;">P</span><span class="mpunct">,</span><span class="mspace" style="margin-right:0.1667em;"></span><span class="mord mathnormal">Q</span></span></span></span>로 치환하는 연습을 수행합니다.</li>

</li>

</ul>
<h3 id="recommended">권장 (Recommended): 술어 논리와 시스템 인바리언트</h3>
<ul>
<li><strong>What + Why</strong>: 개별 인스턴스가 아닌 '집합 전체'에 적용되는 시스템의 핵심 규칙인 인바리언트(Invariant)를 보편 한정사를 통해 기술하는 법을 학습합니다.</li>
<li><strong>What to Learn</strong>:
<ul>
<li>[Concepts]: 술어(Predicate), 한정사(∀, ∃), 정의역(Domain), 자유 변수(Free Var), 바인딩(Binding), 한정사의 부정(Duality).</li>
<li>[Skills]: "모든 유저는 유효한 토큰을 가져야 한다"와 같은 전역 규칙 명세, 한정사의 Scope 설정, 중첩 한정사 분석.</li>
<li>[Failure Modes]: ∀x ∃y와 ∃y ∀x의 순서를 뒤바꾸어 '어떤' 것이 '모든' 것에 대응되는 물리적 의미를 오해함.</li>
</ul>
</li>
<li><strong>How to Learn</strong>:
<ol>
<li>술어 함수 <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>P</mi><mo stretchy="false">(</mo><mi>x</mi><mo stretchy="false">)</mo></mrow><annotation encoding="application/x-tex">P(x)</annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mord mathnormal" style="margin-right:0.13889em;">P</span><span class="mopen">(</span><span class="mord mathnormal">x</span><span class="mclose">)</span></span></span></span>를 정의하고 <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>x</mi></mrow><annotation encoding="application/x-tex">x</annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal">x</span></span></span></span>의 범위(Domain)를 명시적으로 확정합니다.</li>
<li>보편 한정사(∀) 문장의 부정이 왜 존재 한정사(∃)와 NOT의 결합인지 물리적으로 유도합니다.</li>
<li>TLA+나 SQL 제약 조건에서 사용되는 논리적 한정사 표현을 수집합니다.</li>
</ol>
<ul>
<li>[Practice]: "사이트의 모든 작성자는 적어도 하나의 게시글을 소유한다"를 술어 논리로 명세하기.</li>
</ul>
</li>
<li><strong>Validate</strong>: 복잡한 한정사 식이 적용되는 개체의 범위를 정확히 한정하고, 예외 케이스(Counter-example)를 논리적으로 제시할 수 있는가?</li>
</ul>
<h3 id="practical-satsmt-solver">실무 (Practical): SAT/SMT Solver와 정책 검증</h3>
<ul>
<li><strong>What + Why</strong>: 수천 개의 정책이 결합되었을 때 발생하는 로직 상충을 Z3와 같은 Solver를 통해 기계적으로 탐지하고 최적의 설정을 찾는 실무 능력을 배양합니다.</li>
<li><strong>What to Learn</strong>:
<ul>
<li>[Concepts]: 만족 가능성(Satisfiability), 타당성(Validity), DPLL 알고리즘 기초, 유닛 전파(Unit Propagation), SMT(Satisfiability Modulo Theories).</li>
<li>[Tools]: Z3 Solver (Microsoft), Python z3-solver 라이브러리.</li>
<li>[Skills]: 비즈니스 규칙의 SAT 문제 환원(Reduction), UNSAT Core(모순의 원인) 추출, SMT 비트 벡터 산술 활용.</li>
</ul>
</li>
<li><strong>How to Learn</strong>:
<ol>
<li>Z3 Python API를 설치하고 간단한 명제 논리식을 Solver에 입력합니다.</li>
<li><code>solver.check()</code> 결과가 <code>unsat</code>인 경우, 어떤 제약 조건들이 충돌하는지 식별합니다.</li>
<li>AWS IAM 정책이나 Kubernetes RBAC 설정을 논리식으로 치환하여 권한 과잉을 검사합니다.</li>
</ol>
<ul>
<li>[Practice]: 스케줄링 제약 조건(A는 B와 같은 시간에 안 됨 등)들을 SAT 문제로 풀어 최적해 도출하기.</li>
</ul>
</li>
<li><strong>Validate</strong>: Solver의 'Unknown' 결과를 방지하기 위해 제약 조건의 경계를 명확히 하고, 도출된 해가 실제 물리 시스템에서 유효한가?</li>
</ul>
<h3 id="advanced">심화 (Advanced): 메타 논리와 검증의 논리적 한계</h3>
<ul>
<li><strong>What + Why</strong>: 정적 분석 도구나 컴파일러가 잡아낼 수 있는 버그와 잡아낼 수 없는 버그의 논리적 경계인 '결정 불가능성'을 이해하여 검증 거버넌스를 수립합니다.</li>
<li><strong>What to Learn</strong>:
<ul>
<li>[Concepts]: 건전성(Soundness), 완전성(Completeness), 결정 가능성(Decidability), 처치-튜닝 테제, 반결정성(Semi-decidability).</li>
<li>[Skills]: 정적 분석 도구의 오탐(False Positive)과 미탐(False Negative) 원인 분석, 증명 부하(Proof Burden) 추산.</li>
<li>[Failure Modes]: 모든 버그를 수학적으로 잡을 수 있다고 맹신하여 검증 도구의 결과를 비판 없이 수용함.</li>
</ul>
</li>
<li><strong>How to Learn</strong>:
<ol>
<li>술어 논리 추론 체계가 '참인 명제를 항상 찾아낼 수 있는지(완전성)'에 대한 괴델의 이론을 고찰합니다.</li>
<li>튜링 기계의 정지 문제(Halting Problem)와 정적 분석의 한계 사이의 논리적 연결을 추적합니다.</li>
<li>실제 프로젝트에서 '어디까지 정형 검증할 것인가'에 대한 비용 대비 효과(ROI) 분석 기준을 마련합니다.</li>
</ol>
<ul>
<li>[Practice]: 특정 정적 분석 도구가 특정 패턴의 버그를 못 잡는 이유를 결정 불가능성 이론으로 보고서 작성.</li>
</ul>
</li>
<li><strong>Validate</strong>: 분석 도구의 결과를 논리적 한계 관점에서 해석하고, 인간의 개입이 필요한 '논리적 예외' 영역을 명확히 구분할 수 있는가?</li>
</ul>
