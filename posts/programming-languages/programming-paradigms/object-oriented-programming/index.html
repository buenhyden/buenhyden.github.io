객체 지향 프로그래밍 (Object-Oriented Programming)

객체 지향 프로그래밍 (Object-Oriented Programming, OOP) 은 실세계의 사물을 객체로 모델링하여 프로그램을 설계하는 방법론이다. 추상화, 캡슐화, 상속, 다형성이라는 네 가지 핵심 특징을 통해 코드의 재사용성과 유지보수성을 향상시키며, 대규모 소프트웨어 개발에 적합한 프로그래밍 패러다임이다.

3. 요약 문장#

객체 지향 프로그래밍 (OOP, Object-Oriented Programming) 은 데이터와 함수를 ’ 객체 ’ 라는 단위로 묶어, 현실 세계를 모델링하고, 코드의 재사용성·유지보수성·확장성을 높이는 프로그래밍 패러다임입니다 46.

4. 개요#

객체 지향 프로그래밍은 프로그램을 객체라는 단위로 설계하여, 각 객체는 데이터 (속성) 와 함수 (메서드) 를 포함하며, 객체 간 상호작용으로 시스템을 구성합니다. 주요 특징으로는 캡슐화, 상속, 다형성, 추상화가 있으며, 코드의 재사용성과 유지보수성을 높이고, 복잡한 시스템을 효과적으로 관리할 수 있게 합니다. 실무에서는 은행 시스템, 게임, 웹 등 다양한 분야에서 활용됩니다 46.

5. 핵심 개념#

  • 객체 (Object): 데이터 (속성) 와 함수 (메서드) 를 묶은 소프트웨어 모듈
  • 클래스 (Class): 객체를 생성하기 위한 설계도 (템플릿)
  • 캡슐화 (Encapsulation): 객체의 내부 데이터를 외부에서 직접 접근하지 못하게 보호
  • 상속 (Inheritance): 기존 클래스의 속성과 메서드를 상속받아 새로운 클래스 생성
  • 다형성 (Polymorphism): 동일한 메서드 이름으로 다양한 방식의 동작 구현
  • 추상화 (Abstraction): 복잡한 현실을 단순화하여 모델링 65

실무 구현 요소

  • 객체 모델링: 시스템 내 객체 식별 및 속성·메서드 정의
  • 클래스 설계: 객체의 공통 특성을 클래스로 추상화
  • 메시지 전달: 객체 간 상호작용을 메시지로 구현
  • 관계 설계: 상속, 연관, 집합 등 객체 간 관계 정의
  • 설계 패턴 적용: 유지보수 및 확장성을 높이는 패턴 활용 810

6. 조사 내용#

배경#

객체 지향 프로그래밍은 절차지향, 구조적 프로그래밍의 한계를 극복하기 위해 등장했습니다. 대규모 소프트웨어에서 데이터와 함수를 분리해 관리하는 기존 방식은 복잡성 증가와 유지보수 어려움을 초래했고, 객체 단위로 설계함으로써 이 문제를 해결했습니다 13.

목적 및 필요성#

  • 코드 재사용성 및 유지보수성 향상
  • 복잡한 시스템의 모듈화 및 확장성 확보
  • 실세계 모델링을 통한 직관적 설계
  • 변경에 유연하게 대응 가능
    객체 지향 프로그래밍은 대규모 프로젝트에서 필수적입니다 56.

주요 기능 및 역할#

  • 객체 생성 및 관리
  • 데이터와 함수의 결합
  • 객체 간 상호작용 (메시지 전달)
  • 클래스 계층 구조 설계
  • 코드 재사용 및 확장
  • 설계 패턴 적용
    이러한 기능은 소프트웨어의 구조적 안정성과 유연성을 높입니다 [4]11.

특징#

  • 캡슐화: 객체 내부 정보 은닉
  • 상속: 코드 재사용 및 확장
  • 다형성: 동일한 인터페이스로 다양한 구현
  • 추상화: 복잡한 현실 단순화
  • 메시지 전달: 객체 간 통신 65

핵심 원칙#

  • SRP(단일 책임 원칙): 한 클래스는 하나의 책임만 가짐
  • OCP(개방 - 폐쇄 원칙): 확장에는 열려 있고, 변경에는 닫혀 있음
  • LSP(리스코프 치환 원칙): 하위 클래스는 상위 클래스를 대체 가능
  • ISP(인터페이스 분리 원칙): 여러 인터페이스 사용
  • DIP(의존 관계 역전 원칙): 추상화에 의존
    실무에서는 이 원칙들을 준수해 설계 품질을 높입니다 [12][13][14].

class Customer:
    def __init__(self, name):
        self.name = name
        self.accounts = []

    def open_account(self, initial_balance=0):
        account = Account(initial_balance)
        self.accounts.append(account)
        return account

class Account:
    def __init__(self, balance=0):
        self.balance = balance
        self.transactions = []

    def deposit(self, amount):
        self.balance += amount
        self.transactions.append(Transaction(amount, "deposit"))

    def withdraw(self, amount):
        if self.balance >= amount:
            self.balance -= amount
            self.transactions.append(Transaction(amount, "withdraw"))
            return True
        return False

class Transaction:
    def __init__(self, amount, type_):
        self.amount = amount
        self.type = type_
        self.date = "2025-06-14"

3. 200 자 내외 요약#

객체 지향 프로그래밍은 현실 세계의 개체를 클래스와 인스턴스로 모델링하여 캡슐화, 상속, 다형성 등을 통해 재사용성, 유연성, 유지보수성을 높이는 소프트웨어 패러다임입니다. 특히 대규모·복잡한 시스템 설계에 유리하며, 협업 환경에서도 구조적 이점을 제공합니다. (techtarget.com, en.wikipedia.org)

4. 250 자 내외 개요#

OOP 는 데이터와 메서드를 묶어 객체 (Object) 로 구성하고, 이를 분류하는 설계 단위인 클래스 (Class) 를 통해 시스템을 구현합니다. 이론적 기반은 1960~70 년대 Simula 와 Smalltalk 이며, 주요 특징은 캡슐화, 추상화, 상속, 다형성입니다. 이러한 원칙을 통해 코드 재사용성과 유지 보수성을 극대화하며, SOLID 및 GRASP 같은 설계 원칙과 디자인 패턴에 기반한 견고한 아키텍처를 구현할 수 있습니다. 그러나 과도한 계층화는 복잡도를 증가시키며, 퍼포먼스와 학습 난이도에서 단점이 발생할 수 있습니다.

5. 핵심 개념#

  • 클래스 (Class) & 객체 (Object): 구조와 행위를 정의하는 템플릿과, 실행 시 메모리상 인스턴스
  • 캡슐화 (Encapsulation): 데이터와 메서드를 묶고 접근 제어 (public/private) 로 내부 구현 숨김
  • 추상화 (Abstraction): 구현 숨기고 인터페이스만 제공
  • 상속 (Inheritance): 공통 특성과 기능을 부모 - 자식 관계로 재사용 (en.wikipedia.org)
  • 다형성 (Polymorphism): 동일 인터페이스로 다른 구현을 호출 가능

배경 (Background)#

객체 지향 프로그래밍은 1960~70 년대 스몰토크 (Smalltalk) 와 시뮬라 (Simula) 언어에서 출발했습니다.
최초의 객체 지향 언어 Simula 는 시뮬레이션을 위해 클래스와 객체 개념을 도입하였고, Smalltalk 는 메시지 기반 상호작용을 통해 OOP 사고방식을 주류로 만들었습니다.

목적 및 필요성 (Purpose and Need)#

  • 복잡성 관리: 현실세계 유사 객체 모델링을 통해 복잡한 시스템을 계층화·모듈화
  • 코드 재사용성: 상속과 다형성을 통한 기능 공유 및 확장
  • 유지보수성 향상: 캡슐화를 통한 내부 구조 은닉, API 중심 설계
  • 협업 효율: 명확한 책임 분리, 모듈 간 역할 정의

주요 기능 및 역할 (Key Features & Responsibilities)#

  • 클래스와 객체: 시스템 요소 추상화
  • 캡슐화: 내부 상태 보호 및 인터페이스 제공
  • 추상화: 중요한 속성만 노출
  • 상속: 부모의 속성·메서드 재사용
  • 다형성: 같은 메시지에 다른 응답

특징 (Characteristics)#

  • 모듈화: 기능 단위 객체 구성
  • 동적 바인딩: 실행 시점 메서드 결정
  • 유연한 구조: 확장 가능, 테스트 용이
  • 설계 원칙: SOLID, GRASP, 디자인 패턴

필수 구성 요소#

  1. 클래스 (Class)

    • 설계 템플릿: 속성 (field) + 메서드 (method) 정의

  2. 객체/인스턴스 (Object/Instance)

    • 메모리에 생성된 실제 실행 단위

  3. 인터페이스 / 추상 클래스 (Interface / Abstract Class)

    • 공통 행위 정의, 구체 구현은 하위 클래스에서 담당

  4. 패키지 / 모듈 (Package / Module)

    • 클래스 단위 논리 그룹화, 패키지 간 의존성 최소화

  5. 메시지 (호출)–바인딩 (Message–Binding)

    • 런타임 시 메서드 호출을 결정하는 바인딩 기법

  • 리플렉션 (Reflection)
    • 실행 시 타입 정보 메타데이터 조사 및 동적 객체 생성/호출

    • 객체 간 의존성을 런타임 또는 컴파일 타임에 주입

from abc import ABC, abstractmethod

# 전략(Strategy) 인터페이스
class AttackStrategy(ABC):
    @abstractmethod
    def attack(self, attacker, target):
        pass

# 구체 전략
class MeleeAttack(AttackStrategy):
    def attack(self, attacker, target):
        damage = attacker.strength
        target.hp -= damage
        print(f"{attacker.name} attacks {target.name} for {damage} HP!")

class MagicAttack(AttackStrategy):
    def attack(self, attacker, target):
        damage = attacker.intelligence * 1.5
        target.hp -= damage
        print(f"{attacker.name} casts spell on {target.name} for {damage} HP!")

# 캐릭터 클래스
class Character:
    def __init__(self, name, hp, strength, intelligence, attack_strategy: AttackStrategy):
        self.name = name
        self.hp = hp
        self.strength = strength
        self.intelligence = intelligence
        self._attack_strategy = attack_strategy

    def attack(self, target):
        self._attack_strategy.attack(self, target)

    def change_strategy(self, new_strategy: AttackStrategy):
        self._attack_strategy = new_strategy

# 실행 예시
orc = Character("Orc", hp=100, strength=15, intelligence=5, attack_strategy=MeleeAttack())
mage = Character("Mage", hp=80, strength=5, intelligence=20, attack_strategy=MagicAttack())

orc.attack(mage)  # Orc melee
mage.attack(orc)  # Mage magic

# 전략 전환
orc.change_strategy(MagicAttack())
orc.attack(mage)  # Orc magic

  • OOP 클래스/인터페이스 구조
  • 다형성 + 전략 패턴
  • 실행 중 전략 교체 가능 → 유연성 증가

  • 필요 최소한으로 모킹: 외부 의존만 모킹하고, 핵심 로직은 실제 객체로 테스트 (reddit.com)
  • Mock 설정 책임 최소화: 테스트가 mock 설정 세부에 의존하면 리팩토링 시 테스트가 자주 깨짐
  • 명명 규칙 준수: MethodName_StateUnderTest_ExpectedBehavior (learn.microsoft.com)

2. DI(Dependency Injection) 적용 전략#

  • 의존성 역전 원칙 (DIP): 하위 모듈이 아닌 상위 모듈도 추상화에 의존하도록 설계
  • DI 유형별 사용 (위키피디아 기준):
    • 생성자 주입 (Constructor Injection): 기본 의존성 주입 방식
    • 세터 주입 (Setter Injection): 선택적 의존성 시 사용
    • 메소드 주입 (Method Injection): 일시적 의존성에 적합
    • 인터페이스 주입 (Interface Injection): 클라이언트가 주입 메서드 구현
  • DI 를 이용한 테스트 구조
    • DI 기반 설계로 mock 주입 간편 → 테스트 격리성 확보 (microsoft.github.io, en.wikipedia.org, stackoverflow.com)
    • 모의 객체 (Mock) 또는 Fake 객체를 생성자/세터/메소드 주입 방식으로 주입

요약#

객체 지향 프로그래밍 (OOP) 은 실세계의 사물을 객체로 모델링하여 프로그램을 설계하는 패러다임으로, 캡슐화, 상속, 다형성, 추상화의 4 가지 특성을 통해 코드의 재사용성, 유지보수성, 확장성을 높이며 복잡한 소프트웨어 시스템의 개발과 관리를 용이하게 합니다.

개요#

객체 지향 프로그래밍은 클래스와 객체를 기본 단위로 하여 프로그램을 구성하는 방법론입니다. SOLID 원칙을 기반으로 한 설계와 다양한 디자인 패턴을 통해 소프트웨어의 품질을 향상시키며, 현대의 대부분 프로그래밍 언어에서 지원됩니다. 은행 시스템, 게임 개발, 웹 애플리케이션 등 다양한 분야에서 활용되고 있습니다.

1 부: 기본 개념과 배경#

핵심 개념#

이론적 핵심 개념#

  • 객체 (Object): 데이터 (속성) 와 메서드 (행동) 를 캡슐화한 기본 단위
  • 클래스 (Class): 객체를 생성하기 위한 템플릿이자 설계도
  • 인스턴스 (Instance): 클래스로부터 생성된 실제 객체
  • 메시지 패싱 (Message Passing): 객체 간 상호작용을 위한 통신 메커니즘
  • 추상화 (Abstraction): 복잡한 시스템을 단순화하여 핵심만 모델링
  • 4 가지 특성: 캡슐화, 상속, 다형성, 추상화

실무 구현 요소#

  • 접근 제어자: public, private, protected 를 통한 정보 은닉
  • 생성자/소멸자: 객체 생명주기 관리
  • 인터페이스와 추상클래스: 계약 정의와 구현 강제
  • 디자인 패턴: 재사용 가능한 설계 솔루션
  • 의존성 주입 (Dependency Injection): 객체 간 결합도 감소
  • 제어 역전 (Inversion of Control): 프레임워크 기반 객체 관리

배경#

객체 지향 프로그래밍은 1960 년대 시뮬라 (Simula) 언어에서 시작되어 1970 년대 스몰토크 (Smalltalk) 에서 체계화되었습니다. 기존의 절차적 프로그래밍의 한계를 극복하기 위해 등장했으며, 복잡성 증가와 소프트웨어 재사용 필요성이 주요 동인이었습니다.

목적 및 필요성#

  1. 복잡성 관리: 대규모 시스템을 관리 가능한 단위로 분할
  2. 코드 재사용: 한 번 작성된 코드의 재활용을 통한 개발 효율성 향상
  3. 유지보수성: 모듈화된 구조로 변경과 확장이 용이
  4. 협업 증진: 명확한 인터페이스를 통한 팀 개발 지원
  5. 품질 향상: 캡슐화를 통한 오류 감소와 안정성 증대

2 부: 핵심 원칙과 특징, 구조#

주요 기능 및 역할#

객체 지향 프로그래밍의 주요 기능은 현실 세계의 복잡한 문제를 객체 간의 상호작용으로 모델링하여 해결하는 것입니다. 각 객체는 명확한 책임을 가지며, 메시지를 통해 협력하여 시스템 전체의 목표를 달성합니다.

특징#

4 가지 핵심 특징#

  1. 캡슐화 (Encapsulation)

    • 데이터와 메서드를 하나의 단위로 묶음
    • 정보 은닉을 통한 내부 구현 보호
    • 접근 제어자를 통한 인터페이스 제어

  2. 상속 (Inheritance)

    • 기존 클래스의 속성과 메서드를 새 클래스가 물려받음
    • 코드 재사용과 계층적 구조 형성
    • IS-A 관계 표현

  3. 다형성 (Polymorphism)

    • 동일한 인터페이스로 다양한 객체 처리
    • 런타임 바인딩을 통한 유연성 제공
    • 메서드 오버라이딩과 오버로딩

  4. 추상화 (Abstraction)

    • 복잡한 시스템의 핵심만 모델링
    • 불필요한 세부사항 제거
    • 인터페이스와 추상클래스 활용

핵심 원칙 (SOLID 원칙)#

SOLID 원칙 상세 분석#

  1. 단일 책임 원칙 (Single Responsibility Principle, SRP)

    • 클래스는 하나의 책임만 가져야 함
    • 변경의 이유는 단 하나여야 함

  2. 개방 - 폐쇄 원칙 (Open-Closed Principle, OCP)

    • 확장에는 열려있고 변경에는 닫혀있어야 함
    • 기존 코드 수정 없이 새 기능 추가

  3. 리스코프 치환 원칙 (Liskov Substitution Principle, LSP)

    • 하위 타입은 상위 타입으로 대체 가능해야 함
    • 상속 관계의 올바른 설계 보장

  4. 인터페이스 분리 원칙 (Interface Segregation Principle, ISP)

    • 클라이언트는 사용하지 않는 인터페이스에 의존하지 않아야 함
    • 응집력 있는 인터페이스 설계

  5. 의존성 역전 원칙 (Dependency Inversion Principle, DIP)

    • 고수준 모듈은 저수준 모듈에 의존하지 않아야 함
    • 추상화에 의존해야 함

필수 구성 요소#

  1. 클래스 (Class)

    • 기능: 객체의 템플릿 역할
    • 역할: 속성과 메서드 정의
    • 특징: 타입 안전성 제공

  2. 객체 (Object)

    • 기능: 실제 데이터와 행동 보유

    • 특징: 상태와 행동 캡슐화

  3. 메서드 (Method)

    • 기능: 객체의 행동 정의
    • 역할: 데이터 조작과 기능 수행
    • 특징: 매개변수와 반환값 지원

  4. 속성 (Attribute)

    • 기능: 객체의 상태 저장
    • 역할: 데이터 보관
    • 특징: 접근 제어 적용

  1. 인터페이스 (Interface)

    • 기능: 계약 정의
    • 역할: 다형성 지원
    • 특징: 구현 강제

  2. 추상 클래스 (Abstract Class)

    • 기능: 공통 구현 제공
    • 역할: 템플릿 메서드 패턴 지원
    • 특징: 부분 구현 허용

  3. 내부 클래스 (Inner Class)

    • 기능: 클래스 내부 캡슐화
    • 역할: 헬퍼 클래스 제공
    • 특징: 외부 클래스 접근 가능

구현 기법#

생성 패턴 (Creational Patterns)#

  1. 싱글톤 (Singleton)

    • 정의: 클래스의 인스턴스를 하나만 생성하도록 보장
    • 구성: private 생성자, static 인스턴스, public 접근 메서드
    • 목적: 전역 접근점 제공, 자원 절약
    • 실제 예시: 데이터베이스 연결 관리자, 로그 관리자

  2. 팩토리 (Factory)

    • 정의: 객체 생성 로직을 별도 클래스로 분리
    • 구성: 팩토리 클래스, 제품 인터페이스, 구체 제품들
    • 목적: 객체 생성의 복잡성 숨김
    • 실제 예시: GUI 컴포넌트 생성, 데이터베이스 드라이버 선택

구조 패턴 (Structural Patterns)#

  1. 어댑터 (Adapter)

    • 정의: 호환되지 않는 인터페이스를 연결
    • 구성: 타겟 인터페이스, 어댑터 클래스, 어댑티 클래스
    • 목적: 레거시 코드와 신규 시스템 통합
    • 실제 예시: 서드파티 라이브러리 통합, API 버전 호환성

  2. 데코레이터 (Decorator)

    • 정의: 기존 객체에 새로운 기능을 동적으로 추가
    • 구성: 컴포넌트 인터페이스, 구체 컴포넌트, 데코레이터들
    • 목적: 상속 없이 기능 확장
    • 실제 예시: 스트림 필터링, UI 컴포넌트 효과 추가

행위 패턴 (Behavioral Patterns)#

  1. 전략 (Strategy)

    • 정의: 알고리즘을 캡슐화하고 교체 가능하게 만듦
    • 구성: 전략 인터페이스, 구체 전략들, 컨텍스트 클래스
    • 목적: 런타임에 알고리즘 변경
    • 실제 예시: 결제 처리 시스템, 정렬 알고리즘 선택

  2. 옵저버 (Observer)

    • 정의: 객체 상태 변화를 여러 객체에 자동 통지
    • 구성: 주제 인터페이스, 옵저버 인터페이스, 구체 구현들
    • 목적: 느슨한 결합으로 이벤트 처리
    • 실제 예시: MVC 패턴, 이벤트 리스너 시스템

from abc import ABC, abstractmethod
from datetime import datetime
from typing import List, Optional

# 추상 클래스 정의
class Account(ABC):
    """계좌 추상 클래스"""
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0):
        self._account_number = account_number  # 캡슐화: private 속성
        self._owner = owner
        self._balance = initial_balance
        self._transactions: List['Transaction'] = []
    
    @property
    def account_number(self) -> str:
        """계좌번호 조회 (읽기 전용)"""
        return self._account_number
    
    @property 
    def balance(self) -> float:
        """잔액 조회"""
        return self._balance
    
    def deposit(self, amount: float) -> bool:
        """입금 처리"""
        if amount <= 0:
            raise ValueError("입금액은 0보다 커야 합니다")
        
        self._balance += amount
        self._add_transaction("DEPOSIT", amount)
        return True
    
    @abstractmethod
    def withdraw(self, amount: float) -> bool:
        """출금 처리 (추상 메서드)"""
        pass
    
    def _add_transaction(self, transaction_type: str, amount: float):
        """거래 내역 추가"""
        transaction = Transaction(transaction_type, amount, self._balance)
        self._transactions.append(transaction)

# 구체 클래스들 (상속)
class SavingsAccount(Account):
    """저축 계좌"""
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0):
        super().__init__(account_number, owner, initial_balance)
        self._interest_rate = 0.02  # 2% 이자율
    
    def withdraw(self, amount: float) -> bool:
        """저축계좌 출금 (잔액 확인)"""
        if amount <= 0:
            raise ValueError("출금액은 0보다 커야 합니다")
        
        if self._balance < amount:
            return False  # 잔액 부족
        
        self._balance -= amount
        self._add_transaction("WITHDRAW", amount)
        return True
    
    def calculate_interest(self) -> float:
        """이자 계산"""
        return self._balance * self._interest_rate

class CheckingAccount(Account):
    """당좌 계좌 (마이너스 허용)"""
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0, 
                 overdraft_limit: float = 1000):
        super().__init__(account_number, owner, initial_balance)
        self._overdraft_limit = overdraft_limit
    
    def withdraw(self, amount: float) -> bool:
        """당좌계좌 출금 (마이너스 한도 확인)"""
        if amount <= 0:
            raise ValueError("출금액은 0보다 커야 합니다")
        
        if (self._balance - amount) < -self._overdraft_limit:
            return False  # 한도 초과
        
        self._balance -= amount
        self._add_transaction("WITHDRAW", amount)
        return True

# 거래 내역 클래스
class Transaction:
    """거래 내역"""
    def __init__(self, transaction_type: str, amount: float, balance_after: float):
        self.transaction_type = transaction_type
        self.amount = amount
        self.balance_after = balance_after
        self.timestamp = datetime.now()
    
    def __str__(self) -> str:
        return f"{self.timestamp}: {self.transaction_type} {self.amount}, 잔액: {self.balance_after}"

# 은행 서비스 클래스 (컴포지션)
class BankService:
    """은행 서비스 관리"""
    def __init__(self):
        self._accounts: dict[str, Account] = {}
    
    def create_account(self, account_type: str, account_number: str, 
                      owner: str, initial_balance: float = 0) -> Account:
        """계좌 생성 (팩토리 패턴)"""
        if account_type == "SAVINGS":
            account = SavingsAccount(account_number, owner, initial_balance)
        elif account_type == "CHECKING":
            account = CheckingAccount(account_number, owner, initial_balance)
        else:
            raise ValueError("지원하지 않는 계좌 유형입니다")
        
        self._accounts[account_number] = account
        return account
    
    def get_account(self, account_number: str) -> Optional[Account]:
        """계좌 조회"""
        return self._accounts.get(account_number)
    
    def transfer(self, from_account: str, to_account: str, amount: float) -> bool:
        """계좌 이체"""
        source = self.get_account(from_account)
        target = self.get_account(to_account)
        
        if not source or not target:
            return False
        
        if source.withdraw(amount):
            target.deposit(amount)
            return True
        return False

# 사용 예시 (다형성 활용)
def demonstrate_polymorphism():
    """다형성 시연"""
    bank = BankService()
    
    # 다양한 타입의 계좌 생성
    savings = bank.create_account("SAVINGS", "SAV001", "김철수", 10000)
    checking = bank.create_account("CHECKING", "CHK001", "이영희", 5000)
    
    # 동일한 인터페이스로 다른 동작 수행 (다형성)
    accounts = [savings, checking]
    
    for account in accounts:
        print(f"계좌: {account.account_number}, 잔액: {account.balance}")
        account.deposit(1000)  # 모든 계좌 타입에서 동일하게 작동
        account.withdraw(500)   # 각 계좌 타입별로 다른 로직 실행
        print(f"처리 후 잔액: {account.balance}")

# 실행
if __name__ == "__main__":
    demonstrate_polymorphism()

객체 지향 프로그래밍 (OOP) 은 소프트웨어 개발에서 현실 세계의 객체를 모델링하여 클래스를 정의하고, 이를 기반으로 객체를 생성하여 프로그램을 구성하는 방식이다. 주요 원칙으로는 캡슐화, 상속, 다형성, 추상화가 있으며, 이러한 원칙들은 코드의 재사용성, 확장성, 유지보수성을 향상시킨다. OOP 는 복잡한 시스템을 모듈화하고, 변경에 유연하게 대응할 수 있도록 설계되어 현대 소프트웨어 개발에서 널리 사용된다.

핵심 개념#

  1. 객체 (Object) 와 클래스 (Class)

    • 클래스 (Class): 객체를 생성하기 위한 청사진으로, 속성과 메서드를 정의한다.
    • 객체 (Object): 클래스의 인스턴스로, 실제 데이터를 가지며 동작을 수행한다.
    • 인스턴스 (Instance): 특정 클래스로부터 생성된 객체

  2. 4 대 특징 (Pillar of OOP)

    • 캡슐화 (Encapsulation): 데이터와 메서드를 하나의 단위로 묶고, 외부로부터의 직접 접근을 제한하여 데이터 보호를 강화한다.
    • 상속 (Inheritance): 기존 클래스의 특성을 새로운 클래스가 물려받아 코드의 재사용성을 높인다.
    • 다형성 (Polymorphism): 동일한 인터페이스를 통해 다양한 형태의 동작을 구현하여 유연성을 제공한다.
    • 추상화 (Abstraction): 복잡한 시스템에서 핵심적인 부분만을 노출하여 사용자가 필요한 정보에만 집중할 수 있도록 한다.

  3. 메시지 패싱 (Message Passing)

    • 객체 간 상호작용의 기본 메커니즘
    • 메서드 호출을 통한 객체 간 통신

핵심 원칙#

SOLID 원칙:

  • 단일 책임 원칙 (Single Responsibility Principle): 클래스는 하나의 책임만 가져야 한다.
  • 개방 - 폐쇄 원칙 (Open-Closed Principle): 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.
  • 리스코프 치환 원칙 (Liskov Substitution Principle): 자식 클래스는 부모 클래스의 기능을 대체할 수 있어야 한다.
  • 인터페이스 분리 원칙 (Interface Segregation Principle): 클라이언트는 자신이 사용하지 않는 인터페이스에 의존하지 않아야 한다.
  • 의존 역전 원칙 (Dependency Inversion Principle): 고수준 모듈은 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 한다.

상속 (Inheritance)#

정의: 기존 (부모, Super) 클래스의 속성과 기능 (메서드) 을 새로운 (자식, Sub) 클래스가 물려받아 재사용하고, 필요에 따라 확장할 수 있게 해주는 객체지향 프로그래밍의 핵심 개념이다.

주요 목적:

  • 코드의 재사용성과 확장성
  • 유지보수성 향상.
    공통 기능을 부모 클래스에 정의하고, 자식 클래스에서 추가 기능이나 수정이 필요할 때만 오버라이딩 (재정의) 하여 사용.

종류:

유형설명특징예시
단일 상속 (Single)하나의 부모 클래스에서 상속가장 단순하고 안전한 형태Animal → Dog
다중 상속 (Multiple)여러 부모 클래스에서 상속다이아몬드 문제 발생 가능Person, Employee → Manager
다단계 상속 (Multilevel)상속의 연쇄적 확장계층적 구조 형성Vehicle → Car → SportsCar
계층적 상속 (Hierarchical)하나의 부모에서 여러 자식 상속공통 기능 분리에 유용Shape → Circle, Square, Triangle
하이브리드 상속 (Hybrid)여러 상속 유형의 조합복잡하지만 유연한 구조다중 + 계층적 상속 결합

예시: 자동차 (Car) 클래스를 부모로 하여 BMW, Benz, Genesis 등 다양한 차종의 자식 클래스를 만들고, 공통 속성 (바퀴 수, 엔진 등) 은 부모에, 각 차종의 특성은 자식 클래스에 구현.
키워드 예시:
- Python: class Child(Parent):
- Java: class Child extends Parent
장점:

  • 코드 중복 제거
  • 일관성 유지
  • 변경 용이성 (부모 클래스만 수정해도 모든 자식 클래스에 반영).
    주의 사항:
  • 잘못 사용하면 강한 결합도를 유발하여 유지보수를 어렵게 함 → 필요 시 컴포지션으로 대체 고려
다형성 (Polymorphism)#

정의: 다형성은 하나의 인터페이스 (메서드, 클래스 등) 로 여러 형태의 동작을 구현할 수 있는 능력. 즉, 동일한 메시지 (메서드 호출) 에 대해 객체에 따라 서로 다르게 반응할 수 있다.

주요 목적: 유연한 코드 구조, 인터페이스 기반 설계, 코드 재사용성 향상.

종류:

  • 컴파일 타임 다형성 (오버로딩, Overloading): 같은 이름의 메서드를 매개변수의 타입이나 개수로 구분하여 여러 개 정의.
  • 런타임 다형성 (오버라이딩, Overriding): 부모 클래스의 메서드를 자식 클래스에서 재정의하여, 실행 시점에 실제 객체의 타입에 따라 동작이 결정.

예시: Animal 이라는 부모 클래스의 makeSound() 메서드를 Dog, Cat 등 자식 클래스에서 각각 다르게 구현. Animal 타입의 변수로 여러 동물 객체를 다룰 수 있음.

캡슐화 (Encapsulation)#

정의: 캡슐화는 객체의 속성과 메서드를 하나의 단위 (클래스) 로 묶고, 외부에서 내부 구현 (데이터, 상태) 에 직접 접근하지 못하도록 은닉하는 원칙이다.

주요 목적:

  • 정보 은닉 (Information Hiding)
  • 의도하지 않은 접근이나 변경 방지
  • 객체의 무결성 보장.

방법:

  • 멤버 변수 (속성) 는 보통 private 으로 선언하고, 외부에서는 public 메서드 (getter/setter) 로만 접근하게 한다.

패턴 적용 예시:

  • DTO (Data Transfer Object)
  • 접근자 (getter)/설정자 (setter)

장점:

  • 오류 발생 범위 축소
  • 유지보수성 향상
  • 객체의 일관성 유지.
추상화 (Abstraction)#

정의: 추상화는 복잡한 현실 세계의 엔티티에서 핵심적인 속성과 동작만을 모델링하여, 불필요한 세부사항은 감추는 개념.

주요 목적:

  • 복잡성 감소
  • 코드의 단순화
  • 재사용성 및 유연성 향상.

구현 방법: 추상 클래스 (abstract class), 인터페이스 (interface) 등을 사용해 공통 동작의 틀만 정의하고, 구체적 구현은 하위 클래스에서 담당.

예시: Animal 이라는 추상 클래스에 makeSound() 라는 추상 메서드만 정의하고, Cat, Dog 등에서 구체적으로 구현.

장점:

  • 코드 품질 및 유지보수성 향상
  • 다양한 객체를 일관된 방식으로 다룰 수 있음.

구조 및 아키텍처#

객체 지향 프로그래밍의 구조는 다음과 같은 계층적 구조를 갖는다:

graph TD
    A[시스템] --> B[패키지/네임스페이스]
    B --> C[클래스]
    C --> D[객체/인스턴스]
    D --> E[메서드]
    D --> F[속성/필드]
    C --> G[상속 관계]
    C --> H[인터페이스]
    H --> I[구현]
구성 요소#
구분구성요소기능 및 역할특징
필수클래스 (Class)객체 생성을 위한 템플릿- 속성과 메서드 정의
- 객체의 구조와 행동 명세
객체 (Object)클래스의 실제 인스턴스- 실행 시점에 메모리에 생성
- 고유한 상태 보유
메서드 (Method)객체의 행동을 정의하는 함수- 데이터 조작 및 연산 수행
- 객체 간 상호작용 메커니즘
속성 (Attribute)객체의 상태를 나타내는 데이터- 인스턴스 변수와 클래스 변수
- 접근 제어자로 가시성 조절
선택인터페이스 (Interface)메서드 시그니처만 정의- 다중 상속 지원
- 계약 (Contract) 역할
추상 클래스일부 구현된 상위 클래스- 공통 기능 제공
- 인스턴스 생성 불가
패키지/네임스페이스클래스들의 그룹화- 이름 충돌 방지
- 논리적 구조화

주요 원리와 작동 원리#

객체 지향 프로그래밍은 클래스와 객체를 중심으로 동작한다. 클래스는 객체를 생성하기 위한 틀이며, 객체는 클래스의 인스턴스로서 실제 데이터를 가지고 동작한다. 객체들은 메시지를 주고받으며 상호작용하며, 이러한 구조를 통해 프로그램이 실행된다.

  1. 메시지 전달 메커니즘
sequenceDiagram
    participant A as 객체 A
    participant B as 객체 B
    A->>B: 메시지 전송
    B-->>B: 메서드 실행
    B->>A: 결과 반환
  1. 상속 계층 구조
graph TD
    A[상위 클래스] --> B[하위 클래스 1]
    A --> C[하위 클래스 2]
    B --> D[하위 클래스 1-1]
    C --> E[하위 클래스 2-1]
  1. 다형성 실현 메커니즘
    • 컴파일 타임 다형성: 메서드 오버로딩
    • 런타임 다형성: 메서드 오버라이딩과 동적 바인딩

구현 기법#

기법 유형기법 이름정의 및 구성 요소목적 및 활용 예시
클래스 기반클래스 설계클래스 정의 → 속성 (필드) 및 메서드 선언 → 인스턴스 생성타입 안정성과 구조화된 설계 제공예: Java, C++ 의 클래스 기반 OOP
프로토타입 기반프로토타입 체인기존 객체를 복제하여 새 객체 생성, 동적 속성 추가 가능유연한 동적 구조 제공예: JavaScript 의 객체 생성 방식
상속 및 구조화상속과 다형성 활용부모 - 자식 클래스 관계 구성, 인터페이스 구현코드 재사용과 확장성 확보예: Shape → Circle, Rectangle
기능 조합믹스인 (Mixin)기능 단위 모듈 정의 → 클래스 또는 객체에 주입다중 상속의 대안, 유연한 기능 확장예: Python, Ruby
객체 포함컴포지션 (Composition)기능 객체를 생성하고 포함시켜 기능 위임런타임 동작 변경과 느슨한 결합예: 전략 패턴, 데코레이터 패턴
접근 제어캡슐화 (Encapsulation)접근 제한자 (private, public 등) + getter/setter 메서드로 데이터 보호외부 접근 제한, 유지보수성 향상예: 클래스 내부 변수 은닉
동작 다양성다형성 (Polymorphism)동일한 인터페이스로 다양한 구현체 실행 가능코드의 유연성 및 확장성 증가예: draw() 메서드를 다양한 도형에서 구현
복잡도 감추기추상화 (Abstraction)핵심 기능만 노출하고 내부 구현 세부사항은 감춤시스템 복잡성 감소, 인터페이스 기반 설계예: Database 인터페이스
설계 최적화디자인 패턴 적용생성, 구조, 행위 패턴 등 문제 해결을 위한 반복 가능한 설계 기법 적용Singleton, Observer, Factory, Strategy 등 패턴을 통한 구조적 문제 해결

상태 관리의 어려움객체 상태 변화에 따른 예측 불가능한 부작용, 디버깅의 어려움 발생불변 객체 (Immutable Object) 사용, 상태 변경 최소화

객체 간 의존성 증가객체들 간의 강한 결합으로 시스템 변경이 어려움의존성 주입 (Dependency Injection), 느슨한 결합 설계
성능 오버헤드객체 생성, 소멸, 다형적 메서드 호출 시의 성능 저하객체 풀링 (Object Pooling), 메서드 인라인 최적화, GC 튜닝

테스트 어려움객체 간 상호작용이 많을 경우 단위 테스트 작성이 어려움의존성 역전 (DIP), 인터페이스 기반 설계로 Mock 테스트 유도

from abc import ABC, abstractmethod
from datetime import datetime
from typing import List, Optional

# 추상화 - 기본 계좌 인터페이스
class Account(ABC):
    """계좌 추상 클래스"""
    
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0):
        self._account_number = account_number  # 캡슐화 - private 속성
        self._owner = owner
        self._balance = initial_balance
        self._transactions: List['Transaction'] = []
    
    @property  # 캡슐화 - getter를 통한 접근
    def account_number(self) -> str:
        return self._account_number
    
    @property
    def owner(self) -> str:
        return self._owner
    
    @property
    def balance(self) -> float:
        return self._balance
    
    @abstractmethod  # 추상화 - 하위 클래스에서 구현 필요
    def calculate_interest(self) -> float:
        pass
    
    def deposit(self, amount: float) -> bool:
        """입금 메서드"""
        if amount <= 0:
            return False
        
        self._balance += amount
        transaction = Transaction(
            self._account_number, 
            'DEPOSIT', 
            amount, 
            self._balance
        )
        self._transactions.append(transaction)
        return True
    
    def withdraw(self, amount: float) -> bool:
        """출금 메서드 - 하위 클래스에서 오버라이드 가능"""
        if amount <= 0 or amount > self._balance:
            return False
        
        self._balance -= amount
        transaction = Transaction(
            self._account_number, 
            'WITHDRAW', 
            amount, 
            self._balance
        )
        self._transactions.append(transaction)
        return True
    
    def get_transaction_history(self) -> List['Transaction']:
        """거래 내역 조회"""
        return self._transactions.copy()

# 상속 - 저축계좌 구현
class SavingsAccount(Account):
    """저축계좌 - Account를 상속"""
    
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0, interest_rate: float = 0.02):
        super().__init__(account_number, owner, initial_balance)
        self._interest_rate = interest_rate
        self._withdrawal_limit = 5  # 월 출금 제한
        self._monthly_withdrawals = 0
    
    def calculate_interest(self) -> float:
        """이자 계산 구현"""
        return self._balance * self._interest_rate
    
    def withdraw(self, amount: float) -> bool:
        """출금 메서드 오버라이드 - 월 출금 제한 추가"""
        if self._monthly_withdrawals >= self._withdrawal_limit:
            print("월 출금 한도를 초과했습니다.")
            return False
        
        if super().withdraw(amount):  # 부모 클래스 메서드 호출
            self._monthly_withdrawals += 1
            return True
        return False
    
    def reset_monthly_withdrawals(self):
        """월 출금 횟수 초기화"""
        self._monthly_withdrawals = 0

# 상속 - 당좌계좌 구현
class CheckingAccount(Account):
    """당좌계좌 - Account를 상속"""
    
    def __init__(self, account_number: str, owner: str, initial_balance: float = 0, overdraft_limit: float = 1000):
        super().__init__(account_number, owner, initial_balance)
        self._overdraft_limit = overdraft_limit
    
    def calculate_interest(self) -> float:
        """당좌계좌는 이자 없음"""
        return 0
    
    def withdraw(self, amount: float) -> bool:
        """출금 메서드 오버라이드 - 마이너스 통장 기능"""
        if amount <= 0:
            return False
        
        # 마이너스 한도까지 출금 허용
        if self._balance - amount >= -self._overdraft_limit:
            self._balance -= amount
            transaction = Transaction(
                self._account_number, 
                'WITHDRAW', 
                amount, 
                self._balance
            )
            self._transactions.append(transaction)
            return True
        
        print("마이너스 한도를 초과합니다.")
        return False

# 거래 내역 클래스
class Transaction:
    """거래 내역을 나타내는 클래스"""
    
    def __init__(self, account_number: str, transaction_type: str, amount: float, balance_after: float):
        self.account_number = account_number
        self.transaction_type = transaction_type
        self.amount = amount
        self.balance_after = balance_after
        self.timestamp = datetime.now()
    
    def __str__(self) -> str:
        return f"{self.timestamp.strftime('%Y-%m-%d %H:%M:%S')} | {self.transaction_type} | {self.amount:,f}원 | 잔액: {self.balance_after:,f}원"

# 은행 시스템 클래스
class BankSystem:
    """은행 시스템 관리 클래스"""
    
    def __init__(self):
        self._accounts: dict[str, Account] = {}
    
    def create_savings_account(self, account_number: str, owner: str, initial_balance: float = 0) -> bool:
        """저축계좌 생성"""
        if account_number in self._accounts:
            return False
        
        self._accounts[account_number] = SavingsAccount(account_number, owner, initial_balance)
        return True
    
    def create_checking_account(self, account_number: str, owner: str, initial_balance: float = 0) -> bool:
        """당좌계좌 생성"""
        if account_number in self._accounts:
            return False
        
        self._accounts[account_number] = CheckingAccount(account_number, owner, initial_balance)
        return True
    
    def get_account(self, account_number: str) -> Optional[Account]:
        """계좌 조회"""
        return self._accounts.get(account_number)
    
    def transfer(self, from_account: str, to_account: str, amount: float) -> bool:
        """계좌 이체"""
        from_acc = self.get_account(from_account)
        to_acc = self.get_account(to_account)
        
        if not from_acc or not to_acc:
            return False
        
        if from_acc.withdraw(amount):
            if to_acc.deposit(amount):
                return True
            else:
                # 입금 실패 시 롤백
                from_acc.deposit(amount)
        
        return False
    
    def calculate_total_interest(self) -> float:
        """전체 계좌 이자 총합 계산 - 다형성 활용"""
        total_interest = 0
        for account in self._accounts.values():
            total_interest += account.calculate_interest()  # 다형성 - 각 계좌 유형별로 다른 구현
        return total_interest

클래스 구조 관리클래스 수가 과도하면 복잡도 증가 및 유지보수성 저하역할 단위로 클래스를 구성하고, 불필요한 분할은 지양

상태 관리 전략상태가 많은 객체는 직렬화, 캐싱, 테스트에서 성능 문제 발생가능한 Stateless 설계 적용, DTO 와 상태 분리

기본 개념객체 (Object)데이터와 메서드를 포함하는 소프트웨어의 실행 단위, 클래스의 인스턴스

OOP 4 대 특성캡슐화 (Encapsulation)데이터와 메서드를 하나로 묶고 외부로부터 숨김으로써 데이터 보호

디자인 패턴싱글톤 (Singleton)하나의 인스턴스만 존재하도록 제한하는 생성 패턴

옵저버 (Observer)객체 상태 변화를 다른 객체에 자동으로 알리는 행위 패턴
컴포지션 (Composition)객체 포함을 통해 기능을 조합하는 방식으로, 상속을 대체할 수 있음
아키텍처 패턴MVC (Model-View-Controller)사용자 인터페이스를 모델, 뷰, 컨트롤러로 분리하는 구조적 설계 패턴
레이어드 아키텍처시스템을 계층 구조로 나누어 책임을 분리하는 설계 방식
마이크로서비스 아키텍처시스템을 독립적으로 배포 가능한 작은 서비스들로 구성하여 확장성과 유지보수성을 높이는 구조

책임 주도 설계 (RDD)객체 간 책임을 명확히 나누고 협력 중심으로 모델링하는 설계 방법론
