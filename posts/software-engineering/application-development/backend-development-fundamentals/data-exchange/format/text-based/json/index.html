Json (JavaScript Object Notation)#

용어 정리#

용어설명

참고 및 출처#

1. 주제의 분류 적절성#

JSON(JavaScript Object Notation, 자바스크립트 객체 표기법) 은 “Computer Science and Engineering” > “Backend Development” > “Backend Fundamentals” > “Data Exchange” > “Format” > “Text-Based” 분류에 매우 적합하다. JSON 은 백엔드 시스템에서 데이터 교환, 저장, API 응답 등 핵심적인 역할을 하는 대표적인 텍스트 기반 데이터 포맷이다 [1][2][3].

2. 요약 설명 (200 자 내외)#

JSON 은 텍스트 기반의 경량 데이터 교환 포맷으로, 다양한 언어와 플랫폼에서 구조화된 데이터를 직관적으로 표현하고 교환할 수 있다. 웹 API, 설정 파일, NoSQL 데이터베이스 등에서 표준적으로 사용된다 [2][3][8].

3. 전체 개요 (200 자 내외)#

JSON 은 키 - 값 쌍과 배열 구조를 활용해 데이터를 표현하는 텍스트 포맷으로, 언어 독립적이며 인간과 기계 모두 읽기 쉽다. REST API, 웹·모바일 앱, 설정 파일, NoSQL 데이터베이스 등 다양한 환경에서 데이터 저장과 교환의 표준으로 자리잡았다 [1][2][6][8].

핵심 개념#

  • **JSON(JavaScript Object Notation, 자바스크립트 객체 표기법)**은 구조화된 데이터를 텍스트로 표현하는 경량 데이터 교환 포맷이다 [1][2][6].
  • **키 - 값 쌍 (Object)**과 배열 (Array) 구조로 데이터 계층을 표현한다.
  • 기본 데이터 타입: 문자열 (String), 숫자 (Number), 불리언 (Boolean), 배열 (Array), 객체 (Object), null[1][6].
  • 언어 및 플랫폼 독립적이며, 다양한 프로그래밍 언어에서 쉽게 파싱 및 생성 가능하다 [2][3][8].
  • 주로 RESTful API, 설정 파일, NoSQL 데이터베이스 (MongoDB 등) 에서 표준 포맷으로 사용된다 [2][8][9].

주요 내용 정리#

목적 및 필요성#

  • 시스템 간 데이터 교환의 표준화 및 효율성 확보 [2][3].
  • API, 웹/모바일 앱, 서버 간 통신 등에서 데이터 구조를 직관적으로 표현 [2][3][8].
  • 설정 파일, 데이터베이스 저장 등 다양한 환경에서 범용적으로 활용 [2][9].

주요 기능 및 역할#

  • 데이터 직렬화 및 역직렬화 (serialize/deserialize)
  • REST API 응답 및 요청 포맷
  • 설정 파일 및 데이터베이스 저장 포맷
  • 다양한 언어 및 플랫폼 간 데이터 호환성 제공 [2][3][8].

특징#

  • 경량, 텍스트 기반, 인간/기계 모두 읽기 쉬움
  • 계층적 구조 (중첩 가능)
  • 스키마 없음 (유연성 높음)
  • 언어 독립적 [1][2][6].

핵심 원칙#

  • 모든 키는 문자열, 값은 문자열, 숫자, 불리언, 배열, 객체, null 중 하나여야 함 [1][5][6].
  • 키 - 값 쌍은 중괄호 ({}) 로, 배열은 대괄호 ([]) 로 감쌈
  • 문자열은 반드시 큰따옴표 (") 사용
  • 데이터 구조는 계층적으로 중첩 가능

주요 원리 및 작동 원리#

  • 데이터는 중괄호 ({}) 로 감싼 객체, 대괄호 ([]) 로 감싼 배열로 표현
  • 각 키 - 값 쌍은 콜론 (:) 으로 구분, 쌍끼리는 쉼표 (,) 로 구분
  • JSON 파서 (parser) 를 통해 문자열 ↔ 객체 간 변환이 가능

구조 및 아키텍처#

구성 요소기능 및 역할

배열 (Array)대괄호 ([]) 로 감싸진 값들의 순서 있는 리스트
키 (Key)문자열, 각 값의 이름 역할
값 (Value)문자열, 숫자, 불리언, 배열, 객체, null 중 하나
데이터 타입JSON 이 지원하는 기본 타입 (문자열, 숫자 등)

장점과 단점#

구분항목설명
✅ 장점경량성텍스트 기반으로 전송 및 저장이 효율적임
범용성다양한 언어 및 플랫폼에서 지원, 호환성 높음
가독성인간과 기계 모두 읽고 쓰기 쉬움
계층적 구조중첩 구조를 통한 복잡한 데이터 표현 가능
⚠ 단점스키마 부재데이터 타입, 구조의 강제성이 없음 (유효성 문제)
대용량 비효율대용량·반복 데이터에선 바이너리 포맷보다 비효율
주석 미지원설정 파일 등에서 주석 사용 불가
보안 이슈입력값 검증 미흡 시 JSON 인젝션 등 취약점 발생

도전 과제#

  • 대용량 데이터 처리 시 성능 저하
  • 스키마 미지원으로 인한 데이터 유효성 검증 필요
  • JSON 인젝션 등 보안 취약점 대응 필요 [7]
  • 주석 미지원으로 인한 설정 파일 관리의 불편함

활용 사례#

시나리오: 쇼핑몰 주문 처리 REST API

  • 사용자가 주문 정보를 입력 → 클라이언트가 JSON 으로 서버에 POST
  • 서버가 주문 처리 후, 결과를 JSON 으로 응답
  • 클라이언트는 응답 데이터를 파싱해 UI 갱신

1. 주제 분류의 적절성 평가#

주제인 JSON(JavaScript Object Notation)은 “Computer Science and Engineering” > “Backend Development” > “Backend Fundamentals” > “Data Exchange” > “Format” > “Text-Based"로 분류하는 것이 적절합니다. JSON은 텍스트 기반의 데이터 교환 형식으로, 백엔드 시스템 간의 데이터 통신에서 핵심적인 역할을 합니다.

2. 요약 설명 (200자 내외)#

JSON은 경량의 텍스트 기반 데이터 교환 형식으로, 사람과 기계 모두 읽고 쓰기 쉬우며, 다양한 프로그래밍 언어에서 지원됩니다. 웹 API, 구성 파일, 데이터 저장 등에서 널리 사용됩니다.

3. 전체 개요 (200자 내외)#

JSON은 키-값 쌍으로 구성된 구조화된 데이터를 표현하는 형식으로, 웹 개발에서 서버와 클라이언트 간의 데이터 교환에 표준으로 자리 잡았습니다. 간결한 문법과 언어 독립성으로 인해 다양한 분야에서 활용되고 있습니다.

4. 핵심 개념#

  • 정의: JSON은 키-값 쌍으로 데이터를 표현하는 텍스트 기반의 데이터 교환 형식입니다.

  • 기원: JavaScript의 객체 리터럴 문법에서 파생되었지만, 언어에 독립적으로 사용됩니다.

  • 표준화: ECMA-404, ISO/IEC 21778:2017, RFC 8259 등에서 정의되어 있습니다.(위키백과)

  • 데이터 타입: 문자열(String), 숫자(Number), 불리언(Boolean), 배열(Array), 객체(Object), null을 지원합니다.(W3C)

  • 문법: 중괄호({})로 객체를, 대괄호([])로 배열을 표현하며, 키는 문자열로 표기하고 콜론(:)으로 값을 연결합니다.

5. 주제와 관련하여 조사할 내용#

목적 및 필요성#

  • 데이터 교환: 서버와 클라이언트 간의 효율적인 데이터 전송을 위해 필요합니다.

  • 구성 파일: 애플리케이션의 설정을 저장하는 데 사용됩니다.

주요 기능 및 역할#

  • 경량성: 간결한 문법으로 데이터의 크기를 줄입니다.

  • 가독성: 사람이 읽고 쓰기 쉬운 구조를 가집니다.

  • 언어 독립성: 다양한 프로그래밍 언어에서 지원됩니다.

특징#

  • 텍스트 기반: UTF-8 인코딩을 사용하는 텍스트 형식입니다.

  • 구조화된 데이터 표현: 중첩된 객체와 배열을 통해 복잡한 데이터를 표현할 수 있습니다.

핵심 원칙#

  • 단순성: 복잡한 문법을 배제하여 사용이 간편합니다.

  • 일관성: 명확한 문법 규칙으로 파싱이 용이합니다.(Medium)

주요 원리 및 작동 원리#

  • 파싱 및 직렬화: JSON 데이터를 객체로 변환하거나, 객체를 JSON 문자열로 변환합니다.

  • 데이터 전송: HTTP 프로토콜을 통해 JSON 데이터를 전송합니다.

구조 및 아키텍처#

  • 객체(Object): 중괄호({})로 감싸며, 키-값 쌍의 집합입니다.

  • 배열(Array): 대괄호([])로 감싸며, 값들의 순서 있는 리스트입니다.

구성 요소#

  • 키(Key): 문자열로 표현되며, 값에 대한 식별자 역할을 합니다.

  • 값(Value): 문자열, 숫자, 불리언, 배열, 객체, null 중 하나입니다.(Sonra)

장점과 단점#

구분항목설명

JSON vs XML구조 비교JSON은 간결하고 빠르며 JavaScript와의 호환성이 높고, XML은 문서 지향적이며 스키마 검증이 강력합니다.

JSON과 Protobuf직렬화 포맷 비교JSON은 사람이 읽기 쉬우나 용량이 크고 느린 반면, Protocol Buffers (Protobuf)는 바이너리 형식으로 훨씬 효율적입니다.
JSON과 BSONMongoDB 포맷 비교BSON은 JSON을 확장한 이진 포맷으로, MongoDB에서 내부적으로 사용되며 성능과 표현력이 개선됨.
보안JSON 인젝션 위험클라이언트 입력을 그대로 JSON으로 직렬화할 경우, XSS 또는 서버 취약점을 유발할 수 있어 반드시 필터링 및 검증 필요
데이터 전송 최적화압축 및 캐싱Gzip 압축, HTTP 캐싱 헤더와 함께 JSON을 사용할 경우 전송 속도를 대폭 향상시킬 수 있음
JSON과 NoSQL비정형 데이터 저장MongoDB 등 NoSQL 데이터베이스는 JSON 유사 문서 구조를 사용하여 유연한 데이터 모델링 가능

  "title": "사용자",
  "type": "object",
  "properties": {
    "id": {
      "type": "integer"
    },
    "name": {
      "type": "string",
      "minLength": 1
    },
    "email": {
      "type": "string",
      "format": "email"
    },
    "isActive": {
      "type": "boolean"
    }
  },
  "required": ["id", "name", "email"]
}

구성 요소 설명#

키설명
$schema사용 중인 JSON Schema 버전 명시
type데이터 타입 (object, array, string 등)
properties필드 정의
required필수 필드 지정
format이메일, URI, 날짜 등 형식 검사 지원

from jsonschema import validate, ValidationError

schema = {
    "type": "object",
    "properties": {
        "username": {"type": "string"},
        "age": {"type": "integer", "minimum": 0}
    },
    "required": ["username"]
}

data = {"username": "john", "age": 30}

try:
    validate(instance=data, schema=schema)
    print("유효한 데이터입니다.")
except ValidationError as e:
    print("검증 오류:", e.message)

const Ajv = require("ajv");
const ajv = new Ajv();

const schema = {
  type: "object",
  properties: {
    title: { type: "string" },
    year: { type: "integer", minimum: 2000 }
  },
  required: ["title"]
};

const validate = ajv.compile(schema);
const valid = validate({ title: "영화 제목", year: 2023 });

if (valid) {
  console.log("데이터 유효함");
} else {
  console.error(validate.errors);
}

기능예시설명

🧠 실무 적용 시 주의 사항#

고려사항설명
Schema 버전 일치$schema에 지정한 Draft 버전과 도구의 지원 버전을 일치시킬 것
기본값 처리JSON Schema 자체는 default 값을 자동으로 적용하지 않음 (도구 별 지원 여부 다름)
스키마 복잡도조건부 구조(if/then/else 등)는 간결하게 작성할 것
성능대형 Schema 검증 시 비용이 발생할 수 있어 병렬 처리 또는 사전 검증 고려
