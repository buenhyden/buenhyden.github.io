Short Polling#

Short polling은 클라이언트와 서버 간의 실시간에 가까운 통신을 구현하기 위한 기본적인 기술이다.

Short polling은 실시간 업데이트가 필요하지만 진정한 실시간성이 중요하지 않은 애플리케이션에서 구현이 간단하고 호환성이 좋은 솔루션이다. 그러나 사용자가 많아지거나 지연 시간이 중요한 애플리케이션에서는 Long Polling, SSE, WebSockets 같은 더 효율적인 기술의 사용을 고려해야 한다.

Short Polling의 개념#

Short polling은 클라이언트가 주기적으로 서버에 HTTP 요청을 보내 새로운 데이터가 있는지 확인하는 방식이다. 클라이언트는 정해진 시간 간격으로 서버에 요청을 보내고, 서버는 그 순간 가지고 있는, 클라이언트가 아직 받지 않은 데이터를 응답한다.

작동 메커니즘#

  1. 클라이언트가 서버에 HTTP 요청을 보낸다.
  2. 서버는 즉시 현재 상태를 응답한다.
  3. 클라이언트는 응답을 처리한 후, 일정 시간(보통 몇 초)을 기다린다.
  4. 일정 시간이 지난 후, 클라이언트는 다시 요청을 보낸다.
  5. 이 과정이 계속 반복된다.

// Short Polling 구현 예시
function startShortPolling() {
    const pollingInterval = 3000; // 3초마다 서버에 요청
    
    function pollServer() {
        fetch('https://api.example.com/updates')
            .then(response => response.json())
            .then(data => {
                // 데이터 처리 로직
                processData(data);
                
                // 다음 요청을 위한 타이머 설정
                setTimeout(pollServer, pollingInterval);
            })
            .catch(error => {
                console.error('폴링 중 오류 발생:', error);
                // 오류 발생 시에도 다음 요청을 예약
                setTimeout(pollServer, pollingInterval);
            });
    }
    
    // 첫 번째 폴링 시작
    pollServer();
}

function processData(data) {
    // 서버로부터 받은 데이터 처리
    if (data.hasNewMessages) {
        // 새 메시지 표시 로직
        displayNewMessages(data.messages);
    }
}

// 폴링 시작
startShortPolling();

// Express.js를 사용한 서버 측 구현
const express = require('express');
const app = express();

let messages = []; // 메시지 저장소
let lastMessageId = 0; // 마지막 메시지 ID 추적

app.get('/updates', (req, res) => {
    // 클라이언트가 마지막으로 받은 메시지 ID
    const lastSeenId = parseInt(req.query.lastId) || 0;
    
    // 클라이언트가 보지 않은 새 메시지 필터링
    const newMessages = messages.filter(msg => msg.id > lastSeenId);
    
    // 응답 전송
    res.json({
        hasNewMessages: newMessages.length > 0,
        messages: newMessages
    });
});

// 새 메시지 추가 엔드포인트
app.post('/message', express.json(), (req, res) => {
    const newMessage = {
        id: ++lastMessageId,
        text: req.body.text,
        timestamp: new Date()
    };
    
    messages.push(newMessage);
    res.status(201).json(newMessage);
});

app.listen(3000, () => {
    console.log('서버가 3000번 포트에서 실행 중입니다.');
});

Short Polling의 장단점#

장점#

  1. 구현 용이성: 기존 HTTP 요청-응답 모델을 사용하므로 구현이 매우 간단하다.
  2. 넓은 호환성: 거의 모든 브라우저와 서버에서 지원된다.
  3. 확장성: 서버 리소스를 효율적으로 사용할 수 있다 (요청이 왔을 때만 응답).
  4. 방화벽 친화적: 표준 HTTP 통신을 사용하므로 대부분의 방화벽에서 문제없이 작동한다.

단점#

  1. 지연 발생: 폴링 간격만큼의 지연이 필연적으로 발생한다.
  2. 불필요한 요청: 새 데이터가 없을 때도 정기적으로 요청을 보내므로 리소스가 낭비된다.
  3. 서버 부하: 사용자가 많아질수록 서버가 처리해야 할 요청 수가 크게 증가한다.
  4. 불필요한 대역폭 사용: 변경사항이 없는 경우에도 HTTP 요청과 응답에 데이터를 소비한다.

최적화 전략#

  1. 적절한 폴링 간격 설정: 애플리케이션의 요구사항에 맞게 폴링 간격을 조정한다.
  2. 응답 크기 최소화: 필요한 데이터만 응답하여 대역폭 사용을 줄인다.
  3. 조건부 요청: ETag나 Last-Modified와 같은 HTTP 헤더를 사용하여 실제 변경사항이 있을 때만 전체 응답을 받는다.
  4. 지수 백오프: 일정 시간 동안 변경사항이 없으면 점진적으로 폴링 간격을 늘린다.

Short Polling vs. Long Polling#

Short polling의 단점을 보완하기 위해 등장한 것이 long polling이다. Long polling에서는 서버가 새 데이터가 생길 때까지 응답을 보류하다가 데이터가 생기면 응답한다. 이는 지연 시간을 줄이고 불필요한 요청을 감소시키지만, 서버 측에서 연결을 더 오래 유지해야 하는 부담이 있다.
