Domain-Driven Design#

아래는 “Domain-Driven Design(도메인 주도 설계, DDD)” 에 대한 체계적이고 심층적인 조사 결과입니다.

Domain-Driven-Design, Software-Engineering, System-Architecture, Microservices

분류 기준인 **“Computer Science and Engineering” > “Software Engineering” > “Design and Architecture” > “Detailed design” > “Design Methodologies”**는 DDD 의 위치를 명확히 반영합니다.

3. 요약 문장#

Domain-Driven Design(DDD) 은 비즈니스 도메인에 집중하여, 도메인 모델을 중심으로 소프트웨어를 설계하는 방법론으로, 복잡한 비즈니스 로직을 효과적으로 관리하고, 개발자와 도메인 전문가의 협업을 강화하여 시스템의 유지보수성과 확장성을 높입니다 13.

4. 개요#

Domain-Driven Design(DDD) 은 비즈니스 도메인과 그 복잡성을 이해하고, 도메인 모델을 중심으로 소프트웨어를 설계하는 방법론입니다. DDD 는 유비쿼터스 언어 (Ubiquitous Language), 바운디드 컨텍스트 (Bounded Context), 도메인 모델, 애그리게이트 등 핵심 개념을 활용하여, 비즈니스 요구사항을 정확히 반영하는 시스템을 구축합니다. 실무에서는 대규모 엔터프라이즈 시스템, 마이크로서비스 등에서 널리 활용되고 있습니다 14.

5. 핵심 개념#

  • 도메인 (Domain): 소프트웨어가 해결하려는 비즈니스 문제 영역
  • 도메인 모델 (Domain Model): 도메인의 핵심 개념과 규칙을 객체로 표현한 모델
  • 유비쿼터스 언어 (Ubiquitous Language): 도메인 전문가와 개발자가 공통으로 사용하는 언어
  • 바운디드 컨텍스트 (Bounded Context): 도메인 모델이 적용되는 명확한 경계
  • 애그리게이트 (Aggregate): 연관된 엔티티와 값 객체를 하나의 논리적 단위로 묶음
  • 엔티티 (Entity): 고유 식별자를 가진 도메인 객체
  • 값 객체 (Value Object): 식별자가 없는 불변 객체
  • 도메인 서비스 (Domain Service): 여러 엔티티와 값 객체를 활용하는 비즈니스 로직
  • 리포지토리 (Repository): 도메인 객체의 영속성 관리
  • 도메인 이벤트 (Domain Event): 도메인 내에서 발생하는 중요한 상태 변화 15

실무 구현 요소

  • 도메인 모델링: 도메인 전문가와 협업하여 도메인 모델 정의
  • 바운디드 컨텍스트 설계: 도메인별 경계 설정 및 컨텍스트 매핑
  • 애그리게이트 및 엔티티 설계: 비즈니스 규칙 중심의 객체 설계
  • 유비쿼터스 언어 적용: 코드, 문서, 커뮤니케이션에 일관된 용어 사용
  • 도메인 서비스 및 리포지토리 구현: 비즈니스 로직과 데이터 접근 분리 14

배경#

DDD 는 복잡한 비즈니스 로직을 가진 대규모 시스템에서, 개발자와 도메인 전문가 간의 언어 장벽을 해소하고, 비즈니스 요구사항을 정확히 반영하기 위해 등장했습니다. 기존의 데이터 중심 설계에서 벗어나, 도메인 중심의 설계를 강조합니다 63.

class Member:
    def __init__(self, name):
        self.name = name

    def create_order(self, product):
        return Order(self, product)

class Order:
    def __init__(self, member, product):
        self.member = member
        self.product = product
        self.status = "CREATED"

    def process(self):
        self.status = "PROCESSED"
        return Delivery(self)

class Delivery:
    def __init__(self, order):
        self.order = order
        self.status = "READY"

    def process(self):
        self.status = "DELIVERED"

3. 200 자 요약#

도메인 주도 설계 (DDD) 는 도메인 전문가와 협업하여 핵심 비즈니스 도메인을 모델링하고, Bounded Context와 Ubiquitous Language를 통해 도메인 중심의 코드 구조와 아키텍처를 구축하는 방법론입니다. 복잡한 비즈니스 로직이 명확해지며 마이크로서비스, CQRS, 이벤트 기반 설계 등과 자연스럽게 결합되어 높은 유지보수성과 확장성을 제공합니다.(geeksforgeeks.org, medium.com, en.wikipedia.org)

4. 250 자 개요#

DDD 는 에릭 에반스 (Eric Evans) 가 2003 년 제안한 방법론으로, 도메인을 단위로 분할하고 각 컨텍스트 내에서 Ubiquitous Language를 사용하는 것을 핵심으로 합니다. 전략적 설계 (Strategic Design) 에서는 Bounded Context, Subdomain, Context Mapping을 통해 복잡한 시스템을 분리·구조화하고, 전술적 설계 (Tactical Design) 에서는 Entity, Value Object, Aggregate, Repository, Factory, Domain Event 등 도메인 모델을 실제 코드에 구현합니다. 이는 복잡 도메인에서 협업성, 명시성, 유지보수성, 확장성 등을 강화하며, CQRS 와 이벤트 기반 아키텍처와 연계되어 현대 마이크로서비스 구현에 적합합니다.(redis.io)

5. 핵심 개념#

(1) 전략적 설계 (Strategic Design)#

  • Ubiquitous Language: 개발자와 도메인 전문가 간 일관된 언어 및 코드/문서 반영 (en.wikipedia.org)
  • Bounded Context: 모델의 일관성과 개념 구분 영역 (geeksforgeeks.org)
  • Context Mapping: 컨텍스트 간 관계 (파트너, 고객 - 공급자, 공유 커널) 맵핑 (geeksforgeeks.org)
  • Subdomain: 핵심 (Core), 지원 (Supporting), 일반 (General) 등 기능별 분류

(2) 전술적 설계 (Tactical Design)#

  • Entity: 고유 식별자 (ID) 를 가지며 상태 변화를 추적 (arxiv.org)
  • Value Object: 불변 (immutable) 이며 비식별 속성 중심 (redis.io)
  • Aggregate / Aggregate Root: 엔티티 (Value Object 포함) 의 일관된 집계 단위 (redis.io)
  • Domain Event: 도메인 내 의미 있는 상태 변화 이벤트 (medium.com)
  • Repository: 도메인 객체 지속성 인터페이스 · 계층 분리 (bytebytego.com)
  • Factory: 복잡한 도메인 객체 생성 책임 분리
  • Service: 엔티티/VO 에 포함되지 않는 도메인 로직 처리

필수 구성 요소#

  1. Bounded Context

    • 독립적으로 의미가 통용되는 도메인 경계
    • 내부에서만 정확한 모델과 언어 사용

  2. Subdomain (Core, Supporting, Generic)

    • 비즈니스 가치나 역할에 따라 분류된 세부 도메인

  3. Context Map

    • 컨텍스트 간 관계 정의 (Shared Kernel, Customer‑Supplier 등)

  4. Domain Layer

    • Entity, Value Object, Aggregate, Domain Service, Domain Event

  5. Application Layer

    • Use Case 구현, 트랜잭션 관리, 명령/쿼리 흐름 조정

  6. Infrastructure Layer

    • 구현 기술 집약 (DB, 메시징, 외부 API 등)

  7. Interface Layer (Presentation / API)

    • 웹, API, UI 계층. 외부 요구와 계약함

선택 구성 요소#

  • Factory: 복잡 객체 생성 책임
  • Repository: Aggregate 저장소
  • Specification: 도메인 규칙 표현
  • Anti‑Corruption Layer (ACL): 외부 시스템과 격리
  • Event Sourcing / CQRS: 이벤트 기반 이력 모델, 읽기/쓰기 모델 분리

# domain/models.py
from dataclasses import dataclass
from datetime import datetime
import uuid

@dataclass(frozen=True)
class OrderItem:
    product_id: str
    quantity: int

@dataclass
class Order:
    id: str
    items: list[OrderItem]
    status: str
    created_at: datetime

    @classmethod
    def create(cls, items: list[OrderItem]):
        if not items:
            raise ValueError("주문 항목 필요")
        return cls(id=str(uuid.uuid4()), items=items, status="CREATED", created_at=datetime.utcnow())

    def place(self):
        if self.status != "CREATED":
            raise ValueError("잘못된 상태")
        self.status = "PLACED"
        # Domain Event 발행 (단순 로그)
        print("OrderPlaced:", self.id)

# application/services.py
class OrderService:
    def __init__(self, repo):
        self.repo = repo

    def create_and_place(self, items):
        order = Order.create(items)
        self.repo.save(order)
        order.place()
        self.repo.save(order)
        return order

# infrastructure/repository.py
class InMemoryOrderRepo:
    def __init__(self):
        self.store = {}

    def save(self, order):
        self.store[order.id] = order

    def get(self, order_id):
        return self.store.get(order_id)

# 실행 예시
if __name__ == "__main__":
    repo = InMemoryOrderRepo()
    svc = OrderService(repo)
    order = svc.create_and_place([OrderItem("prod1", 2)])
    print(order)

    • Entity(Order), Value Object(OrderItem)
    • Factory (create 클래스 메서드)
    • Domain Event 생성 (place)
    • Repository 추상화 (InMemoryOrderRepo)
    • Application Service(OrderService) 가 업무 동작 총괄

3. 요약문 (200 자 내외)#

Domain-Driven Design (DDD, 도메인 주도 설계) 는 복잡한 비즈니스 도메인을 중심으로 소프트웨어를 설계하는 방법론으로, 도메인 전문가와 개발자가 유비쿼터스 언어 (Ubiquitous Language) 를 통해 협력하여 도메인 모델을 구축하고, 바운디드 컨텍스트 (Bounded Context) 로 복잡성을 관리하며, 전략적 설계와 전술적 설계를 통해 유지보수 가능한 소프트웨어를 개발하는 접근법입니다.

4. 전체 내용을 포괄하는 개요 (250 자 내외)#

DDD 는 Eric Evans 가 2003 년 제시한 소프트웨어 설계 방법론으로, 비즈니스 도메인의 복잡성을 효과적으로 관리하기 위해 개발되었습니다. 전략적 설계를 통해 도메인을 바운디드 컨텍스트로 분할하고, 전술적 설계를 통해 엔티티 (Entity), 애그리거트 (Aggregate), 도메인 서비스 등의 구체적 구현을 다룹니다. 현재 마이크로서비스 아키텍처, CQRS (Command Query Responsibility Segregation), 이벤트 소싱 (Event Sourcing) 과 함께 사용되어 현대적인 분산 시스템 구축에 핵심적으로 활용되고 있습니다.

  2. 도메인 모델 (Domain Model): 도메인을 개념적으로 표현한 추상화 모델
  3. 유비쿼터스 언어 (Ubiquitous Language): 도메인 전문가와 개발자가 공통으로 사용하는 언어
  4. 바운디드 컨텍스트 (Bounded Context): 도메인 모델이 일관되게 적용되는 제한된 영역
  5. 애그리거트 (Aggregate): 일관성과 트랜잭션의 경계를 정의하는 연관된 객체들의 집합
  6. 엔티티 (Entity): 고유 식별자를 가진 변경 가능한 도메인 객체
  7. 밸류 오브젝트 (Value Object): 불변의 값을 표현하는 객체
  8. 도메인 서비스 (Domain Service): 특정 엔티티에 속하지 않는 도메인 로직을 담당
  9. 리포지토리 (Repository): 도메인 객체의 영속성을 추상화하는 패턴
  10. 도메인 이벤트 (Domain Event): 도메인에서 발생하는 중요한 비즈니스 사건

5.2 실무 구현 요소#

  1. 이벤트 스토밍 (Event Storming): 도메인 이벤트를 중심으로 바운디드 컨텍스트를 식별하는 워크샵 기법
  2. 컨텍스트 매핑 (Context Mapping): 바운디드 컨텍스트 간의 관계와 통합 방식을 정의
  3. 레이어드 아키텍처 (Layered Architecture): 관심사 분리를 위한 계층형 구조
  4. 헥사고날 아키텍처 (Hexagonal Architecture): 포트와 어댑터 패턴을 활용한 의존성 격리
  5. CQRS 패턴: 명령과 조회의 책임을 분리하여 성능과 확장성 향상
  6. 이벤트 소싱: 상태 변경을 이벤트로 저장하여 감사 추적과 시간 여행 가능
  7. 의존성 역전 원칙 (DIP): 고수준 모듈이 저수준 모듈에 의존하지 않도록 하는 설계 원칙

제 1 부: 기본 개념과 배경#

배경#

DDD 는 기존의 데이터 중심 설계 (Data-Driven Design) 의 한계를 극복하기 위해 등장했습니다. 전통적인 소프트웨어 개발에서는 다음과 같은 문제점들이 있었습니다:

  • 비즈니스 도메인에 대한 이해 부족: 기술적 구현에만 집중하여 실제 비즈니스 요구사항과 괴리
  • 도메인 전문가와 개발자 간 소통 단절: 서로 다른 언어와 관점으로 인한 오해
  • 복잡성 관리 실패: 시스템이 커질수록 유지보수가 어려워지는 구조
  • 변화에 대한 취약성: 비즈니스 변화에 유연하게 대응하지 못하는 경직된 설계

목적 및 필요성#

주요 목적#

  1. 비즈니스 중심의 소프트웨어 설계: 기술보다 도메인을 우선시
  2. 복잡성 관리: 큰 시스템을 관리 가능한 작은 단위로 분할
  3. 의사소통 개선: 모든 이해관계자가 동일한 언어 사용
  4. 변화에 대한 적응력: 비즈니스 변화에 유연하게 대응

필요성#

  • 디지털 전환 가속화: 빠르게 변화하는 비즈니스 환경
  • 시스템 복잡도 증가: 마이크로서비스, 분산 시스템의 확산
  • 도메인 전문성 중요성: 비즈니스 경쟁력의 핵심이 되는 도메인 지식

주요 기능 및 역할#

전략적 설계 (Strategic Design)#

  • 도메인 분할: 큰 도메인을 하위 도메인으로 나누어 관리
  • 바운디드 컨텍스트 식별: 모델의 일관성이 유지되는 경계 정의
  • 컨텍스트 매핑: 바운디드 컨텍스트 간의 관계와 통합 방식 설계

전술적 설계 (Tactical Design)#

  • 도메인 모델링: 엔티티, 밸류 오브젝트, 애그리거트 설계
  • 비즈니스 로직 구현: 도메인 서비스와 도메인 이벤트 활용
  • 아키텍처 패턴 적용: 레이어드, 헥사고날 아키텍처 구현

특징#

핵심 특징#

  1. 도메인 중심성: 비즈니스 도메인이 설계의 중심
  2. 협업 지향: 도메인 전문가와 개발자의 긴밀한 협력
  3. 진화적 설계: 지속적인 학습과 개선을 통한 모델 발전
  4. 명시적 모델링: 암묵적 지식을 명시적 코드로 표현

구별되는 특징#

  • 유비쿼터스 언어: 모든 의사소통에서 동일한 용어 사용
  • 바운디드 컨텍스트: 모델의 적용 범위를 명확히 제한
  • 애그리거트: 일관성 경계를 명확히 정의
  • 이벤트 중심 사고: 비즈니스 이벤트를 중심으로 모델링

핵심 원칙#

DDD 의 3 가지 핵심 원칙#

  1. Focus on the Core Domain (핵심 도메인에 집중)

    • 비즈니스 경쟁 우위를 제공하는 핵심 영역에 집중
    • 제한된 자원을 가장 중요한 부분에 투자

  2. Explore Models in Collaboration (협력을 통한 모델 탐구)

    • 도메인 전문가와 개발자의 지속적 협력
    • 공동 모델링을 통한 깊은 도메인 이해

  3. Speak Ubiquitous Language (유비쿼터스 언어 사용)

    • 모든 이해관계자가 동일한 언어 사용
    • 코드와 대화에서 일관된 용어 유지

설계 원칙#

  1. Model-Driven Design: 모델과 구현의 긴밀한 연결
  2. Hands-On Modelers: 모델러가 직접 코드 작성 참여
  3. Refactoring Toward Deeper Insight: 지속적 리팩토링을 통한 깊은 통찰

각 계층의 구성요소와 역할#
1. 프레젠테이션 계층 (Presentation Layer)#

구성요소:

  • 컨트롤러 (Controller): HTTP 요청/응답 처리
  • 뷰 (View): 사용자 인터페이스
  • DTO (Data Transfer Object): 계층 간 데이터 전송

기능과 역할:

  • 사용자 요청 해석 및 응답 생성
  • 입력 데이터 유효성 검증
  • 응용 계층과의 인터페이스 제공
2. 응용 계층 (Application Layer)#

구성요소:

  • 응용 서비스 (Application Service): 사용 사례 조정
  • 명령 핸들러 (Command Handler): 명령 처리
  • 쿼리 핸들러 (Query Handler): 조회 처리

기능과 역할:

  • 비즈니스 워크플로 조정
  • 트랜잭션 경계 관리
  • 도메인 객체 간 협력 조정
3. 도메인 계층 (Domain Layer) - 핵심 계층#

필수 구성요소:

  • 엔티티 (Entity): 고유 식별자를 가진 비즈니스 객체
  • 밸류 오브젝트 (Value Object): 불변의 값 표현
  • 애그리거트 (Aggregate): 일관성 경계 정의
  • 도메인 서비스 (Domain Service): 엔티티에 속하지 않는 비즈니스 로직
  • 리포지토리 인터페이스 (Repository Interface): 영속성 추상화

선택 구성요소:

  • 도메인 이벤트 (Domain Event): 비즈니스 사건 표현
  • 팩토리 (Factory): 복잡한 객체 생성 로직
  • 명세 (Specification): 비즈니스 규칙 표현

기능과 역할:

  • 핵심 비즈니스 로직 구현
  • 비즈니스 규칙과 제약사항 적용
  • 도메인 지식의 명시적 표현
4. 인프라스트럭처 계층 (Infrastructure Layer)#

구성요소:

  • 리포지토리 구현체 (Repository Implementation): 데이터 영속성 구현
  • 외부 서비스 어댑터 (External Service Adapter): 외부 API 연동
  • 메시징 시스템: 이벤트 발행/구독

기능과 역할:

  • 기술적 세부사항 구현
  • 외부 시스템과의 통신
  • 데이터 저장 및 조회

구성 요소#

전술적 설계의 주요 구성요소#

1. 엔티티 (Entity)#
  • 정의: 연속성과 고유 식별자를 가진 객체
  • 특징: 생명주기 동안 상태 변경 가능, 동등성은 식별자로 판단
  • 예시: User, Order, Product
2. 밸류 오브젝트 (Value Object)#
  • 정의: 값으로만 식별되는 불변 객체
  • 특징: 동등성은 모든 속성으로 판단, 불변성 보장
  • 예시: Money, Address, Email
3. 애그리거트 (Aggregate)#
  • 정의: 일관성과 트랜잭션 경계를 정의하는 객체 집합
  • 특징: 애그리거트 루트를 통해서만 외부 접근 허용
  • 예시: Order 애그리거트 (Order + OrderItem)
4. 도메인 서비스 (Domain Service)#
  • 정의: 특정 엔티티에 속하지 않는 도메인 로직
  • 특징: 상태를 갖지 않음, 순수한 비즈니스 로직
  • 예시: PriceCalculationService, TransferService
5. 리포지토리 (Repository)#
  • 정의: 도메인 객체의 컬렉션을 시뮬레이션하는 패턴
  • 특징: 영속성 기술의 추상화, 도메인 계층에 인터페이스 위치
  • 예시: OrderRepository, UserRepository

구현 기법#

1. 이벤트 스토밍 (Event Storming)#

목적: 도메인 이벤트를 중심으로 비즈니스 프로세스 이해 및 바운디드 컨텍스트 식별

구성 요소:

  • 도메인 이벤트 (주황색): " 주문이 생성됨 “, " 결제가 완료됨 "
  • 커맨드 (파란색): " 주문 생성 “, " 결제 처리 "
  • 액터 (노란색): " 고객 “, " 관리자 "
  • 애그리거트 (노란색): " 주문 “, " 상품 "
  • 바운디드 컨텍스트 (점선): 관련 요소들의 그룹화

class Account:
    def __init__(self, account_id):
        self.id = account_id
        self.balance = 0
        self.version = 0
        
    def deposit(self, amount):
        event = MoneyDeposited(self.id, amount, datetime.now())
        self.apply(event)
        return event
        
    def apply(self, event):
        if isinstance(event, MoneyDeposited):
            self.balance += event.amount
        elif isinstance(event, MoneyWithdrawn):
            self.balance -= event.amount
        self.version += 1
        
    @classmethod
    def from_events(cls, account_id, events):
        account = cls(account_id)
        for event in events:
            account.apply(event)
        return account

장점#

# 밸류 오브젝트 (Value Objects)
@dataclass(frozen=True)  # 불변성 보장
class Money:
    """금액을 표현하는 밸류 오브젝트"""
    amount: Decimal
    currency: str = "KRW"
    
    def add(self, other: 'Money') -> 'Money':
        if self.currency != other.currency:
            raise ValueError("다른 통화는 더할 수 없습니다")
        return Money(self.amount + other.amount, self.currency)
    
    def multiply(self, multiplier: int) -> 'Money':
        return Money(self.amount * multiplier, self.currency)

@dataclass(frozen=True)
class CustomerId:
    """고객 ID를 표현하는 밸류 오브젝트"""
    value: UUID
    
    @classmethod
    def generate(cls) -> 'CustomerId':
        return cls(uuid4())

@dataclass(frozen=True)
class Address:
    """배송 주소를 표현하는 밸류 오브젝트"""
    street: str
    city: str
    postal_code: str
    country: str = "KR"
    
    def __post_init__(self):
        if not self.street or not self.city or not self.postal_code:
            raise ValueError("주소 정보가 불완전합니다")

# 도메인 이벤트 (Domain Events)
@dataclass
class DomainEvent:
    """도메인 이벤트 기본 클래스"""
    event_id: UUID
    occurred_at: datetime
    aggregate_id: UUID

@dataclass
class OrderCreated(DomainEvent):
    """주문 생성 이벤트"""
    customer_id: UUID
    total_amount: Money

@dataclass
class OrderItemAdded(DomainEvent):
    """주문 항목 추가 이벤트"""
    product_id: UUID
    quantity: int
    price: Money

# 열거형 (Enums)
class OrderStatus(Enum):
    DRAFT = "DRAFT"
    CONFIRMED = "CONFIRMED"
    PAID = "PAID"
    SHIPPED = "SHIPPED"
    DELIVERED = "DELIVERED"
    CANCELLED = "CANCELLED"

# 엔티티 (Entities)
class OrderItem:
    """주문 항목 엔티티"""
    def __init__(self, product_id: UUID, quantity: int, unit_price: Money):
        if quantity <= 0:
            raise ValueError("수량은 0보다 커야 합니다")
        if unit_price.amount <= 0:
            raise ValueError("가격은 0보다 커야 합니다")
            
        self.product_id = product_id
        self.quantity = quantity
        self.unit_price = unit_price
    
    def calculate_total(self) -> Money:
        """항목별 총액 계산"""
        return self.unit_price.multiply(self.quantity)
    
    def change_quantity(self, new_quantity: int):
        """수량 변경"""
        if new_quantity <= 0:
            raise ValueError("수량은 0보다 커야 합니다")
        self.quantity = new_quantity

class Order:
    """주문 애그리거트 루트"""
    def __init__(self, customer_id: CustomerId, shipping_address: Address):
        self.id = uuid4()
        self.customer_id = customer_id
        self.shipping_address = shipping_address
        self.status = OrderStatus.DRAFT
        self.created_at = datetime.now()
        self.items: List[OrderItem] = []
        self._domain_events: List[DomainEvent] = []
    
    def add_item(self, product_id: UUID, quantity: int, unit_price: Money):
        """주문 항목 추가"""
        if self.status != OrderStatus.DRAFT:
            raise ValueError("확정된 주문은 수정할 수 없습니다")
        
        # 기존 항목 확인
        existing_item = self._find_item(product_id)
        if existing_item:
            existing_item.change_quantity(existing_item.quantity + quantity)
        else:
            item = OrderItem(product_id, quantity, unit_price)
            self.items.append(item)
        
        # 도메인 이벤트 추가
        self._add_domain_event(OrderItemAdded(
            event_id=uuid4(),
            occurred_at=datetime.now(),
            aggregate_id=self.id,
            product_id=product_id,
            quantity=quantity,
            price=unit_price
        ))
    
    def remove_item(self, product_id: UUID):
        """주문 항목 제거"""
        if self.status != OrderStatus.DRAFT:
            raise ValueError("확정된 주문은 수정할 수 없습니다")
        
        self.items = [item for item in self.items if item.product_id != product_id]
    
    def calculate_total(self) -> Money:
        """주문 총액 계산"""
        if not self.items:
            return Money(Decimal('0'))
        
        total = self.items[0].calculate_total()
        for item in self.items[1:]:
            total = total.add(item.calculate_total())
        return total
    
    def confirm_order(self):
        """주문 확정"""
        if self.status != OrderStatus.DRAFT:
            raise ValueError("이미 확정된 주문입니다")
        if not self.items:
            raise ValueError("주문 항목이 없습니다")
        
        self.status = OrderStatus.CONFIRMED
        
        # 주문 생성 이벤트 발행
        self._add_domain_event(OrderCreated(
            event_id=uuid4(),
            occurred_at=datetime.now(),
            aggregate_id=self.id,
            customer_id=self.customer_id.value,
            total_amount=self.calculate_total()
        ))
    
    def ship_order(self):
        """주문 배송"""
        if self.status != OrderStatus.PAID:
            raise ValueError("결제된 주문만 배송 가능합니다")
        self.status = OrderStatus.SHIPPED
    
    def _find_item(self, product_id: UUID) -> Optional[OrderItem]:
        """상품 ID로 주문 항목 찾기"""
        return next((item for item in self.items if item.product_id == product_id), None)
    
    def _add_domain_event(self, event: DomainEvent):
        """도메인 이벤트 추가"""
        self._domain_events.append(event)
    
    def get_domain_events(self) -> List[DomainEvent]:
        """도메인 이벤트 조회"""
        return self._domain_events.copy()
    
    def clear_domain_events(self):
        """도메인 이벤트 초기화"""
        self._domain_events.clear()

# 도메인 서비스 (Domain Services)
class OrderDomainService:
    """주문 도메인 서비스"""
    
    @staticmethod
    def can_apply_discount(order: Order, discount_percentage: Decimal) -> bool:
        """할인 적용 가능 여부 판단"""
        return (order.status == OrderStatus.DRAFT and 
                order.calculate_total().amount >= Decimal('100000') and
                discount_percentage <= Decimal('0.3'))
    
    @staticmethod
    def calculate_shipping_fee(order: Order) -> Money:
        """배송료 계산"""
        total = order.calculate_total()
        if total.amount >= Decimal('50000'):
            return Money(Decimal('0'))  # 무료배송
        return Money(Decimal('3000'))   # 기본 배송료

# 리포지토리 인터페이스 (Repository Interface)
class OrderRepository(ABC):
    """주문 리포지토리 인터페이스"""
    
    @abstractmethod
    def save(self, order: Order) -> None:
        """주문 저장"""
        pass
    
    @abstractmethod
    def find_by_id(self, order_id: UUID) -> Optional[Order]:
        """ID로 주문 조회"""
        pass
    
    @abstractmethod
    def find_by_customer_id(self, customer_id: CustomerId) -> List[Order]:
        """고객 ID로 주문 목록 조회"""
        pass

@dataclass
class CreateOrderCommand:
    """주문 생성 커맨드"""
    customer_id: UUID
    shipping_address: Address
    items: List[dict]  # [{"product_id": UUID, "quantity": int, "unit_price": Decimal}]

@dataclass
class AddOrderItemCommand:
    """주문 항목 추가 커맨드"""
    order_id: UUID
    product_id: UUID
    quantity: int
    unit_price: Decimal

class OrderApplicationService:
    """주문 응용 서비스"""
    
    def __init__(self, order_repository: OrderRepository):
        self.order_repository = order_repository
    
    def create_order(self, command: CreateOrderCommand) -> UUID:
        """주문 생성"""
        customer_id = CustomerId(command.customer_id)
        order = Order(customer_id, command.shipping_address)
        
        # 주문 항목 추가
        for item_data in command.items:
            order.add_item(
                product_id=item_data["product_id"],
                quantity=item_data["quantity"],
                unit_price=Money(Decimal(str(item_data["unit_price"])))
            )
        
        self.order_repository.save(order)
        return order.id
    
    def add_order_item(self, command: AddOrderItemCommand):
        """주문 항목 추가"""
        order = self.order_repository.find_by_id(command.order_id)
        if not order:
            raise ValueError("주문을 찾을 수 없습니다")
        
        order.add_item(
            command.product_id,
            command.quantity,
            Money(Decimal(str(command.unit_price)))
        )
        
        self.order_repository.save(order)
    
    def confirm_order(self, order_id: UUID):
        """주문 확정"""
        order = self.order_repository.find_by_id(order_id)
        if not order:
            raise ValueError("주문을 찾을 수 없습니다")
        
        order.confirm_order()
        self.order_repository.save(order)
        
        # 이벤트 처리 (실제로는 이벤트 버스로 발행)
        events = order.get_domain_events()
        for event in events:
            print(f"이벤트 발행: {event}")
        order.clear_domain_events()

# 3. 인프라스트럭처 계층 (Infrastructure Layer)
# ============================================================================

class InMemoryOrderRepository(OrderRepository):
    """인메모리 주문 리포지토리 구현체"""
    
    def __init__(self):
        self._orders: dict[UUID, Order] = {}
    
    def save(self, order: Order) -> None:
        """주문 저장"""
        self._orders[order.id] = order
    
    def find_by_id(self, order_id: UUID) -> Optional[Order]:
        """ID로 주문 조회"""
        return self._orders.get(order_id)
    
    def find_by_customer_id(self, customer_id: CustomerId) -> List[Order]:
        """고객 ID로 주문 목록 조회"""
        return [order for order in self._orders.values() 
                if order.customer_id == customer_id]

# ============================================================================
# 4. 프레젠테이션 계층 (Presentation Layer)
# ============================================================================

class OrderController:
    """주문 컨트롤러"""
    
    def __init__(self, order_service: OrderApplicationService):
        self.order_service = order_service
    
    def create_order(self, request_data: dict) -> dict:
        """주문 생성 API"""
        try:
            command = CreateOrderCommand(
                customer_id=UUID(request_data["customer_id"]),
                shipping_address=Address(**request_data["shipping_address"]),
                items=request_data["items"]
            )
            
            order_id = self.order_service.create_order(command)
            return {"success": True, "order_id": str(order_id)}
            
        except Exception as e:
            return {"success": False, "error": str(e)}

설계 방법론전략적 설계 (Strategic Design)큰 그림에서 도메인을 이해하고 바운디드 컨텍스트를 식별하는 설계 접근법

2. 200 자 요약#

Domain-Driven Design(DDD) 는 복잡한 비즈니스 도메인을 소프트웨어에 정확히 반영하기 위해 도메인 전문가와 개발자가 협력하여 도메인 모델을 설계하고, 유비쿼터스 언어와 바운디드 컨텍스트, 엔티티, 값 객체, 애그리게이트 등 핵심 개념을 활용하는 소프트웨어 개발 방법론입니다 14.

3. 250 자 개요#

Domain-Driven Design(DDD) 는 소프트웨어 개발에서 도메인 (비즈니스 영역) 의 복잡성을 효과적으로 해결하기 위해 도메인 모델링을 중심에 두는 접근법입니다.
핵심은 도메인 전문가와 개발자가 유비쿼터스 언어를 통해 소통하며, 바운디드 컨텍스트별로 명확한 모델을 구축하고, 엔티티, 값 객체, 애그리게이트, 도메인 서비스, 도메인 이벤트 등 다양한 전술적 패턴을 적용하는 것입니다.
DDD 는 복잡한 비즈니스 로직을 효과적으로 관리하고, 유지보수성과 확장성이 뛰어난 시스템을 구축하는 데 매우 유용합니다 14[20].

4. 핵심 개념#

  • 도메인 (Domain): 소프트웨어가 해결하고자 하는 비즈니스 영역 또는 문제 공간 14.
  • 도메인 모델 (Domain Model): 도메인의 개념, 규칙, 로직을 코드로 표현한 추상화 구조. 엔티티, 값 객체, 애그리게이트 등으로 구성 15.
  • 유비쿼터스 언어 (Ubiquitous Language): 개발자와 도메인 전문가가 공유하는 공통 언어로, 모델, 코드, 문서 등 모든 소통에 사용 14[20].
  • 바운디드 컨텍스트 (Bounded Context): 특정 도메인 모델과 언어가 일관되게 적용되는 경계. 복잡한 시스템을 명확하게 분리하여 관리 14[20].
  • 엔티티 (Entity): 고유 식별자를 가지며, 상태와 행위를 갖는 객체. 예: 주문, 사용자 1.
  • 값 객체 (Value Object): 식별자 없이 속성 값만으로 구분되는 불변 객체. 예: 주소, 금액 1.
  • 애그리게이트 (Aggregate): 연관된 엔티티와 값 객체의 집합. 일관성 경계를 제공하며, 루트 엔티티를 통해서만 접근 1.
  • 도메인 서비스 (Domain Service): 엔티티/값 객체에 속하지 않는 도메인 로직을 담당하는 서비스 6.
  • 도메인 이벤트 (Domain Event): 도메인 내에서 발생한 의미 있는 사건 [5][13].
  • 리포지토리 (Repository): 도메인 객체의 영속성 (저장/조회) 추상화 1.

4. 핵심 개념#

  • 도메인 (Domain): 소프트웨어가 해결하려는 특정 문제 영역이나 비즈니스 컨텍스트를 의미합니다.(this-circle-jeong.tistory.com)

  • 도메인 모델 (Domain Model): 도메인의 개념, 규칙, 데이터를 추상화한 모델로, 비즈니스 로직을 명확하게 표현합니다.(f-lab.kr)

  • 유비쿼터스 언어 (Ubiquitous Language): 도메인 전문가와 개발자가 공유하는 공통 언어로, 도메인 모델의 개념과 용어를 일관되게 사용합니다.(f-lab.kr)

  • 바운디드 컨텍스트 (Bounded Context): 특정 도메인 모델이 적용되는 명확한 경계를 정의하여, 모델의 일관성과 명확성을 유지합니다.(f-lab.kr)

  • 전략적 설계 (Strategic Design): 도메인을 이해하고, 바운디드 컨텍스트와 도메인 간의 관계를 정의하는 설계 단계입니다.(f-lab.kr)

  • 전술적 설계 (Tactical Design): 도메인 모델을 구현하기 위한 구체적인 설계 단계로, 엔티티, 값 객체, 애그리거트, 리포지토리 등의 패턴을 포함합니다.(f-lab.kr)

5.1 배경 및 목적#

DDD 는 복잡한 비즈니스 로직을 효과적으로 반영하기 위해 도메인 중심으로 소프트웨어를 설계하는 방법론입니다. 기존의 데이터 중심 설계 방식에서 벗어나, 도메인 전문가와 개발자가 협업하여 도메인 모델을 구축하고, 유비쿼터스 언어를 통해 의사소통의 효율성을 높입니다.(velog.io, f-lab.kr)

5.2 주요 기능 및 역할#

  • 도메인 모델링: 비즈니스 로직을 도메인 모델로 추상화하여 소프트웨어에 반영합니다.(f-lab.kr)

  • 유비쿼터스 언어 사용: 도메인 전문가와 개발자가 공통의 언어를 사용하여 의사소통의 효율성을 높입니다.

  • 바운디드 컨텍스트 정의: 도메인 모델의 적용 범위를 명확히 하여 모델의 일관성과 명확성을 유지합니다.

11.1 CQRS (Command Query Responsibility Segregation) 와 DDD#

개념 설명:
CQRS 는 " 명령 (Command)” 과 " 조회 (Query)” 를 책임 단위로 분리하는 아키텍처 패턴입니다. DDD 에서 복잡한 도메인의 상태 변경과 조회를 분리함으로써, 도메인 로직은 더 명확해지고 성능 최적화도 가능합니다.

구조 다이어그램 (Mermaid):

graph TD
    UI --> CommandHandler
    UI --> QueryHandler
    CommandHandler --> DomainModel
    DomainModel --> EventStore
    QueryHandler --> ReadModelDB
요소설명
CommandHandler도메인 상태를 변경하는 명령 처리 책임
QueryHandler별도 DB 또는 Projection 을 통해 조회 처리
ReadModelDB조회 전용 데이터 저장소 (예: Redis, View Table 등)
EventStore도메인 변경을 이벤트로 저장 (Event Sourcing 과 함께 사용됨)

11.2 Event Sourcing 과 DDD#

개념 설명:
Event Sourcing 은 도메인 객체의 상태를 변경 이벤트 (event) 들의 이력으로 저장하고, 현재 상태는 이 이벤트들을 순차적으로 리플레이하여 생성하는 방식입니다.

장점:

  • 상태 이력 추적이 가능

  • 복구 및 감사 (Audit) 용이

  • 비동기 확장성 확보

단점 및 주의점:

  • 상태 조회가 복잡해질 수 있음 → Snapshot 도입 고려

  • 이벤트 스키마 변경이 까다로움 → 이벤트 버전 관리 체계 도입 필요

11.3 Clean Architecture 와 DDD 결합#

Clean Architecture 는 계층 구조를 중심으로 책임을 분리하며, 내부 (core) 와 외부 (infrastructure) 를 명확히 구분합니다. DDD 의 도메인 계층은 가장 중심 (Core Domain Layer) 에 위치합니다.

Domain-Driven Design (DDD) 종합 분석#

1. 주제 분류 적절성 분석#

주제 분류 적절성: 제시된 분류 “Computer Science and Engineering” > “System and Software Architecture” > “Modeling” > “Domain Modeling” 은 적절합니다. DDD 는 본질적으로 비즈니스 도메인을 중심으로 한 소프트웨어 모델링 접근법이므로 Domain Modeling 분류가 핵심을 정확히 나타냅니다.

2. 주제 요약 (200 자 내외)#

Domain-Driven Design(DDD) 은 비즈니스 도메인을 중심으로 복잡한 소프트웨어를 설계하는 방법론입니다. 도메인 전문가와 개발자 간의 협업을 통해 Ubiquitous Language 를 구축하고, Bounded Context 로 시스템을 분할하여 비즈니스 요구사항과 소프트웨어 설계를 일치시킵니다.

3. 전체 개요 (250 자 내외)#

DDD 는 Eric Evans 가 2003 년 도입한 소프트웨어 설계 접근법으로, 복잡한 비즈니스 도메인을 효과적으로 모델링하는 것을 목표로 합니다. Strategic Design(전략적 설계) 과 Tactical Design(전술적 설계) 로 구분되며, 도메인 모델을 중심으로 한 아키텍처를 통해 유지보수성과 확장성을 향상시킵니다. 마이크로서비스 아키텍처와도 자연스럽게 연계됩니다.

제 1 부: 기본 개념 및 원리#

핵심 개념#

기본 개념#
  • Domain (도메인): 소프트웨어 시스템이 해결하고자 하는 특정 문제 영역
  • Ubiquitous Language (유비쿼터스 언어): 도메인 전문가와 개발팀이 공유하는 공통 어휘로, 모든 의사소통과 코드에서 일관되게 사용
  • Bounded Context (바운디드 컨텍스트): 특정 모델이 명확하게 정의되고 적용되는 경계
  • Domain Model (도메인 모델): 비즈니스 도메인의 핵심 개념과 관계를 포착하는 소프트웨어 추상화
심화 개념#
  • Aggregate (애그리게이트): 데이터 변경 시 하나의 단위로 취급되는 도메인 객체들의 클러스터
  • Entity (엔티티): 속성이 아닌 고유 식별자로 정의되는 객체
  • Value Object (값 객체): 속성으로만 정의되며 개념적 식별자가 없는 불변 객체
  • Repository (리포지토리): 데이터 저장소에서 도메인 객체를 검색하는 메서드를 제공하는 객체

배경#

DDD 는 2003 년 Eric Evans 의 저서 “Domain-Driven Design: Tackling Complexity in the Heart of Software” 를 통해 소개되었습니다. 복잡한 소프트웨어 개발에서 기술적 측면보다는 비즈니스 도메인에 집중해야 한다는 철학에서 출발했습니다.

전통적인 소프트웨어 개발에서는 데이터베이스 스키마나 최신 프레임워크가 중심이 되어 비즈니스 요구사항과 괴리가 발생하는 문제가 있었습니다. DDD 는 이러한 문제를 해결하기 위해 Extreme Programming(XP) 방법론과 결합하여 지속적인 리팩토링과 진화하는 도메인 모델을 강조합니다.

목적 및 필요성#

주요 목적#
  1. 비즈니스 - 기술 격차 해소: 비즈니스와 소프트웨어 개발 간의 격차를 줄이고 기술 팀과 도메인 전문가 간의 명확한 소통 촉진
  2. 복잡성 관리: 복잡한 비즈니스 로직을 캡슐화하는 도메인 모델을 통해 비즈니스 현실과 코드 간의 격차 해소
  3. 시스템 모듈화: 대규모 도메인을 여러 Bounded Context 로 분할하여 명시적 관계 설정

  • 레거시 시스템의 한계: 단일 통합 모델의 한계로 인한 Big Ball of Mud 아키텍처 방지

주요 기능 및 역할#

Strategic Design (전략적 설계)#
  • Subdomain 식별: 도메인을 응집력 있는 기능 단위로 분할
  • Bounded Context 정의: 모델의 적용 경계를 명확히 설정하여 팀 자율성 확보
  • Context Map 작성: Bounded Context 간의 관계와 데이터 흐름 매핑
Tactical Design (전술적 설계)#
  • 도메인 모델 구현: Entity, Value Object, Aggregate 패턴을 통한 도메인 로직 구현
  • Repository 패턴: 데이터 접근 로직과 도메인 모델 분리
  • Domain Service: 여러 엔티티에 걸친 비즈니스 로직 모델링

특징#

    • Aggregate Root: 애그리게이트 외부에서의 모든 접근이 이를 통해서만 이루어지는 진입점

  2. Bounded Context: 특정 모델이 명확하게 정의되고 적용되는 범위

  3. Ubiquitous Language: 도메인 전문가와 개발팀이 공유하는 명확한 어휘

선택 구성요소#
  1. Repository: Aggregate 생명주기 관리를 위한 데이터 접근 추상화
  2. Domain Service: 단일 엔티티에 속하지 않는 도메인 로직 캡슐화
  3. Factory: 복잡한 도메인 객체 생성 로직 캡슐화
  4. Domain Events: 도메인 내에서 발생한 중요한 사건을 나타내는 객체

        self._total = 0
        self._items = []
    
    def add_item(self, item):
        self._items.append(item)
        self._calculate_total()

        self._total = sum(item.get_subtotal() for item in self._items)

복잡한 소프트웨어 시스템을 개발하기 위한 접근 방식으로, 비즈니스 도메인을 중심으로 소프트웨어를 설계하고 개발하는 방법론.
이 패턴은 Eric Evans 가 2003 년에 출간한 책 “Domain-Driven Design: Tackling Complexity in the Heart of Software” 에서 처음 소개되었다.

DDD 의 핵심 개념#

  1. 유비쿼터스 언어 (Ubiquitous Language)
    DDD 에서는 개발자와 도메인 전문가 사이의 의사소통을 위해 공통의 언어를 사용하는 것을 강조한다.
    이 언어는 코드, 문서, 대화 등 모든 곳에서 일관되게 사용되어야 한다.
    이를 통해 도메인 지식을 정확히 소프트웨어에 반영할 수 있다.

  2. 경계 지어진 컨텍스트 (Bounded Context)
    복잡한 도메인을 여러 개의 작은 하위 도메인으로 나누고, 각 하위 도메인 내에서 일관된 모델을 유지하는 것을 의미한다.
    이를 통해 대규모 시스템의 복잡성을 관리할 수 있다.

  3. 도메인 모델 (Domain Model)
    비즈니스 도메인의 핵심 개념과 규칙을 표현하는 객체 모델이다.
    이 모델은 지속적으로 발전하며, 도메인의 복잡성을 단순화하고 이해하기 쉬운 구조로 변환한다.

DDD 의 장점#
  1. 비즈니스 도메인에 집중: 개발자들이 비즈니스 도메인의 복잡성을 이해하고 실제 요구사항을 반영하는 소프트웨어를 개발할 수 있다.
  2. 의사소통과 협업 강화: 도메인 전문가와 개발자 간의 의사소통을 원활하게 한다.
  3. 유연하고 확장 가능한 설계: 변화하는 요구사항에 유연하게 대응할 수 있는 구조를 제공한다.
  4. 복잡성 관리: 복잡한 비즈니스 도메인을 단순하고 이해하기 쉬운 구조로 변환한다.
DDD 의 단점#
  1. 학습 곡선: 개발자들이 DDD 개념을 이해하고 적용하는 데 시간이 필요할 수 있다.
  2. 복잡성 증가: 작은 프로젝트에서는 오히려 복잡성을 증가시킬 수 있다.
  3. 도메인 전문가 참여 필수: 효과적인 DDD 적용을 위해서는 도메인 전문가의 지속적인 참여가 필요하다.

  1. 엔티티 (Entity)
    고유한 식별자를 가지며, 생명주기 동안 지속적으로 변경될 수 있는 객체이다.

  2. 값 객체 (Value Object)
    속성만으로 식별되며, 변경 불가능한 객체이다. 주로 개념적 완전성을 표현하는 데 사용된다.

  3. 집합체 (Aggregate)
    관련된 객체들의 집합으로, 하나의 단위로 취급된다.
    집합체 루트 (Aggregate Root) 를 통해서만 접근이 가능하다.

  4. 도메인 서비스 (Domain Service)
    엔티티나 값 객체에 속하지 않는 도메인 로직을 포함하는 객체이다.

  5. 리포지토리 (Repository)
    영속성 저장소에 대한 추상화 계층으로, 도메인 객체의 저장과 조회를 담당한다.

DDD 의 계층 구조#

DDD 는 일반적으로 다음과 같은 4 계층 구조를 따르다.

  1. 프레젠테이션 계층 (Presentation Layer)
  2. 응용 계층 (Application Layer)
  3. 도메인 계층 (Domain Layer)
  4. 인프라스트럭처 계층 (Infrastructure Layer)

이 구조를 통해 각 계층은 단일 책임을 가지며, 하위 계층에 대한 의존성만을 가진다.

# Value Object
@dataclass(frozen=True)
class Money:
    amount: int
    currency: str

    def add(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("Currency mismatch")
        return Money(self.amount + other.amount, self.currency)

    def subtract(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("Currency mismatch")
        return Money(self.amount - other.amount, self.currency)

@dataclass
class Product:
    id: UUID
    name: str
    price: Money
    stock: int

    @classmethod
    def create(cls, name: str, price: Money, stock: int) -> "Product":
        return cls(id=uuid4(), name=name, price=price, stock=stock)

    def decrease_stock(self, quantity: int):
        if self.stock < quantity:
            raise ValueError("Insufficient stock")
        self.stock -= quantity
