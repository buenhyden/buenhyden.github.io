Domain Event#

도메인 이벤트(Domain Event)는 마이크로서비스 아키텍처(MSA)에서 데이터 일관성을 유지하고 시스템 간 의존성을 줄이며 확장성을 높이는 데 중요한 개념이다.
도메인 이벤트는 주로 도메인 주도 설계(DDD) 의 개념에서 비롯되었으며, 비즈니스 로직과 시스템 상태 변화의 핵심을 나타낸다.
이를 통해 분산 시스템에서 데이터 일관성과 비즈니스 흐름을 효과적으로 관리할 수 있다.

도메인 이벤트는 도메인 내에서 발생한 중요한 상태 변화를 나타내는 객체이다.
이는 과거에 발생한 사건을 기술하며, 다른 모듈이나 시스템이 해당 이벤트를 구독하고 적절히 반응할 수 있도록 설계된다. 예를 들어, 전자상거래 시스템에서 “주문이 생성됨(Order Created)“이나 “결제가 완료됨(Payment Completed)” 같은 사건이 도메인 이벤트로 표현될 수 있다.

도메인 이벤트는 MSA 환경에서 데이터 일관성과 확장성을 유지하는 데 매우 유용한 패턴이다. 이를 통해 각 서비스가 독립적으로 동작하면서도 필요한 경우 협력할 수 있도록 설계할 수 있다. 적절히 활용하면 복잡한 분산 시스템에서도 안정적이고 유연한 아키텍처를 구축할 수 있다.

도메인 이벤트의 역할과 중요성#

도메인 이벤트는 다음과 같은 역할을 수행한다:

  1. 서비스 간 통신: 한 서비스에서 발생한 중요한 상태 변화를 다른 서비스에 전달하여, 시스템 전체의 일관성을 유지한다.
  2. 최종 일관성(Eventual Consistency): 즉각적인 일관성을 보장하기 어려운 분산 시스템에서, 시간이 지남에 따라 데이터의 일관성을 확보할 수 있도록 한다.
  3. 비동기 처리: 이벤트를 비동기적으로 처리함으로써, 서비스 간의 결합도를 낮추고 시스템의 확장성과 유연성을 높인다.

주요 특징#

  1. 과거형으로 표현: 도메인 이벤트는 이미 발생한 사건을 나타내므로 과거형으로 기술된다(예: “OrderPlaced”, “PaymentProcessed”).
  2. 불변성: 이벤트는 발생 시점의 상태를 기록하며, 이후 수정되거나 삭제되지 않는다.
  3. 비즈니스 중심: 도메인 이벤트는 비즈니스적으로 중요한 상태 변화를 모델링한다.

도메인 이벤트의 구성 요소#

  1. 이벤트 주체: 도메인 객체(엔티티, 값 객체, 애그리게이트 등)가 이벤트를 생성한다.
  2. 이벤트 핸들러(Event Handler): 이벤트를 처리하는 컴포넌트로, 특정 작업(알림 발송, 데이터 동기화 등)을 수행한다.
  3. 이벤트 디스패처(Event Dispatcher): 이벤트를 핸들러에 전달하는 역할을 한다.

도메인 이벤트의 설계 및 구현#

  1. 도메인 이벤트 정의: 도메인 이벤트는 상태 변화와 관련된 데이터를 포함해야 한다. 예를 들어, 주문 생성 이벤트는 주문 ID, 고객 ID, 주문 시간 등의 정보를 포함할 수 있다.

2. 이벤트 발행: 도메인 객체가 상태 변경 시 관련 이벤트를 생성하고 발행한다. 
	
	```java
	public class Order {
	    private String id;
	    private String status;
	
	    public void createOrder(String customerId) {
	        this.id = UUID.randomUUID().toString();
	        this.status = "CREATED";
	        DomainEventPublisher.publish(new OrderCreatedEvent(this.id, customerId, LocalDateTime.now()));
	    }
	}
	```

3. 이벤트 처리: 이벤트 핸들러가 해당 이벤트를 구독하고 필요한 작업을 수행한다. 

	```java
	public class NotificationHandler {
	    @EventListener
	    public void handleOrderCreated(OrderCreatedEvent event) {
	        System.out.println("Sending notification for order: " + event.getOrderId());
	    }
	}
	```

### 도메인 이벤트의 장점

1. **느슨한 결합**: 시스템 간 의존성을 줄여 독립적인 확장이 가능하다. 
2. **확장성**: 새로운 기능 추가 시 기존 코드에 영향을 주지 않고 새로운 핸들러만 추가하면 된다. 
3. **비동기 처리**: 비동기 방식으로 처리하면 성능과 응답 속도를 개선할 수 있다.
4. **데이터 일관성**: 분산 트랜잭션 없이 이벤추얼 컨시스턴시(Eventual Consistency)를 유지할 수 있다.

### 도메인 이벤트와 통합 이벤트의 차이점

| **특징**      | **도메인 이벤트**                                   | **통합 이벤트**                                                   |
| ------------- | --------------------------------------------------- | ----------------------------------------------------------------- |
| **목적**      | 동일한 도메인의 내부 상태 변화를 알림               | 다른 마이크로서비스나 외부 시스템에 상태 변화를 알림              |
| **범위**      | 단일 바운디드 컨텍스트(Bounded Context) 내에서 사용 | 여러 마이크로서비스 또는 외부 시스템 간 통신                      |
| **처리 방식** | 메모리 내에서 동기 또는 비동기로 처리               | 메시지 큐(Kafka, RabbitMQ 등)를 사용하여 비동기로 처리            |
| **예시**      | `OrderCreatedEvent`가 주문 서비스 내부에서만 사용   | `OrderCreatedIntegrationEvent`가 결제 서비스나 배송 서비스에 전달 |

### 도메인 이벤트 활용 사례

1. **전자상거래 플랫폼**
	- 주문 생성 시 재고 확인 및 결제 요청을 트리거
	- 배송 준비 완료 시 고객에게 알림 전송
1. **금융 시스템**
	- 계좌 이체 완료 후 거래 내역 기록 및 알림 발송
2. **물류 관리**
	- 배송 상태 변경 시 실시간 추적 정보 업데이트

### 구현 시 고려사항

1. **핸들러의 멱등성(Idempotency)**: 동일한 이벤트가 여러 번 처리되더라도 결과가 동일해야 한다. 
2. **이벤트 저장소(Event Store)**: 발생한 이벤트를 영구적으로 저장하여, 필요에 따라 재처리하거나 시스템의 상태를 재구성할 수 있도록 한다. 
3. **성능 최적화**: 비동기 메시징 시스템(Kafka, RabbitMQ 등)을 활용해 성능 저하를 방지한다. 
4. **테스트 가능성**: 테스트와 디버깅을 용이하게 하기 위해 명확하고 단순한 설계를 유지한다. 
5. **트랜잭션 관리**: 트랜잭션 경계를 명확히 하고, 트랜잭션이 커밋된 후에만 이벤트를 발행해야 한다.
