도메인 주도 개발 (Domain-Driven Development, DDD)#

복잡한 소프트웨어를 개발할 때 비즈니스 도메인을 중심으로 설계와 개발을 진행하는 방법론
도메인이란 소프트웨어가 해결하고자 하는 핵심 비즈니스 영역을 의미

주요 단계#

  1. 전략적 설계(Strategic Design):
    • 핵심 도메인 식별
    • 바운디드 컨텍스트 정의
    • 컨텍스트 맵 작성
  2. 전술적 설계(Tactical Design):
    • 엔티티, 값 객체, 집합체 등의 도메인 모델 요소 설계
    • 도메인 서비스 정의
    • 리포지토리 및 팩토리 구현
  3. 구현 및 지속적 개선:
    • 도메인 모델을 코드로 구현
    • 지속적인 리팩토링과 모델 개선

특징#

  1. 도메인 중심 접근: DDD는 비즈니스 도메인을 소프트웨어 설계의 핵심으로 삼습니다.
  2. 유비쿼터스 언어: 개발자와 도메인 전문가 간의 공통 언어를 사용하여 의사소통을 개선합니다.
  3. 바운디드 컨텍스트: 도메인 모델의 적용 범위를 명확히 정의합니다.
  4. 모델 주도 설계: 도메인 모델을 중심으로 소프트웨어를 설계합니다.

장점#

  1. 비즈니스 목표와의 연계: 소프트웨어가 실제 비즈니스 요구사항을 더 잘 반영합니다.
  2. 복잡성 관리: 큰 시스템을 관리 가능한 바운디드 컨텍스트로 나눕니다.
  3. 유지보수성 향상: 도메인 모델이 코드에 직접 반영되어 유지보수가 용이합니다.
  4. 협업 개선: 도메인 전문가와 개발자 간의 의사소통이 향상됩니다.

단점#

  1. 초기 학습 곡선: DDD 개념과 실践에 익숙해지는 데 시간이 필요합니다.
  2. 복잡한 도메인에서의 어려움: 매우 복잡한 도메인에서는 모델링이 어려울 수 있습니다.
  3. 초기 개발 속도 저하: 도메인 모델링에 시간이 소요되어 초기 개발 속도가 느려질 수 있습니다.

핵심 요소#

  1. 엔티티(Entity): 고유한 식별자를 가지는 도메인 객체.
  2. 값 객체(Value Object): 속성만으로 정의되는 불변 객체.
  3. 집합체(Aggregate): 관련된 객체들의 집합.
  4. 도메인 서비스(Domain Service): 특정 엔티티에 속하지 않는 도메인 로직.
  5. 리포지토리(Repository): 도메인 객체의 저장소.
  6. 팩토리(Factory): 복잡한 객체의 생성을 담당.

적합한 프로젝트 유형#

  1. 복잡한 비즈니스 로직을 가진 프로젝트: DDD는 복잡한 도메인을 효과적으로 모델링할 수 있습니다.
  2. 장기적인 유지보수가 필요한 프로젝트: DDD는 코드의 유지보수성을 향상시킵니다.
  3. 도메인 전문가와의 긴밀한 협업이 필요한 프로젝트: DDD는 도메인 전문가와 개발자 간의 협업을 강화합니다.
  4. 마이크로서비스 아키텍처를 사용하는 프로젝트: DDD의 바운디드 컨텍스트 개념은 마이크로서비스 설계에 적합합니다.
  5. 비즈니스 규칙이 자주 변경되는 프로젝트: DDD는 변화하는 비즈니스 요구사항에 유연하게 대응할 수 있습니다.
