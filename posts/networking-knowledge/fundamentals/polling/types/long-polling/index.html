Long Polling#

Long polling은 전통적인 short polling의 한계를 극복하기 위해 발전된 웹 통신 기법으로, 실시간에 가까운 데이터 전송을 가능하게 한다. 이 기술은 특히 웹소켓(WebSocket)이 등장하기 전에 실시간 웹 애플리케이션 구현에 널리 사용되었다.

Long polling은 WebSockets의 대중화 이전에 실시간 웹 애플리케이션의 핵심 기술이었으며, 오늘날에도 특정 상황에서 유용한 접근 방식이다. 특히 WebSockets 지원이 제한된 환경이나, 단순한 실시간 요구사항을 가진 애플리케이션에서 여전히 가치 있는 솔루션이다.

최신 웹 애플리케이션에서는 WebSockets가 선호되는 경향이 있지만, Long polling은 폴백(fallback) 메커니즘으로 구현되어 WebSockets를 지원하지 않는 환경에서도 실시간에 가까운 경험을 제공할 수 있다.

Long Polling의 개념#

Long polling은 클라이언트가 서버에 HTTP 요청을 보내고, 서버는 새로운 정보가 있을 때까지 응답을 보류하는 방식이다. 서버는 새로운 이벤트가 발생하거나 타임아웃이 될 때까지 연결을 열어둔다. 이 방식은 클라이언트가 지속적으로 서버에 요청을 보내는 short polling과 달리, 서버에서 실제 데이터 변경이 있을 때만 응답이 이루어지므로 불필요한 요청과 응답을 줄일 수 있다.

작동 메커니즘#

  1. 클라이언트가 서버에 HTTP 요청을 보낸다.
  2. 서버는 응답을 즉시 보내지 않고, 새 데이터가 생길 때까지 연결을 유지한다.
  3. 새 데이터가 생기면 서버는 응답을 완료하고 연결을 종료한다.
  4. 클라이언트는 응답을 받은 즉시 다음 요청을 보내 다시 프로세스를 시작한다.
  5. 일정 시간(타임아웃) 동안 새 데이터가 없으면 서버는 빈 응답을 보내고, 클라이언트는 새 요청을 시작한다.

// 클라이언트 측 Long Polling 구현
function startLongPolling() {
    // 서버에 요청 보내기
    function poll() {
        fetch('https://api.example.com/events', {
            method: 'GET',
            headers: {
                'Content-Type': 'application/json',
                'Last-Event-ID': lastEventId // 마지막으로 받은 이벤트 ID
            },
            // 타임아웃 설정 (브라우저 기본값보다 길게)
            timeout: 60000 // 60초
        })
        .then(response => response.json())
        .then(data => {
            if (data && data.events && data.events.length > 0) {
                // 새 이벤트 처리
                processEvents(data.events);
                
                // 마지막 이벤트 ID 업데이트
                lastEventId = data.events[data.events.length - 1].id;
            }
            
            // 응답을 받은 즉시 다음 요청 시작 (재귀적 호출)
            poll();
        })
        .catch(error => {
            console.error('Long polling 오류:', error);
            
            // 오류 발생 시 잠시 대기 후 재시도
            setTimeout(poll, 5000);
        });
    }
    
    // 첫 요청 시작
    let lastEventId = 0; // 마지막으로 받은 이벤트 ID 추적
    poll();
}

function processEvents(events) {
    // 받은 이벤트 처리 로직
    events.forEach(event => {
        console.log('새 이벤트:', event);
        // 이벤트 유형에 따른 처리
        switch(event.type) {
            case 'message':
                displayNewMessage(event.data);
                break;
            case 'notification':
                showNotification(event.data);
                break;
            // 기타 이벤트 유형 처리
        }
    });
}

// Long Polling 시작
startLongPolling();

const express = require('express');
const app = express();

// 활성 연결 저장소
const connections = [];
// 이벤트 저장소
let events = [];
let nextEventId = 1;

app.get('/events', (req, res) => {
    // 클라이언트의 마지막 이벤트 ID
    const lastEventId = parseInt(req.headers['last-event-id']) || 0;
    
    // 클라이언트가 아직 받지 않은 이벤트 확인
    const newEvents = events.filter(event => event.id > lastEventId);
    
    if (newEvents.length > 0) {
        // 새 이벤트가 있으면 즉시 응답
        return res.json({ events: newEvents });
    }
    
    // 새 이벤트가 없으면 연결 유지
    const timeout = setTimeout(() => {
        // 30초 타임아웃 후 빈 응답
        res.json({ events: [] });
        // 연결 목록에서 제거
        const index = connections.findIndex(c => c.id === connectionId);
        if (index !== -1) connections.splice(index, 1);
    }, 30000); // 30초 타임아웃
    
    // 연결 식별자 생성
    const connectionId = Date.now();
    
    // 활성 연결 목록에 추가
    connections.push({
        id: connectionId,
        res,
        timeout
    });
    
    // 클라이언트 연결 종료 시 정리
    req.on('close', () => {
        clearTimeout(timeout);
        const index = connections.findIndex(c => c.id === connectionId);
        if (index !== -1) connections.splice(index, 1);
    });
});

// 새 이벤트 생성 (예: 채팅 메시지)
app.post('/message', express.json(), (req, res) => {
    const newEvent = {
        id: nextEventId++,
        type: 'message',
        data: req.body,
        timestamp: new Date()
    };
    
    // 이벤트 저장
    events.push(newEvent);
    
    // 최대 100개 이벤트만 유지 (메모리 관리)
    if (events.length > 100) {
        events = events.slice(-100);
    }
    
    // 대기 중인 모든 연결에 이벤트 전송
    connections.forEach(connection => {
        // 타임아웃 취소
        clearTimeout(connection.timeout);
        // 응답 전송
        connection.res.json({ events: [newEvent] });
    });
    
    // 연결 목록 초기화
    connections.length = 0;
    
    // 성공 응답
    res.status(201).json({ success: true });
});

app.listen(3000, () => {
    console.log('서버가 3000번 포트에서 실행 중입니다.');
});

Long Polling의 장점#

  1. 지연 시간 감소: 이벤트가 발생하면 서버가 즉시 클라이언트에 알려주므로 short polling보다 지연 시간이 적다.
  2. 불필요한 요청 감소: 새 데이터가 있을 때만 실제 데이터가 전송되므로 리소스 사용이 효율적이다.
  3. 서버 부하 감소: 데이터 변경이 없으면 불필요한 요청-응답 사이클이 발생하지 않는다.
  4. 표준 HTTP 프로토콜 사용: 특별한 프로토콜이나 포트가 필요 없어 대부분의 환경에서 호환성이 좋다.
  5. 방화벽 친화적: 일반적인 HTTP 통신을 사용하므로 대부분의 방화벽 설정에서 문제없이 작동한다.

Long Polling의 단점#

  1. 서버 리소스 점유: 연결이 오래 유지되므로 서버 자원(특히 스레드)을 더 많이 소비한다.
  2. 확장성 제한: 동시 연결 수가 많아지면 서버 성능이 저하될 수 있다.
  3. 메시지 순서 보장 문제: 여러 개의 동시 요청이 있을 경우 메시지 순서가 보장되지 않을 수 있다.
  4. 연결 타임아웃 관리: 프록시, 로드 밸런서, 방화벽 등이 유휴 연결을 끊을 수 있어 적절한 타임아웃 관리가 필요하다.
  5. 복잡한 에러 처리: 연결이 끊어진 경우의 재연결 로직 등 구현이 short polling보다 복잡하다.

최적화 전략#

  1. 적절한 타임아웃 설정: 서버 측 타임아웃을 네트워크 인프라(로드 밸런서, 프록시 등)의 타임아웃보다 짧게 설정한다.
  2. 재연결 전략: 지수 백오프(exponential backoff) 알고리즘을 사용하여 연결 실패 시 점진적으로 재시도 간격을 늘린다.
  3. 연결 풀링: 서버 측에서 연결 풀을 관리하여 리소스 사용을 최적화한다.
  4. 클러스터링과 로드 밸런싱: 여러 서버 인스턴스 간에 로드를 분산하고, 세션 정보를 공유한다.
  5. 하이브리드 접근: 상황에 따라 short polling과 long polling을 혼합하여 사용한다.

Long Polling vs. 다른 실시간 통신 기술#

  1. Short Polling: 주기적으로 서버에 요청을 보내는 방식이다. Long polling에 비해 구현이 간단하지만, 불필요한 요청이 많고 지연 시간이 길다.

  2. Server-Sent Events (SSE): 서버에서 클라이언트로 단방향 실시간 이벤트 스트림을 제공한다. Long polling보다 구현이 간단하고 효율적이지만, 브라우저 지원과 양방향 통신에 제한이 있다.

  3. WebSockets: 양방향 실시간 통신을 위한 전용 프로토콜로, 가장 효율적이고 지연 시간이 적지만 구현이 복잡하고 일부 네트워크 환경(특히 기업 방화벽)에서 제한될 수 있다.

적용 사례#

Long polling은 다음과 같은 상황에서 효과적이다:

  1. 채팅 애플리케이션: 메시지가 불규칙적으로 발생하며 실시간 전달이 중요한 경우
  2. 알림 시스템: 사용자에게 즉시 알림을 전달해야 하는 경우
  3. 협업 도구: 여러 사용자가 동시에 작업하는 문서 편집기 등
  4. 주식 거래 플랫폼: 시세 변동을 실시간으로 전달해야 하는 경우
  5. 게임: 실시간 상호작용이 필요한 웹 기반 게임

구현 시 고려사항#

  1. 서버 측 리소스 관리: 각 연결이 서버 리소스를 소비하므로 적절한 타임아웃과 연결 제한을 설정한다.
  2. 클라이언트 연결 관리: 네트워크 변경이나 브라우저 탭 전환 시에도 안정적으로 작동하도록 설계한다.
  3. 메시지 중복 및 손실 처리: 각 메시지에 고유 ID를 부여하고 순서를 추적한다.
  4. 보안 고려사항: 인증과 권한 부여를 통해 연결된 클라이언트의 접근 권한을 관리한다.
  5. 프록시와 로드 밸런서: 중간 인프라가 장기 연결을 지원하도록 설정합니다.한다.
