Abstract Factory Pattern

구체적인 클래스를 지정하지 않고도 관련된 객체들의 집합을 생성할 수 있게 해주는 생성 패턴

Abstract Factory Pattern#

구체적인 클래스를 지정하지 않고도 관련된 객체들의 집합을 생성할 수 있게 해주는 생성 패턴

특징#

  • 제품군의 생성을 캡슐화합니다
  • 관련된 객체들이 함께 동작하도록 보장합니다
  • 구체적인 클래스에 의존하지 않고 인터페이스에 의존합니다
  • 제품군을 쉽게 교체할 수 있습니다

사용사례#

  • 여러 제품군 중 하나를 선택해 시스템을 설정해야 할 때
  • 관련된 객체들이 함께 사용되어야 할 때
  • 제품에 대한 클래스 라이브러리를 제공하고 구현이 아닌 인터페이스만 노출하고 싶을 때

실제 예:

  • 크로스 플랫폼 UI 컴포넌트 생성
  • 다양한 데이터베이스 시스템 지원
  • 여러 운영체제에 대한 서비스 구현
  • 다양한 게임 캐릭터와 아이템 생성
  • 문서 포맷 변환 시스템

장점#

  • 제품군의 일관성 보장
  • 구체적인 클래스와의 결합도 감소
  • 제품군 교체가 용이
  • 새로운 제품 추가가 용이

단점#

  • 코드가 복잡해질 수 있음
  • 새로운 종류의 제품을 추가하기 어려움
  • 불필요한 추상화로 인한 오버헤드 가능성

주의사항 및 고려사항#

  • 확장성 고려
    • 새로운 제품군 추가를 고려한 설계
    • 인터페이스 설계 시 신중한 고려
  • 복잡성 관리
    • 적절한 추상화 수준 유지
    • 명확한 책임 분리
  • 성능 최적화
    • 객체 생성 비용 고려
    • 캐싱 전략 검토
  • 테스트 용이성
    • 목 객체 생성 방법 고려
    • 테스트 시나리오 설계

from abc import ABC, abstractmethod

# 추상 제품 클래스들
class Button(ABC):
    @abstractmethod
    def render(self):
        pass

    @abstractmethod
    def handle_click(self):
        pass

class Checkbox(ABC):
    @abstractmethod
    def render(self):
        pass

    @abstractmethod
    def toggle(self):
        pass

# 구체적인 제품 클래스들 - Light Theme
class LightButton(Button):
    def render(self):
        return "밝은 테마 버튼을 렌더링합니다"

    def handle_click(self):
        return "밝은 테마 버튼 클릭을 처리합니다"

class LightCheckbox(Checkbox):
    def render(self):
        return "밝은 테마 체크박스를 렌더링합니다"

    def toggle(self):
        return "밝은 테마 체크박스 상태를 토글합니다"

# 구체적인 제품 클래스들 - Dark Theme
class DarkButton(Button):
    def render(self):
        return "어두운 테마 버튼을 렌더링합니다"

    def handle_click(self):
        return "어두운 테마 버튼 클릭을 처리합니다"

class DarkCheckbox(Checkbox):
    def render(self):
        return "어두운 테마 체크박스를 렌더링합니다"

    def toggle(self):
        return "어두운 테마 체크박스 상태를 토글합니다"

# 추상 팩토리
class UIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button:
        pass

    @abstractmethod
    def create_checkbox(self) -> Checkbox:
        pass

# 구체적인 팩토리들
class LightThemeFactory(UIFactory):
    def create_button(self) -> Button:
        return LightButton()

    def create_checkbox(self) -> Checkbox:
        return LightCheckbox()

class DarkThemeFactory(UIFactory):
    def create_button(self) -> Button:
        return DarkButton()

    def create_checkbox(self) -> Checkbox:
        return DarkCheckbox()

# 클라이언트 코드
class Application:
    def __init__(self, factory: UIFactory):
        self.factory = factory
        self.button = None
        self.checkbox = None

    def create_ui(self):
        self.button = self.factory.create_button()
        self.checkbox = self.factory.create_checkbox()

    def paint(self):
        print(self.button.render())
        print(self.checkbox.render())

# 사용 예시
if __name__ == "__main__":
    # 밝은 테마 사용
    print("밝은 테마 적용:")
    app = Application(LightThemeFactory())
    app.create_ui()
    app.paint()

    print("\n어두운 테마 적용:")
    # 어두운 테마로 전환
    app = Application(DarkThemeFactory())
    app.create_ui()
    app.paint()

// 추상 제품 클래스들
class Vehicle {
    constructor() {
        if (this.constructor === Vehicle) {
            throw new Error("추상 클래스는 인스턴스화할 수 없습니다.");
        }
    }

    getSpecifications() {
        throw new Error("추상 메서드는 구현해야 합니다.");
    }
}

class Engine {
    constructor() {
        if (this.constructor === Engine) {
            throw new Error("추상 클래스는 인스턴스화할 수 없습니다.");
        }
    }

    getDetails() {
        throw new Error("추상 메서드는 구현해야 합니다.");
    }
}

// 구체적인 제품 클래스들 - 스포츠카
class SportsCar extends Vehicle {
    getSpecifications() {
        return "스포츠카: 고성능, 2인승";
    }
}

class SportsEngine extends Engine {
    getDetails() {
        return "스포츠카 엔진: V8, 500hp";
    }
}

// 구체적인 제품 클래스들 - SUV
class SUV extends Vehicle {
    getSpecifications() {
        return "SUV: 실용성, 7인승";
    }
}

class SUVEngine extends Engine {
    getDetails() {
        return "SUV 엔진: V6, 280hp";
    }
}

// 추상 팩토리
class VehicleFactory {
    createVehicle() {
        throw new Error("추상 메서드는 구현해야 합니다.");
    }

    createEngine() {
        throw new Error("추상 메서드는 구현해야 합니다.");
    }
}

// 구체적인 팩토리들
class SportsCarFactory extends VehicleFactory {
    createVehicle() {
        return new SportsCar();
    }

    createEngine() {
        return new SportsEngine();
    }
}

class SUVFactory extends VehicleFactory {
    createVehicle() {
        return new SUV();
    }

    createEngine() {
        return new SUVEngine();
    }
}

// 클라이언트 코드
class CarDealer {
    constructor(factory) {
        this.factory = factory;
    }

    orderCar() {
        const vehicle = this.factory.createVehicle();
        const engine = this.factory.createEngine();

        console.log("새로운 차량 주문:");
        console.log(vehicle.getSpecifications());
        console.log(engine.getDetails());
    }
}

// 사용 예시
function main() {
    console.log("스포츠카 주문:");
    const sportsCarDealer = new CarDealer(new SportsCarFactory());
    sportsCarDealer.orderCar();

    console.log("\nSUV 주문:");
    const suvDealer = new CarDealer(new SUVFactory());
    suvDealer.orderCar();
}

main();

참고 및 출처#

1. 주제의 분류가 적절한지에 대한 조사#

Abstract Factory Pattern(추상 팩토리 패턴) 은 “Computer Science and Engineering > Software Design and Architecture > Software Design Patterns > GoF > Creational Design Patterns” 분류에 완벽히 부합합니다. GoF(Gang of Four) 에서 정의한 23 가지 디자인 패턴 중 하나로, 객체 생성 방법을 추상화하여 관련 객체군의 생성과 결합을 책임지는 대표적인 생성 (Creational) 패턴입니다 [1][2][6].

2. 200 자 요약#

추상 팩토리 패턴은 관련된 객체들의 집합을 구체 클래스를 지정하지 않고 생성할 수 있는 인터페이스를 제공하는 생성 패턴입니다. 이 패턴은 제품군 간의 일관성과 확장성을 보장하며, 클라이언트 코드와 객체 생성 로직을 분리해 코드의 유연성과 유지보수성을 높입니다 [1][2][3][9].

3. 250 자 개요#

추상 팩토리 패턴은 제품군 (관련 객체들) 을 생성하는 팩토리 인터페이스를 정의하고, 구체 팩토리에서 각 제품군에 맞는 객체를 생성하도록 위임하는 생성 패턴입니다. 클라이언트는 추상 팩토리와 추상 제품 인터페이스만을 사용해 객체를 생성하므로, 구체 클래스에 의존하지 않고 다양한 제품군을 쉽게 교체하거나 확장할 수 있습니다. GUI, 데이터베이스, 크로스플랫폼 등에서 제품군별 일관성, 테스트 용이성, 코드 유연성을 확보할 수 있습니다 [1][2][3][9][16][18].

핵심 개념#

  • 정의: 추상 팩토리 패턴은 관련된 객체들의 집합을 구체 클래스를 지정하지 않고 생성할 수 있는 인터페이스를 제공하는 생성 패턴입니다 [1][2][3][9][16][18].
  • 목적 및 필요성: 제품군 간의 일관성 보장, 객체 생성 로직과 사용 코드 분리, 교체 및 확장 용이성 [1][2][3][18].
  • 주요 기능 및 역할:
    • 제품군별 객체 생성 인터페이스 제공
    • 구체 팩토리에서 제품군에 맞는 객체 생성
    • 클라이언트는 추상 타입만 사용
  • 특징: 느슨한 결합, 일관성, 확장성, 테스트 용이성, 제품군 교체의 유연성 [1][2][3][9][18][23].
  • 핵심 원칙: DIP(의존성 역전 원칙), OCP(개방/폐쇄 원칙), SRP(단일 책임 원칙), 인터페이스 분리 [1][2][3][23].

의도 (Intent)#

구체 클래스를 지정하지 않고, 관련된 객체들의 집합 (제품군) 을 생성할 수 있는 인터페이스를 제공한다 [1][2][3][18].

다른 이름 (Also Known As)#

  • Factory of Factories(팩토리의 팩토리)[16]
  • Kit Pattern

동기 (Motivation / Forces)#

  • 제품군별 일관성 보장
  • 객체 생성 로직과 사용 코드 분리
  • 크로스플랫폼, 테마, DB 등 다양한 환경에서 제품군 교체의 유연성 확보 [1][2][3][18].

적용 가능성 (Applicability)#

  • 제품군별로 관련 객체를 함께 생성해야 할 때
  • 클라이언트 코드가 구체 클래스를 몰라도 되게 하고 싶을 때
  • 제품군 교체가 빈번하거나, 확장성이 중요한 시스템 [1][2][3][18][23].

주요 원리 및 작동 원리#

  1. 클라이언트는 AbstractFactory 인터페이스를 통해 객체 생성
  2. 구체 팩토리 (ConcreteFactory) 는 제품군별 객체를 생성해 반환
  3. 클라이언트는 추상 제품 타입만 사용하며, 구체 클래스는 몰라도 됨
  4. 제품군 교체 시 ConcreteFactory 만 교체하면 됨

# Abstract Products
class Button:
    def click(self): pass

class Checkbox:
    def check(self): pass

# Concrete Products
class WinButton(Button):
    def click(self): print("Windows Button Clicked")

class MacButton(Button):
    def click(self): print("Mac Button Clicked")

class WinCheckbox(Checkbox):
    def check(self): print("Windows Checkbox Checked")

class MacCheckbox(Checkbox):
    def check(self): print("Mac Checkbox Checked")

# Abstract Factory
class GUIFactory:
    def create_button(self): pass
    def create_checkbox(self): pass

# Concrete Factories
class WinFactory(GUIFactory):
    def create_button(self): return WinButton()
    def create_checkbox(self): return WinCheckbox()

class MacFactory(GUIFactory):
    def create_button(self): return MacButton()
    def create_checkbox(self): return MacCheckbox()

# Client
def client_code(factory: GUIFactory):
    button = factory.create_button()
    checkbox = factory.create_checkbox()
    button.click()
    checkbox.check()

client_code(WinFactory())  # Windows Button Clicked / Windows Checkbox Checked
client_code(MacFactory())  # Mac Button Clicked / Mac Checkbox Checked

Abstract Factory Pattern (추상 팩토리 패턴) 에 대해 실무 중심으로 심층 조사 및 분석한 내용을 아래에 정리하였습니다.

1. 주제의 분류 적절성 검토#

  • 현재 분류
    “Computer Science and Engineering” > “Software Design and Architecture” > “Software Design Patterns” > “GoF” > “Creational Design Patterns”

  • 검토 결과
    ✅ 적절합니다. Abstract Factory 는 객체 생성을 캡슐화하여 객체군 (Family of Objects) 을 생성할 수 있도록 하는 대표적인 생성 (Creational) 디자인 패턴입니다.

2. 요약 설명 (200 자 내외)#

추상 팩토리 패턴은 관련된 객체군을 일관되게 생성하기 위한 패턴으로, 클라이언트는 객체 생성 로직에 의존하지 않고 인터페이스를 통해 제품군을 생성할 수 있다. UI 테마, 플랫폼 독립 객체 생성에 유리하다.

3. 개요 설명 (250 자 내외)#

Abstract Factory Pattern 은 관련 객체들을 그룹으로 생성하는 팩토리 인터페이스를 정의하고, 구체 팩토리에서 이를 구현하여 일관된 객체군을 생성할 수 있도록 하는 생성 패턴이다. 이 패턴을 사용하면 제품군 간 일관성을 유지할 수 있으며, 런타임 시 제품군을 쉽게 교체할 수 있다. 예를 들어 UI 컴포넌트를 다크 모드/라이트 모드로 생성하거나, 운영체제별 버튼과 체크박스를 생성할 때 유용하다.

4. 핵심 개념#

항목설명
핵심 아이디어객체군 생성을 캡슐화하여 클라이언트가 객체 생성 클래스를 직접 참조하지 않음

추상 팩토리 패턴 (Abstract Factory Pattern) 심화 분석#

1. 주제 분류 검증#

제시된 분류 “Computer Science and Engineering” > “Software Design and Architecture” > “Software Design Patterns” > “GoF” > “Creational Design Patterns” 는 적절합니다. 추상 팩토리 패턴은 1995 년 GoF(Gang of Four) 에 의해 정의된 23 가지 디자인 패턴 중 생성 패턴 (Creational Pattern) 에 속하는 핵심 디자인 패턴입니다.

2. 주요 개념 요약#

추상 팩토리 패턴은 구체적인 클래스를 지정하지 않고도 관련된 객체들의 패밀리 (제품군) 를 생성할 수 있게 하는 생성 패턴으로, 객체 생성과 조합을 캡슐화하여 시스템의 유연성과 확장성을 제공하는 핵심 소프트웨어 아키텍처 설계 기법입니다.

3. 전체 개요#

추상 팩토리 패턴은 서로 관련성이 있거나 독립적인 여러 객체의 군을 생성하기 위한 인터페이스를 제공하는 디자인 패턴입니다. 이 패턴은 구체적인 클래스의 이름을 명시하지 않고도 객체의 집합을 생성할 수 있게 하며, 클라이언트 코드와 구체적인 구현부를 분리시켜 시스템의 독립성과 유지보수성을 향상시킵니다.

4. 핵심 개념#

기본 개념#

  • 추상 팩토리 (Abstract Factory): 관련된 객체군을 생성하는 인터페이스
  • 제품군 (Product Family): 함께 사용되도록 설계된 관련 객체들의 집합
  • 구체적 팩토리 (Concrete Factory): 특정 제품군을 생성하는 구현체
  • 클라이언트 분리: 구체적인 구현부로부터 독립적인 클라이언트 코드

심화 개념#

  • 팩토리의 팩토리: 여러 팩토리를 관리하는 상위 수준의 추상화
  • 제품 일관성 (Product Consistency): 같은 패밀리 내 객체들의 호환성 보장
  • 런타임 구성: 실행 시점에서의 동적 팩토리 선택
  • 확장성 캡슐화: 새로운 제품군 추가 시 기존 코드 변경 최소화

  • 시스템 독립성: 객체 생성·구성·표현 방식과 무관한 시스템 구축

  4. 설정 캡슐화: 제품군 선택 로직의 중앙 집중화

주요 역할#

7. 특징 및 핵심 원칙#

주요 특징#

  • 계층적 구조: 추상 팩토리 → 구체적 팩토리 → 제품
  • 인터페이스 기반: 추상화를 통한 느슨한 결합
  • 런타임 결정: 실행 시점에서의 팩토리 선택
  • 제품군 관리: 여러 관련 객체의 일괄 처리

핵심 원칙#

  1. 개방 - 폐쇄 원칙 (OCP): 확장에는 열려있고 수정에는 닫혀있음
  2. 의존성 역전 원칙 (DIP): 추상화에 의존, 구체화에 의존하지 않음
  3. 단일 책임 원칙 (SRP): 각 팩토리는 하나의 제품군만 담당
  4. 인터페이스 분리 원칙 (ISP): 필요한 메소드만 노출

9. 구조 및 아키텍처#

필수 구성요소#

1. Abstract Factory (추상 팩토리)#
  • 기능: 제품군 생성을 위한 인터페이스 정의
  • 역할: 각 제품에 대한 생성 메소드 선언
  • 특징: 구체적인 구현을 포함하지 않음
2. Concrete Factory (구체적 팩토리)#
  • 기능: 특정 제품군의 실제 생성 구현
  • 역할: 추상 팩토리 인터페이스의 구현체
  • 특징: 하나의 제품군에 대한 모든 제품 생성
3. Abstract Product (추상 제품)#
  • 기능: 제품 타입별 인터페이스 정의
  • 역할: 제품군 내 각 제품의 공통 인터페이스
  • 특징: 구체적 제품들의 상위 타입
4. Concrete Product (구체적 제품)#
  • 기능: 실제 제품의 구현
  • 역할: 추상 제품 인터페이스의 구현체
  • 특징: 특정 제품군에 속하는 실제 객체

선택 구성요소#

5. Client (클라이언트)#
  • 기능: 추상 팩토리를 통한 객체 생성 및 사용
  • 역할: 구체적 구현에 의존하지 않는 코드 작성

10. 구현 기법#

1. 기본 구현 기법#

  • 정의: 인터페이스와 구체적 구현을 통한 표준 구현
  • 구성: 추상 팩토리 인터페이스 + 구체적 팩토리 클래스들
  • 목적: 기본적인 제품군 생성 패턴 제공
  • 실제 예시: GUI 컴포넌트 시스템에서 Windows/Mac 스타일별 위젯 생성

2. 매개변수화된 팩토리 기법#

  • 정의: 매개변수를 통해 생성할 제품 타입을 결정하는 기법
  • 구성: 단일 메소드 + 타입 매개변수
  • 목적: 팩토리 메소드 수 감소 및 유연성 증대
  • 실제 예시: 게임에서 레벨별 적 캐릭터 생성 (type: “boss”, “minion”, “elite”)

3. 팩토리 등록 기법#

  • 정의: 런타임에 팩토리를 등록하고 검색하는 동적 기법
  • 구성: 팩토리 레지스트리 + 동적 팩토리 선택
  • 목적: 플러그인 아키텍처 지원 및 동적 확장성
  • 실제 예시: 데이터베이스 드라이버 시스템에서 벤더별 커넥션 팩토리 등록

4. 싱글톤 팩토리 기법#

  • 정의: 싱글톤 패턴과 결합된 팩토리 구현
  • 구성: 싱글톤으로 구현된 팩토리 인스턴스
  • 목적: 메모리 효율성 및 전역 접근성
  • 실제 예시: 로깅 시스템에서 환경별 로거 팩토리 (dev/prod/test)

22. GoF 패턴 상세 분석#

패턴 이름과 분류#

  • 이름: Abstract Factory Pattern (추상 팩토리 패턴)
  • 분류: Creational Pattern (생성 패턴)
  • 별칭: Kit Pattern

의도 (Intent)#

관련성이 있거나 상호 의존적인 객체들의 군을 생성하기 위한 인터페이스를 제공하되, 구체적인 클래스는 명시하지 않는다.

다른 이름 (Also Known As)#

  • Kit Pattern
  • Family Factory Pattern
  • Toolkit Pattern

동기 (Motivation / Forces)#

  • 제품군 일관성: 관련된 제품들이 함께 사용되어야 하는 제약
  • 플랫폼 독립성: 다양한 환경에서 동일한 인터페이스 제공 필요
  • 확장성 요구: 새로운 제품군 추가 시 기존 코드 영향 최소화
  • 구현 은닉: 클라이언트가 구체적인 구현에 의존하지 않도록 하는 필요성

적용 가능성 (Applicability)#

  1. 시스템 독립성: 객체 생성, 구성, 표현 방식과 무관한 시스템 구축
  2. 제품군 선택: 여러 제품군 중 하나를 선택하여 시스템 구성
  3. 제품 일관성: 관련 제품들이 함께 사용되도록 제약 적용
  4. 라이브러리 제공: 구현이 아닌 인터페이스만 노출하는 라이브러리

구조 (Structure)#

앞서 제시한 클래스 다이어그램 참조

참여자 (Participants)#

  1. AbstractFactory: 추상 제품 객체 생성 연산의 인터페이스 정의
  2. ConcreteFactory: 구체적 제품 객체 생성 연산 구현
  3. AbstractProduct: 제품 객체 타입의 인터페이스 정의
  4. ConcreteProduct: 구체적 팩토리가 생성할 제품 객체 정의
  5. Client: AbstractFactory 와 AbstractProduct 인터페이스만 사용

협력 (Collaboration)#

  • 런타임 인스턴스: ConcreteFactory 클래스의 인스턴스가 런타임에 생성
  • 제품 생성: 구체적 팩토리가 특정 구현을 갖는 제품 객체 생성
  • 팩토리 선택: 서로 다른 제품 객체를 위해 다른 구체적 팩토리 사용
  • 책임 위임: AbstractFactory 가 제품 생성 책임을 ConcreteFactory 에 위임

결과 (Consequences)#

장점#
  1. 구체적 클래스 분리: 클라이언트와 구현 클래스 간 분리
  2. 제품군 쉬운 교체: 구체적 팩토리 변경만으로 제품군 교체
  3. 제품 간 일관성: 한 번에 관련된 제품군 사용 보장
단점#
  1. 새로운 제품 지원 어려움: 새 제품 추가 시 모든 팩토리 수정 필요

구현 (Implementation)#

구현 이슈들#
  1. 팩토리를 싱글톤으로: 애플리케이션에서 하나의 ConcreteFactory 인스턴스만 필요
  2. 제품 생성: 각 ConcreteFactory 에서 제품 생성 방법 구현
  3. 확장 가능한 팩토리: 새로운 종류의 제품 추가를 위한 유연한 설계

// 추상 팩토리
public interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// 구체적 팩토리 - Windows
public class WindowsFactory implements GUIFactory {
    public Button createButton() {
        return new WindowsButton();
    }
    
    public Checkbox createCheckbox() {
        return new WindowsCheckbox();
    }
}

// 구체적 팩토리 - Mac
public class MacFactory implements GUIFactory {
    public Button createButton() {
        return new MacButton();
    }
    
    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}

// 추상 제품
public interface Button {
    void paint();
}

public interface Checkbox {
    void paint();
}

// 구체적 제품
public class WindowsButton implements Button {
    public void paint() {
        System.out.println("Windows 스타일 버튼 렌더링");
    }
}

public class MacButton implements Button {
    public void paint() {
        System.out.println("Mac 스타일 버튼 렌더링");
    }
}

// 클라이언트 코드
public class Application {
    private Button button;
    private Checkbox checkbox;
    
    public Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }
    
    public void paint() {
        button.paint();
        checkbox.paint();
    }
}

// 사용 예시
public class Demo {
    public static void main(String[] args) {
        GUIFactory factory;
        String osName = System.getProperty("os.name").toLowerCase();
        
        if (osName.contains("mac")) {
            factory = new MacFactory();
        } else {
            factory = new WindowsFactory();
        }
        
        Application app = new Application(factory);
        app.paint();
    }
}

관련 패턴 (Related Patterns)#

협력 관계#
  • Factory Method: 추상 팩토리는 팩토리 메소드들의 집합으로 구현되는 경우가 많음
  • Prototype: 구체적 팩토리에서 프로토타입을 사용하여 제품 생성 가능
  • Singleton: 구체적 팩토리는 보통 싱글톤으로 구현됨
대안 관계#
  • Builder: 복잡한 객체를 단계별로 생성 vs 관련 객체군을 즉시 생성
  • Factory Method: 하나의 제품 vs 관련된 제품군
구조적 관계#
  • Facade: 서브시스템 객체 생성 방식만 숨기고 싶을 때 추상 팩토리 대신 사용 가능

용어 정리#

용어설명
제품군 (Product Family)함께 사용되도록 설계된 관련 객체들의 집합
구체적 팩토리 (Concrete Factory)추상 팩토리 인터페이스를 구현하여 특정 제품군을 생성하는 클래스
팩토리 메소드 (Factory Method)객체 생성을 담당하는 메소드로, 구체적인 클래스를 명시하지 않음
런타임 바인딩 (Runtime Binding)실행 시점에 사용할 구체적 팩토리를 결정하는 방식
크로스 플랫폼 (Cross-Platform)여러 플랫폼에서 동일하게 동작할 수 있는 소프트웨어 특성
의존성 역전 (Dependency Inversion)구체적 구현에 의존하지 않고 추상화에 의존하는 설계 원칙
결합도 (Coupling)클래스나 모듈 간의 상호 의존성 정도
응집도 (Cohesion)하나의 모듈 내에서 구성요소들이 관련성을 갖는 정도
