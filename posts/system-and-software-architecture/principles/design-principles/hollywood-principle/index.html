Hollywood Principle

고수준 컴포넌트가 저수준 컴포넌트를 언제 어떻게 호출할지 제어하되, 저수준 컴포넌트는 고수준 컴포넌트를 직접 호출하지 않도록 하는 것이다

Hollywood Principle#

1. 주제의 분류 적절성 분석#

Hollywood Principle(헐리우드 원칙) 은 “Don’t call us, we’ll call you(먼저 연락하지 마세요, 저희가 연락드리겠습니다)” 라는 문구로 요약되는 소프트웨어 설계 원칙입니다. 이 원칙은 컴포넌트 간의 느슨한 결합 (loose coupling) 과 제어의 역전 (Inversion of Control, IoC) 을 강조하며, 프레임워크/라이브러리 설계, 의존성 관리, 확장성 등 시스템 및 소프트웨어 아키텍처의 핵심 설계 원칙과 밀접하게 연관되어 있습니다. 따라서 “System and Software Architecture > Principles > Design Principles” 분류에 매우 적합합니다 [1][2][3][17][18].

1. 주제 분류의 적절성 분석#

“Hollywood Principle” 는 소프트웨어 아키텍처에서 제어 흐름과 컴포넌트 간의 의존성 관리를 다루므로, “System and Software Architecture > Principles > Design Principles” 분류에 적합합니다. 이는 IoC, 의존성 역전 원칙 (DIP) 등과 밀접한 관련이 있으며, 다양한 디자인 패턴 (예: Observer, Template Method) 에서 활용됩니다.

분류 적절성 분석#

제시된 분류 “Computer Science and Engineering” > “System and Software Architecture” > “Principles” > “Design Principles” 는 매우 적절합니다. 근거는 다음과 같습니다:

  • Hollywood Principle 은 소프트웨어 설계의 근본적인 원칙 중 하나임
  • 시스템 아키텍처 전반에 영향을 미치는 구조적 설계 패러다임임
  • SOLID 원칙과 함께 현대 소프트웨어 개발의 핵심 설계 원칙으로 인정받음
  • 컴퓨터 과학의 이론적 배경과 소프트웨어 공학의 실무적 적용을 모두 포괄함

요약문 (200 자)#

Hollywood Principle 은 ’ 우리에게 전화하지 마세요, 저희가 전화드리겠습니다 ’ 라는 할리우드 오디션 문구에서 유래된 소프트웨어 설계 원칙입니다. 전통적인 프로그래밍에서 애플리케이션이 라이브러리를 호출하는 방식과 달리, 프레임워크가 애플리케이션 코드를 호출하는 제어의 역전 (IoC) 개념을 구현합니다. 의존성 주입, 이벤트 기반 프로그래밍, 관찰자 패턴 등을 통해 느슨한 결합과 높은 확장성을 제공합니다.

3. 전체 개요 (250 자 내외)#

Hollywood Principle 은 소프트웨어 설계에서 제어 흐름을 고수준 컴포넌트가 관리하고, 저수준 컴포넌트는 호출을 기다리는 방식을 의미합니다. 이는 **Inversion of Control (IoC)**의 핵심 개념으로, 느슨한 결합과 높은 응집도를 통해 유지보수성과 확장성을 향상시킵니다. 이 원칙은 Observer, Template Method, Strategy 등의 디자인 패턴과 이벤트 기반 아키텍처에서 널리 활용됩니다.(Obsidian)

3. 전체 개요 (250 자 내외)#

Hollywood Principle 은 객체지향 설계 및 프레임워크 설계에서 널리 쓰이는 원칙으로, “Don’t call us, we’ll call you” 라는 문구로 대표됩니다. 이 원칙은 저수준 (구현) 모듈이 고수준 (프레임워크, 추상화) 모듈을 직접 호출하는 것이 아니라, 고수준 모듈이 저수준 모듈을 필요할 때 호출하도록 구조를 설계합니다. 이를 통해 의존성 부패 (Dependency Rot) 를 방지하고, 시스템의 유연성, 확장성, 테스트 용이성을 높입니다. 대표적으로 Inversion of Control, Dependency Injection, Observer, Template Method, Strategy 패턴 등에서 적용됩니다 [1][2][3][17][18][20].

개요 (250 자)#

Hollywood Principle 은 소프트웨어 시스템에서 제어 흐름을 역전시키는 핵심 설계 원칙으로, 프레임워크나 외부 시스템이 애플리케이션 코드를 호출하는 방식을 구현합니다. 이 원칙은 의존성 주입, 이벤트 기반 아키텍처, 콜백 메커니즘을 통해 컴포넌트 간 느슨한 결합을 실현하며, Spring Framework, React.js, Node.js 등 현대 소프트웨어 개발에서 광범위하게 활용됩니다. 코드의 재사용성, 테스트 용이성, 확장성을 크게 향상시키지만, 복잡성 증가와 디버깅 어려움이라는 트레이드오프가 존재합니다.

2. 핵심 개념#

이론과 실무를 아우르는 필수 개념들#

  1. 제어의 역전 (Inversion of Control, IoC)

    • 프로그램의 제어 흐름이 애플리케이션 코드가 아닌 외부 프레임워크에서 결정되는 원칙
    • 전통적인 “call” 방식에서 “callback” 방식으로의 패러다임 전환

  2. 콜백 메커니즘 (Callback Mechanism)

    • 특정 이벤트나 조건 발생 시 자동으로 호출되는 함수나 메서드
    • 비동기 프로그래밍과 이벤트 처리의 핵심 메커니즘

  3. 의존성 주입 (Dependency Injection, DI)

    • 객체가 필요한 의존성을 생성하지 않고 외부에서 제공받는 방식
    • 객체 생성과 사용의 책임을 분리하여 결합도를 낮춤

  4. 이벤트 기반 아키텍처 (Event-driven Architecture)

    • 이벤트의 발생과 처리를 중심으로 시스템을 설계하는 방식
    • Publisher-Subscriber 패턴의 기반이 되는 아키텍처 스타일

  5. 느슨한 결합 (Loose Coupling)

    • 컴포넌트 간 의존성을 최소화하여 변경에 대한 영향을 줄이는 설계 방식
    • 높은 응집도 (High Cohesion) 와 함께 소프트웨어 품질의 핵심 지표

  6. 프레임워크 vs 라이브러리 구분

    • 프레임워크: 애플리케이션 코드를 호출하는 주체
    • 라이브러리: 애플리케이션 코드에 의해 호출되는 대상

  7. Publisher-Subscriber 패턴

    • 이벤트 발행자와 구독자 간의 비동기 통신을 지원하는 패턴
    • 일대다 통신과 느슨한 결합을 동시에 실현

4. 핵심 개념#

  • 목적: 제어의 역전을 통해 느슨한 결합과 높은 응집도를 달성하여, 유지보수성과 확장성을 향상시킵니다.

    • Observer 패턴: Subject 가 상태 변경 시 Observer 에게 알림을 보냄.

    • 이벤트 기반 아키텍처: 이벤트 발생 시 등록된 핸들러가 호출됨.(Medium)

4. 핵심 개념#

  • 정의: Hollywood Principle 은 “Don’t call us, we’ll call you” 라는 구호로, 저수준 모듈이 고수준 모듈을 직접 호출하지 않고, 고수준 모듈이 저수준 모듈을 필요할 때 호출하는 구조를 의미합니다 [1][2][3][17][18].
  • 목적: 느슨한 결합, 의존성 부패 방지, 확장성 및 테스트 용이성 확보.
  • 주요 특징: 제어의 역전 (Inversion of Control), 추상화에 의존, 의존성 주입 (Dependency Injection) 과 밀접한 연관.
  • 적용 영역: 프레임워크 설계, 이벤트 기반 시스템, 디자인 패턴 (Observer, Template Method, Strategy 등)[2][5][17][20].
  • 실무 활용: Spring DI, 이벤트 리스너, 플러그인 시스템, 콜백 구조 등.

5. 상세 조사 및 정리#

배경#

  • 소프트웨어 설계에서 의존성 부패 (Dependency Rot) 와 복잡한 결합 구조를 방지하기 위해 등장 [1][8][12][19].
  • 프레임워크와 라이브러리의 차이, 제어 흐름의 주체를 명확히 하기 위해 도입 [3][18][20].

목적 및 필요성#

  • 컴포넌트 간 결합도 감소, 유지보수성 및 확장성 향상 [1][2][17][18].
  • 시스템 구조의 명확화와 테스트 용이성 확보.

배경 및 필요성#

Hollywood Principle 은 1980 년대 후반 객체지향 프로그래밍의 발전과 함께 등장했습니다. Martin Fowler 가 1988 년으로 그 기원을 추적하였으며, 할리우드 오디션에서 흔히 듣는 “Don’t call us, we’ll call you” 문구에서 이름이 유래되었습니다.

필요성:

  • 코드의 재사용성 향상
  • 컴포넌트 간 결합도 감소
  • 테스트 가능성 증대
  • 확장성과 유지보수성 개선
  • 관심사의 분리 실현

주요 기능 및 역할#

  1. 제어 흐름 관리: 프레임워크가 애플리케이션의 실행 흐름을 제어
  2. 의존성 관리: 객체 간 의존성을 외부에서 주입하여 관리
  3. 이벤트 처리: 비동기적 이벤트 발생과 처리를 조율
  4. 생명주기 관리: 객체의 생성, 초기화, 소멸을 프레임워크가 관리
  5. 확장점 제공: 플러그인이나 확장 모듈을 위한 훅 (Hook) 제공

주요 기능 및 역할#

  • 고수준 컴포넌트가 저수준 컴포넌트의 실행 시점과 방식을 제어.
  • 저수준 컴포넌트는 등록만 하고, 호출은 고수준 컴포넌트가 담당 [1][2][5][17][18].

특징#

  • 느슨한 결합 (Loose Coupling) 강조.
  • 제어의 역전 (Inversion of Control, IoC) 실현.
  • 테스트 용이성, 확장성, 코드 품질 향상.

특징#

  • 비침입적 (Non-intrusive): 비즈니스 로직에 프레임워크 코드가 침투하지 않음
  • 선언적 (Declarative): 설정이나 어노테이션을 통한 선언적 프로그래밍
  • 확장가능 (Extensible): 새로운 구현체를 쉽게 추가할 수 있는 구조
  • 테스트 친화적: Mock 객체를 쉽게 주입하여 단위 테스트 용이

핵심 원칙#

  1. “Don’t call us, we’ll call you”: 능동적 호출에서 수동적 대기로의 전환
  2. 제어권 위임: 애플리케이션 코드가 제어권을 프레임워크에 위임
  3. 인터페이스 기반 설계: 구체 클래스 대신 인터페이스에 의존
  4. 관심사 분리: 비즈니스 로직과 인프라스트럭처 코드의 분리

핵심 원칙/주요 원리#

  • 제어의 역전 (IoC): 실행 흐름의 주체를 고수준 모듈로 전환 [1][2][14][17][18].
  • 추상화에 의존: 구체 구현이 아닌 인터페이스/추상 클래스에 의존 [1][2][19].
  • 의존성 주입 (DI): 외부에서 의존성을 주입받아 결합도 최소화 [1][14][17].

핵심 차이점#

전통적 방식:

  • 애플리케이션 코드가 라이브러리 함수를 직접 호출
  • 제어 흐름이 위에서 아래로 (Top-down)
  • 애플리케이션이 언제, 어떤 함수를 호출할지 결정

Hollywood Principle:

  • 프레임워크가 애플리케이션 코드를 호출
  • 제어 흐름이 프레임워크 중심 (Framework-driven)
  • " 우리에게 전화하지 마세요, 저희가 전화드리겠습니다 "

    Callback->>Framework: Return control/result
    Framework->>Client: Optional: notify completion

Hollywood Principle 의 작동 원리는 다음과 같은 단계로 구성됩니다:

  1. 등록 단계 (Registration): 클라이언트 코드가 프레임워크에 콜백이나 핸들러를 등록
  2. 대기 단계 (Waiting): 클라이언트는 능동적으로 호출하지 않고 대기
  3. 트리거 단계 (Triggering): 프레임워크 내부에서 특정 이벤트나 조건 발생
  4. 호출 단계 (Invocation): 프레임워크가 등록된 콜백을 호출
  5. 실행 단계 (Execution): 클라이언트의 비즈니스 로직 실행
  6. 반환 단계 (Return): 제어권이 프레임워크로 돌아감

작동 원리#

  • 저수준 컴포넌트 (플러그인, 리스너 등) 는 고수준 컴포넌트에 등록만 함.
  • 고수준 컴포넌트 (프레임워크, 이벤트 디스패처 등) 가 필요할 때 저수준 컴포넌트를 호출 [1][2][5][17][18].

핵심 원칙 및 작동 원리#

Hollywood Principle 은 **제어의 역전 (Inversion of Control)**을 통해 고수준 컴포넌트가 제어 흐름을 관리하고, 저수준 컴포넌트는 호출을 기다리는 방식을 채택합니다. 이로 인해 컴포넌트 간의 결합도가 낮아지고, 시스템의 유연성과 확장성이 향상됩니다.

구조 및 아키텍처#

이 원칙은 다양한 디자인 패턴과 아키텍처에서 구현됩니다:

  • Observer 패턴: Subject 가 상태 변경 시 등록된 Observer 에게 알림을 보냄.

  • Template Method 패턴: 상위 클래스가 알고리즘의 구조를 정의하고, 하위 클래스가 세부 구현을 제공.

  • 이벤트 기반 아키텍처: 이벤트 발생 시 등록된 핸들러가 호출됨.(Medium)

필수 구성요소#

  1. Framework/Container (프레임워크/컨테이너)

    • 기능: 전체 시스템의 제어권을 가지고 있는 핵심 구성요소
    • 역할: 콜백 호출, 이벤트 발생, 생명주기 관리의 중심
    • 특징: 애플리케이션 코드의 실행 흐름을 결정

  2. Callback Registry (콜백 레지스트리)

    • 기능: 등록된 콜백 함수들을 저장하고 관리
    • 역할: 콜백의 등록, 해제, 조회 기능 제공
    • 특징: 동적으로 콜백을 추가/제거할 수 있는 유연성

  3. Event Manager (이벤트 매니저)

    • 기능: 이벤트의 발생, 전파, 처리를 담당
    • 역할: 이벤트 리스너 관리, 이벤트 큐 관리
    • 특징: 비동기 이벤트 처리 지원

  4. Lifecycle Manager (생명주기 매니저)

    • 기능: 객체의 생성, 초기화, 소멸 과정 관리
    • 역할: 의존성 주입 시점 결정, 리소스 관리
    • 특징: 객체의 전체 생명주기에 대한 제어권 보유

  1. Configuration Manager (설정 매니저)

    • 기능: 시스템 설정과 구성 정보 관리
    • 역할: 외부 설정 파일 로딩, 설정 값 제공
    • 특징: 런타임에 설정 변경 가능

  2. Security Manager (보안 매니저)

    • 기능: 보안 정책 적용과 권한 검증
    • 역할: 접근 제어, 인증, 인가 처리
    • 특징: 횡단 관심사로서 AOP 적용 가능

    • 기능: 시스템 성능과 상태 모니터링

시스템 구성 요소#

  1. OrderService (주문 서비스): 주문 생성 및 관리
  2. EventPublisher (이벤트 발행자): 시스템 이벤트 발행
  3. PaymentService (결제 서비스): 결제 처리
  4. InventoryService (재고 서비스): 재고 관리
  5. EmailService (이메일 서비스): 알림 이메일 발송
  6. EventBus (이벤트 버스): 이벤트 중개 및 라우팅

Hollywood Principle 의 역할#

  1. 이벤트 기반 아키텍처: 각 서비스는 직접 호출하지 않고 이벤트를 통해 통신
  2. 느슨한 결합: 서비스 간 직접적인 의존성 없이 독립적 운영
  3. 확장성: 새로운 서비스 추가 시 기존 코드 변경 없이 이벤트 구독만 추가
  4. 비동기 처리: 블로킹 없이 여러 작업을 동시에 처리
Hollywood Principle 적용 효과#
1. 느슨한 결합 (Loose Coupling)#
  • 각 서비스는 다른 서비스를 직접 호출하지 않음
  • 이벤트 버스를 통한 간접적 통신으로 의존성 최소화
2. 확장성 (Scalability)#
  • 새로운 서비스 추가 시 기존 코드 변경 불필요
  • 이벤트 구독만으로 새로운 기능 추가 가능
3. 비동기 처리 (Asynchronous Processing)#
  • 여러 서비스가 동시에 작업 수행
  • 전체 처리 시간 단축과 사용자 경험 향상
4. 장애 격리 (Fault Isolation)#
  • 한 서비스의 장애가 다른 서비스에 직접 영향 없음
  • 시스템 전체의 안정성 향상

Workflow 상세 설명#

  1. 사용자가 주문 요청을 보냄
  2. 주문 컨트롤러가 요청을 받아 주문 서비스에 전달
  3. 주문 서비스가 " 주문 생성 " 이벤트를 이벤트 버스에 발행
  4. Hollywood Principle 적용: 이벤트 버스가 등록된 서비스들을 호출
  5. 각 서비스가 독립적으로 작업 수행 (결제, 재고 차감, 이메일 발송)
  6. 작업 완료 후 결과를 다시 이벤트로 알림
  7. 최종적으로 사용자에게 주문 완료 응답 전송

5. 추가 학습 내용#

할리우드 원칙은 " 먼저 연락하지 마세요. 저희가 연락드리겠습니다 (Don’t call us, we’ll call you)" 라는 문구에서 이름을 따온 소프트웨어 설계 원칙이다.
이 표현은 할리우드 오디션에서 배우들에게 자주 하던 말에서 유래했다. 소프트웨어 설계에서 이 원칙은 제어 흐름의 역전을 의미하며, 고수준 컴포넌트가 저수준 컴포넌트를 직접 호출하는 대신, 상위 수준의 컴포넌트가 하위 수준의 컴포넌트를 직접 호출하지 않고, 추상화된 인터페이스를 통해 상호작용하도록 하는 구조를 만드는 것을 권장한다.

이 원칙은 템플릿 메서드, 옵저버, 의존성 주입과 같은 다양한 디자인 패턴의 기반이 되며, 현대적인 프레임워크와 리액티브 프로그래밍, 함수형 프로그래밍에서도 널리 활용된다.

할리우드 원칙의 핵심은 고수준 컴포넌트가 저수준 컴포넌트를 언제 어떻게 호출할지 제어하되, 저수준 컴포넌트는 고수준 컴포넌트를 직접 호출하지 않도록 하는 것이다. 이를 통해 의존성 부패를 방지하고, 시스템의 모듈성과 유지보수성을 향상시킬 수 있다.

프레임워크와 라이브러리의 차이점 중 하나는 바로 이 할리우드 원칙의 적용 여부이다. 프레임워크는 할리우드 원칙을 따르며 제어 흐름을 관리하지만, 라이브러리는 개발자가 직접 호출하는 방식으로 사용된다.

할리우드 원칙을 효과적으로 적용하려면 추상화, 인터페이스, 콜백, 이벤트 등의 메커니즘을 이해하고 활용해야 한다. 또한 이 원칙이 모든 상황에 적합한 것은 아니므로, 시스템의 복잡성과 요구사항을 고려하여 적절히 적용하는 것이 중요하다.

할리우드 원칙의 정의와 목적#

할리우드 원칙은 소프트웨어 시스템에서 의존성 부패 (dependency rot) 를 방지하기 위한 설계 접근법이다.
의존성 부패란 여러 컴포넌트 간의 복잡한 상호 의존 관계가 형성되어 시스템이 이해하기 어렵고 유지보수하기 힘들어지는 상태를 말한다.

이 원칙의 주요 목적은 다음과 같다:

  1. 의존성 역전: 전통적인 호출 계층 구조를 뒤집어 저수준 컴포넌트가 고수준 컴포넌트에 의존하게 한다.
  2. 결합도 감소: 컴포넌트 간의 직접적인 의존성을 줄여 결합도를 낮춘다.
  3. 제어 흐름 중앙화: 프레임워크나 고수준 컴포넌트가 전체 제어 흐름을 관리하게 한다.
  4. 확장성 향상: 기존 코드를 수정하지 않고 새로운 기능을 추가할 수 있게 한다.

할리우드 원칙의 역사와 배경#

할리우드 원칙이라는 용어는 1980 년대 말에서 1990 년대 초에 객체지향 프로그래밍 커뮤니티에서 처음 사용되기 시작했다. 리차드 스윗 (Richard Sweet) 이 Smalltalk 언어의 MVC(Model-View-Controller) 패턴에 대해 설명하면서 이 용어를 사용했고, 나중에 랄프 존슨 (Ralph Johnson) 과 브라이언 푸트 (Brian Foote) 가 이를 더 널리 알렸다.

이 원칙은 제어 역전 (Inversion of Control, IoC) 의 한 형태로, SOLID 원칙 중 의존성 역전 원칙 (Dependency Inversion Principle, DIP) 과 밀접한 관련이 있다. 하지만 할리우드 원칙은 특히 프레임워크와 라이브러리의 관계에서 이 개념을 설명할 때 자주 사용된다.

할리우드 원칙 vs. 제어 역전#

할리우드 원칙과 제어 역전 (IoC) 은 종종 같은 개념으로 혼용되지만, 미세한 차이가 있다:

  1. 제어 역전 (IoC): 더 넓은 개념으로, 프로그램의 제어 흐름이 사용자 코드에서 프레임워크나 컨테이너로 이동하는 모든 기법을 포함한다. 의존성 주입 (DI), 서비스 로케이터, 이벤트 기반 프로그래밍 등이 모두 IoC 의 형태이다.

  2. 할리우드 원칙: IoC 의 특정 형태로, 특히 " 너는 나를 호출하지 말고, 내가 너를 호출할게 " 라는 관점에 초점을 맞춘다. 주로 콜백, 템플릿 메서드, 옵저버 패턴 등의 형태로 구현된다.

    protected void processSpecificData() {
        System.out.println("CSV 데이터 처리");
    }
}

class XMLDataProcessor extends DataProcessor {
    @Override
    protected void processSpecificData() {
        System.out.println("XML 데이터 처리");
    }

function fetchData(url, callback) {
    // 데이터 가져오기
    fetch(url)
        .then(response => response.json())
        .then(data => {
            // 데이터를 가져온 후 콜백 호출
            callback(data);

            console.error('Error:', error);
        });

// 옵저버 인터페이스
interface Observer {
    void update(String event, Object data);
}

// 주제(Subject) 클래스
class EventManager {
    private Map<String, List<Observer>> observers = new HashMap<>();
    
    public void subscribe(String eventType, Observer observer) {
        observers.computeIfAbsent(eventType, k -> new ArrayList<>()).add(observer);
    }
    
    public void unsubscribe(String eventType, Observer observer) {
        if (observers.containsKey(eventType)) {
            observers.get(eventType).remove(observer);
        }
    }
    
    public void notify(String eventType, Object data) {
        if (observers.containsKey(eventType)) {
            for (Observer observer : observers.get(eventType)) {
                observer.update(eventType, data);
            }
        }
    }
}

// 구체적인 주제
class Editor {
    private EventManager events;
    private File file;
    
    public Editor() {
        this.events = new EventManager();
    }
    
    public void openFile(String filePath) {
        this.file = new File(filePath);
        events.notify("open", file);
    }
    
    public void saveFile() {
        // 파일 저장 로직
        events.notify("save", file);
    }
    
    public EventManager getEventManager() {
        return events;
    }
}

// 구체적인 옵저버
class LoggingListener implements Observer {
    private String logFileName;
    
    public LoggingListener(String logFileName) {
        this.logFileName = logFileName;
    }
    
    @Override
    public void update(String eventType, Object data) {
        System.out.println("로그 작성: " + eventType + ", 데이터: " + data + ", 로그 파일: " + logFileName);
    }
}

// 클라이언트 코드
public class Main {
    public static void main(String[] args) {
        Editor editor = new Editor();
        
        LoggingListener logger = new LoggingListener("log.txt");
        editor.getEventManager().subscribe("open", logger);
        editor.getEventManager().subscribe("save", logger);
        
        editor.openFile("test.txt");
        editor.saveFile();
    }
}

이 예제에서 Editor(주제, 고수준 컴포넌트) 는 파일 열기와 저장의 전체 흐름을 제어하고, LoggingListener(옵저버, 저수준 컴포넌트) 는 이벤트가 발생했을 때 알림을 받아 로깅 작업을 수행한다. 옵저버는 주제에 등록되지만, 주제가 적절한 시점에 옵저버를 호출하는 방식으로 제어 흐름이 역전된다.

        return new MessageClient(service);

interface Visitor {

// 요소 인터페이스
interface Shape {

}

// 구체적인 요소
class Circle implements Shape {
    private double radius;
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    public double getRadius() {
        return radius;
    }
    
    @Override
    public void accept(Visitor visitor) {
        visitor.visit(this);  // 할리우드 원칙 적용
    }
}

class Rectangle implements Shape {
    private double width;

class AreaCalculator implements Visitor {
    @Override
    public void visit(Circle circle) {
        double area = Math.PI * circle.getRadius() * circle.getRadius();
        System.out.println("원의 면적: " + area);
    }
    
    @Override
    public void visit(Rectangle rectangle) {
        double area = rectangle.getWidth() * rectangle.getHeight();
        System.out.println("사각형의 면적: " + area);
    }
    
    @Override
    public void visit(Triangle triangle) {
        double area = 0.5 * triangle.getBase() * triangle.getHeight();
        System.out.println("삼각형의 면적: " + area);
    }
}

프레임워크는 할리우드 원칙을 따른다.

할리우드 원칙의 장점#

할리우드 원칙을 적용하면 다양한 이점이 있다:

  1. 의존성 감소
    고수준 컴포넌트가 저수준 컴포넌트를 직접 호출하는 대신, 저수준 컴포넌트를 호출할 수 있는 방법을 제공함으로써 의존성을 줄인다. 이는 시스템의 결합도를 낮추고 유지보수성을 향상시킨다.

  2. 제어 흐름 중앙화
    프로그램의 제어 흐름을 중앙화하여 전체 흐름을 더 쉽게 이해하고 관리할 수 있게 한다. 중앙화된 제어 흐름은 디버깅과 테스트가 더 쉽다.

  3. 확장성 향상
    시스템의 핵심 구조를 변경하지 않고도 새로운 기능을 추가할 수 있어 확장성이 향상된다. 예를 들어, 템플릿 메서드 패턴에서는 새로운 알고리즘 변형을 추가하기 위해 새로운 하위 클래스만 추가하면 된다.

  4. 관심사 분리
    고수준 정책 (제어 흐름) 과 저수준 세부사항 (구체적인 구현) 을 분리할 수 있어 각각을 독립적으로 발전시킬 수 있다. 이는 코드의 모듈성을 향상시키고 변경 사항의 영향 범위를 제한한다.

  5. 병렬 개발 촉진
    인터페이스나 추상 클래스가 명확하게 정의되면, 여러 개발자가 고수준 컴포넌트와 저수준 컴포넌트를 동시에 개발할 수 있다. 이는 대규모 프로젝트에서 개발 효율성을 높인다.

  6. 코드 재사용성 향상
    공통 로직은 고수준 컴포넌트에 한 번만 구현하고, 변화하는 부분만 저수준 컴포넌트에서 구현함으로써 코드 재사용성이 향상된다.

할리우드 원칙의 도전과 주의사항#

할리우드 원칙을 적용할 때는 몇 가지 도전과 주의사항이 있다:

  1. 복잡성 증가
    추상화 계층이 추가되면 시스템의 전체적인 복잡성이 증가할 수 있다. 특히 작은 프로젝트에서는 이러한 복잡성이 불필요할 수 있다.

    추가적인 추상화 계층이나 간접 호출로 인해 성능 오버헤드가 발생할 수 있다. 대부분의 경우 이 오버헤드는 무시할 만하지만, 성능이 중요한 시스템에서는 고려해야 할 사항이다.

  4. 학습 곡선
    할리우드 원칙과 관련 패턴을 효과적으로 사용하려면 일정한 학습 곡선이 필요하다. 팀 전체가 이러한 개념에 익숙하지 않다면, 초기에는 생산성이 저하될 수 있다.

// 이벤트 버스
class EventBus {
    private Map<Class<?>, List<EventListener<?>>> listeners = new HashMap<>();
    
    public <T> void register(Class<T> eventType, EventListener<T> listener) {
        listeners.computeIfAbsent(eventType, k -> new ArrayList<>()).add(listener);
    }
    
    public <T> void post(T event) {
        List<EventListener<?>> eventListeners = listeners.getOrDefault(event.getClass(), Collections.emptyList());
        for (EventListener<?> listener : eventListeners) {
            ((EventListener<T>) listener).onEvent(event);
        }
    }
}

// 이벤트 리스너 인터페이스
interface EventListener<T> {
    void onEvent(T event);
}

// 구체적인 이벤트 리스너
class OrderProcessor implements EventListener<OrderEvent> {
    @Override
    public void onEvent(OrderEvent event) {
        System.out.println("주문 처리: " + event.getOrderId() + ", 상태: " + event.getStatus());
        // 주문 처리 로직
    }
}

class NotificationService implements EventListener<OrderEvent> {
    @Override
    public void onEvent(OrderEvent event) {
        System.out.println("알림 전송: " + event.getOrderId() + ", 상태: " + event.getStatus());
        // 알림 전송 로직
    }
}

// 이벤트 발생
public class Main {
    public static void main(String[] args) {
        EventBus eventBus = new EventBus();
        
        // 리스너 등록
        eventBus.register(OrderEvent.class, new OrderProcessor());
        eventBus.register(OrderEvent.class, new NotificationService());
        
        // 이벤트 발생
        eventBus.post(new OrderEvent("ORD-123", "CREATED"));
        eventBus.post(new OrderEvent("ORD-123", "PAID"));
    }
}

이 예제에서 EventBus(고수준 컴포넌트) 는 이벤트 발생 시 적절한 리스너를 호출하는 제어 흐름을 관리한다. 리스너 (저수준 컴포넌트) 는 자신이 언제 호출될지 모르며, EventBus 에 등록되어 적절한 이벤트가 발생했을 때 호출된다.
