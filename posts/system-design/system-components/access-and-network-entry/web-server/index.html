4. 개요 (250 자)#

웹 서버는 인터넷상에서 웹 사이트와 웹 애플리케이션을 호스팅하고 클라이언트 요청에 대한 응답을 제공하는 핵심 시스템입니다. HTTP/HTTPS 프로토콜을 사용하여 브라우저, 모바일 앱 등 다양한 클라이언트와 통신하며, 정적 파일 서빙부터 동적 콘텐츠 생성까지 광범위한 기능을 제공합니다. Apache, Nginx, IIS 등 다양한 구현체가 존재하며, 로드 밸런싱, 보안, 캐싱 등의 고급 기능을 통해 확장 가능하고 안정적인 웹 서비스를 구현할 수 있습니다.

4. 전체 개요 (250 자 이내)#

Web Server 는 클라이언트의 HTTP/HTTPS 요청을 수신하여 정적 파일 (html, css, js 등) 을 제공하거나, 애플리케이션 서버와 연계해 동적 응답을 반환하는 핵심 시스템 컴포넌트이다. Apache HTTP Server, NGINX, Microsoft IIS, Traefik, AWS CloudFront 등이 주요 구현체이며, 현대 웹 아키텍처에서 보안, 확장성, 성능 최적화의 출발점으로 작용한다. 또한, 로드 밸런싱, SSL 종료, 요청 라우팅 등의 고급 기능도 수행할 수 있다.

4. 개요 (250 자 내외)#

웹 서버는 인터넷, 인트라넷 등 각종 네트워크 환경에서 HTTP, HTTPS 프로토콜을 통해 클라이언트로부터 요청을 받아 정적 파일 (HTML, CSS, JS, 이미지 등) 및 동적 콘텐츠 (애플리케이션 연동, API 등) 를 응답하는 서버 소프트웨어다. Apache HTTP Server, Nginx, Microsoft IIS(Internet Information Services) 등 여러 종류가 있으며, 대규모 서비스, 기업 시스템, 클라우드 등에 필수적으로 활용된다. 실제 운영 환경에서는 보안, 고가용성, 확장성, 성능 최적화를 위해 다양한 확장 모듈과 복합 구조로 구성된다.

    여러 클라이언트 동시 요청 처리 위해 프로세스, 스레드, 논블로킹 I/O 사용.

웹 서버의 핵심 개념들을 이론과 실무 관점에서 분석하면 다음과 같습니다:

1. HTTP 서버 (HTTP Server)#

HTTP 프로토콜을 구현하여 클라이언트 - 서버 통신을 처리하는 소프트웨어입니다. 요청 메시지를 파싱하고 적절한 응답을 생성하는 역할을 담당합니다.

2. 요청 - 응답 모델 (Request-Response Model)#

클라이언트가 요청을 보내면 서버가 응답을 반환하는 동기식 통신 패턴입니다. 상태 비저장 (Stateless) 특성을 가집니다.

3. 정적 콘텐츠 서빙 (Static Content Serving)#

HTML, CSS, JavaScript, 이미지 등 미리 생성된 파일을 클라이언트에게 전달하는 기능입니다.

4. 동적 콘텐츠 생성 (Dynamic Content Generation)#

서버 사이드 스크립트나 애플리케이션을 통해 실시간으로 콘텐츠를 생성하여 제공하는 기능입니다.

5. 가상 호스팅 (Virtual Hosting)#

하나의 웹 서버에서 여러 도메인이나 웹사이트를 호스팅하는 기술입니다.

실무 구현을 위한 연관성 분석:

  • HTTP 서버는 네트워크 소켓 프로그래밍과 프로토콜 파싱 기술이 필요
  • 요청 - 응답 모델은 멀티스레딩, 비동기 처리 기법과 밀접한 관련
  • 정적 콘텐츠 서빙은 파일 시스템 I/O 최적화와 캐싱 전략이 중요
  • 동적 콘텐츠 생성은 애플리케이션 서버와의 연동 및 템플릿 엔진 활용이 필요
  • 가상 호스팅은 DNS 설정과 서버 구성 관리 기술이 연관

4. 핵심 개념#

웹 서버의 기본 개념#

웹 서버 (Web Server) 는 HTTP(HyperText Transfer Protocol) 또는 HTTPS(HTTP Secure) 프로토콜을 사용하여 클라이언트 (주로 웹 브라우저) 의 요청을 처리하고 응답을 제공하는 소프트웨어 또는 하드웨어 시스템입니다.

클라이언트 - 서버 모델#

웹 서버는 클라이언트 - 서버 아키텍처의 핵심 구성 요소로, 클라이언트가 요청 (request) 을 보내면 서버가 이를 처리하여 응답 (response) 을 반환합니다. 이 모델은 웹의 기본 작동 원리를 형성합니다.

HTTP 프로토콜#

웹 서버 통신의 기반이 되는 HTTP 는 상태가 없는 (stateless) 프로토콜로, 각 요청은 독립적으로 처리됩니다. HTTP 메서드 (GET, POST, PUT, DELETE 등) 를 통해 다양한 작업을 수행할 수 있습니다.

정적 Vs 동적 콘텐츠#
  • 정적 콘텐츠 (Static Content): HTML, CSS, JavaScript, 이미지 등 변경되지 않는 파일
  • 동적 콘텐츠 (Dynamic Content): 사용자 요청이나 데이터베이스 상호작용에 따라 실시간으로 생성되는 콘텐츠
가상 호스팅 (Virtual Hosting)#

하나의 물리적 서버에서 여러 웹사이트를 호스팅하는 기술로, 이름 기반 (Name-based) 과 IP 기반 (IP-based) 가상 호스팅이 있습니다.

리버스 프록시 (Reverse Proxy)#

클라이언트와 서버 사이에 위치하여 클라이언트의 요청을 적절한 백엔드 서버로 전달하고, 서버의 응답을 클라이언트에게 반환하는 중간 서버입니다.

로드 밸런싱 (Load Balancing)#

다수의 서버에 트래픽을 분산하여 시스템의 확장성, 가용성, 성능을 향상시키는 기술입니다.

웹 캐시 (Web Cache)#

자주 접근하는 데이터를 임시 저장하여 반복적인 요청의 처리 시간을 단축하는 메커니즘입니다.

웹 보안 기초#

SSL/TLS 암호화, 접근 제어, DDoS 방어 등 웹 서버의 보안을 유지하기 위한 핵심 개념입니다.

웹 서버 성능 지표#

요청 처리 시간, 동시 연결 수, 처리량 (throughput) 등 웹 서버의 성능을 평가하는 주요 지표입니다.

등장 및 발전 배경#

웹 서버는 1989 년 팀 버너스리 (Tim Berners-Lee) 가 CERN 에서 World Wide Web 을 발명하면서 시작되었습니다. 초기 CERN HTTPd 에서 시작하여 1995 년 Apache HTTP Server 의 등장으로 급속히 발전했습니다.

주요 발전 단계:

  1. 1 세대 (1989-1995): CERN HTTPd, NCSA HTTPd - 기본적인 HTTP 기능
  2. 2 세대 (1995-2000): Apache HTTP Server - 모듈화, 확장성
  3. 3 세대 (2000-2010): IIS, Lighttpd - 성능 최적화, 이벤트 기반 처리

3. 모듈화 아키텍처#

플러그인 방식의 모듈 시스템을 통해 기능을 확장할 수 있습니다. 이는 Apache 의 mod_rewrite, mod_ssl 등을 통해 달성됩니다.

4. 동시성 처리#

멀티스레딩, 멀티프로세싱, 이벤트 기반 처리를 통해 동시 다중 요청을 효율적으로 처리합니다.

5. 캐싱 지원#

정적 파일 캐싱, HTTP 캐시 헤더 처리를 통해 성능을 최적화합니다.

1. 멀티프로세싱 모델 (Multi-Processing Model)#

정의: 각 요청을 별도의 프로세스에서 처리하는 방식

구성:

  • 마스터 프로세스: 요청 분배 및 워커 프로세스 관리
  • 워커 프로세스: 실제 요청 처리

목적: 프로세스 격리를 통한 안정성 확보

실제 예시:

  • Apache MPM prefork 모듈
  • 각 프로세스가 독립적인 메모리 공간 사용
  • 한 프로세스 장애가 다른 프로세스에 영향을 주지 않음

2. 멀티스레딩 모델 (Multi-Threading Model)#

정의: 하나의 프로세스 내에서 여러 스레드로 요청을 처리하는 방식

구성:

  • 메인 스레드: 요청 수신 및 분배
  • 워커 스레드 풀: 요청 처리

목적: 메모리 효율성과 컨텍스트 스위칭 비용 감소

실제 예시:

  • Apache MPM worker 모듈
  • IIS 의 스레드 풀 모델
  • 스레드 간 메모리 공유로 인한 효율성

3. 이벤트 기반 모델 (Event-Driven Model)#

정의: 비동기 I/O 와 이벤트 루프를 사용하는 단일 스레드 모델

구성:

  • 이벤트 루프: 비동기 이벤트 처리
  • 콜백 함수: 이벤트 완료 시 실행

목적: 높은 동시성과 낮은 리소스 사용량

실제 예시:

  • Nginx 의 epoll/kqueue 기반 처리
  • Node.js 의 이벤트 루프
  • 수만 개의 동시 연결 처리 가능

4. 하이브리드 모델 (Hybrid Model)#

정의: 여러 처리 방식을 조합한 모델

구성:

  • 멀티프로세싱 + 멀티스레딩
  • 이벤트 기반 + 스레드 풀

목적: 각 모델의 장점을 결합하여 최적화

실제 예시:

  • Apache MPM event 모듈
  • 이벤트 기반 연결 관리 + 스레드 기반 요청 처리

1. 동시성 모델 (Concurrency Models)#

정의: 여러 클라이언트 요청을 동시에 처리하는 방법을 결정하는 모델입니다.

구성:

2. I/O 처리 기법 (I/O Handling Techniques)#

정의: 네트워크 및 파일 시스템 I/O 작업을 처리하는 기법입니다.

구성:

  • 블로킹 I/O(Blocking I/O): I/O 작업이 완료될 때까지 스레드 대기
  • 논블로킹 I/O(Non-blocking I/O): I/O 작업의 완료 여부를 주기적으로 확인
  • 비동기 I/O(Asynchronous I/O): I/O 작업 완료 시 알림 수신
  • 제로 카피 (Zero-copy): 데이터 복사 최소화 기법

목적: I/O 작업의 효율성을 높여 서버 성능을 향상시킵니다.

실제 예시:

3. 캐싱 전략 (Caching Strategies)#

정의: 자주 요청되는 콘텐츠를 임시 저장하여 빠르게 제공하는 기법입니다.

구성:

  • 메모리 캐시 (Memory Cache): RAM 에 콘텐츠 저장
  • 디스크 캐시 (Disk Cache): 파일 시스템에 콘텐츠 저장
  • 분산 캐시 (Distributed Cache): 여러 서버에 캐시 분산

목적: 반복적인 연산과 I/O 작업을 줄여 응답 시간을 단축하고 서버 부하를 감소시킵니다.

4. 부하 분산 기법 (Load Balancing Techniques)#

정의: 여러 서버에 트래픽을 분산하는 기법입니다.

구성:

  • 라운드 로빈 (Round Robin): 순차적으로 서버에 요청 분배
  • 가중치 기반 (Weighted): 서버 용량에 따른 가중치 부여
  • 최소 연결 (Least Connections): 활성 연결이 가장 적은 서버로 라우팅
  • IP 해시 (IP Hash): 클라이언트 IP 기반 라우팅
  • URL 해시 (URL Hash): 요청 URL 기반 라우팅

목적: 시스템 부하를 균등하게 분산하고 가용성과 확장성을 향상시킵니다.

  • Nginx 의 upstream 모듈: HTTP 부하 분산

# Apache의 mod_rewrite를 사용한 URL 재작성 예시
<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteBase /
    
    # www 없는 도메인을 www 있는 도메인으로 리디렉션
    RewriteCond %{HTTP_HOST} ^example\.com [NC]
    RewriteRule ^(.*)$ http://www.example.com/$1 [L,R=301]
    
    # 제품 ID를 사용한 SEO 친화적인 URL
    RewriteRule ^product/([0-9]+)/([a-zA-Z0-9-]+)$ product.php?id=$1&name=$2 [NC,L]
</IfModule>

마이크로서비스 환경에서의 웹 서버#

마이크로서비스 아키텍처에서 웹 서버는 API 게이트웨이 역할을 하며 다음과 같은 기능을 제공합니다:

  1. 라우팅: 요청을 적절한 마이크로서비스로 전달
  2. 인증/인가: 중앙화된 보안 처리
  3. 요청 변환: 클라이언트 요청을 내부 형식으로 변환
  4. 응답 통합: 여러 서비스의 응답을 하나로 통합
  5. 서비스 발견: 동적으로 서비스 인스턴스 찾기
  6. 로드 밸런싱: 마이크로서비스 인스턴스 간 부하 분산
  7. 회로 차단 (Circuit Breaking): 장애 서비스 격리

서버리스 아키텍처와 웹 서버#

서버리스 아키텍처에서는 전통적인 웹 서버의 역할이 변화합니다:

  1. 기능적 분해: 모놀리식 웹 서버가 아닌 개별 함수로 분해
  2. 이벤트 기반 처리: 요청에 따라 필요한 함수만 활성화
  3. 인프라 추상화: 물리적 서버 관리 제거, 클라우드 제공업체가 관리
  4. 자동 확장: 트래픽에 따른 자동 확장 및 축소
  5. 비용 최적화: 사용한 리소스에 대해서만 비용 지불

서버리스 환경에서도 API 게이트웨이가 전통적인 웹 서버의 일부 기능 (라우팅, 인증 등) 을 수행하지만, 백엔드 로직은 AWS Lambda, Azure Functions 등의 FaaS(Function as a Service) 서비스에서 실행됩니다.

4. 웹 서버와 애플리케이션 서버의 차이#

웹 개발에서 종종 혼동되는 개념이므로 차이점을 명확히 알아보겠습니다:

특성웹 서버애플리케이션 서버
주요 목적정적 콘텐츠 제공 및 HTTP 요청 처리비즈니스 로직 실행 및 동적 콘텐츠 생성
프로토콜주로 HTTP/HTTPS다양한 프로토콜 (HTTP, RMI, RPC 등)
콘텐츠 유형주로 정적 콘텐츠 (HTML, CSS, JS, 이미지 등)동적 콘텐츠 및 애플리케이션 기능
예시Apache, Nginx, IISTomcat, JBoss/WildFly, WebLogic, WebSphere

실제 환경에서는 웹 서버와 애플리케이션 서버가 함께 작동하는 경우가 많습니다. 웹 서버가 정적 콘텐츠를 처리하고, 동적 콘텐츠 요청은 애플리케이션 서버로 전달하는 구조입니다.

웹 서버와 애플리케이션 서버의 관계#

웹 서버와 애플리케이션 서버는 자주 함께 사용되지만 다른 역할을 수행합니다:

  • 웹 서버: HTTP 요청 처리, 정적 콘텐츠 제공, 요청 라우팅
  • 애플리케이션 서버: 비즈니스 로직 실행, 데이터베이스 연동, 동적 콘텐츠 생성

이 두 서버는 다음과 같은 방식으로 연동됩니다:

  1. 리버스 프록시 방식: 웹 서버가 클라이언트 요청을 받아 애플리케이션 서버로 전달
  2. 모듈 통합 방식: 웹 서버에 애플리케이션 서버 모듈 설치 (예: Apache with mod_php)
  3. 독립 실행 방식: 애플리케이션 서버가 자체 HTTP 요청 처리 기능 포함

3. 주요 웹 서버 소프트웨어#

현재 시장에서 가장 많이 사용되는 웹 서버 소프트웨어들을 살펴보겠습니다:

Apache HTTP Server#

  • 가장 오래되고 널리 사용되는 웹 서버 중 하나
  • 크로스 플랫폼 지원 (Windows, Linux, macOS 등)
  • 모듈 기반 아키텍처로 확장성이 뛰어남
  • .htaccess 파일을 통한 디렉토리 수준의 구성 가능
  • MPM(Multi-Processing Module) 을 통한 다양한 처리 모델 지원

Nginx (엔진엑스)#

  • 높은 동시성과 낮은 메모리 사용량으로 유명
  • 이벤트 기반 비동기 아키텍처
  • 리버스 프록시, 로드 밸런서, HTTP 캐시 기능
  • 정적 콘텐츠 제공에 특히 뛰어남
  • 마이크로서비스 아키텍처에서 API 게이트웨이로 자주 사용

Microsoft IIS (Internet Information Services)#

  • Windows 서버 환경에 통합된 웹 서버
  • .NET 애플리케이션과의 우수한 통합
  • Windows 인증과의 원활한 통합
  • GUI 기반 관리 도구
  • PowerShell 을 통한 자동화 지원

LiteSpeed#

  • Apache 호환성을 제공하면서 더 높은 성능 제공
  • 내장 캐싱과 보안 기능
  • 낮은 리소스 사용량과 높은 동시성

Caddy#

  • 자동 HTTPS 가 기본 제공되는 최신 웹 서버
  • 간단한 구성 파일 형식
  • 개발 환경에서 사용하기 쉬움
  • HTTP/2 와 HTTP/3 지원

// Node.js Express 서버 예시
const express = require('express');
const cluster = require('cluster');
const numCPUs = require('os').cpus().length;

// 클러스터 구성으로 멀티프로세싱
if (cluster.isMaster) {
    console.log(`마스터 프로세스 ${process.pid} 시작`);
    
    // CPU 코어 수만큼 워커 프로세스 생성
    for (let i = 0; i < numCPUs; i++) {
        cluster.fork();
    }
    
    // 워커 프로세스 종료 시 재시작
    cluster.on('exit', (worker, code, signal) => {
        console.log(`워커 프로세스 ${worker.process.pid} 종료`);
        cluster.fork();
    });
} else {
    // 워커 프로세스에서 Express 앱 실행
    const app = express();
    const PORT = process.env.PORT || 3000;
    
    // 미들웨어 설정
    app.use(express.json({ limit: '10mb' }));
    app.use(express.urlencoded({ extended: true }));
    
    // 헬스 체크 엔드포인트
    app.get('/health', (req, res) => {
        res.status(200).json({ 
            status: 'healthy',
            timestamp: new Date().toISOString(),
            uptime: process.uptime()
        });
    });
    
    // API 라우트 예시
    app.get('/api/products', async (req, res) => {
        try {
            // Redis 캐시 확인
            const cachedProducts = await redis.get('products:list');
            if (cachedProducts) {
                return res.json(JSON.parse(cachedProducts));
            }
            
            // 데이터베이스에서 조회
            const products = await db.query('SELECT * FROM products WHERE active = true');
            
            // Redis에 캐시 저장 (5분 TTL)
            await redis.setex('products:list', 300, JSON.stringify(products));
            
            res.json(products);
        } catch (error) {
            console.error('상품 조회 오류:', error);
            res.status(500).json({ error: '서버 내부 오류' });
        }
    });
    
    // 서버 시작
    app.listen(PORT, () => {
        console.log(`워커 프로세스 ${process.pid}가 포트 ${PORT}에서 실행 중`);
    });
}

웹 서버의 역할 및 워크플로우#

  1. 초기 요청 처리:

    • 사용자가 전자상거래 사이트 접속
    • CDN 이 정적 자산 (이미지, CSS, 자바스크립트) 제공
    • 동적 요청은 가장 가까운 에지 서버로 전달

  2. 리버스 프록시 및 보안:

    • Nginx 웹 서버가 SSL/TLS 종료 처리
    • 보안 헤더 추가 (HSTS, CSP, XSS 보호 등)
    • 기본적인 DDoS 방어 및 요청 속도 제한
    • URL 라우팅 및 리라이팅

  3. 정적 콘텐츠 제공:

    • HTML, CSS, JavaScript, 제품 이미지 등 정적 콘텐츠 직접 제공
    • 효율적인 파일 전송 (sendfile, 압축, 캐싱)
    • 브라우저 캐싱 설정 (Expires, Cache-Control 헤더)

  4. 동적 요청 프록싱:

    • 제품 검색, 장바구니 처리, 결제 등 동적 요청을 애플리케이션 서버로 전달
    • 세션 고정성 (sticky sessions) 관리
    • 업스트림 서버 상태 모니터링 및 장애 감지

  5. 부하 분산:

    • 요청 유형에 따른 지능적 라우팅
    • 서버 부하 및 성능 지표 기반 트래픽 분산
    • 자동 서버 추가/제거 (오토스케일링) 지원

  6. 성능 최적화:

    • 응답 캐싱 (제품 목록, 인기 상품 등)
    • HTTP/2 지원으로 연결 효율성 향상
    • 콘텐츠 압축 (Gzip, Brotli)

# 기본 HTTP 서버 설정
http {
    # 기본 설정
    include       mime.types;
    default_type  application/octet-stream;
    sendfile      on;
    tcp_nopush    on;
    tcp_nodelay   on;
    
    # 압축 설정
    gzip  on;
    gzip_vary on;
    gzip_min_length 1000;
    gzip_types text/plain text/css application/json application/javascript;
    
    # 캐싱 설정
    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=STATIC:10m inactive=24h max_size=1g;
    
    # 메인 서버 블록
    server {
        listen       443 ssl http2;
        server_name  shop.example.com;
        
        # SSL 설정
        ssl_certificate     /etc/nginx/ssl/example.com.crt;
        ssl_certificate_key /etc/nginx/ssl/example.com.key;
        ssl_protocols       TLSv1.2 TLSv1.3;
        ssl_ciphers         HIGH:!aNULL:!MD5;
        
        # 보안 헤더
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
        add_header X-Content-Type-Options nosniff;
        add_header X-Frame-Options SAMEORIGIN;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://trusted-cdn.com;";
        
        # 정적 콘텐츠
        location /static/ {
            alias /var/www/static/;
            expires 30d;
            add_header Cache-Control "public, no-transform";
            proxy_cache STATIC;
            proxy_cache_valid 200 302 7d;
        }
        
        # 제품 이미지
        location /images/ {
            proxy_pass https://storage-backend.example.com;
            proxy_cache STATIC;
            proxy_cache_valid 200 302 24h;
            proxy_cache_use_stale error timeout invalid_header updating http_500;
        }
        
        # API 요청
        location /api/ {
            proxy_pass http://api_backend;
            proxy_http_version 1.1;
            proxy_set_header Connection "upgrade";
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_read_timeout 60s;
        }
        
        # 기본 요청(SPA 지원)
        location / {
            try_files $uri $uri/ /index.html;
            expires -1;
        }
    }
    
    # 업스트림 서버 그룹
    upstream api_backend {
        least_conn;
        server api1.example.internal:8080 max_fails=3 fail_timeout=30s;
        server api2.example.internal:8080 max_fails=3 fail_timeout=30s;
        server api3.example.internal:8080 max_fails=3 fail_timeout=30s backup;
    }
}

카테고리용어설명
네트워크HTTP (HyperText Transfer Protocol)웹에서 클라이언트와 서버 간 데이터 전송을 위한 응용 계층 프로토콜
네트워크HTTPS (HTTP Secure)SSL/TLS 암호화를 적용한 보안 HTTP 프로토콜
네트워크TCP (Transmission Control Protocol)신뢰성 있는 데이터 전송을 보장하는 전송 계층 프로토콜
네트워크SSL/TLS (Secure Sockets Layer/Transport Layer Security)네트워크 통신 보안을 위한 암호화 프로토콜
아키텍처MPM (Multi-Processing Module)Apache 웹 서버의 요청 처리 방식을 정의하는 모듈

아키텍처FastCGICGI 의 성능을 개선한 프로토콜, 프로세스 재사용을 통한 최적화

성능CDN (Content Delivery Network)전 세계에 분산된 서버를 통해 콘텐츠를 빠르게 전달하는 네트워크
성능압축 (Compression)데이터 크기를 줄여 네트워크 전송 효율을 높이는 기술

클라우드PaaS (Platform as a Service)애플리케이션 개발과 배포 플랫폼을 제공하는 클라우드 서비스
클라우드SaaS (Software as a Service)완성된 소프트웨어를 서비스로 제공하는 클라우드 모델

용어 정리#

용어설명
CDN(Content Delivery Network)전 세계 여러 위치에 분산된 서버 네트워크를 통해 사용자에게 가까운 위치에서 콘텐츠를 제공하는 서비스
리버스 프록시 (Reverse Proxy)클라이언트와 백엔드 서버 사이에서 요청을 중개하고, 응답을 클라이언트에게 전달하는, 백엔드 서버를 보호하고 최적화하는 서버
SSL/TLS(Secure Sockets Layer/Transport Layer Security)웹 통신 암호화를 위한 프로토콜로, 데이터 전송 중 보안을 제공
로드 밸런싱 (Load Balancing)여러 서버에 네트워크 트래픽을 분산하여 서비스의 가용성과 확장성을 향상시키는 기술
가상 호스팅 (Virtual Hosting)하나의 물리적 서버에서 여러 웹사이트를 호스팅하는 기술
에지 서버 (Edge Server)사용자에게 가까운 네트워크 경계 (에지) 에 위치한 서버로, CDN 의 일부로 작동하여 지연 시간을 최소화
HTTP/2병렬 요청 처리, 헤더 압축 등을 지원하는 HTTP 프로토콜의 두 번째 주요 버전
HTTP/3QUIC 프로토콜 기반의 HTTP 프로토콜 최신 버전으로, 연결 설정 시간 단축 및 네트워크 변경에 대한 복원력 향상
QUIC(Quick UDP Internet Connections)Google 이 개발한 UDP 기반 전송 프로토콜로, TCP 의 한계를 극복하고 연결 설정 시간 단축
WebSocket웹 브라우저와 서버 간 양방향 실시간 통신을 지원하는 프로토콜
마이크로서비스 (Microservices)애플리케이션을 독립적으로 배포 가능한 작은 서비스로 분할하는 아키텍처 접근 방식
FastCGI웹 서버와 동적 콘텐츠 생성 프로그램 간의 인터페이스로, CGI 의 성능 향상 버전
WAF(Web Application Firewall)웹 애플리케이션에 특화된 방화벽으로, HTTP 트래픽을 모니터링하고 악의적인 요청 차단
HSTS(HTTP Strict Transport Security)웹사이트가 HTTPS 만을 통해 접근하도록 강제하는 보안 정책
서버리스 (Serverless)개발자가 서버 인프라를 관리하지 않고 기능을 배포할 수 있는 클라우드 컴퓨팅 모델
