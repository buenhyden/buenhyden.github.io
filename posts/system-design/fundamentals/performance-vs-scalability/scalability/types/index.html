Types

트래픽 증가나 데이터 처리량이 급증할 때 적절한 확장 전략을 선택하는 것은 시스템의 성공적인 운영에 핵심적인 역할을 한다.

Types of Scaling#

시스템 확장은 서비스의 성능과 안정성을 유지하기 위해 필수적인 작업이다. 특히, 트래픽 증가나 데이터 처리량이 급증할 때 적절한 확장 전략을 선택하는 것은 시스템의 성공적인 운영에 핵심적인 역할을 한다.

Horizontal Scaling (수평 확장)#

Horizontal Scaling은 시스템의 용량을 늘리기 위해 서버나 노드(인스턴스)를 추가하거나 제거하는 방식이다. 이를 “Scaling Out/In"이라고도 부른다.

특징#

  • 여러 대의 서버가 동일한 작업을 병렬로 처리하며, 로드 밸런서를 통해 트래픽을 분산한다.
  • 클라우드 환경에서 흔히 사용되며, 특히 마이크로서비스 아키텍처와 잘 맞는다.
  • Stateless 애플리케이션(상태를 서버에 저장하지 않는 구조)에서 효과적이다.

장점#

  • 무제한 확장 가능성: 이론적으로 필요한 만큼 노드를 추가할 수 있어 큰 트래픽을 처리할 수 있다.
  • 고가용성 및 내결함성: 다수의 노드가 있기 때문에 하나의 노드가 실패해도 시스템 전체에 영향을 미치지 않는다.
  • 다운타임 없음: 새로운 노드를 추가해도 기존 시스템은 계속 작동한다.
  • 비용 효율성: 클라우드 환경에서 사용한 만큼만 비용을 지불하므로 유연한 비용 관리가 가능한다.

단점#

  • 복잡한 아키텍처: 다중 노드를 관리하기 위한 로드 밸런서, 네트워크 설정 등이 필요하다.
  • 데이터 일관성 문제: 분산 환경에서는 데이터 동기화가 어려울 수 있습니다. 이를 해결하기 위해 분산 데이터베이스나 캐싱 솔루션이 필요하다.
  • 네트워크 지연: 노드 간 통신이 많아질 경우 성능 저하가 발생할 수 있다.

사용 사례#

  • 대규모 트래픽을 처리해야 하는 웹 애플리케이션(예: 전자상거래 플랫폼, 소셜 네트워크).
  • 데이터베이스 샤딩이나 분산 처리가 필요한 경우.
  • 클라우드 네이티브 애플리케이션.

구현 방법#

  1. 로드 밸런싱
    로드 밸런서는 수평적 확장의 핵심 구성 요소로, 여러 서버 인스턴스에 트래픽을 분산시킨다.
    주요 로드 밸런싱 알고리즘:

    • 라운드 로빈: 순차적으로 각 서버에 요청 분배
    • 가중치 기반: 서버 용량에 따라 다른 비율로 요청 분배
    • 최소 연결: 가장 적은 활성 연결을 가진 서버로 요청 라우팅
    • IP 해시: 클라이언트 IP 주소를 기반으로 일관된 서버 선택

    인기 있는 로드 밸런서:

    • AWS Elastic Load Balancer
    • NGINX
    • HAProxy
    • Kubernetes Ingress Controller

  2. 오토 스케일링
    오토 스케일링은 부하에 따라 자동으로 인스턴스 수를 조정하는 메커니즘이다.
    주요 스케일링 트리거:

    • CPU 사용률
    • 메모리 사용량
    • 요청 대기열 길이
    • 사용자 정의 메트릭

    클라우드 플랫폼의 오토 스케일링 서비스:

    • AWS Auto Scaling
    • Azure Virtual Machine Scale Sets
    • Google Cloud Autoscaler
    • Kubernetes Horizontal Pod Autoscaler

    # AWS Auto Scaling 정책 예시 (Python/Boto3)
import boto3

client = boto3.client('autoscaling')

# 오토 스케일링 그룹 생성
response = client.create_auto_scaling_group(
    AutoScalingGroupName='my-app-asg',
    LaunchConfigurationName='my-app-launch-config',
    MinSize=2,                  # 최소 인스턴스 수
    MaxSize=10,                 # 최대 인스턴스 수
    DesiredCapacity=2,          # 초기 희망 용량
    AvailabilityZones=['ap-northeast-2a', 'ap-northeast-2c'],
    VPCZoneIdentifier='subnet-1234abcd,subnet-5678efgh',
    HealthCheckType='ELB',
    HealthCheckGracePeriod=300
)

# CPU 사용률 기반 스케일링 정책 설정
response = client.put_scaling_policy(
    AutoScalingGroupName='my-app-asg',
    PolicyName='cpu-tracking-policy',
    PolicyType='TargetTrackingScaling',
    TargetTrackingConfiguration={
        'PredefinedMetricSpecification': {
            'PredefinedMetricType': 'ASGAverageCPUUtilization'
        },
        'TargetValue': 70.0,     # CPU 사용률 70%가 목표
        'ScaleInCooldown': 300,  # 스케일 인 쿨다운 기간(초)
        'ScaleOutCooldown': 60   # 스케일 아웃 쿨다운 기간(초)
    }
)

  3. 스테이트리스(Stateless) 설계
    수평적 확장을 효과적으로 구현하려면 애플리케이션이 스테이트리스해야 한다. 즉, 서버가 클라이언트 세션 정보를 로컬에 저장하지 않아야 한다.
    스테이트리스 설계를 위한 전략:

    • 세션 정보를 Redis와 같은 외부 저장소에 저장
    • JWT(JSON Web Tokens)와 같은 클라이언트 측 토큰 활용
    • 분산 캐시 시스템 구현

  4. 데이터베이스 샤딩(Database Sharding)
    대규모 데이터를 처리하기 위해 데이터베이스를 여러 샤드(조각)로 분할하는 기술이다.
    샤딩 전략:

    • 수평적 샤딩(Horizontal Sharding): 동일한 스키마의 다른 데이터 행을 다른 데이터베이스 서버에 저장
    • 수직적 샤딩(Vertical Sharding): 다른 테이블 또는 스키마를 다른 데이터베이스 서버에 저장
    • 기능적 샤딩(Functional Sharding): 기능에 따라 데이터를 다른 데이터베이스 서버에 저장

    샤딩 구현 기술:

    • MongoDB 샤딩 클러스터
    • MySQL 클러스터
    • PostgreSQL 파티셔닝
    • 커스텀 샤딩 미들웨어

Vertical Scaling (수직 확장)#

Vertical Scaling은 기존 서버의 자원(CPU, 메모리, 스토리지 등)을 업그레이드하여 성능을 향상시키는 방식이다. 이를 “Scaling Up/Down"이라고도 부른다.

특징#

  • 기존 하드웨어나 가상 머신(VM)의 성능을 강화하여 더 많은 부하를 처리한다.
  • 단일 서버 기반으로 동작하며, 주로 레거시 시스템이나 상태 기반(stateful) 애플리케이션에서 사용된다.

장점#

  • 간단한 구현: 새로운 서버를 추가할 필요 없이 기존 서버를 업그레이드하면 된다.
  • 관리 용이성: 단일 서버만 관리하면 되므로 복잡성이 낮다.
  • 빠른 성능 향상: 즉각적인 리소스 증대로 성능 병목현상을 해결할 수 있다.

단점#

  • 확장 한계: 하드웨어 업그레이드는 물리적 한계(CPU 코어 수, 메모리 용량 등)에 도달할 수 있다.
  • 단일 장애 지점(Single Point of Failure): 서버가 다운되면 전체 시스템이 중단될 위험이 있다.
  • 다운타임 발생 가능성: 업그레이드 작업 중 서비스 중단이 불가피할 수 있다.

사용 사례#

  • 레거시 애플리케이션이나 단일 데이터베이스 시스템.
  • 예측 가능한 트래픽 증가 시.
  • 초기 스타트업 환경에서 간단한 확장이 필요한 경우.

구현 방법#

  1. 하드웨어 업그레이드
    물리적 서버의 경우, 직접적인 하드웨어 업그레이드를 수행한다.
    일반적인 업그레이드 항목:

    • CPU 처리 능력 증가(코어 수, 클럭 속도)
    • RAM 용량 증가
    • 고속 SSD/NVMe 스토리지로 교체
    • 네트워크 인터페이스 업그레이드

  2. 가상 머신 리소스 조정
    클라우드 환경에서는 가상 머신의 리소스를 조정한다.
    주요 클라우드 제공업체 인스턴스 유형:

    • AWS EC2 인스턴스 유형(t3.micro에서 m5.24xlarge로 업그레이드)
    • Azure 가상 머신 크기
    • Google Cloud 머신 유형

    // AWS SDK를 사용한 인스턴스 유형 변경 예시 (Node.js)
const AWS = require('aws-sdk');
const ec2 = new AWS.EC2({ region: 'ap-northeast-2' });

// 인스턴스 중지 (유형 변경 전 필요)
const stopInstance = async (instanceId) => {
  const params = { InstanceIds: [instanceId] };

  try {
    await ec2.stopInstances(params).promise();
    console.log(`인스턴스 ${instanceId} 중지 요청 성공`);

    // 인스턴스가 완전히 중지될 때까지 대기
    await ec2.waitFor('instanceStopped', params).promise();
    console.log(`인스턴스 ${instanceId} 중지 완료`);
    return true;
  } catch (err) {
    console.error(`인스턴스 중지 오류: ${err.message}`);
    return false;
  }
};

// 인스턴스 유형 변경 (수직적 확장)
const changeInstanceType = async (instanceId, newInstanceType) => {
  const params = {
    InstanceId: instanceId,
    InstanceType: newInstanceType  // 예: 't2.micro'에서 'm5.xlarge'로
  };

  try {
    await ec2.modifyInstanceAttribute(params).promise();
    console.log(`인스턴스 ${instanceId}의 유형이 ${newInstanceType}으로 변경됨`);
    return true;
  } catch (err) {
    console.error(`인스턴스 유형 변경 오류: ${err.message}`);
    return false;
  }
};

// 인스턴스 시작
const startInstance = async (instanceId) => {
  const params = { InstanceIds: [instanceId] };

  try {
    await ec2.startInstances(params).promise();
    console.log(`인스턴스 ${instanceId} 시작 요청 성공`);
    return true;
  } catch (err) {
    console.error(`인스턴스 시작 오류: ${err.message}`);
    return false;
  }
};

// 수직적 확장 실행 함수
const performVerticalScaling = async (instanceId, newInstanceType) => {
  // 1. 인스턴스 중지
  const stopped = await stopInstance(instanceId);
  if (!stopped) return false;

  // 2. 인스턴스 유형 변경
  const changed = await changeInstanceType(instanceId, newInstanceType);
  if (!changed) return false;

  // 3. 인스턴스 재시작
  const started = await startInstance(instanceId);
  return started;
};

// 사용 예시
performVerticalScaling('i-1234567890abcdef0', 'm5.2xlarge')
  .then(result => console.log(`수직적 확장 ${result ? '성공' : '실패'}`))
  .catch(err => console.error(`오류 발생: ${err.message}`));

  3. 데이터베이스 확장
    데이터베이스 서버는 특별한 관심이 필요한 확장 대상이다.
    데이터베이스 수직적 확장 전략:

    • 인스턴스 리소스 증가
    • 데이터베이스 최적화(인덱싱, 쿼리 튜닝)
    • 읽기 전용 복제본 추가
    • 캐싱 레이어 구현(Redis, Memcached)

  4. 소프트웨어 최적화하드웨어 업그레이드 외에도 소프트웨어 최적화를 통해 효율적인 수직적 확장이 가능하다.
    최적화 영역:

    • 코드 프로파일링 및 병목 현상 제거
    • 알고리즘 개선
    • 메모리 사용 최적화
    • 데이터베이스 쿼리 튜닝

Horizontal Vs Vertical Scaling 비교#

확장성 측면#

수평적 확장:

  • 이론적으로 무제한 확장 가능
  • 필요에 따라 인스턴스를 계속 추가할 수 있음
  • 클라우드 환경에서 특히 효과적

수직적 확장:

  • 하드웨어 한계에 제약됨
  • 단일 머신의 최대 사양 이상으로 확장 불가능
  • 확장 한계가 명확하게 존재

비용 효율성#

수평적 확장:

  • 일반적으로 저가형 상용 하드웨어 활용 가능
  • 필요에 따라 점진적 투자 가능
  • 클라우드 환경에서 비용 최적화에 유리

수직적 확장:

  • 고성능 하드웨어는 비용이 기하급수적으로 증가할 수 있음
  • 대규모 업그레이드 시 일시적인 큰 투자 필요
  • 사용하지 않는 리소스에 대한 비용 낭비 가능성

복잡성#

수평적 확장:

  • 로드 밸런싱, 세션 관리, 데이터 일관성 등의 복잡한 문제 해결 필요
  • 분산 시스템 설계 지식 요구
  • 네트워크 지연 및 분산 시스템 문제 고려 필요

수직적 확장:

  • 구현이 상대적으로 간단
  • 아키텍처 변경 최소화
  • 단일 지점 장애(SPOF) 위험 존재

성능 특성#

수평적 확장:

  • 더 큰 동시성 처리 능력
  • 전체 시스템 처리량 증가
  • 개별 요청 처리 시간은 크게 변화 없음

수직적 확장:

  • 개별 요청 처리 속도 향상
  • 더 높은 동시성 처리 가능
  • 단일 서버의 성능 최적화

가용성 및 내결함성#

수평적 확장:

  • 여러 인스턴스로 인한 내재적 중복성
  • 일부 서버 실패에도 서비스 계속 가능
  • 고가용성 설계에 적합

수직적 확장:

  • 단일 지점 장애 취약성
  • 서버 다운 시 전체 서비스 중단 위험
  • 고가용성을 위한 추가 솔루션 필요

실제 시나리오에 따른 적용#

웹 애플리케이션 티어#

프론트엔드 서버:

  • 수평적 확장 선호: 사용자 요청을 여러 서버에 분산하여 동시 접속 처리
  • 구현: 로드 밸런서 뒤에 여러 웹 서버 배치, CDN 활용

애플리케이션 서버:

  • 혼합 접근법: 초기에는 수직적 확장 후 필요에 따라 수평적 확장
  • 구현: 오토 스케일링 그룹, 컨테이너 오케스트레이션(Kubernetes)

데이터베이스:

  • 초기에는 수직적 확장: 단일 강력한 데이터베이스 서버
  • 성장 시 수평적 확장 고려: 읽기 복제본, 샤딩 도입
  • 구현: 마스터-슬레이브 복제, 데이터베이스 샤딩

마이크로서비스 아키텍처#

마이크로서비스는 본질적으로 수평적 확장에 적합한다.

각 서비스별 확장 전략:

  • 트래픽이 많은 서비스는 더 많은 인스턴스로 수평 확장
  • 리소스 집약적 서비스는 더 강력한 노드로 수직 확장
  • 서비스 메시(Service Mesh)를 통한 트래픽 관리

빅데이터 처리#

데이터 처리 파이프라인:

  • 수평적 확장 선호: 분산 처리 프레임워크 활용(Hadoop, Spark)
  • 구현: 클러스터에 더 많은 노드 추가

데이터 저장:

  • 수평적 확장 필수: 분산 파일 시스템, NoSQL 데이터베이스
  • 구현: HDFS, Cassandra, MongoDB 샤딩

각 접근 방식의 이상적인 사용 사례#

수평적 확장이 적합한 경우#

  • 웹 서버 및 API 서버: 요청 처리가 간단하고 병렬화가 용이함
  • 마이크로서비스: 독립적으로 확장 가능한 작은 서비스
  • 분산 데이터 처리: 빅데이터 처리, 배치 작업
  • 고가용성이 중요한 시스템: 중복성을 통한 장애 허용
  • 트래픽 패턴이 가변적인 서비스: 클라우드 환경에서 자동 확장

수직적 확장이 적합한 경우#

  • 데이터베이스 서버(초기 단계): 복잡한 트랜잭션 처리, 데이터 일관성 유지
  • 메모리 집약적 애플리케이션: 인메모리 데이터베이스, 캐싱 서버
  • CPU 집약적 작업: 비디오 인코딩, 과학적 계산
  • 레거시 애플리케이션: 수평적 확장을 위한 재설계가 어려운 경우
  • 단순한 아키텍처 선호: 관리 복잡성 최소화 필요

실제 구현 시 고려사항#

클라우드 네이티브 환경에서의 확장#

클라우드 환경은 두 확장 방식 모두를 손쉽게 구현할 수 있는 유연성을 제공합니다.

AWS 환경 예시:

  • 수평적 확장: EC2 Auto Scaling 그룹, ECS 서비스, EKS 클러스터
  • 수직적 확장: EC2 인스턴스 타입 변경, RDS 인스턴스 클래스 업그레이드
  • 하이브리드 접근법: 다양한 워크로드에 맞는 인스턴스 패밀리 선택

컨테이너 및 오케스트레이션#

컨테이너 기술은 수평적 확장을 크게 단순화했다.

Kubernetes에서의 확장:

  • 수평적 확장: Horizontal Pod Autoscaler(HPA)를 통한 자동 Pod 수 조정
  • 수직적 확장: Vertical Pod Autoscaler(VPA)를 통한 리소스 요청 조정
  • 클러스터 확장: Cluster Autoscaler를 통한 노드 수 자동 조정

# Kubernetes HPA(Horizontal Pod Autoscaler) 설정 예시
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
  namespace: default
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2          # 최소 Pod 수
  maxReplicas: 10         # 최대 Pod 수
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU 사용률 70%에서 스케일링
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80  # 메모리 사용률 80%에서 스케일링
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 스케일 다운 전 안정화 기간
    scaleUp:
      stabilizationWindowSeconds: 60   # 스케일 업 전 안정화 기간

비용 최적화#

확장 전략은 비용에 직접적인 영향을 미친다.

비용 효율적인 확장 전략:

  • 적절한 리소스 프로비저닝: 과도한 프로비저닝 방지
  • 자동 스케일링 구현: 수요에 따른 용량 조정
  • 예약 인스턴스/약정 사용 할인: 예측 가능한 워크로드에 적용
  • 스팟 인스턴스/선점형 VM: 내결함성 있는 워크로드에 활용
  • 하이브리드 확장 전략: 기본 부하는 예약 인스턴스로, 변동성 부하는 스팟 인스턴스로 처리

모니터링 및 알림#

효과적인 확장을 위해 포괄적인 모니터링이 필수적이다.

핵심 모니터링 영역:

  • 리소스 사용률: CPU, 메모리, 디스크, 네트워크
  • 애플리케이션 성능: 응답 시간, 오류율, 처리량
  • 사용자 경험 메트릭: 페이지 로드 시간, 사용자 상호작용 지연
  • 비즈니스 메트릭: 트랜잭션 볼륨, 활성 사용자 수

인기 모니터링 도구:

  • Prometheus + Grafana
  • AWS CloudWatch
  • Datadog
  • New Relic
  • Dynatrace

최신 트렌드와 발전#

서버리스 아키텍처#

서버리스 컴퓨팅은 수평적 확장의 다음 진화 단계로 볼 수 있다.

서버리스의 확장 특성:

  • 자동 확장 및 프로비저닝
  • 코드 실행 단위까지의 미세한 확장
  • 부하에 따른 비용 지불(사용한 만큼 지불)

서버리스 플랫폼:

  • AWS Lambda
  • Azure Functions
  • Google Cloud Functions
  • Cloudflare Workers

자동화된 확장 결정#

머신러닝을 활용한 스마트 확장 결정이 늘어나고 있다.

AI 기반 자동 확장:

  • 예측적 확장(트래픽 패턴 예측)
  • 이상 탐지 기반 확장
  • 비즈니스 메트릭 연계 확장

멀티 클라우드 및 하이브리드 확장#

단일 클라우드 제공업체에 의존하지 않는 확장 전략이 인기를 얻고 있다.

멀티 클라우드 확장 이점:

  • 벤더 종속 방지
  • 지역별 최적화
  • 비용 최적화
  • 재해 복구 강화

구현 기술:

  • Kubernetes Federation
  • 클라우드 중립 오케스트레이션 도구
  • 멀티 클라우드 관리 플랫폼
