API(Application Programming Interface)s#

두 애플리케이션이 서로 통신할 수 있게 해주는 소프트웨어 중개자.
이는 데이터와 기능을 안전하고 제어된 방식으로 공유할 수 있게 해주는 메커니즘이다.

레스토랑에서…
- 손님(클라이언트) = API를 사용하는 프로그램
- 웨이터(API) = 주문을 주방에 전달하고 음식을 가져오는 인터페이스
- 주방(서버) = 실제 서비스를 제공하는 시스템

주요 특징#

  1. 추상화: API는 복잡한 내부 로직을 숨기고 간단한 인터페이스를 제공한다.
  2. 표준화: API는 일관된 방식으로 데이터를 주고받을 수 있게 한다.
  3. 모듈성: API를 통해 애플리케이션을 독립적인 모듈로 분리할 수 있다.
  4. 확장성: API를 사용하면 기존 시스템에 새로운 기능을 쉽게 추가할 수 있다.

엔드포인트를 설계할 때 주의해야 할 주요 사항#

  1. 명사 사용: 동사 대신 명사를 사용하여 리소스를 표현한다.
    • 좋은 예: /users, /products
    • 나쁜 예: /getUsers, /createProduct
  2. 복수형 사용: 컬렉션을 나타낼 때는 복수형 명사를 사용한다.
    • 예: /users, /posts
  3. 계층 구조 반영: 관계를 표현할 때 중첩을 사용한다.
    • 예: /posts/{postId}/comments
  4. 불필요한 정보 제거:
    • 마지막 슬래시(/) 제거
    • 대문자 사용 피하기
    • 밑줄(_) 대신 하이픈(-) 사용
  5. HTTP 메서드 활용: 리소스에 대한 행위는 HTTP 메서드로 표현한다.
    • GET, POST, PUT, DELETE 등을 적절히 사용
  6. 명확한 매개변수 사용: 필터링, 정렬, 페이지네이션을 위한 매개변수는 목적을 명확히 나타내는 이름을 사용한다.
    • 예: /users?name=John&age=25&sort=asc&page=2

적절한 API 응답 코드를 사용하는 방법#

  1. 표준 HTTP 상태 코드 사용:
    가능한 한 표준 HTTP 상태 코드를 사용하여 일관되고 잘 이해되는 사용자 경험을 제공한다.
  2. 일관성 유지:
    API 전체에서 상태 코드를 일관되게 사용하여 클라이언트가 각 코드의 의미를 쉽게 이해할 수 있도록 한다.
  3. 적절한 코드 선택:
    응답의 성격에 따라 적절한 상태 코드를 선택한다.
  4. 상세한 오류 정보 제공:
    오류 상태 코드 반환 시 클라이언트가 문제를 진단하고 해결할 수 있도록 상세한 오류 정보를 함께 제공한다.
  5. 상태 코드 과부하 방지:
    하나의 상태 코드를 다른 맥락에서 다른 의미로 사용하지 않도록 한다.
  6. API 문서화:
    사용된 상태 코드와 그 의미를 명확히 문서화하여 개발자들이 쉽게 이해하고 사용할 수 있도록 한다.

파라미터 이름을 선택할 때 고려할 사항#

  1. 설명적인 이름 사용:
    파라미터의 목적과 기능을 명확히 나타내는 이름을 선택한다.
  2. 의미 기반 이름 사용:
    파라미터의 타입보다는 의미에 기반한 이름을 사용하는 것을 고려한다.
  3. 약어와 숫자 인덱스 사용 지양:
    이해하기 어려운 약어나 의미 없는 숫자 인덱스 대신 완전한 단어를 사용한다.
  4. 간단하고 설명적인 이름 사용:
    너무 길지 않으면서도 파라미터의 역할을 잘 설명하는 이름을 선택한다.
  5. 오해의 소지가 있는 이름 피하기:
    파라미터의 실제 기능과 일치하지 않는 이름은 사용하지 않는다.
  6. 컨텍스트 반영:
    파라미터가 사용되는 맥락을 고려하여 이름을 선택한다.
  7. 필터링, 정렬, 페이지네이션을 위한 명확한 이름 사용:
    예를 들어, ’tags’, ‘sort’, ‘page’ 등의 이름을 사용하여 각 기능을 명확히 나타낸다.

gRPC API

gRPC API gRPC(gRPC Remote Procedure Call)는 Google에서 개발한 오픈소스 원격 프로시저 호출(RPC) 시스템. 이 시스템은 효율적이고 빠른 서비스 간 통신을 제공하며, 다양한 프로그래밍 언어와 플랫폼에서 사용할 수 있다. gRPC는 특히 다음과 같은 상황에서 유용하다: 마이크로서비스 아키텍처 서비스 간 효율적인 통신 강력한 타입 안정성 뛰어난 성능 실시간 통신이 필요한 시스템 채팅 애플리케이션 실시간 모니터링 게임 서버 모바일 애플리케이션 효율적인 데이터 전송 배터리 사용량 최적화 네트워크 대역폭 절약 gRPC의 주요 특징 Protocol Buffers 사용: gRPC는 데이터 직렬화를 위해 Protocol Buffers를 사용합니다. 이는 JSON이나 XML보다 더 작고 빠른 데이터 포맷을 제공합니다. RESTful API

RESTful API(Representational State Transfer API) 웹 서비스를 설계하고 구현하기 위한 아키텍처 스타일 RESTful API는 HTTP 프로토콜을 기반으로 하며, 클라이언트와 서버 간의 통신을 위한 표준화된 방식을 제공한다. RESTful API의 주요 특징 자원 중심 구조: URI를 통해 자원을 명확하게 표현합니다.
