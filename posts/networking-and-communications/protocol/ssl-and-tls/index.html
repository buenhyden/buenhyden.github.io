SSL(Secure Sockets Layer)과 TLS(Transport Layer Security)#

인터넷 상에서 데이터를 안전하게 전송하기 위한 암호화 프로토콜.
TLS는 SSL의 후속 버전으로, 보안성과 성능이 향상되었다.

SSL의 역사와 개념#

SSL은 1995년 Netscape에 의해 개발되었다.
주요 목적은 웹 브라우징 시 데이터의 기밀성과 무결성을 보장하는 것.
SSL은 다음과 같은 버전으로 발전했다:

  1. SSL 1.0 (내부적으로만 사용)
  2. SSL 2.0 (1995년 공개)
  3. SSL 3.0 (1996년 공개)

TLS의 등장과 발전#

TLS는 SSL 3.0을 기반으로 1999년 IETF(Internet Engineering Task Force)에 의해 개발되었다.
TLS는 SSL의 보안 취약점을 개선하고 더 강력한 암호화 알고리즘을 도입했다.
TLS의 주요 버전은 다음과 같다:

  1. TLS 1.0 (1999년)
  2. TLS 1.1 (2006년)
  3. TLS 1.2 (2008년)
  4. TLS 1.3 (2018년)

OSI 7계층과 TCP/IP 모델에서의 위치#

  • OSI 7계층:
    전송 계층(4계층)과 응용 계층(7계층) 사이에서 동작
  • TCP/IP 모델:
    전송 계층과 응용 계층 사이에서 동작

주요 특징#

  1. 데이터의 기밀성, 무결성, 인증을 제공한다.
  2. 공개키 암호화와 대칭키 암호화를 모두 사용한다.
  3. 디지털 인증서를 통해 서버의 신원을 확인한다.
  4. 다양한 암호화 알고리즘과 프로토콜 버전을 지원한다.

장점#

  1. 데이터 전송의 보안성 강화
  2. 서버 인증을 통한 신뢰성 확보
  3. 검색 엔진 최적화(SEO) 향상
  4. 규정 준수 지원 (예: PCI DSS)
  5. 사용자 신뢰도 증가

SSL과 TLS의 차이점#

  1. 버전:
    • SSL은 1.0, 2.0, 3.0 버전이 있으며 모두 취약점이 발견되어 더 이상 사용되지 않는다.
    • TLS는 1.0, 1.1, 1.2, 1.3 버전이 있으며, 현재는 1.2와 1.3이 주로 사용된다.
  2. 보안성:
    • TLS가 SSL보다 더 강력한 암호화 알고리즘과 보안 기능을 제공한다.
  3. 핸드셰이크 과정:
    • TLS는 SSL보다 더 효율적이고 빠른 핸드셰이크 과정을 가진다.
  4. 암호 스위트:
    • TLS는 더 다양하고 안전한 암호 스위트를 지원한다.
  5. 메시지 인증:
    • SSL은 MAC(Message Authentication Code)를 사용하지만, TLS는 HMAC(Hash-based Message Authentication Code)를 사용한다.

동작 방식#

  1. 핸드셰이크(handshake) 과정:
    1. 클라이언트 헬로: 클라이언트가 지원하는 암호화 방식 등을 서버에 전송한다.
    2. 서버 헬로: 서버가 선택한 암호화 방식과 인증서를 클라이언트에 전송한다.
    3. 인증서 검증: 클라이언트가 서버의 인증서를 검증한다.
    4. 키 교환: 안전한 통신을 위한 대칭 키를 교환한다.
    5. 암호화 통신 시작: 협상된 암호화 방식으로 데이터를 암호화하여 통신한다.
  2. 데이터 전송: 생성된 세션 키를 사용하여 데이터를 암호화하고 전송한다.
  3. 연결 종료: 통신이 완료되면 세션을 종료하고 세션 키를 폐기한다.

SSL(Secure Sockets Layer)과 TLS(Transport Layer Security)는 네트워크 통신을 안전하게 암호화하기 위한 프로토콜입니다. 두 프로토콜의 주요 차이점과 관련 개념들에 대해 설명하겠습니다.

인증서 체인#

인증서 체인은 디지털 인증서의 신뢰성을 확립하는 계층적 구조이다.
일반적으로 다음과 같은 구조로 이루어진다:

  1. 루트 인증서: 최상위 신뢰 기관의 자체 서명 인증서
  2. 중간 인증서: 루트 인증 기관이 서명한 인증서
  3. 최종 사용자 인증서: 웹사이트나 서비스에 발급된 인증서
    완전한 인증서 체인을 제공하는 것이 중요하며, 이를 통해 브라우저는 인증서의 유효성을 검증할 수 있다.

암호 스위트#

암호 스위트는 SSL/TLS 연결에서 사용되는 암호화 알고리즘의 조합.
주요 구성 요소는 다음과 같다:

  1. 키 교환 알고리즘 (예: RSA, ECDHE)
  2. 인증 알고리즘 (예: RSA, ECDSA)
  3. 대칭 암호화 알고리즘 (예: AES, ChaCha20)
  4. 메시지 인증 코드 알고리즘 (예: SHA-256, POLY1305)
    최신 TLS 버전에서는 보안성이 높은 암호 스위트를 우선적으로 사용한다.

전방 비밀성#

전방 비밀성(Forward Secrecy)은 현재 세션의 키가 노출되더라도 과거 세션의 통신 내용을 해독할 수 없도록 하는 보안 속성이다.
ECDHE나 DHE와 같은 키 교환 방식을 사용하여 구현된다.

HTTPS#

HTTPS는 HTTP 프로토콜에 SSL/TLS를 적용한 것이다.
웹 브라우저와 서버 간의 모든 통신을 암호화하여 데이터의 기밀성과 무결성을 보장한다.

성능 최적화#

SSL/TLS 성능 최적화를 위한 주요 방법들:

  1. 세션 재개: 이전 연결의 정보를 재사용하여 핸드셰이크 과정을 단축한다.
  2. OCSP Stapling: 인증서 상태 확인 과정을 최적화한다.
  3. HTTP/2 지원: 연결 다중화를 통해 성능을 향상시킨다.
  4. 적절한 암호 스위트 선택: 보안과 성능의 균형을 고려한다.
  5. 하드웨어 가속: 암호화 작업을 전용 하드웨어로 처리하여 성능을 개선한다.

추가적으로 알아야 할 내용:

  1. SSL과 TLS의 차이: TLS는 SSL의 후속 버전으로, 더 강화된 보안 기능을 제공합니다.
  2. 인증서 체인: 루트 인증서, 중간 인증서, 최종 엔티티 인증서로 구성된 체인을 통해 신뢰성을 확보합니다.
  3. 암호 스위트: 다양한 암호화 알고리즘의 조합을 지원하며, 클라이언트와 서버가 협상하여 사용할 암호 스위트를 결정합니다.
  4. 전방 비밀성: 세션 키가 노출되더라도 과거의 통신 내용을 해독할 수 없도록 하는 기능을 제공합니다.
  5. HTTPS: SSL/TLS는 HTTPS 프로토콜의 기반이 되며, 웹 브라우징 보안에 필수적입니다.
  6. 성능 최적화: TLS 1.3에서는 핸드셰이크 과정을 간소화하여 연결 속도를 개선했습니다.
