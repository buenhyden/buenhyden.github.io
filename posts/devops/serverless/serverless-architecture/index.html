Serverless Architecture#

Serverless Architecture는 개발자가 서버 관리 없이 애플리케이션 로직에 집중할 수 있도록 설계된 클라우드 컴퓨팅 모델로, 이벤트 기반의 자동 확장성과 사용한 만큼만 비용을 지불하는 구조가 핵심이다.

Serverless Architecture는 빠른 개발 속도와 비용 효율성으로 스타트업 및 이벤트 기반 애플리케이션에 적합하나, 장기 실행 작업이나 높은 제어력이 필요한 경우에는 EC2 등의 전통적 아키텍처와의 조합이 필요하다.
AWS Lambda + API Gateway + DynamoDB 조합이 가장 널리 사용되며, Serverless Framework를 통해 멀티 클라우드 환경 구축이 가능하다.

Serverless Architecture의 핵심 개념#

  1. 기본 원리

    • No Server Management: 서버 프로비저닝, 패치, 확장을 클라우드 공급자(AWS, Google Cloud 등)가 전담한다. 개발자는 코드 작성에만 집중.
    • Event-Driven: API 호출, 파일 업로드, 데이터베이스 변경 등 이벤트 발생 시 코드 실행 (예: AWS Lambda 함수 트리거).

  2. 구성 요소

요소설명예시 서비스
FaaS함수 단위 실행AWS Lambda, Google Cloud Functions
BaaS백엔드 서비스 통합Firebase, AWS DynamoDB
API Gateway함수를 HTTP 엔드포인트로 노출AWS API Gateway

주요 특징#

  1. 관리형 서비스 활용:
    • 서버리스 아키텍처에서는 인증, 데이터베이스, 스토리지, 메시징 등 다양한 기능을 클라우드 제공업체의 관리형 서비스로 대체한다. 예를 들어, 인증 기능은 Auth0나 Amazon Cognito와 같은 서비스를 사용하여 구현할 수 있다.
  2. 이벤트 중심 컴퓨팅:
    • 서버리스 애플리케이션은 이벤트에 반응하여 실행되는 함수로 구성된다. 이러한 함수는 HTTP 요청, 데이터베이스 변경, 파일 업로드 등 다양한 이벤트에 의해 트리거되며, 필요할 때만 실행되어 자원을 효율적으로 사용한다.
  3. 자동 확장 및 축소:
    • 클라우드 제공업체는 애플리케이션의 부하에 따라 컴퓨팅 자원을 자동으로 확장하거나 축소한다. 따라서 트래픽 급증 시에도 안정적인 서비스 제공이 가능하며, 사용량이 적을 때는 자원 사용을 최소화하여 비용을 절감할 수 있다.
  4. 종량제 요금제:
    • 서버리스 모델에서는 실제로 함수가 실행된 시간과 사용된 자원에 대해서만 비용이 부과된다. 따라서 미사용 상태에서의 비용 부담이 없으며, 효율적인 비용 관리가 가능하다.

주요 장점과 단점#

장점#

  • 개발 생산성 향상:
    • 인프라 관리 부담이 줄어들어 개발자는 비즈니스 로직 구현에 집중할 수 있으며, 개발 주기가 단축된다.
  • 비용 효율성:
    • 사용한 만큼만 비용을 지불하므로, 리소스 낭비를 최소화하고 비용 효율적인 운영이 가능하다.
  • 확장성:
    • 클라우드 제공업체의 자동 확장 기능을 통해 애플리케이션은 트래픽 변화에 유연하게 대응할 수 있다.

단점#

  • 제한된 제어권:
    • 인프라에 대한 세부적인 제어가 어려워 특정 요구사항을 충족하기 어려울 수 있다.
  • 벤더 종속성:
    • 특정 클라우드 제공업체의 서비스에 의존하게 되어, 벤더 종속성이 발생할 수 있다.
  • 콜드 스타트 지연:
    • 함수가 오랜 시간 호출되지 않으면 초기 실행 시 지연이 발생할 수 있다.

주요 사용 사례#

  1. 웹 애플리케이션 백엔드:
    • 서버리스 아키텍처를 활용하여 웹 애플리케이션의 백엔드를 구축하면, 확장성과 비용 효율성을 높일 수 있다.
  2. 데이터 처리 파이프라인:
    • 이벤트 중심의 데이터 처리 작업에 서버리스 함수를 사용하여 효율적인 데이터 파이프라인을 구성할 수 있다.
  3. 챗봇 및 AI 서비스:
    • 서버리스 함수를 활용하여 챗봇의 메시지 처리나 AI 모델의 추론 작업을 효율적으로 수행할 수 있다.

성능 최적화 기법#

  1. 콜드 스타트 완화

    • 프로비저닝된 동시성: AWS Lambda에서 최소 인스턴스 유지.
    • 함수 압축: Webpack으로 코드 크기 최소화.

  2. 보안 전략

    • 최소 권한 원칙: IAM 역할에 세분화된 권한 부여.
    • Secrets 관리: AWS Secrets Manager로 API 키 암호화.

적용 시 고려사항#

  • 적합한 경우: 변동성 큰 트래픽, 마이크로서비스, 프로토타입 개발.
  • 비적합: 장시간 배치 작업(예: 비디오 렌더링).
